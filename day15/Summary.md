## 1. 고급 JOIN 활용 (Outer Join의 이해)
- **필요성 인지:** 한쪽 테이블에 정보가 없는 사원 정보도 함께 조회하고 싶을 때가 있다.
- **Outer Join:** 이러한 경우 '아우터 조인(Outer Join)'을 사용한다.
- **조인 유형 이해:** 조인 시 왼쪽 테이블을 기준으로 모든 정보를 가져올지(Left Join), 오른쪽 테이블을 기준으로 모든 정보를 가져올지(Right Join), 또는 양쪽 모두 가능한 경우(Full Outer Join)를 고려해야 한다.
- **제한 사항 보완:** 일부 환경에서는 Full Outer Join이 제공되지 않을 수 있으므로, 집합 연산자(UNION 등)를 활용하여 유사한 기능을 구현할 수 있다.

## 2. 서브쿼리(Subquery)의 활용
- **배경:** 단일 쿼리만으로는 원하는 복잡한 정보를 얻기 어려울 때가 있다.
- **목적:** 서브쿼리는 메인 쿼리 내부에 또 다른 쿼리를 넣어 원하는 정보를 추출하는 기법이다.
- **활용 사례:** '스위스는 어떤 부서에 속하며, 그 부서의 평균 급여는 얼마인가?'와 같이, 한 번의 쿼리로 해결하기 어려운 연계 정보를 얻을 때 유용하다.
- **사용 범위:** 서브쿼리는 `WHERE` 절뿐만 아니라 `FROM` 절에서도 사용할 수 있다.
- **결과 형태:** 서브쿼리 실행 결과는 단일 로우만 나올 수도 있고, 여러 개의 로우가 나올 수도 있다.

## 3. 비교 연산자의 확장 (`ANY`, `ALL`)
- **기본 비교:** 일반적인 비교는 `=`(같다), `>`,`<` 등의 단일 값 비교에 국한된다.
- **다중 비교의 필요성:** 서브쿼리 결과가 여러 줄(여러 개의 값)이 나올 경우, 이를 비교하기 위해 `ANY`와 `ALL` 연산자가 필요하다.
- **작동 방식:**
    *   **`IN`:** 값의 동등성 비교 (`= ANY`와 유사).
    *   **`ANY`:** 결과 중 **하나라도** 조건을 만족하면 참으로 간주한다. (예: 특정 값이 최소값보다 크기만 하면 포함)
    *   **`ALL`:** 결과의 모든 조건을 만족해야 참으로 간주한다. (예: 특정 값이 최대값보다 커야만 포함)
- **핵심 차이:** `ANY`는 최소값(Minimum) 비교에 활용될 수 있으며, `ALL`은 최대값(Maximum) 비교에 활용될 수 있다는 차이가 있다.

## 4. 데이터 무결성을 위한 제약 조건(Constraints)
- **정의:** 테이블을 정의할 때 데이터의 정확성과 무결성을 유지하기 위해 부여하는 규칙이다.
- **종류 및 역할:**
    *   **기본 키 (Primary Key):** 테이블의 행을 고유하게 식별하는 키.
    *   **외래 키 (Foreign Key):** 다른 테이블의 기본 키와 관계를 맺어 테이블 간의 연관 관계를 유지한다.
    *   **NULL 제약:** 해당 컬럼에는 결측값(NULL)이 들어오지 않게 강제한다.
    *   **Unique 제약:** 컬럼의 값이 중복되지 않게 한다.
    *   **도메인 제약:** 컬럼에 들어갈 수 있는 값의 범위를 제한한다.
- **성능과 영향:**
    *   제약 조건은 데이터 입력 시마다 검사를 수행하므로, 매우 많은 데이터를 처리할 때는 속도가 느려질 수 있다.
    *   따라서 대용량 데이터 이동(예: 백업 복사) 시에는 성능 향상을 위해 일시적으로 제약 조건을 해제(OFF)하기도 한다.

## 5. 데이터 조회 성능 향상을 위한 인덱스(Index)
- **개념:** 데이터 조회(읽기) 성능을 높이기 위해 사용하는 도구이다. (책의 목차상으로는 인덱스, 실제 내용상으로는 검색 성능 향상 관련 내용이 많아 이를 통합하여 설명함.)
- **효과:** 데이터베이스가 데이터를 검색할 때, 전체 데이터를 순차적으로 검색하는 대신 인덱스를 이용하면 훨씬 빠르게 원하는 데이터를 찾아낼 수 있다.
- **구조적 원리:** 인덱스는 데이터를 정렬된 **트리(Tree)** 형태로 저장한다. 이 트리 구조 덕분에 데이터의 양과 관계없이 검색 범위를 줄여 빠르고 효율적으로 검색이 가능하다.
- **사용 시점:** 시스템이 느려지거나 성능 문제가 발견되었을 때, 어떤 컬럼에 인덱스를 적용하는 것이 가장 효과적일지 고려해 볼 필요가 있다.
- **주의 사항:** 인덱스를 잘못 걸 경우, 데이터 삽입, 수정, 삭제(DML 작업) 시마다 인덱스까지 조정해야 하므로 오히려 시스템의 속도를 저하시킬 수 있다. 인덱스 적용 여부를 신중하게 판단하는 것이 중요하다.

## 6. 데이터베이스 설계 및 모델링 (ERD 및 객체 매핑)
- **DB 설계 프로세스:** 데이터베이스를 설계할 때는 요구사항 및 데이터 처리 요건을 분석하는 것에서 시작한다.
- **개념적 설계 (ERD):** 분석된 요구사항을 바탕으로 개념적으로 데이터를 설계하는 단계가 **개체-관계 다이어그램(ERD, Entity-Relationship Diagram)**이다.
- **객체 지향 관점과의 비교:** ERD는 자바 등 객체지향 언어에서 클래스(Class)를 설계하는 것과 유사하게 생각할 수 있다.
- **관계형 모델링의 핵심:** 실질적인 데이터 구조를 설계할 때는, 모든 정보를 하나의 테이블에 담기보다는 연관된 정보들을 여러 개의 독립적인 테이블로 분리하고(정규화), 이 테이블들 사이에 관계(외래 키)를 설정하는 것이 중요하다. (예: '통장 정보'와 '입출금 내역'을 하나의 테이블에 담기보다 분리해야 효율적이다.)

## 1. 요구사항 분석: 필요한 데이터 정의하기
*   **개념:** 실제로 구현하고자 하는 애플리케이션(예: 은행 시스템)을 제시하고, 이 애플리케이션이 동작하기 위해 어떤 데이터와 정보가 필요한지 구체적으로 질문하고 정의하는 과정입니다.
*   **과정:**
    *   "무엇을 만들 것인가?"에 대한 정의부터 시작하여, 시스템이 수행할 모든 기능(작업, Use Case)을 목록화합니다.
    *   예를 들어, '은행' 시스템을 만든다면 단순히 '통장'만 필요한 것이 아니라, '소유자 정보', '계좌 번호', '잔액', 그리고 '입출금 기록(History)'에 대한 데이터까지 모두 고려해야 합니다.
    *   **핵심 질문:** 사용자가 이 기능을 사용하려면 어떤 데이터를 저장해야 할까?

## 2. 데이터 구조화 및 모델링: 데이터 분류 및 관계 설정
*   **데이터 그룹화:** 1단계에서 정의된 방대한 데이터들을 논리적으로 그룹화하여 테이블(테이블)로 분리하는 작업이 필요합니다. (예: '통장 정보'와 '고객 정보'는 별개의 테이블로 분리).
*   **중복 방지 원칙:** 만약 모든 정보를 하나의 테이블에 묶어 저장할 경우, 소유자 이름이나 주소 같은 정보가 여러 번 중복되어 저장될 위험이 크므로, 공통적인 정보는 별도의 마스터 테이블로 분리해야 합니다.
*   **관계 정의 (Cardinality):** 테이블 간의 논리적 관계를 설정하는 것이 중요합니다.
    *   **1:N 관계 (일대다):** 고객 1명이 통장 여러 개를 가질 수 있다. (고객 → 통장)
    *   **N:M 관계 (다대다):** 사용자(User)와 역할(Role)처럼, 한 명의 사용자가 여러 권한을 갖고, 하나의 권한이 여러 사용자에게 부여될 수 있는 관계는 중간 테이블을 활용해야 합니다. (예: User_Role 테이블 생성)

## 3. 데이터베이스 설계 과정: 개념적 → 물리적 구현 단계
*   **개념적 설계 (Conceptual Design):** 필요한 데이터를 정의하고 테이블을 설계하며, 테이블 간의 관계를 그림(ERD: Entity Relationship Diagram)으로 표현하는 단계입니다.
    *   **목표:** '무엇을 저장할지'와 '서로 어떻게 연결될지'를 정의합니다.
    *   **도구:** ER 다이어그램을 사용하며, 1:1, 1:N 등 관계의 종류를 시각화합니다.
*   **논리적 설계 (Logical Design):** 개념적 설계를 실제 데이터베이스 구조로 구체화하는 단계입니다.
    *   **핵심 작업:** 정규화(Normalization)를 수행하여 중복을 제거하고 데이터 무결성을 높입니다.
    *   **기술 적용:** 기본 키(Primary Key)와 외래 키(Foreign Key)를 지정하여 테이블 간의 참조 관계를 명확히 합니다.
*   **물리적 설계 (Physical Design):** 실제 데이터베이스 시스템에 맞게 성능과 구현을 최적화하는 단계입니다.
    *   **고려 요소:** 조회 속도, 저장 구조 등 성능(Performance)을 고려하여 테이블을 최적화합니다.
    *   **최종 단계:** DDL(Data Definition Language)을 사용하여 실제 데이터베이스에 테이블 스키마를 생성하고 적용합니다.

## 4. 고급 모델링 기법 및 객체-관계 매핑 이해
*   **복합적인 관계 처리 (N:M):** 다대다 관계를 처리하기 위해 중간 연결 테이블(Mapping Table)을 반드시 사용해야 합니다. (예: `User`와 `Role`을 연결하는 `User_Role` 테이블).
*   **객체 vs. 관계형 데이터베이스:**
    *   **관계형 DB (RDB):** 데이터를 테이블, 컬럼, 레코드로 구조화하며, 관계성을 중심으로 합니다.
    *   **객체(Object):** 현실 세계의 개념이나 클래스처럼 속성과 메서드를 갖는 단위입니다.
    *   **문제점:** 전통적으로 개발자들은 객체의 설계(객체 지향 관점)를 관계형 DB의 구조에 맞추어 설계하는 경향이 있었습니다. 이로 인해 두 관점 간의 불일치(불편함)가 발생했습니다.
    *   **해결책 (ORM/JPA):** 자바 객체와 관계형 DB 테이블 간의 차이점(Mapping)을 개발자가 직접 신경 쓰지 않아도 알아서 변환하고 값을 채워주는 기술이 필요하며, 이것이 ORM(Object-Relational Mapping)이며, 대표적인 예가 **JPA**입니다.

## 5. 프로젝트 진행 및 학습 태도
*   **실습의 중요성:** 데이터베이스 설계 과정은 책상에서 배운 지식을 실제 애플리케이션 기획에 적용하며 반복적으로 연습해야만 체득할 수 있습니다.
*   **성장의 관점:** 완벽하게 모든 것을 배운 후에 프로젝트를 시작하려 하기보다는, 아는 지식으로 작은 것부터 직접 '만들어보는' 과정(토이 프로젝트)이 훨씬 빠르고 효과적으로 학습 목표와 부족한 부분을 파악하는(메타인지) 데 도움을 줍니다.
*   **협업의 자세:** 팀 프로젝트에서는 조장이나 특정 인원에게 모든 책임을 맡기기보다, 팀원 개개인이 가진 강점을 파악하고 서로의 역할을 분담하며 협력하는 자세가 중요합니다.

## 1. 서비스 기획 및 화면 프로토타이핑 (데이터 설계의 시작)

*   **프로토타이핑의 중요성:** 프로젝트 기획 단계에서 실제 기능을 구현하기 전에, 만들고자 하는 애플리케이션의 화면을 미리 그려보는(화면 프로토타이핑) 것이 매우 중요함.
*   **기획 순서:** 구체적인 아이디어가 부족할 경우, 기능 구현보다 **화면 구성부터** (어떤 화면이 필요하고, 버튼을 눌렀을 때 어떤 화면으로 이동할지) 스케치하는 것을 추천함.
*   **효과:**
    *   **커뮤니케이션 원활화:** 말로만 설명하는 것보다 그림으로 구체화하면 팀원 간의 오해를 줄이고 효율적인 의사소통이 가능함.
    *   **데이터 흐름 파악:** 필요한 화면을 통해 어떤 정보(예: 이메일, 이름, 암호)가 필요하고, 그 정보를 저장하기 위해 어떤 데이터베이스 테이블이 필요한지 역추적할 수 있음.
*   **프로젝트 규모 설정:**
    *   처음부터 너무 큰 규모의 프로젝트를 잡는 것보다는, **개인이 잘 알고 있는 비즈니스**나 **작은 규모의 주제**부터 시작하여 익숙해진 뒤 점진적으로 규모를 키워나가는 것이 학습에 효과적임.

## 2. 데이터베이스 구조 설계 원칙 (테이블 관계 및 정규화)

*   **테이블 관계 설정:** 개체들 사이의 관계(예: 회원과 게시판, 회원과 권한)를 설계할 때, 이를 기반으로 테이블들을 생성해야 함.
    *   **외래 키(Foreign Key) 사용:** 한 테이블의 컬럼이 다른 테이블의 기본 키를 참조하는 관계(1:N)를 설정하는 것이 기본적인 설계 방법임.
    *   **다대다(M:N) 관계 처리:** 회원과 권한처럼 여러 개가 서로 연결될 수 있는 다대다 관계의 경우, 이들을 연결하는 **별도의 매핑 테이블**을 생성해야 함.
*   **데이터 무결성 및 최적화:**
    *   **인덱스(Index) 활용:** 데이터베이스의 무결성을 지키는 **유니크 제약 조건(Unique Constraint)**은 기본적으로 인덱스를 생성하는 효과가 있으며, 이는 데이터 검색 속도를 최적화하는 데 도움을 줌.
    *   **계획적인 설계:** 비즈니스가 어떻게 동작하는지 이해한 후, 논리적 분석을 거쳐 테이블을 설계하고, 그 위에 필요한 제약 조건과 인덱스를 추가하는 과정을 거쳐야 함.

## 3. 트랜잭션(Transaction)의 개념과 중요성

*   **정의:** 트랜잭션이란 데이터베이스에서 **하나의 논리적 작업 단위**로 처리되어야 하는 일련의 작업들을 의미함. (예: 계좌 이체)
*   **핵심 예시 (계좌 이체):** 계좌 이체는 'A 통장에서 출금'과 'B 통장으로 입금'이라는 두 개의 물리적인 작업으로 이루어지지만, 비즈니스 관점에서는 '하나의 작업(트랜잭션)'으로 묶여야 함.
*   **트랜잭션의 필수 조건:** 트랜잭션은 중간에 오류가 발생하더라도 모든 작업이 정상적으로 완료되었을 때만 그 결과를 반영(Commit)하고, 어느 하나라도 실패하면 처음 상태로 모두 되돌려야(Rollback) 하기 때문에 필수적임.
*   **트랜잭션 제어 (SQL 명령어):**
    *   **COMMIT:** 트랜잭션이 성공적으로 완료되었음을 시스템에 알리고, 데이터의 변경 사항을 영구적으로 반영(DB에 기록)시키는 명령.
    *   **ROLLBACK:** 트랜잭션 중간에 오류가 발생하여 작업이 중단되었을 때, 지금까지의 모든 변경 사항을 취소하고 트랜잭션 시작 전의 상태로 되돌리는 명령.
    *   **주의:** 트랜잭션 제어 명령은 데이터가 입력, 수정, 삭제 등 **실제로 변경되는 작업**에서 사용되며, 단순히 조회(SELECT)하는 작업은 트랜잭션 제어 대상이 아님.

## 4. 트랜잭션 제어 및 동시성 문제 이해 (Commit, Rollback, 격리)

*   **자동 커밋(Auto-Commit)의 이해:**
    *   기본적으로 DBMS는 자동 커밋(`AUTOCOMMIT=ON`) 모드로 설정되어 있어, 데이터를 변경하는 명령을 실행할 때마다 즉시 커밋이 발생하여 다른 세션(Session)에서도 변경 내용을 볼 수 있음.
    *   하지만 중요한 비즈니스 작업(예: 계좌 이체)의 경우, 작업 중간에 실패할 수 있으므로, **자동 커밋 기능을 끄고** (Auto-Commit = OFF) 모든 작업이 끝난 후 한 번에 커밋 또는 롤백을 처리해야 함.
*   **세션(Session)과 격리(Isolation):**
    *   데이터베이스에 접속하는 모든 프로그램(워크벤치, 커맨드 라인 등)은 독립적인 세션을 형성하며, 각 세션은 독립된 작업 공간을 가짐.
    *   트랜잭션이 진행되는 동안 변경된 데이터는 해당 세션에만 임시로 반영되며, 다른 세션에서는 아직 변경 사항이 보이지 않음.
*   **동시성 문제 (Concurrency Issues):**
    *   **잠금(Locking):** 한 세션이 특정 레코드에 트랜잭션을 걸어 작업하는 동안, 다른 세션이 해당 레코드에 접근하여 데이터를 변경하거나 조회하는 것을 막는 메커니즘이 작동함.
    *   **데드락(Deadlock):** 두 개 이상의 세션이 서로 상대방이 잠근 리소스(Lock)를 기다리느라 무한정 멈춰버리는 교착 상태를 의미함. DBMS는 이러한 상태를 감지하면 강제로 작업을 중단시키거나 오류(Timeout)를 발생시켜 시스템이 멈추는 것을 방지함.

## 1. 애플리케이션 설계 및 데이터 모델링 과정
*   **요구사항 분석의 중요성:**
    *   어떤 애플리케이션이 필요한지 정한 후, 이 기능이 수행해야 할 모든 요구사항(어떤 내용을 보여줄지, 어떤 기능이 필요한지)을 분석하는 단계가 필수적입니다.
    *   요구사항 분석 과정을 통해 필요한 데이터들이 자연스럽게 도출됩니다.
*   **데이터 도출 및 시각화 방법:**
    *   **비즈니스 이해도가 높은 경우:** 데이터 도출이 비교적 쉽습니다.
    *   **비즈니스 이해도가 낮은 경우:** 고객과의 커뮤니케이션을 통해 필요한 요건을 정의하고 데이터를 뽑아내는 과정이 필요합니다.
    *   **화면 프로토타이핑 활용:** 실제 구현할 애플리케이션의 화면(와이어프레임)을 먼저 그려보는 '화면 프로토타이핑'을 하면, 이 화면에 어떤 데이터들이 보여야 하는지를 정의함으로써 필요한 데이터들을 효과적으로 도출할 수 있습니다.
*   **데이터 모델링 기법 (ERD vs. 클래스 다이어그램):**
    *   **ER 다이어그램 (Entity Relationship Diagram):** 데이터를 정의하고, 어떤 테이블들로 묶는지를 정의하는 과정입니다.
    *   **클래스 다이어그램 (Class Diagram):** 애플리케이션이 필요한 객체들(Entity)을 정의하는 과정입니다.
    *   **차이점 및 연관성:**
        *   관계형 DB는 테이블로 표현되지만, 클래스에서는 타입으로 다른 객체(예: 부서 객체) 자체를 포함할 수 있어 모델링에 더 유연성이 있습니다.
        *   현실적으로 두 설계 방식(객체와 테이블)을 함께 사용하는 경우도 많습니다. 데이터베이스의 레코드(Row) 하나가 클래스(객체) 하나에 대응되므로, 객체 구조를 따르는 것이 데이터 처리(입력/수정)가 간편할 때가 많습니다.
*   **정제화(Refinement):**
    *   초기 설계 단계에서 테이블 간에 데이터가 중복되는 부분이 있는지, 혹은 테이블이 너무 작게 쪼개져 있는 부분은 없는지 등을 점검하고 수정하는 정교화 과정이 필요합니다.

## 2. 트랜잭션 이해 및 데이터 무결성 관리
*   **트랜잭션의 정의 및 중요성:**
    *   트랜잭션은 **'나눌 수 없는 작업 단위'**를 의미합니다. 하나의 기능 수행 과정(예: 게시글 작성)이 데이터베이스에 여러 개의 데이터를 입력하거나 수정하는 일련의 작업을 포함합니다.
    *   트랜잭션은 **전부 성공하거나 (COMMIT) 전부 실패해야 (ROLLBACK)** 합니다. 만약 중간에 하나라도 실패하면, 이전에 성공했던 작업까지 모두 취소되어야 합니다 (롤백).
*   **트랜잭션의 원자적 특성 (ACID 속성):**
    *   **원자성(Atomicity):** 모든 작업이 성공하거나, 아니면 모든 작업이 취소되어야 합니다.
    *   **일관성(Consistency):** 트랜잭션의 시작 전후로 데이터의 무결성(유효성)이 유지되어야 합니다. (예: 게시글이 없는데 사진만 남는 상황 방지).
    *   **격리성(Isolation):** 동시에 여러 트랜잭션이 실행될 때, 서로 간에 간섭이 없어야 독립적으로 작업이 수행될 수 있습니다.
    *   **지속성(Durability):** 트랜잭션이 완료되면 영구적으로 저장되어야 합니다.
*   **동시성 문제와 교착상태 (Deadlock):**
    *   여러 세션이 동시에 DB에 접근할 때, 만약 A 세션이 B 자원을 기다리고 있고, 동시에 B 세션도 A 자원을 기다리는 상태가 되면(서로의 자원을 점유하며 대기), 아무도 작업을 진행할 수 없게 됩니다. 이를 **교착상태**라고 하며, DBMS는 이를 감지하고 적절한 오류를 발생시켜 작업을 정리해 줍니다.
*   **트랜잭션 제어 흐름:**
    *   `INSERT`, `UPDATE`, `DELETE` 등 데이터가 바뀌는 작업(DML)은 기본적으로 트랜잭션 제어 범위에 속합니다.
    *   `CREATE TABLE` 등 데이터 구조를 정의하는 작업(DDL)은 예외적으로 트랜잭션과 독립적이어서, 중간에 발생해도 강제적으로 커밋(COMMIT)이 될 수 있습니다.
*   **프레임워크의 활용 (Spring Framework):**
    *   실제 애플리케이션 개발 시, 트랜잭션 처리는 매우 복잡하므로 스프링과 같은 프레임워크를 사용합니다.
    *   개발자가 특정 함수에 `@Transactional`과 같이 **선언적 방식**으로 트랜잭션 적용을 선언만 해주면, 프레임워크가 내부적으로 트랜잭션 시작, 성공 시 커밋, 실패 시 롤백 코드를 대신 처리해 줍니다.

## 3. 데이터베이스 스키마 수정 (DDL) 및 제약조건 활용
*   **ALTER TABLE 구문 활용:**
    *   `ALTER TABLE` 구문을 사용하여 테이블의 구조를 수정할 수 있습니다.
    *   **컬럼 추가:** `ADD COLUMN`을 사용해 컬럼을 추가할 수 있습니다.
    *   **컬럼 수정:** 컬럼의 타입이나 이름 등을 변경할 수 있습니다.
    *   **삭제:** 컬럼을 삭제하는 것도 가능합니다.
*   **제약조건 (Constraint) 추가:**
    *   `ALTER TABLE ... ADD CONSTRAINT` 구문을 사용하여 제약조건을 추가합니다.
    *   **제약조건 이름 지정:** 제약조건은 중복될 수 없으므로, 이름(예: `fk_board_user`)을 명확하게 지정하여 추가하는 것이 좋습니다.
    *   **이점:** 제약조건에 이름을 부여하면, 나중에 이 제약조건을 특정 목적(비활성화, 삭제 등)으로 사용할 때 데이터 딕셔너리 전체를 조회할 필요 없이 바로 접근할 수 있어 관리 효율성이 높아집니다.

## 1. JDBC의 정의 및 데이터베이스 연동의 목적
JDBC(Java Database Connectivity)는 자바 프로그램이 데이터베이스(DB)에 접속하여 데이터를 조작할 수 있도록 하는 표준 API입니다.

- **핵심 목적:** 프로그램 내부에서 데이터베이스에 데이터를 **입력, 수정, 조회, 삭제 (CRUD)**하는 기능을 수행하기 위함입니다.
- **기능적 역할:** 자바 프로그램과 데이터베이스 시스템 간의 '연결(Connectivity)'을 담당하는 브릿지 역할을 합니다.
- **접속 방식의 일관성:** 데이터베이스를 사용하든 (Workbench, CMD 등), 자바 프로그램을 이용하든, DB에 접근하기 위해서는 **반드시 접속 과정**과 **정보(ID, 비밀번호, 호스트, 포트)**를 제공하는 과정이 필수적입니다.

## 2. 데이터베이스 접속의 필수 원리 이해
데이터베이스를 사용하는 과정은 어떤 환경에서든 기본 원칙이 동일합니다.

- **필수 과정:**
    1.  **접속(Connection):** DB에 접속하기 위한 초기 과정이 필요합니다.
    2.  **쿼리 작성 및 실행:** SQL(QL) 문을 작성하고, 이를 DB에 실행 요청합니다.
    3.  **결과값 처리:** DB가 반환한 결과값(Result Set)을 받아 프로그램에서 활용합니다.
- **데이터베이스 벤더의 구현:** 각 DBMS 벤더(MySQL, Oracle 등)들은 이 접속, 쿼리 수행, 결과 반환 등 DB 상호작용에 필요한 **구현체**를 갖추고 있습니다.

## 3. 표준 인터페이스의 중요성 (추상화의 개념)
다양한 DB 벤더들이 각자 다른 방식으로 시스템을 구현하고 있기 때문에, 표준화가 이루어지지 않는다면 사용자는 매우 불편을 겪을 것입니다.

- **문제점:** 각 DBMS 벤더가 자신들만의 방식으로 DB 접근 방법을 구현하고 있어, 하나의 애플리케이션에서 여러 DB를 사용하기가 어렵습니다.
- **해결책: 인터페이스(Interface) 도입:**
    - 자바는 `java.sql`과 같은 표준 인터페이스를 제공하여 **통일된 규격**을 제시합니다. (예: `java.util.List` 인터페이스처럼)
    - **역할 분담:** 인터페이스는 ‘무엇을 할 수 있는지’에 대한 **규격(계약)**만 정의하고, 실제 동작 방식은 각 DBMS 벤더에게 맡깁니다.
    - **사용자 이점:** 개발자는 DB가 MySQL인지, Oracle인지 알 필요 없이, 이 표준 인터페이스만 사용하면 되기 때문에 높은 이식성과 사용 편의성을 확보할 수 있습니다.

## 4. 프로젝트 환경에서의 의존성 관리 (Maven/Gradle)
JDBC를 사용하려면 해당 DB 벤더가 표준 인터페이스를 따르도록 구현한 **드라이버 라이브러리(구현체)**가 필요합니다. 이 라이브러리를 프로젝트에 통합하는 과정이 복잡하고 오류 발생률이 높습니다.

- **수동 관리의 어려움:**
    - 과거 방식대로라면, 개발자가 직접 각 DB의 공식 페이지에 방문하여 드라이버 JAR 파일을 다운로드하고, 프로젝트의 클래스패스(Classpath)에 수동으로 경로를 추가해야 했습니다.
    - 여러 라이브러리를 다루는 협업 환경에서는 버전 충돌이나 설정 오류가 발생하기 매우 쉽습니다.
- **빌드 도구의 필요성 (Maven/Gradle):**
    - 이러한 불편함을 해결하기 위해 Maven이나 Gradle 같은 **빌드 도구**를 사용합니다.
    - **원리:** 개발자가 빌드 파일(예: `build.gradle`)에 "이 라이브러리가 필요하다"라는 의존성(Dependency)을 명시적으로 선언하기만 하면, 빌드 도구가 자동으로 해당 라이브러리들을 **Maven 중앙 저장소(Repository)** 등에서 다운로드하고, 프로젝트에 사용할 수 있도록 환경을 완벽하게 설정해 줍니다.
    - **결과:** 개발자는 복잡한 수동 설정 과정 없이 코드 작성에만 집중할 수 있게 됩니다.

2. **표준화의 필요성:** 여러 DB 벤더의 차이를 극복하기 위해 자바가 표준 인터페이스를 제공.
3. **현대적 개발 환경:** DB 드라이버 라이브러리 같은 외부 의존성 관리를 위해 Maven 또는 Gradle 같은 빌드 도구를 사용해야 함.

## 1. JDBC 프로그래밍 환경 설정 및 패키지 구조
- **패키지 사용의 중요성:** 애플리케이션 개발 시, 클래스를 패키지(Package)로 관리해야 나중에 클래스를 찾지 못하는 문제를 방지할 수 있습니다.
- **작업 목표:** 라이온DB 등 특정 패키지 이름을 지정하여 JDBC 클래스들을 구조화하고 사용하는 것이 좋습니다.
- **최우선 과제:** 데이터베이스에 안정적으로 접속하는 테스트를 가장 먼저 진행해야 합니다.

## 2. 데이터베이스 연결 (Connection)
- **접속 과정의 정의:** 데이터베이스에 접속하는 행위는 'Connection 객체'를 얻어내는 과정으로 추상화됩니다.
- **접속 메커니즘:** 자바의 `java.sql` 패키지에 있는 `DriverManager` 클래스의 `getConnection()` 메서드를 사용합니다.
- **필수 정보:** 이 메서드를 사용하려면 DB 접속에 필요한 세 가지 정보(URL, 사용자명(User ID), 비밀번호(Password))를 반드시 준비해야 합니다.
- **URL의 구성:** DBMS(데이터베이스 관리 시스템)마다 URL 형식이 다릅니다. (예: MySQL은 `mysql://`로 시작)
- **접속 성공의 의미:** `Connection` 객체를 성공적으로 얻어내는 것은 '접속이 잘 되었음'을 의미합니다.
- **예외 처리:** 접속 시에는 `SQLException`과 같은 예외가 발생할 수 있으므로, 반드시 `try-catch` 구문을 이용한 예외 처리가 필요합니다.
- **자원 관리:** 연결(Connection)과 같은 자원은 사용 후 반드시 닫아주는(Close) 것이 좋습니다.

## 3. SQL 구문 추상화 객체 이해: Statement vs. PreparedStatement
- **SQL 구문의 추상화:** SQL 쿼리문(Query) 자체도 추상화된 객체가 필요합니다.
    *   접속 추상화: `Connection`
    *   쿼리 추상화: `Statement`
- **Statement 객체:**
    *   쿼리문 전체를 문자열 형태로 전달하여 사용합니다. (예: `"SELECT * FROM table"`)
    *   쿼리문 내에 변수를 포함할 경우, 문자열 연결(Concatenation)을 통해 쿼리를 완성해야 합니다. (코드 가독성 및 유지보수성이 떨어짐)
- **PreparedStatement 객체 (권장):**
    *   가장 효율적이며 안전한 방법으로, 쿼리문 내의 값이 들어갈 자리를 `물음표(?)`로 표시합니다.
    *   실제 데이터 값은 나중에 별도로 값을 바인딩(set)하여 넣어줍니다. (예: `?`에 이름을 넣는 방식)
    *   **장점:**
        *   **성능:** DBMS가 쿼리문의 구조(문장)를 미리 해석(Prepare)해두기 때문에, 같은 쿼리를 반복적으로 실행할 때 재해석 과정 없이 빠르게 실행할 수 있습니다.
        *   **안전성:** SQL 인젝션(SQL Injection) 공격으로부터 안전합니다. (데이터 타입별로 값을 강제하여 처리)

## 4. SQL 문 실행 (CRUD) 및 트랜잭션 관리
- **SQL 문 실행 단계:** 데이터베이스에 데이터를 입력하거나 수정/삭제하는 과정은 반드시 순서가 있습니다. (접속 $\rightarrow$ 쿼리문 작성 $\rightarrow$ 실행)
- **쿼리문 작성 및 실행의 차이:**
    *   **`PreparedStatement` 활용:** 가장 먼저 쿼리문(`?`)을 작성하고, 이후 `Connection` 객체에서 `prepareStatement()`를 통해 쿼리 객체를 얻어냅니다.
    *   **값 바인딩:** `setString()`, `setInt()` 등의 메서드를 사용하여 물음표 순서와 데이터 타입에 맞춰 값을 설정합니다.
- **실행 메서드의 구분:** SQL 문의 종류에 따라 사용해야 하는 실행 메서드가 다릅니다.
    *   `executeQuery()`: `SELECT` 문을 실행할 때 사용합니다. 결과를 담는 `ResultSet` 객체가 반환됩니다.
    *   `executeUpdate()`: `INSERT`, `UPDATE`, `DELETE` 문을 실행할 때 사용합니다. 실행된 행의 개수를 나타내는 `int` 값이 반환됩니다.
- **트랜잭션(Transaction) 관리:**
    *   **Auto Commit의 이해:** 기본적으로 `Connection`은 `autoCommit = true`로 설정되어 있어, 명령(Statement)을 실행할 때마다 즉시 DB에 반영(Commit)됩니다.
    *   **수동 트랜잭션 관리:** 여러 개의 작업을 하나로 묶어 원자적으로 처리(전체 성공 또는 전체 실패)하고 싶다면, `autoCommit`을 `false`로 설정해야 합니다.
        *   모든 작업이 성공했을 경우: `conn.commit()`을 호출하여 데이터를 반영합니다.
        *   작업 중 오류가 발생했을 경우: `catch` 블록에서 `conn.rollback()`을 호출하여 모든 변경 사항을 취소(롤백)해야 합니다.

## 5. 다음 단계 및 코드 리팩토링 방향
- **데이터 조회 (SELECT):** `ResultSet` 객체를 얻어와 원하는 데이터 필드를 순회하며 자바 객체로 옮겨오는 과정이 필요합니다. (대용량 데이터 처리 시 성능 고려 필요)
- **코드 구조 개선 (리팩토링):**
    *   현재는 모든 로직을 `main` 메서드 안에 구현하고 있으나, 실제 서비스에서는 기능을 클래스 단위의 메서드로 분리해야 합니다.
    *   데이터베이스 접속, CRUD 등의 공통 로직이 반복되는 경우, 이를 전담하는 다른 클래스로 추출하여 중복을 제거하고 코드를 구조화하는 것이 중요합니다.
