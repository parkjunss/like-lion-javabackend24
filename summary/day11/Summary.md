## 1. 강의 시작 및 학습 자세 점검

*   **일상 회고 습관:** 잠자리에 들기 전, 오늘 하루 동안 잘했던 점이나 배우고 뿌듯했던 경험을 긍정적으로 회고하는 습관을 들이는 것이 좋습니다.
*   **지속적인 노력의 중요성:** 코딩을 배우는 것 자체가 하나의 과정이며, 당장의 어려움에 좌절하지 않고 꾸준히 개념을 붙잡고 나아가는 끈기가 중요합니다.

## 2. 개념적 사고의 중요성: 넓고 깊게 생각하기
*   **전체적인 연결고리 파악:** 학습하는 각 개념(예: 컬렉션, 인터페이스)을 개별적인 요소로 분리해서 생각하지 말고, 큰 그림에서 유기적으로 연결되는 시스템으로 이해해야 합니다.
*   **구현 원리 질문:** 단순히 API 사용법을 암기하는 것을 넘어, "왜 이 자료구조는 이렇게 설계했을까?", "왜 이 인터페이스가 필요했을까?"와 같은 근본적인 질문을 던지며 구조를 파악하는 것이 중요합니다.
*   **추상화와 일반화의 관점:** 모든 시스템을 추상화된 개념(인터페이스)을 통해 접근하는 관점을 유지해야 합니다. (예: 모든 움직이는 객체에 '날다'라는 인터페이스를 적용하는 것처럼)

## 3. 핵심 프로그래밍 개념 정리 (자료구조, 제네릭, 비교 기준)
**A. 자료구조 (Data Structure)**
*   **정의:** 대량의 데이터를 효율적으로 관리하고 처리하는 방법을 체계적으로 정리해 놓은 것을 의미합니다.
*   **선택 기준:** 데이터를 어떻게 관리할지(목적)에 따라 적절한 자료구조를 선택하는 것이 핵심입니다.
    *   **List (리스트):** 순서(인덱스)가 있고, 중복된 객체 저장이 가능합니다.
    *   **Set (셋):** 순서가 없고, 중복을 허용하지 않습니다.
    *   **Map (맵):** Key와 Value 쌍으로 저장하여 키를 통해 빠르게 값을 검색할 수 있습니다.
*   **효율성 고려:** 데이터 양이 많아지거나 특정 검색/조회 기능이 필요할 때, 단순히 리스트를 쓰는 것보다 셋이나 맵을 사용해야 성능(시간 복잡도)이 크게 개선될 수 있습니다.

**B. 제네릭 (Generics)**
*   **발생 배경:** 자바의 초기 `Object` 타입으로 모든 객체를 담을 경우, 실제 객체의 특징(예: 문자열의 메서드)을 사용할 수 없고, 매번 강제 형변환(Casting)이 필요하여 번거롭고 에러 발생 가능성이 높았습니다.
*   **해결책:** 제네릭은 "다용도 박스"와 같이 **타입이 정해지지 않은 상태로 설계도(가상 타입)**를 만들되, **실제 객체를 생성할 때 타입을 명시하여** 사용할 수 있게 합니다.
*   **장점:** 코드의 안정성을 높이고 사용의 편리성(타입 안전성)을 확보합니다.

**C. 객체 비교 기준 (Comparable vs. Comparator)**
*   **비교 필요성:** 객체들은 비교(Comparison)가 필요할 때가 있습니다 (예: 정렬). 하지만 누가 크다고 할 기준이 명확하지 않습니다.
*   **`Comparable` (비교 가능):** 객체 *자체*가 자신을 비교할 기준을 가지고 있을 때 구현합니다. (예: `String`은 알파벳 순으로 비교 가능).
*   **`Comparator` (비교자):** 객체가 아니라 *사용자*가 임의로 비교 기준을 정의할 때 사용합니다. (예: 사람 객체는 나이로 비교할지, 키로 비교할지 결정하는 기준).
*   **시스템 설계 관점:** 이 두 인터페이스의 존재 덕분에 Java의 `Collections.sort()` 같은 유틸리티 클래스가 유연하게 동작할 수 있습니다.

## 4. 실습 및 프로젝트 확장 방향
*   **자바 API의 설계 관점 이해:** `java.time` 패키지(LocalDate, LocalTime 등)처럼, 시대 변화에 따라 기존 클래스(Date)의 한계를 보완하며 더 안정적이고 확장성이 높은 추상 클래스를 설계하는 과정을 이해하는 것이 중요합니다.
    *   **결합도 감소:** 핵심 개념은 **'결합도(Coupling)'**를 낮추는 것입니다. 구현체를 추상 클래스나 인터페이스를 통해 사용함으로써, 실제 구현체가 바뀌어도 코드를 최소한으로 수정할 수 있게 만듭니다.
*   **토이 프로그래밍 (Toy Programming):**
    *   가장 효과적인 학습 방법은 하나의 흥미로운 주제(예: 은행 시스템, 도서 관리 시스템)를 정하여 지속적으로 프로젝트를 확장하고 개선하는 것입니다.
    *   새로 배운 개념이나 API(예: 상수를 활용한 코드, 새로운 자료구조)가 현재 진행 중인 프로젝트에 어떻게 적용될 수 있을지 고민하며 코드를 '장난감처럼' 꾸준히 키워 나가는 것이 개념 이해에 가장 도움이 됩니다.

## 1. 예외(Exception)의 이해: 개념 및 필요성
- **예외의 정의:** 예외(Exception)란 프로그램의 정상적인 흐름에서 벗어나 예측하지 못한 상황이나 오류가 발생했을 때 이를 의미한다. (예: 출근 과정에서 커피를 쏟거나, 코드에서 배열 인덱스 범위를 벗어나는 경우 등)
- **필요성 (Why Exception Handling?):**
    - 예외 처리가 없다면, 예상치 못한 오류 발생 시 프로그램이 갑작스럽게 종료(Crash)된다.
    - 예외 처리는 프로그램의 **안정성**과 **신뢰성**을 보장하여, 예외 상황이 발생하더라도 프로그램이 정상적인 흐름을 유지하며 안전하게 동작을 계속할 수 있도록 돕는 기법이다.
- **예외와 에러의 구분:**
    - **Error:** 복구할 수 없는 치명적인 오류로, 보통 시스템 수준의 문제입니다.
    - **Exception:** 프로그래밍을 통해 개발자가 예측하고 처리할 수 있는 문제 상황이다.
- **예외의 종류:** 크게 런타임 예외(Runtime Exception)와 체크드 예외(Checked Exception)로 나뉜다.

## 2. 예외 처리의 기본 메커니즘: `try-catch`
- **`try` 블록:** 예외가 발생할 가능성이 있는 코드를 이 블록 안에 묶는다. (예: 인덱스 접근 시 예외가 발생할 수 있다고 예상되는 코드를 배치)
- **`catch` 블록:** `try` 블록에서 특정 종류의 예외가 발생했을 때 실행되는 코드 블록이다.
    - 예외 처리를 통해 프로그램이 중단되는 대신, `catch` 블록 내의 로직으로 흐름을 우회하여 정상적으로 다음 작업을 수행할 수 있게 한다.
    - **여러 개의 `catch` 사용:** 발생하는 예외의 타입별로 다른 처리를 원할 때 여러 개의 `catch` 블록을 순차적으로 사용할 수 있다.
- **예외 정보 활용:**
    - **`e.printStackTrace()`:** 예외가 발생한 원인(메시지)과 위치를 포함한 상세한 추적 스택(Stack Trace) 정보를 출력하여 디버깅에 활용한다.
    - **`e.getMessage()`:** 예외의 발생 원인에 대한 간결한 문자열 메시지를 얻을 수 있다.
- **제어 흐름 이해:**
    - 예외가 발생하지 않으면, `catch` 블록은 실행되지 않고, `try` 블록 다음의 코드가 순차적으로 실행된다.
    - 예외가 발생하면, 프로그램은 현재 실행 흐름을 멈추고 가장 먼저 발견되는 적절한 `catch` 블록으로 제어권을 넘겨 실행한다.

## 3. 프로그램 안정성 보장: `finally` 블록
- **`finally` 블록의 역할:** `try` 또는 `catch` 블록의 실행 결과와 무관하게 **반드시 실행되어야 하는** 코드를 작성할 때 사용한다.
- **필수 사용 상황:** 자원(Resource)을 사용하고 해제하는 과정 (예: 데이터베이스 연결, 파일 입출력, 네트워크 접속)에서 반드시 연결을 끊거나(Close) 종료해야 하는 로직을 배치한다.
- **작동 원리:**
    - `try` 블록에서 예외가 발생하여 `catch`가 실행되든,
    - `try` 블록이 정상적으로 모두 실행되든,
    - `finally` 블록의 내용은 예외 발생 여부와 관계없이 무조건 실행된다.

## 4. 자원 관리와 자동 처리: `try-with-resources`
- **자원 누수(Resource Leak) 방지:** 데이터베이스 연결, 파일 스트림 등은 사용 후 반드시 닫아줘야 시스템 자원 고갈이나 문제를 막을 수 있다.
- **`try-with-resources`의 도입:** 반복적으로 `finally` 블록을 사용하여 자원 해제(Close) 코드를 작성하는 번거로움을 줄이기 위해 자바에서 문법적으로 추가된 기능이다.
- **원리:** `try` 괄호 안에 자원을 선언하는 객체를 포함하면, 이 객체가 **`AutoCloseable` 인터페이스**를 구현했는지 확인하고, 블록이 끝나는 순간 (예외 발생 여부와 무관하게) 해당 자원을 자동으로 `close()` 메서드를 호출하여 닫아준다.
- **활용:** 자원을 다루는 코드를 작성할 때 이 구문을 사용하면, 자원 해제(Close) 로직을 명시적으로 적지 않아도 되므로 코드가 간결해지고 안정성이 높아진다.

## 1. 예외(Exception)의 기본 개념 이해
- **예외의 정의:** 프로그램이 정상적으로 동작하는 과정 중에 발생하는 비정상적인 상황을 의미합니다. (예: 배열의 인덱스 범위를 벗어남, 잘못된 타입의 데이터를 입력함 등)
- **예외 객체의 역할:** 예외 객체 자체는 특별한 기능을 수행하기보다는, '지금 어떤 예외 상황이 발생했다'는 사실을 알려주는 신호탄 역할을 합니다.
- **발생 원인:** 자바와 같은 언어 환경에서 미리 정의된 규칙을 벗어나는 상황이 발생했을 때, 시스템이 이를 인식하고 예외 객체를 발생시켜 개발자에게 이를 알려줍니다.

## 2. 에러(Error)와 예외(Exception)의 종류 및 구분
- **컴파일 에러 (Compile Error):** 문법적 오류(Syntax Error)나 타입 오류(Type Error) 등 코드가 문법적으로 잘못되어 컴파일러가 아예 실행 파일을 만들지 못하게 막는 오류입니다. (예: `String` 변수에 정수 값을 넣으려는 시도)
- **런타임 에러 (Runtime Error):** 코드는 문법적으로 완벽하나, 프로그램이 실제로 실행되는 과정에서 발생하는 오류입니다.
    *   **Error (에러):** 심각도가 높아 프로그래머가 직접적으로 수습하기 어려운, 시스템 레벨의 문제 (예: 메모리 부족, 스택 오버플로우 등).
    *   **Exception (예외):** 프로그래머가 적절한 로직을 통해 처리하거나 예측할 수 있는, 상대적으로 덜 심각한 오류 상황입니다. (예: 0으로 나누기, 파일 찾기 실패 등)

## 3. 예외 처리의 기본 문법 및 흐름 제어
- **예외 처리의 필요성:** 예외 처리를 하지 않으면 프로그램이 비정상적으로 종료되어 데이터 손실, 보안 취약점 노출, 사용자 경험 저하 등이 발생할 수 있습니다.
- **Try-Catch 블록:**
    *   `try`: 예외가 발생할 가능성이 있는 코드를 이 블록 안에 묶어 실행합니다.
    *   `catch`: `try` 블록에서 특정 종류의 예외가 발생했을 때, 해당 예외를 잡아서(Catch) 정의된 로직을 통해 처리합니다.
    *   `finally` (언급되지는 않았으나 필수 지식): 예외 발생 여부와 관계없이 반드시 실행되어야 하는 코드를 처리합니다.
- **다중 Catch:** 하나의 블록 내에서 여러 개의 예외가 발생할 수 있으므로, 여러 개의 `catch` 블록을 순차적으로 작성하여 발생한 예외의 종류에 따라 다르게 처리할 수 있습니다. (🚨 **주의:** 가장 구체적인 예외를 먼저, 그리고 가장 포괄적인 예외를 나중에 배치해야 합니다.)

## 4. 예외 전파 및 처리 책임의 원칙
- **예외 전파 (Throwing):** 현재 메서드에서 발생한 예외를 처리하지 않고, 이 예외를 호출한 상위 메서드 쪽으로 넘겨주는 행위입니다.
- **`throws` 키워드 (선언):** 메서드의 시그니처(Signature)에서 "이 메서드는 예외를 던질 수 있다"고 미리 선언합니다. 이 예외는 호출한 쪽에서 반드시 처리해야 함을 명시합니다.
- **`throw` 키워드 (실행):** 코드 블록 내에서 "지금 당장 이 예외를 발생시키겠다"고 강제로 던지는 행위입니다.
- **예외 처리의 주체:**
    *   **자체 처리 (Try-Catch):** 예외가 발생했을 때 현재 메서드 내에서 직접 해결하는 방법입니다.
    *   **책임 위임 (Throwing):** 현재 메서드에서 해결할 수 없는 예외는 호출한 쪽(메인 로직)에서 처리하도록 상위 단계로 던져주는 것이 좋습니다.
    *   **원칙:** 개발자 스스로 처리할 수 있는 범위까지는 직접 처리하고, 더 상위 레벨의 맥락적 이해가 필요한 문제는 전파하는 것이 구조적으로 바람직합니다.

## 5. 체크드(Checked)와 언체크드(Unchecked) 예외의 이해
- **체크드 예외 (Checked Exception):**
    *   **특징:** 컴파일 시점(컴파일러)에 "이 예외는 반드시 처리해야 한다"는 강제성을 부여합니다.
    *   **발생 배경:** 외부 환경(I/O, 네트워크, 파일 접근 등)과 관련된 예측 가능한 문제들이 주로 여기에 속하며, 예외 처리가 누락되면 컴파일 자체가 불가능합니다.
    *   **예시:** `FileNotFoundException` 등
- **언체크드 예외 (Unchecked Exception):**
    *   **특징:** 컴파일 시점에서는 처리를 강제하지 않습니다.
    *   **발생 배경:** 주로 프로그래머의 논리적 실수나 구현상의 부주의로 인해 발생하는 내부적인 문제들이 많습니다.
    *   **예시:** `NullPointerException`, `ArithmeticException` (0으로 나누기) 등이 대표적입니다.
- **핵심 이해:** 모든 예외는 `Throwable`을 상속받으며, 중간에 `RuntimeException`이라는 개념을 통해 체크드/언체크드가 구분됩니다. (🚨 **주의:** 언체크드 예외라 할지라도, 실제 프로그램 실행 시에는 예외 처리가 안 되면 프로그램이 종료됩니다.)

## 1. 예외 처리의 개념 및 필요성 (Exception Concept)
*   **예외의 정의:** 프로그램이 정상적으로 수행되는 과정에서 예측하지 못한 비정상적인 상황(Unexpected Events)이 발생하는 것. (예: 범위를 벗어난 인덱스 접근, 파일명 오타 등)
*   **예외 발생의 문제:** 예외가 처리되지 않으면 시스템 전체가 예고 없이 종료되어 심각한 오류를 유발한다.
*   **예외 처리의 목적:** 프로그램 코드 작성 시부터 잠재적 오류 상황을 예측하고, 예외가 발생하더라도 시스템이 종료되지 않도록 사전에 대비책(로직)을 마련하는 것.
*   **애플리케이션 설계 관점:** Java API 등 상용 클래스들은 개발자가 사용 시 반드시 예외 처리를 하도록 설계되어 있음. (필수적인 사용 단계가 존재함)

## 2. 기본적인 예외 처리 구문 (try-catch-finally)
*   **`try` 블록:** 예외가 발생할 가능성이 있는 실제 실행 코드를 묶어주는 블록.
*   **`catch` 블록:** `try` 블록에서 발생한 예외를 포착하여 처리하는 블록.
    *   여러 개의 `catch` 블록을 사용할 수 있으며, 상속 개념을 활용하여 구체적인 예외부터 일반화된 예외 순서로 처리하는 것이 좋다.
*   **`finally` 블록:** 예외 발생 여부와 관계없이 (발생했든, 처리했든, 안 했든) *반드시* 실행되어야 하는 정리 로직(Cleanup Code)을 담는 블록.
    *   예외 처리 시 가장 중요한 자원 해제(Resource Release)를 보장하는 역할을 한다.
*   **예외 발생 방식 (Throwing):**
    *   **Catch:** 예외를 포착하여 처리하는 것.
    *   **Throw (수동 발생):** 프로그래머가 명시적으로 예외 객체를 생성하여 강제로 발생시키는 행위.
    *   **Throws (선언):** 메서드가 사용할 때 "이 예외를 발생시킬 수 있음"을 호출자에게 미리 알려주는 것 (컴파일 상의 계약).

## 3. 예외의 종류 및 사용자 정의 예외
*   **예외의 분류:**
    *   **Checked Exception (체크 예외):** 컴파일 시점부터 개발자가 예외 처리를 **반드시** 해야 한다고 컴파일러가 강제하는 예외. (주로 외부 환경과 관련된 예외: 파일 I/O 등)
    *   **Unchecked Exception (런타임 예외):** 실행(Runtime) 시점에 예외가 발생하는 것. 컴파일러가 처리를 강제하지 않으며, 주로 프로그래밍 로직상의 오류가 원인이다. (예: `NullPointerException`)
*   **사용자 정의 예외 (Custom Exception):**
    *   자바가 제공하는 일반적인 예외 외에, 특정 도메인이나 프로젝트 규칙(Rule)을 위반했을 때 발생하는 예외를 개발자가 직접 정의하여 사용한다.
    *   **핵심 목적:** 예외 객체 자체가 기능을 하는 것이 아니라, **"이 예외가 발생한 이유"를 명확하게 알려주고 코드의 가독성을 높이기 위함** (의미 부여).
    *   **정의 방법:** 기본 예외 클래스(예: `RuntimeException` 또는 `Exception`)를 상속받아 클래스를 정의하면 된다.
    *   **활용:** 생성자를 오버로딩하여 예외가 발생했을 때 상황에 맞는 사용자 지정 메시지를 포함할 수 있다.

## 4. 자원 관리 및 최신 문법 (try-with-resources)
*   **자원 해제의 어려움:** 파일 스트림(File Stream)이나 데이터베이스 연결(DB Connection) 등 시스템 자원(Resource)을 사용할 경우, 코드가 길어지더라도 반드시 `finally` 블록을 이용해 자원을 닫아주어야 한다. (자원을 열고, 사용하고, 닫는 과정 반복)
*   **`try-with-resources` 문법:**
    *   Java 7에서 도입된 문법으로, 자원(Resource)을 선언하는 시점과 `try` 블록을 결합하여 코드를 간결하게 만든다.
    *   `try` 선언부에서 자원을 생성할 때, 해당 자원이 `AutoCloseable` 인터페이스를 구현했는지 확인하고, `try` 블록이 끝나면 *자동으로* `close()` 메서드를 호출하여 자원을 해제해 준다.
    *   **원칙:** 자원이 필요하면 `try-with-resources`를 사용하며, 자원은 반드시 `AutoCloseable`을 구현해야 한다.
*   **주의 사항 (Clean Code):** `catch` 블록을 단순히 비워두는 것(Empty Catch Block)은 아무런 처리도 하지 않는 최악의 습관이므로 지양해야 한다. (예외를 무시하는 행위가 될 수 있음)

## 1. 예외 처리의 필요성과 개념
*   **예외 처리의 목적:** 프로그램이 실행되는 도중 예상치 못한 오류(예외)가 발생할 수 있음을 대비하여, 프로그램이 갑자기 종료되는 것을 막고 안정적으로 작동하도록 설계하는 과정입니다.
*   **예외의 분류:** 예외는 크게 두 가지로 나뉩니다.
    *   **Checked Exception (체크드 예외):** 컴파일 시점에서 개발자가 반드시 예외 발생 가능성을 인지하고 처리하도록 강제하는 예외입니다 (예: 파일 입출력 오류).
    *   **Unchecked Exception (언체크드 예외):** 런타임에 발생하는 예외로, 주로 개발자의 실수(프로그래밍 오류)로 인해 발생합니다 (예: NullPointerException).
*   **예외 처리의 방법:**
    *   **`try-catch`:** 특정 코드 블록(`try`)에서 오류가 발생했을 때, 해당 예외(`catch`)를 잡아서 대응하는 방식입니다.
    *   **`finally`:** 예외 발생 여부와 관계없이 프로그램 종료 직전에 **무조건 실행**되는 코드 블록입니다. (주로 자원 해제에 사용).

## 2. 실행 흐름 제어: `try-catch-finally`의 동작 원리
*   **실행 순서의 원칙:** 프로그램 코드는 항상 메인(Main) 함수에서 시작하여 **한 줄 한 줄 순차적으로** 실행됩니다.
*   **예외 발생 시 흐름:**
    1.  실행 도중 예외가 발생하면, 해당 예외가 **가장 먼저 매칭되는 `catch` 블록**으로 제어가 넘어갑니다.
    2.  예외 처리가 성공적으로 이루어지면, 이후의 코드는 계속 실행됩니다.
    3.  예외가 발생하든, 처리가 완료되든, `finally` 블록은 **무조건** 마지막에 실행됩니다.
*   **핵심 원리:** 특정 `try` 블록 내에서 예외가 발생하면, 그 이후의 코드는 실행되지 않고 예외 처리(Catch) 단계로 건너뛰게 됩니다.

## 3. 자원 관리의 자동화: `try-with-resources`와 `AutoCloseable`
*   **자원 누수의 위험성:** DB 연결, 파일 스트림, 네트워크 연결 등 자원은 유한하며, 사용 후에는 반드시 닫아주어야(Close) 합니다. 이를 잊어버릴 경우 시스템 자원에 문제가 발생합니다.
*   **`try-with-resources` (자바 7+):** 자원을 사용한 후 반드시 닫는 과정(리소스 해제)을 개발자가 일일이 `finally` 블록에 명시할 필요 없이, 문법적으로 **자동**으로 처리해 주는 기능입니다.
*   **작동 원리:**
    *   `try` 키워드 괄호 안에 자원 객체를 선언하면, 해당 객체는 내부적으로 `AutoCloseable` 인터페이스를 구현하고 있어야 합니다.
    *   `try` 블록이 끝나거나, 예외가 발생하여 빠져나올 때, 자바가 자동으로 이 객체의 `close()` 메서드를 호출합니다.
*   **`AutoCloseable`의 중요성:** 이 인터페이스를 구현한 객체들만이 `try-with-resources` 구문 내에서 자원으로 사용될 수 있으며, 이는 자원 관리의 약속이자 통일된 방법입니다.

## 4. 예외 객체의 설계 및 활용 (Custom Exception)
*   **예외를 던지는 목적 (`throw`):**
    *   **개발자 의도에 따른 예외 발생:** 시스템의 로직상 문제가 생길 수 있는 상황(예: 잔액보다 큰 출금 시도)에서, 자바가 자동으로 예외를 발생시키지 않더라도 **개발자가 강제로** 특정 예외를 던져서 오류 상황을 명시하는 것이 중요합니다.
    *   **문서화된 계약:** 메서드가 동작하는 과정에서 예외가 발생할 수 있음을 사용자에게 알려주기 위해 메서드 시그니처에 `throws`를 명시합니다.
*   **사용자 정의 예외 (Custom Exception) 설계:**
    *   **이름의 중요성:** 예외 객체 자체가 무언가 특별한 기능을 수행하기보다는, **"지금 어떤 상황에서, 어떤 이유로 예외가 발생했는지"** 그 이름(메시지)을 명확하게 전달하는 것이 핵심입니다.
    *   **구현 방법:** 기존의 표준 예외 클래스(Exception)를 상속받아 새로운 예외 클래스를 정의합니다.
    *   **계층적 구조:** 상위 개념의 예외(예: `BankingException`)를 추상 클래스로 정의하고, 이를 상속받아 구체적인 예외(예: `InsufficientBalanceException`)를 만들면 체계적인 예외 구조를 갖출 수 있습니다.
*   **예외 처리의 선택권:**
    *   **`throws`의 의미:** 메서드를 사용하는 쪽(호출자)에게 "이 기능을 쓰려면 예외를 반드시 처리해야 한다"는 **책임을 위임**하는 개념입니다.

## 5. 프로젝트 설계 및 구현 가이드라인
*   **시스템 설계 관점:** 좋은 객체 지향 프로그램은 단순히 기능을 구현하는 것을 넘어, 예외가 발생할 수 있는 모든 경계를 생각하고 대비하는 과정이 포함되어야 합니다.
*   **개선 방안 (프로젝트 발전):**
    *   요구사항을 단순히 만족시키는 데 그치지 않고, **예외 처리 로직**을 추가하여 프로그램의 견고함(Robustness)을 높이는 것이 중요합니다.
    *   실제 사용 가능한 모든 객체와 기능에 대해, '어떤 상황에서' 예외가 발생할 수 있는지 예측하고 이를 예외 클래스로 설계하는 연습이 필요합니다.

## 1. 학습 환경 설정 및 도구 사용법 (Workbench)
- **도구 설치 범위 제한:** 데이터베이스 전체를 로컬에 다운로드 받을 필요가 없으므로, 편의를 위해 **워크벤치(Workbench)**만 설치하는 것을 권장합니다.
- **설치 대상:** 데스크탑과 도커 데스크탑을 준비하되, 당장은 워크벤치 설치까지만 완료하는 것이 적절합니다.

## 2. 데이터베이스와 SQL의 이해 (DBMS의 필요성)
- **SQL의 역할:** SQL은 데이터베이스에 "질문"할 수 있게 해주는 언어입니다. (예: "이 안에 뭐가 들어있어?", "너는 누구니?") 이는 프로그래밍 언어라기보다는 데이터에 대한 질문을 하는 언어에 가깝습니다.
- **데이터베이스(DB)의 필요성:** 프로그램이 동작하는 동안 엄청난 양의 데이터가 생성되는데, 이 데이터들을 효과적으로 관리하기 위해 데이터베이스가 필요합니다.
- **DBMS의 정의:** 데이터베이스를 관리하는 시스템을 **DBMS(Database Management System)**라고 합니다. MySQL이나 Oracle 같은 도구들이 이에 해당합니다.
- **지속성(Persistence) 확보:** 일반적인 메모리 기반의 데이터는 컴퓨터가 꺼지면 휘발되어 사라집니다. 데이터베이스는 시스템이 꺼져도 데이터가 유실되지 않도록 **영구적으로(persistence)** 데이터를 보존하고 관리하는 기능을 합니다.

## 3. 주간 과제 및 학습 방향 안내

- **학습 방향:** 객체 설계 등을 꾸준히 시도하여 과제를 제출하면 강사 피드백을 받을 수 있으니, 스스로 학습 내용을 발전시키는 데 참고 바랍니다.
- **다음 강의:** 데이터베이스 관련 내용은 내일부터 이어서 학습할 것이며, 앞선 내용을 놓치지 않도록 꾸준히 따라오는 것이 중요합니다.
