## 1. 프로그래밍 학습의 태도와 이해의 중요성
*   **배움에 대한 관점:**
    *   자바 세상이나 우리 세상에 우리가 보거나 듣지 못한 수많은 '기계(개체)'들이 존재하듯이, 기술도 계속 발전하며 새로운 개념을 배울 기회가 많습니다.
    *   모든 개체를 다 알 필요는 없지만, 자주 사용되는 핵심 개념과 필요한 부분은 미리 알아두는 것이 중요합니다.
*   **성장의 원동력:**
    *   처음 배울 때 어려운 개념(예: I/O, 스레드)을 만났을 때 좌절하기보다는, 관심을 가지고 알아보려는 노력을 하는 것이 중요합니다.
    *   **학습 효과 극대화:** 단순히 지식을 습득하는 것보다, 공부한 내용을 다른 사람에게 **설명**하려고 노력할 때 이해도가 가장 높아집니다.
    *   **지식 체화:** AI 등을 통해 쉽게 답을 얻는 지식도, 반드시 스스로 이해하고 자신의 언어로 설명할 때 비로소 '내 것'으로 체화할 수 있습니다.

## 2. 효과적인 학습을 위한 스터디(회고) 방식 안내
*(이는 수강생들에게 현재 진행 중인 스터디(회고조) 운영 방식에 대한 가이드입니다.)*
*   **회고 시간의 중요성:** 회고(스터디) 시간은 학습 과정에서 매우 귀중한 시간입니다.
*   **회고 활동 목표 및 방법:**
    *   **분담 및 기록:** 그룹원들이 돌아가면서 역할을 분담하고, 학습한 내용을 빠짐없이 기록하는 것이 중요합니다.
    *   **활동 내용:** 자신이 공부한 내용, 조사한 자료, 미니 프로젝트 진행 내용 등을 발표할 때, **실패했던 경험**이나 "처음에 이렇게 이해했는데 나중에 잘못되었다고 느낀 부분" 등을 함께 나누는 것이 좋습니다.
    *   **제출 방식:** 모든 팀은 날짜별/팀별로 회고 기록을 게시판에 남겨야 합니다. (이는 강사가 학습 진도와 어려움을 파악하는 데 도움이 됩니다.)
*   **학습 밀도 높이기:**
    *   단순히 혼자 공부하려고 하는 것이 아니라, "이 내용을 **나중에 팀원에게 설명해야 한다**"는 목표를 가지고 공부하면 학습의 밀도가 훨씬 높아집니다.

## 3. 멀티스레딩(Multi-threading)의 개념 및 구조
*   **단일 스레드 환경의 한계:**
    *   기존 프로그램은 메인 스레드에서 순차적으로 실행되는 **단일 스레드** 환경이었습니다. I/O 작업(예: 키보드 입력)이 있을 때 프로그램 전체가 멈추는 경험을 하게 됩니다.
*   **스레드의 등장 배경 및 정의:**
    *   현실의 컴퓨터(운영체제)는 웹 브라우저, IDE, 채팅 등 여러 프로그램이 동시에 동작하는 **멀티태스킹 환경**입니다.
    *   프로세스(Process)는 독립적인 메모리를 사용하는 큰 단위의 작업 흐름이며, 한 프로세스 내에서 여러 개의 **스레드(Thread)**를 만들어 사용할 수 있습니다.
    *   스레드는 프로세스 내에서 여러 개의 수행 흐름(Execution Flow)을 가질 수 있게 하며, 각각의 스택 영역을 독립적으로 가지고 동작합니다.
*   **스레드 동작의 원리:**
    *   여러 스레드가 동시에 작업한다고 느끼는 것은, 실제로는 여러 흐름이 작고 빠른 단위로 쪼개져 돌아가면서 일을 처리하는 것과 같습니다.
    *   **스레드 라이프 사이클:** 스레드는 `New` $\rightarrow$ `Ready` (대기) $\rightarrow$ `Running` (실행) $\rightarrow$ `Waiting`/`Blocked` (일시 정지) $\rightarrow$ `Terminated` (종료)와 같은 상태 변화를 거칩니다.
    *   **동기화된 흐름 제어:** 스레드는 정해진 순서대로 동작하지 않으며, 이 비동기성 때문에 여러 스레드가 공유 자원을 사용할 때 데이터가 유실될 위험(Race Condition)이 있습니다.

## 4. 스레드 구현 원리 및 동작 방식 심화
*   **스레드 구현의 약속:**
    *   다중 스레드 환경을 쉽게 구현하기 위해, 자바는 **'약속'**이라는 개념을 사용합니다. (강의에서 '약속'이라는 단어를 여러 번 강조함)
    *   **핵심 원리:** 직접 스레드를 상속받는 방식 외에도, 스레드 객체에 구현된 **Runnable 인터페이스**를 활용하여 구현하는 것이 일반적입니다.
*   **핵심 메서드 이해:**
    *   **`Runnable`:** 실제로 수행하고자 하는 로직을 이 `run()` 메서드 안에 구현하는 것이 핵심입니다.
    *   **`Thread.start()` vs `run()`:**
        *   `run()`을 직접 호출해서는 스레드가 동작하지 않습니다. 이는 일반적인 메서드 호출일 뿐입니다.
        *   반드시 `start()` 메서드를 호출해야만 스레드 생성이 시작됩니다. `start()`가 실제 스레드 실행 흐름을 관리하고 `run()`을 호출하는 역할을 합니다.
*   **고급 동기화 및 통신 메커니즘:**
    *   **동기화 (Synchronization):** 여러 스레드가 공유 자원(변수나 객체)을 동시에 접근하여 문제가 생기는 것을 막기 위해 사용합니다.
        *   `synchronized` 키워드를 사용하거나, 더 세부적인 제어를 위해 `synchronized 블록`을 사용합니다.
    *   **데드락 (Deadlock):** 두 개 이상의 스레드가 각각 다른 자원(열쇠)을 잡고 상대방이 가진 자원을 기다리면서, 결국 아무도 진행하지 못하고 멈추는 상태입니다. (자원 획득 순서를 통일하거나 타임아웃 처리로 해결합니다.)
    *   **생산자-소비자 문제 (Wait/Notify):** 한 스레드가 자원(예: 붕어빵)을 생산했을 때, 이 자원을 기다리고 있는 다른 스레드를 깨워야 할 때 사용합니다.
        *   `wait()`: 자원이 없을 때 스레드를 일시 정지(대기)시킵니다.
        *   `notify()`/`notifyAll()`: 대기하고 있던 스레드를 깨워서 다시 실행 가능한 상태(Ready)로 만듭니다.
    *   **조인 (Join):** 메인 스레드가 다른 스레드의 작업이 모두 끝날 때까지 **강제로 기다리게** 할 때 사용합니다.
    *   **스레드 풀 (Thread Pool):** 스레드를 생성하는 것은 비용이 많이 드는 작업입니다. 이를 개선하기 위해, 미리 여러 개의 스레드를 준비(Pool)해두고 재사용하는 방식입니다. (웹 서버가 요청을 빠르게 처리하는 원리입니다.)
*   **결론적 학습 태도:**
    *   앞서 배운 스레드의 개념들은 **'이해의 영역'**입니다. (왜 이런 구조가 필요한가?)
    *   `run()`을 구현하고 `start()`를 사용하는 것은 **'약속된 사용법'**입니다. (이해하려 노력하기보다, 구조를 따라가서 사용하면 됩니다.)

## 1. 람다식 복습 및 목적 이해
- **간편한 구문:** 람다식은 인터페이스를 구현하거나 객체를 생성할 때 사용하던 복잡한 코드를 훨씬 간결하게 작성할 수 있도록 도와주는 문법입니다.
- **목적:** 코드를 더욱 간편하고 효율적으로 사용하기 위해 추가된 문법입니다.

## 2. 함수형 프로그래밍의 개념과 자바의 접근 방식
- **함수형 프로그래밍이란?**
    - 코드를 '어떻게' 할 것인가(절차 지향)에 집중하기보다, '무엇을' 할 것인지(선언형)를 정의하는 방식입니다.
    - 순수 함수(Pure Function)들을 조합하여 소프트웨어를 만드는 패러다임입니다.
- **자바의 한계 (OOP의 제약):**
    - 자바는 철학적으로 모든 것을 객체(Class)로 간주하고 객체 안에 기능(메서드)을 담는 구조를 고수합니다. 따라서 기능만 따로 독립적으로 동작하거나 함수 자체가 1급 객체(변수에 담거나 함수의 인자로 전달될 수 있는 개념)로 취급되는 것이 어렵습니다.
- **람다식의 필요성:**
    - 함수형 프로그래밍의 장점(독립적이고 간편한 기능 처리)을 활용하고 싶지만, 자바 문법상으로는 반드시 객체(인스턴스)로 감싸야 하는 과정이 복잡합니다.
    - 람다식은 이러한 **복잡한 익명 객체 생성 과정**을 대신해 주어, 마치 함수만 단독으로 사용하는 것처럼 사용할 수 있게 해주는 문법적 편의 기능입니다.

## 3. 함수형 프로그래밍의 핵심 이론 (순수 함수와 부작용)
- **순수 함수 (Pure Function):**
    - **정의:** 함수의 실행이 외부 상태에 영향을 주지 않으며, 동일한 입력값에는 항상 동일한 출력을 반환하는 함수입니다.
    - **장점:** 함수 자체가 독립적이기 때문에, 쓰레드(Thread) 안전성을 보장받아 병렬 처리 등을 동기화 없이 안전하게 진행할 수 있습니다.
- **부작용 (Side Effect) 제거:**
    - 일반적으로 프로그램은 변수 값 변경, 자료 구조의 제자리 수정, 필드 값 설정 등 '부수 효과'를 발생시킬 수 있습니다.
    - 함수형 프로그래밍은 이러한 부수 효과를 최대한 제거한 **순수 함수**만을 사용하며, 이를 구현하는 것이 목표입니다.
- **1급 객체(First-Class Citizen):**
    - 함수를 변수에 저장하거나, 다른 함수의 인자로 전달하거나, 함수의 반환 값으로 사용할 수 있는 개념입니다.
    - 자바가 이 개념을 완벽히 지원하지 않기 때문에, 람다식은 이를 우회적으로 사용자가 간편하게 접근할 수 있도록 돕는 것입니다.

## 4. 람다식의 동작 원리와 활용 (구문 및 특징)
- **원리 이해:**
    - 람다식은 문법적인 설탕(Syntactic Sugar)과 같습니다. 실제로 코드를 실행하는 내부 동작은 여전히 **특정 함수적 인터페이스를 구현한 익명 객체**를 생성하는 과정입니다.
    - 람다식의 사용 목적은 개발자가 객체를 직접 정의하고 생성하는 번거로움을 해소하는 것입니다.
- **함수적 인터페이스 (Functional Interface):**
    - 메서드가 단 하나만 존재하는 인터페이스를 의미합니다. 람다식은 이 구조를 가지는 인터페이스(예: `Runnable`)와 결합하여 사용될 때 가장 효과적입니다.
    - 컴파일러는 람다식을 보고 이 인터페이스가 필요한 구조인지 파악합니다.
- **✅ 람다식 구문 정리:**
    * **형식:** `(매개변수 리스트) -> { 구현부 }`
    * **매개변수:**
        * 매개변수가 하나일 경우 괄호를 생략할 수 있습니다. (예: `i` 대신 `(i)`)
        * 매개변수가 여러 개일 경우 괄호는 필수입니다. (예: `(a, b)`)
        * 매개변수가 없을 수도 있습니다. (예: `() -> { 구현부 }`)
    * **반환 타입 및 구현부:**
        * 구현부가 단일 표현식으로 이루어지면 자동으로 반환(return)되어, `return` 키워드와 세미콜론(;)이 생략됩니다.
        * 구현부가 여러 문장으로 이루어져야 할 경우, `{}`를 사용하여 블록 형태로 묶고, 마지막에는 반드시 `return` 키워드를 명시해야 합니다.
    * **타입 추론:** 매개변수 타입을 명시하지 않아도, 컴파일러가 문맥을 통해 타입을 추론할 수 있으므로 생략 가능합니다.
- **핵심 결론:** 람다식은 복잡한 익명 객체 생성을 단순화하여, 개발자가 함수의 '무엇을' 하는지에 집중할 수 있게 돕는 강력한 도구입니다. (어렵게 받아들이지 말고, '편의 기능'으로만 이해하는 것이 중요합니다.)

## 1. 람다식(Lambda Expression)의 개념 및 역할
- **정의:** 람다식은 익명 함수(Anonymous function)를 간편하게 표현할 수 있도록 자바에서 제공하는 문법입니다.
- **존재 이유:** 자바는 메서드 하나만 독립적으로 동작하는 것이 제한적이었기 때문에, 이러한 기능들을 더 간편하게 사용할 수 있도록 추가된 문법입니다.
- **핵심 역할:** 개발자가 직접 인터페이스를 구현하고 객체를 생성하는 복잡한 과정을 거치지 않고도, 함수처럼 동작하는 논리(로직) 자체를 코드로 바로 넣어줄 수 있게 합니다.
- **비교:** 자바스크립트에서도 유사한 개념을 사용하며, 람다식은 인터페이스를 통해 구현체(Implementation)를 제공하는 방식으로 동작합니다.

## 2. 함수형 인터페이스(Functional Interface)를 이용한 실습 및 활용
- **필요성:** 람다식을 사용하려면, 해당 람다식이 구현할 계약(Contract) 역할을 하는 '함수형 인터페이스'가 먼저 정의되어 있어야 합니다.
- **실습 예시:** `PlusFunction`과 같은 함수형 인터페이스를 정의하고, 이 인터페이스를 통해 덧셈, 뺄셈, 곱셈, 나눗셈 등 다양한 연산을 람다식으로 구현해 보는 과정을 진행합니다.
- **강조점:** 특정 함수형 인터페이스를 사용하는 경우, 해당 인터페이스에 `Functional` 같은 접두사를 붙여주면 컴파일러가 이 인터페이스가 함수형 인터페이스임을 명확하게 체크할 수 있어 코드 안정성 확보에 도움이 됩니다.

## 3. 람다식의 핵심 문법 구조 및 동작 원리
- **타입 추론 (Type Inference):** 람다식 내부에서는 명시적으로 매개변수의 타입을 선언할 필요가 없습니다. 컴파일러가 인터페이스의 정의를 바탕으로 매개변수 타입을 자동으로 추론하여 코드를 간소화합니다.
- **리턴 타입 생략:** 만약 람다식이 별도의 `return` 문 없이 값을 반환하지 않는 경우, 명시적인 리턴 타입을 생략할 수 있습니다.
- **간편화의 원리:** 람다식은 결국 특정 함수형 인터페이스를 구현하는 객체(Implementation Object)를 간편하게 생성하여 사용하는 방식이며, 이를 통해 자바의 모든 메서드 호출을 람다식의 형태로 볼 수 있습니다.

## 4. 자바 표준 함수형 인터페이스 심층 이해
- **표준 라이브러리:** 자바는 개발자가 매번 함수형 인터페이스를 직접 정의할 필요가 없도록, `java.util.function` 패키지에 매우 다양한 표준 인터페이스들을 미리 정의해 두었습니다.
- **주요 인터페이스 및 역할:**
    - **Consumer:** 값을 받아서 사용만 하고 리턴 값은 없는(void) 로직 구현에 사용됩니다. (예: 출력을 처리할 때)
    - **Supplier:** 매개변수는 없고, 값을 생성해서 리턴하는 로직 구현에 사용됩니다. (예: 새로운 객체를 생성할 때)
    - **Function:** 값을 입력으로 받아서 다른 타입의 값을 리턴하는 로직 구현에 사용됩니다. (가장 범용적으로 사용됨)
    - **Predicate:** 값을 입력으로 받아서 그 값이 특정 조건을 만족하는지 여부를 판단하여 `boolean` 값을 리턴하는 로직 구현에 사용됩니다. (조건 검사에 사용)
    - **Operator 계열 (BiPredicate, BinaryConsumer 등):** 두 개의 값을 입력으로 받아 특정 연산을 수행하는 로직 구현에 사용됩니다.

## 5. 람다식이 자바 API에서 사용되는 원리 (Stream API 중심)
- **근본 원리:** 자바의 스트림(Stream) API와 같은 고차원적인 API들은 '무엇을 해야 할지(What to do)'에 대한 로직을 인자로 받아 처리합니다.
- **역할 분담:**
    - **API:** "이 리스트를 다뤄라"라는 틀을 제공합니다.
    - **람다식/함수형 인터페이스:** "이 항목들을 **어떻게** 필터링 할지(Predicate)", "이 값들을 **무엇으로 변환**할지(Function)", "이 값들을 **어떻게 취합**할지(Consumer)"라는 구체적인 동작(로직)을 제공합니다.
- **결론:** 람다식은 복잡한 객체 생성 과정 없이도, 특정 목적을 수행하는 '기능' 그 자체를 메소드의 인자로 전달할 수 있게 함으로써, 자바 코드를 훨씬 간결하고 가독성이 높게 만들어주는 핵심 요소입니다.

## 1. 객체지향 프로그래밍(OOP)과 기능적 패러다임의 차이
- **OOP의 특징:** 모든 요소(값, 행위)를 객체라는 단위로 묶어 다룹니다. 행위(함수)는 독립적으로 존재할 수 없고, 반드시 클래스라는 '껍데기' 안에 메서드 형태로 포함되어 동작해야 합니다.
- **OOP의 한계:** 기능 하나만 사용하더라도 해당 기능을 클래스에 묶어야 하므로 코드가 복잡하고 장황해지는 경향이 있습니다.
- **기능형 프로그래밍(FP)의 특징:** 함수 자체를 1급 객체(First-class object)로 취급합니다. 따라서 함수를 변수에 저장하거나, 함수의 매개변수로 전달하거나, 함수가 다른 함수를 반환하는 것이 가능합니다.

## 2. Java가 기능적 코드를 지원하는 방법 (함수형 인터페이스와 람다식)
- **목적:** OOP의 제약을 받으면서도 FP와 같이 간결하게 함수를 사용하고 싶을 때의 해결책입니다.
- **함수형 인터페이스 (Functional Interface):**
    - 인터페이스가 **단 하나의 추상 메서드**만 정의되어 있는 형태의 인터페이스입니다.
    - 이 단일 메서드 덕분에, 이 인터페이스를 구현하는 객체를 만들 때 코드가 단순화됩니다.
- **람다식 (Lambda Expression):**
    - 기존에 인터페이스를 구현할 때 필요했던 복잡한 익명 객체(Anonymous Object) 생성 과정을 획기적으로 줄여줍니다.
    - 코드를 훨씬 간편하고 짧게 작성할 수 있게 해줍니다.
    - **핵심 원리:** 람다식으로 표현할 수 있는 기능은 반드시 **단일 추상 메서드를 가진 인터페이스**가 바탕이 되어야 합니다.
    - **타입 추론 (Type Inference):** 람다식 사용 시, Java가 개발자가 명시하지 않아도 매개변수나 반환 타입을 스스로 유추하여 처리해 줍니다.

## 3. 표준화된 함수형 인터페이스 (`java.util.function` 패키지)
- **배경:** 개발자가 필요할 때마다 기능 인터페이스를 직접 정의하는 것은 번거롭기 때문에, Java는 자주 사용되는 함수 형태들을 미리 정의하여 제공합니다.
- **주요 인터페이스 및 역할:**
    - **`Consumer<T>` (소비자):** 입력값 $T$를 받아서 무언가를 처리만 하고, **반환 값(return)은 없습니다.** (주로 사이드 이펙트 발생)
    - **`Function<T, R>` (함수):** 입력값 $T$를 받아서 처리한 후, 특정 타입 $R$을 **반환합니다.** (가장 일반적인 변환 형태)
    - **`Predicate<T>` (조건문):** 입력값 $T$를 받아서 조건을 검사하고, **Boolean 값(true/false)을 반환**합니다. (필터링, 검증에 사용)
- **핵심 개념:** 인터페이스는 단지 **'계약(Contract)'**과 같습니다. 해당 인터페이스를 사용한다는 것은, 해당 인터페이스의 메서드 시그니처(매개변수와 반환 타입)를 가진 객체를 제공하겠다는 약속과 같습니다.

## 4. 실전 적용: 스트림과 함수형 코딩의 이해
- **Stream의 활용:** 자바의 Stream API는 위에서 설명한 함수형 인터페이스들을 가장 많이 활용하는 예시입니다.
    - `forEach()` 메서드: `Consumer` 타입의 객체를 요구합니다. (반환 값 없음, 처리만 수행)
    - `filter()` 메서드: `Predicate` 타입의 객체를 요구합니다. (조건 체크, Boolean 반환)
    - `map()` 메서드: `Function` 타입의 객체를 요구합니다. (값 변환 및 새로운 값 반환)
- **개념 정리:**
    - 스트림의 메서드가 특정 함수형 인터페이스 타입(`Predicate`, `Consumer` 등)을 매개변수로 요구하는 것은, 그 메서드를 호출하는 시점에 사용자가 그 **인터페이스의 규칙을 따르는 객체**를 넣어줘야 한다는 것을 의미합니다.
    - 복잡하게 생각할 필요 없이, 각 인터페이스의 역할(무엇을 받아서 무엇을 하는지)만 이해하면, 코드에서 필요한 기능을 람다식으로 간단하게 '구현'하여 넣어주기만 하면 됩니다.
- **결론:** 람다식과 함수형 인터페이스 덕분에, 복잡한 객체 정의 없이도 함수형 프로그래밍의 장점(간결성, 함수를 값처럼 취급하는 유연성)을 객체지향 언어인 Java에서 구현할 수 있게 되었습니다.

## 1. Stream API의 개념 및 기본 사용법
*   **스트림의 정의 및 추상화:**
    *   기존의 반복문(예: `for` 루프)을 사용하여 리스트(List)의 데이터를 순차적으로 꺼내 쓰는 방식보다 추상화된 형태로 데이터를 처리합니다.
    *   데이터가 마치 '물처럼 흘러가는' 형태로 데이터를 처리한다고 이해할 수 있습니다.
*   **`stream()` 메서드:**
    *   컬렉션(Collection)에 대해 `.stream()` 메서드를 호출하면, 해당 데이터들이 스트림 타입으로 변환됩니다.
*   **기본 처리 흐름:**
    *   **`stream()`** $\rightarrow$ **`filter()`** (데이터 필터링) $\rightarrow$ **`forEach()`** 또는 **`collect()`** (최종 처리)
    *   스트림은 데이터가 한 번 흘러가면 다시 받아서 쓸 수 없습니다 (일회성 데이터 흐름).

## 2. 핵심 스트림 연산자 이해 (Filter, Map, Collect, Sort)
*   **`filter()`:**
    *   스트림에 포함된 요소들을 조건에 따라 걸러내는 역할을 합니다.
    *   조건(Predicate)을 만족하는 요소만 살아남고, 그렇지 않은 요소는 다음 단계로 넘어가지 않습니다.
*   **`map()`:**
    *   스트림의 각 요소(값)를 입력받아 특정 함수(Function)를 수행한 후, 변환된 결과 값으로 바꿔서 스트림 형태로 다시 출력해 줍니다. (값의 변환 역할)
    *   **핵심:** `map`은 값을 변환(transformation)하는 역할을 하며, 변환된 값이 다시 스트림 형태로 다음 단계로 전달됩니다.
*   **`collect()`:**
    *   스트림 연산을 거쳐 나온 최종 결과 값들을 특정 형태로 수집하여 반환해 줍니다.
    *   가장 일반적인 예는 `List` 형태로 결과를 모아내는 것입니다.
*   **`sorted()`:**
    *   스트림의 요소를 정렬합니다.
    *   정렬을 위해 `Comparator` 인터페이스를 사용하며, `Comparator` 역시 값을 두 개 받아 정수 값(정렬 순서)을 리턴하는 함수형 인터페이스입니다.
*   **`forEach()` (출력):**
    *   스트림의 각 요소에 대해 특정 작업을 수행할 때 사용합니다. (주로 출력이나 부수 효과 발생 시 사용)

## 3. 자바 8 문법 심화: 람다 표현식과 메서드 참조
*   **람다 표현식 (Lambda Expression):**
    *   함수형 인터페이스(Functional Interface)의 구현을 매우 간편하고 짧게 작성할 수 있게 해주는 문법입니다.
    *   원래는 별도의 클래스를 만들거나 객체(인스턴스)를 생성해야 했던 역할을, 간단한 코드 블록으로 대체합니다.
    *   **핵심:** 람다식이 들어가는 코드는 결국 해당 인터페이스를 구현한 객체가 들어와야 한다는 것을 이해해야 합니다.
*   **메서드 참조 (Method Reference):**
    *   특정 메서드를 참조하여 구현체를 간편하게 사용하고 싶을 때 사용합니다.
    *   예: `System.out::println`
    *   이는 메서드가 호출되는 기본적인 사용 방식과 다르지 않습니다.
*   **공통점 이해:**
    *   람다 표현식이나 메서드 참조는 **"이 자리에 이 인터페이스를 구현한 객체를 넣어줘"**라는 자바가 정한 **약속**에 따라 사용되는 문법입니다.
    *   이해하는 핵심은, 람다/메서드 참조를 사용하든, 직접 `new` 키워드를 이용해 객체를 만들든, 최종적으로는 같은 기능을 하는 객체를 전달한다는 점입니다.

## 4. 스트림 이해의 핵심 관점
*   **통합적인 시각:** 스트림을 배울 때, 각 연산자를 개별적인 기능으로만 생각하면 어렵습니다.
*   **근본 원리 이해가 중요:** 스트림 코드를 무작정 외우기보다, '데이터 컬렉션을 어떻게 처리(Filter, Transform, Reduce)하는가'라는 근본적인 로직을 먼저 생각하고, 그 로직을 스트림 문법으로 바꾼다고 생각하는 것이 가장 중요합니다.

## 1. 문제 해결 접근 방식 및 전략
- **문제 분할:** 문제를 한 번에 처리하려고 하기보다는 작은 단위로 쪼개서 생각하는 것이 가장 효과적이다. (Divide and Conquer)
- **단계적 접근:** 'Grouping(그룹화)'을 먼저 수행한 다음, 그 결과로 얻은 그룹에 대해 'Calculation(계산)'을 수행하는 방식으로 문제를 해결해야 한다.
- **예시 문제 풀이 흐름:**
    - **그룹화:** 사원의 부서별로 리스트를 만들거나, 특정 조건에 맞는 사원들을 한데 묶는 작업을 먼저 수행한다.
    - **통계 계산:** 그룹화된 데이터를 이용해 평균값, 합계 등을 구한다. (예: 부서별 평균 급여 구하기)

## 2. 데이터 구조 활용: 부서별 그룹화 및 통계 계산
- **목표:** 사원들을 부서별로 묶고, 각 부서별 평균 급여를 계산하는 방법 학습.
- **구현 방법 (Map 사용):**
    1. **데이터 구조 선택:** 부서명(String)을 키(Key)로, 해당 부서 사원 리스트(List)를 값(Value)으로 가지는 `Map` 자료구조를 활용한다.
    2. **그룹화 로직:** 전체 사원 리스트를 순회하며, 사원 객체의 `getDepartment()` 메서드를 이용해 부서명을 추출한다.
    3. **Map 처리:**
        - 만약 해당 부서명이 `Map`에 없다면(First Encounter), 새로운 빈 리스트를 생성하여 `Map`에 추가한다.
        - 만약 해당 부서명이 이미 존재한다면, 기존 리스트를 가져와서 현재 사원 객체를 추가한다.
    4. **통계 계산:**
        - `Map`에 저장된 리스트를 순회하며 총 급여(Sum)를 누적한다.
        - 최종적으로, 총 급여를 해당 부서 리스트의 크기(사이즈)로 나누어 평균 급여를 계산한다.
- **핵심 메서드:**
    - `map.containsKey()`: 키(부서명) 존재 여부 체크.
    - `map.get(key)`: 특정 키에 해당하는 값을 가져옴.
    - `map.keySet()`: Map에 저장된 모든 키(부서명)들만 셋(Set) 형태로 얻어옴.

## 3. 컬렉션 활용 심화: 필터링, 정렬, 최댓값/최솟값 찾기
- **조건 필터링 (Filtering):**
    - 전체 사원 리스트를 순회하며, 특정 조건(예: 급여가 500만 원 이상)을 만족하는 사원만 걸러내어 새로운 리스트를 만들 수 있다.
- **정렬 (Sorting):**
    - 사원 전체 리스트를 특정 기준(예: 급여)에 따라 정렬해야 할 경우, `Collections.sort()` 메서드를 사용한다.
    - 이 함수는 **Comparator**를 두 번째 인자로 받으며, 급여와 같은 특정 속성만 비교하도록 Comparator를 구현해야 한다.
- **극값 찾기 (Max/Min):**
    - 전체 리스트를 순회하며, 현재까지 발견된 최댓값(Max Salary)을 변수로 초기 설정한다.
    - 루프를 돌면서 다음 사원의 급여가 현재 `Max Salary`보다 큰지 비교하고, 크다면 `Max Salary` 값을 갱신해 나간다.
    - 이 과정을 거치면 전체 사원 중 최고 급여를 효율적으로 찾아낼 수 있다.

## 4. 스트림(Stream)과 프로그래밍 개념 정리
- **스트림의 역할:**
    - 스트림은 리스트나 컬렉션에 들어있는 복잡한 데이터를 순회하며 원하는 값들을 추출하거나 변형하는 과정을 훨씬 간단하게 처리하도록 도와주는 개념이다.
    - 반복문(`for`문)으로 모든 과정을 처리하는 것이 원리는 같지만, 스트림을 사용하면 코드가 훨씬 간결해지고 가독성이 높아진다.
- **학습 권장 사항:**
    - **기본 문법 숙달:** 스트림을 배우기 전이라도, 모든 고급 기능은 결국 **기본 문법(반복문, 비교, 검색 등)**의 응용이다. 기초 문법에 익숙해지도록 다양한 실습 문제(예: 인터페이스 구현, 문자열 변환기 등)를 풀어보는 것이 중요하다.
    - **스트림 접근:** 스트림을 배우는 것은 난이도가 있지만, 꾸준히 접하고 익숙해지면 논리적으로 이해할 수 있게 된다.
