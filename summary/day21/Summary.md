## 1. 한 달 회고 및 학습 동기 부여
- 지난 한 달간의 학습 경험과 피로도를 함께 공유하며 서로를 격려하는 시간을 가졌다.
- 학습의 어려움은 당연하며, 특히 전공자라 할지라도 학교 수업에서 얻기 힘든 깊이 있는 지식이므로 큰 어려움을 느낄 수 있다.
- 이 과정은 단기간에 모든 것을 터득할 수 있는 내용이 아니므로, 좌절하거나 어려움을 겪는 것은 자연스러운 현상으로 인식해야 한다.

## 2. 심화 학습을 위한 학습 태도와 방법론
- **학습의 어려움 인지:** 배우는 내용들이 짧은 시간 안에 습득하기 매우 어렵다. 학교 수업은 과목을 분산하여 가르치기 때문에 깊이 있는 실습 기회가 부족하다.
- **능동적인 학습 태도 필수:**
    - **시간 확보:** '수업 시간 8시간만으로 충분하다'는 생각은 욕심이며 위험하다. 반드시 수업 시간 외에 별도로 학습 시간을 확보해야 한다.
    - **공백 메우기:** 수업을 들으면서 이해하지 못한 부분(공백)이 생기는 것은 자연스럽다. 이를 혼자서 메우려 하기보다는, AI, 팀원, 멘토 등 다양한 자원을 활용하여 질문하고 이해하는 과정을 거쳐야 한다.
- **효율적인 실습(회고 활동) 활용:**
    - 진도가 느리다고 회고 활동 자체를 포기해서는 안 된다.
    - 회고 시간은 단순히 내용을 발표하는 시간이 아니라, **학습 내용을 발표하고 정리하며 집중도를 높이는 시간**으로 반드시 적극 활용해야 한다.
    - 진도가 느린 경우, '이 부분은 어려워서 이해 못 했으니, 대신 **지금 배운 A 문법**에 대해서 깊이 있게 공부했다'는 식으로 자신의 학습 목표와 성과를 설정하여 참여해야 한다.

## 3. 자바 핵심 문법: 람다 표현식(Lambda Expression)의 이해
- **배경:** 자바 초기에 인터페이스를 구현할 때, 메서드가 하나만 있는(기능 하나만 가진) 인터페이스를 사용하려면 코드가 복잡한 익명 내부 클래스를 생성해야 했다.
- **도입 목적:** 람다 표현식은 이처럼 단순한 인터페이스를 구현하고 객체로 사용하는 과정의 복잡성을 해소하고, 코드를 간편하고 직관적으로 작성하기 위해 도입되었다.
- **핵심 원리 (개념적 이해):**
    - 람다식이 간편하게 보이는 것은 사실 내부적으로 컴파일러/JVM이 해당 인터페이스를 구현한 객체를 생성하고 사용하고 있기 때문이다.
    - 람다식이 사용되려면, 해당 기능을 추상화하는 **'함수형 인터페이스(Functional Interface)'**가 반드시 정의되어 있어야 한다.
- **활용 방법:**
    - 람다 표현식은 여러 클래스 정의나 복잡한 익명 객체 생성 과정 없이, 필요한 기능(메서드)만 간결하게 정의하여 사용할 수 있게 한다.
    - 자바 API에 이미 '공급자(Supplier)', '소비자(Consumer)', '함수(Function)', '판별자(Predicate)' 등처럼 자주 사용되는 패턴의 인터페이스를 미리 정의해 두었다.

## 4. 과정 복습 및 다음 학습 로드맵
- **오프라인/쉬는 기간 활용:** 다가오는 휴가 기간을 단순히 쉬는 시간으로 보내기보다, 학습 공백을 채우는 **'절호의 찬스'**로 활용하는 것이 중요하다.
- **실습 프로젝트 추천:** 가장 좋은 복습 방법은 **애플리케이션을 하나 만들어보는 것**이다. 간단한 '친구 관리'나 '도서 관리'와 같은 기능을 가진 웹 서비스를 목표로 하여, 5일 동안 하루 조금씩 코딩에 시간을 투입하는 것이 효과적이다.
- **추가 학습 자원 (참고):**
    - **Web Application Full Stack:** 웹 프로그래밍의 기초, DB 연결(SQL, JDBC), 프론트엔드 개념(HTTP 프로토콜 등)을 복습할 수 있는 강의를 활용한다.
    - **서블릿 이해의 중요성:** 현재 사용하는 스프링 프레임워크가 내부적으로 서블릿을 기반으로 동작한다는 원리를 이해하면, 스프링의 작동 방식과 웹 요청/응답 객체(Request/Response)의 개념을 깊이 이해하는 데 큰 도움이 된다.
    - **자바 기본기 강화:** 부족한 부분이 있다면 프로그래머스 등 온라인 플랫폼에서 IO, 람다, 스레드 등 핵심 주제를 복습하고, 기본적인 SQL 및 JDBC 개념을 점검한다.

5. 스트림의 처리 흐름 (Intermediate vs. Terminal Operations)

---

## 1. 스트림(Stream)의 개념 및 필요성
*   **스트림이란?**
    *   Java 8에서 도입된 개념으로, 컬렉션(Collection)이나 배열과 같은 데이터 소스의 데이터를 '스트림(Stream)'이라는 형태로 추상화한 것입니다.
    *   데이터가 물이 흐르듯이 연속적으로 처리되는 개념을 가져와, 복잡한 데이터 처리 로직을 훨씬 간결하고 직관적인 함수형 프로그래밍 스타일로 작성할 수 있게 돕는 혁신적인 도구입니다.
*   **목적:**
    *   기존 방식처럼 반복문이나 인덱스(0번째, 1번째...)를 사용하여 데이터를 하나하나 꺼내 처리하는 방식 대신, 전체 데이터를 흐름(Stream)으로 보고 일괄 처리할 수 있도록 합니다.
*   **컬렉션 vs. 스트림:**
    *   **컬렉션 (Collection):** 데이터를 **저장**하는 그릇(데이터 구조)입니다.
    *   **스트림 (Stream):** 데이터를 **처리**하는 과정(흐름) 그 자체입니다.

## 2. 전통적 방식과 스트림을 이용한 데이터 처리 비교
*   **전통적 방식 (For/While Loop):**
    *   특정 조건을 만족하는 데이터를 찾거나 (예: 짝수만 출력), 새로운 데이터 목록을 만들 경우, 반복문과 조건문(`if`)을 중첩하여 사용해야 하므로 코드가 복잡해집니다.
*   **스트림 방식:**
    *   `Stream` API를 사용하면 `.filter()`와 같은 메서드만으로 원하는 조건을 걸러내거나, 데이터를 변환하는 과정(예: 대문자로 변환)을 순차적이고 간결하게 처리할 수 있습니다.
*   **코드의 간결성:**
    *   스트림은 메서드 체이닝(Method Chaining)을 지원합니다. 즉, 중간 과정에서 나온 스트림 객체를 변수에 저장하지 않고, 다음 메서드에 바로 연결하여 코드를 짧고 읽기 쉽게 만들 수 있습니다.

## 3. 스트림의 주요 핵심 연산자 (Intermediate Operations)
*   **`filter()` (필터링):**
    *   특정 조건을 만족하는 데이터만 걸러내는 기능을 수행합니다.
    *   사용할 때 `Predicate<T>`라는 함수형 인터페이스를 필요로 합니다.
    *   **람다식 활용:** `Predicate` 구현체(객체)를 직접 생성하기보다, 간결하게 람다 표기법을 사용하여 조건을 지정합니다. (예: `x -> x % 2 == 0`)
*   **`map()` (변환):**
    *   스트림의 요소 하나하나에 특정 함수를 적용하여, 새로운 타입이나 값을 반환하도록 변환합니다. (1:1 매핑)
    *   예시: 문자열을 대문자로 변환하거나, 객체 속성 값을 추출할 때 사용합니다.
*   **`sorted()` (정렬):**
    *   스트림의 요소들을 정렬된 순서로 재배열합니다.
*   **`collect()` (수집):**
    *   스트림의 모든 요소를 모아서 최종적인 결과물(예: `List`, `Set`, Map 등)로 변환하는 기능을 담당하는 핵심 중간 연산자입니다. (최종 연산으로도 볼 수 있음)

## 4. 스트림의 특성 및 생성 방법
*   **스트림의 3가지 주요 특성:**
    1.  **지연 계산 (Lazy):** 스트림에 정의된 연산들은 즉시 실행되지 않습니다. 실제로 값을 소비(Consumption)하는 **최종 연산**이 호출될 때만 데이터 처리가 시작됩니다.
    2.  **단일 사용 (Single Use):** 스트림은 한 번 처리되어 소비(Consumed)되면 그 상태가 소멸됩니다. 따라서 스트림을 재사용하려면 반드시 원본 데이터 소스(배열, 리스트 등)부터 다시 스트림을 생성해야 합니다.
    3.  **병렬 처리 지원:** 대용량 데이터를 처리할 때, 스트림은 병렬 처리(Parallel Stream)를 간단하게 수행할 수 있도록 지원하여 성능 향상에 유리합니다.
*   **스트림 생성 방법 (Source):**
    *   **컬렉션 기반:** `List`, `Set` 등 컬렉션 객체에 내장된 `.stream()` 메서드를 사용합니다.
    *   **배열 기반:** `Arrays.stream(배열)`을 사용합니다.
    *   **범위 기반:** `IntStream.range(시작, 끝)` 또는 `IntStream.rangeClosed(시작, 끝)`를 사용하여 숫자의 범위 스트림을 생성합니다.
    *   **원소 리스트 기반:** `Stream.of(값1, 값2, ...)`를 사용하여 특정 값들로 스트림을 만듭니다.
    *   **외부 소스:** 파일 I/O의 경우, `Files.lines()` 등을 사용하여 파일의 내용 전체를 스트림 형태로 가져올 수 있습니다.

## 5. 스트림의 처리 흐름 (Intermediate vs. Terminal Operations)
*   **중간 연산 (Intermediate Operations):**
    *   `filter()`, `map()`, `sorted()` 등.
    *   **특징:** 연산이 끝난 후에도 다시 **스트림(Stream)**을 반환합니다. (Stream $\rightarrow$ Stream)
    *   **역할:** 데이터 처리를 위한 일련의 과정을 연결합니다.
*   **최종 연산 (Terminal Operations):**
    *   `forEach()`, `collect()`, `count()`, `reduce()` 등.
    *   **특징:** 스트림의 처리를 마무리하고 그 결과를 외부 타입으로 반환합니다. (Stream $\rightarrow$ T, T는 스트림이 아님)
    *   **역할:** 데이터 처리의 최종 목표를 달성합니다.

> **💡 핵심 정리:**
> 스트림은 데이터를 한 번의 흐름(Stream)으로 추상화하여, 중간 연산(`filter`, `map` 등)으로 원하는 대로 조작한 뒤, 최종 연산(`collect`, `forEach` 등)을 통해 결과를 얻어내는 과정입니다. 이 구조가 코드를 매우 간결하고 읽기 쉽게 만듭니다.

## 1. Stream API의 기본 원리: 지연 계산 (Lazy Evaluation)
Stream API를 사용하여 데이터 컬렉션에 함수형 연산을 적용하는 과정을 설명합니다.

*   **스트림의 작동 원리:** Stream은 단순히 데이터가 흐르는 '물줄기' 또는 '컨베이어 벨트'와 같은 개념으로 이해할 수 있습니다. 데이터를 한 번에 처리하는 것이 아니라, 일련의 단계를 거쳐 흐르게 됩니다.
*   **지연 연산 (Lazy Evaluation)의 핵심:** Stream의 중간 연산(Intermediate Operation)은 실제로 연산을 수행하지 않고 **계획만 세워둡니다.**
    *   실제 연산(Calculation)은 최종 연산(Terminal Operation, 예: `forEach`, `collect`)이 호출될 때만 한꺼번에, 그리고 순차적으로 시작됩니다.
    *   (예시: `filter()` 이후 아무리 많은 중간 연산을 거쳐도, `forEach` 같은 최종 연산이 호출되기 전까지는 아무런 일이 일어나지 않습니다.)
*   **작동 방식의 이점:** 이 덕분에 데이터 흐름을 매우 효과적으로 제어하고 최적화된 방식으로 처리가 가능합니다.

## 2. 필수 Stream 연산자 이해하기 (Filter, Map, Distinct, Sorted)
스트림의 핵심적인 중간 연산자들을 학습하고, 각 연산자가 어떤 역할을 하는지 파악합니다.

*   **Filter (필터링):** 특정 조건에 맞는 요소만 걸러내는 역할을 합니다. (예: 짝수만 남기기)
*   **Map (변환):** 스트림의 각 요소를 다른 형태로 변환하는 1:1 함수 역할을 합니다. (예: 모든 문자열을 대문자로 변환)
*   **Distinct (중복 제거):** 스트림에서 중복되는 요소를 제거하고 고유한 요소만 남깁니다.
*   **Sorted (정렬):** 스트림의 요소들을 특정 기준에 따라 정렬합니다. 정렬 조건을 줄 수 있으며, 비교(Comparable)를 구현한 객체나 Comparator를 사용하여 정렬 기준을 지정할 수 있습니다.
*   **주의사항:** 스트림 연산을 통해 필터링된 결과를 다른 곳에 사용하려면, 다시 스트림 형태로 데이터를 받거나 적절히 변환해야 합니다.

## 3. Stream 연산의 마무리와 흐름 제어 (ForEach vs. Peek)
최종 연산을 통해 스트림의 처리 흐름을 제어하는 방법을 배웁니다.

*   **Terminal Operation (최종 연산):** 스트림의 처리를 끝내는 연산입니다. (예: `forEach`, `collect`)
*   **`forEach()`:** 스트림의 모든 요소에 대해 지정된 작업을 수행하며, 이 연산을 호출하면 스트림 처리가 **완전히 종료**됩니다.
*   **`peek()`:** 최종 연산은 아니지만, 연산이 진행되는 중간에 특정 요소에 접근하여 부가적인 작업(Side Effect)을 수행하고 싶을 때 사용합니다.
    *   `peek()`은 요소의 흐름을 유지하면서 디버깅이나 로깅 등의 목적으로 사용되며, 이후에도 스트림 체이닝(Method Chaining)을 이어갈 수 있습니다.
*   **스트림을 중간 연산으로 사용하기:** 최종 연산이 아닌 `peek()`을 사용하여 스트림의 흐름을 중간에 멈추지 않게 유지하는 것이 중요합니다.

## 4. 데이터 조건 검색 및 Null 처리 (Matching & Optional)
특정 조건에 대한 검색 연산과 안전한 코드를 작성하기 위한 패턴을 학습합니다.

*   **조건 검색 메서드:**
    *   `allMatch()`: 스트림의 모든 요소가 주어진 조건을 만족하는지 (Boolean) 검사합니다. (AND 연산과 유사)
    *   `anyMatch()`: 스트림의 요소 중 단 하나라도 주어진 조건을 만족하는지 (Boolean) 검사합니다. (OR 연산과 유사)
    *   `noneMatch()`: 스트림의 모든 요소가 주어진 조건을 만족하지 않는지 (Boolean) 검사합니다.
*   **Optional 클래스 사용:** 자바 API에서 **NullPointerException(NPE)**을 사전에 방지하기 위해 도입된 클래스입니다.
    *   데이터를 검색했는데 결과가 없을 경우 (Null이 될 수 있는 경우), `Optional`은 이를 명시적으로 다루게 합니다.
    *   `Optional`은 단순히 값을 요청하는 것이 아니라, "값이 있을 수 있다"는 것을 시스템에 알려주며, `ifPresent()`, `orElse()`, `map()` 등의 메서드를 사용하여 안전하게 값이 있는지 없는지부터 처리하도록 합니다.
    *   **사용 원칙:** 결과 값을 받을 때부터 일반 타입(String) 대신 `Optional` 타입을 사용하는 것이 안전한 코드 설계의 핵심입니다.

## 5. 집계 연산 (Counting, Min, Max)
스트림에서 데이터의 통계적인 요약 값을 얻는 방법을 다룹니다.

*   **주요 연산자:**
    *   `count()`: 스트림의 전체 요소 개수를 계산합니다.
    *   `min()` / `max()`: 스트림 요소의 최소값 또는 최대값을 찾습니다. (이 역시 결과로 `Optional`을 반환하므로, 값이 없을 수 있음을 염두에 두어야 합니다.)
*   **주의사항:** 집계 연산(예: `min`, `max`)을 수행하는 경우에도 결과값이 존재하지 않을 수 있으므로, 반드시 `Optional`을 사용하여 안전하게 값에 접근하는 것이 좋습니다.

***

**💡 학습 요약 및 실습 팁:**
*   스트림은 복잡해 보이지만, 결국 컬렉션 데이터를 **물 흐르듯이(Stream)** 단계를 거쳐 처리하는 과정으로 이해하면 쉽습니다.
*   코드 암기보다는, **"왜 (When) 이 연산이 실행되는지"** (특히 지연 연산의 작동 시점)와 **"어떤 문제를 방지하기 위해 (Why) Optional을 쓰는지"**에 초점을 맞추어 이해하는 것이 중요합니다.
*   실습 시에는 직접 코드를 입력하는 것보다, *내가 예상하는 결과*와 *실제 나온 결과*를 비교하며 흐름을 예측하는 연습을 하는 것이 가장 효과적입니다.

## 1. 스트림(Stream)의 이해와 도입 배경
*   **기존 컬렉션 처리 방식의 한계:** 기존에는 반복문(`for`, `while` 등)을 사용하여 컬렉션의 데이터를 순회하며, 원하는 조건에 맞는 데이터를 찾거나 필터링할 때마다 새로운 자료구조를 만들어야 하는 비효율성이 있었습니다.
*   **스트림의 정의:** 스트림은 컬렉션에 담긴 데이터를 마치 '흐르는 데이터(Data Flow)'처럼 추상화하여 다루는 방식입니다. (공장의 컨베이어 벨트 비유 사용)
*   **스트림의 장점:** 기존의 명시적인 반복문 사용을 줄이고, 함수형 프로그래밍 개념을 도입하여 코드를 간결하고 선언적으로 작성할 수 있게 합니다.

## 2. 스트림의 동작 원리 및 핵심 개념
*   **메서드 체이닝(Method Chaining):** 스트림은 여러 중간 연산(Intermediate Operation)들이 스트림을 반환하는 구조로 설계되어 있습니다. 이 덕분에 중간 연산들을 `점(.)`으로 계속 연결하여 파이프라인처럼 데이터를 처리할 수 있습니다.
*   **지연 연산(Lazy Evaluation):** 스트림은 코드를 작성할 때 즉시 연산을 수행하지 않습니다.
    *   사용자가 스트림에 여러 중간 연산(필터링, 매핑 등)을 설정할 때, 실제로 연산은 **최종 연산(Terminal Operation)**이 호출될 때 비로소 시작됩니다.
    *   따라서 성능 최적화가 가능하며, 필요할 때만 작업을 수행하는 구조입니다.

## 3. 람다식(Lambda Expression)과 함수형 인터페이스
*   **람다식의 역할:** 자바는 메서드가 매개변수로 들어가는 것을 기본 문법상 제한하고 있습니다. 람다식은 이러한 제약을 극복하고, 마치 독립적인 함수처럼 간결하게 함수 로직을 구현할 수 있도록 문법적으로 제공된 기능입니다.
*   **기반 원리:** 람다식 자체는 돌아다닐 수 없으며, 컴파일러는 내부적으로 람다식을 구현한 객체(함수형 인터페이스를 구현한)를 생성합니다.
*   **함수형 인터페이스:** 스트림 API를 사용하기 위해서는 `java.util.function` 패키지 등에 정의된 미리 만들어진 함수형 인터페이스(예: `Consumer`, `Function` 등)가 존재합니다. 핵심은 **'메서드가 수행할 일(기능)'** 자체입니다.

## 4. NullPointerException 방지: Optional
*   **문제 인식:** 자바 프로그래밍에서 가장 흔하고 골치 아픈 오류 중 하나는 `NullPointerException`입니다. 이는 값이 실제로 `null`일 때 객체의 메서드를 호출하려고 시도할 때 발생합니다.
*   **Optional의 도입:** `Optional`은 값을 담는 컨테이너(Wrapper)와 같습니다. 단순히 값이 `null`인지 아닌지를 확인하는 것을 넘어, 값이 존재할 때와 존재하지 않을 때를 대비한 다양한 처리 방법을 제공합니다.
*   **주요 기능:**
    *   `get()`: 값이 있을 경우 결과를 가져옵니다.
    *   `ifPresent()`: 값이 존재할 때만 특정 로직을 실행합니다.
    *   `orElse()`: 값이 없을 경우 기본값(Default Value)을 지정하여 사용합니다.

## 5. 스트림의 연산 분류 및 주요 메서드
스트림의 메서드는 수행하는 역할에 따라 크게 두 가지로 분류됩니다.

*   **중간 연산 (Intermediate Operations):**
    *   스트림을 반환하는 메서드들입니다.
    *   예: `filter()` (특정 조건을 만족하는 데이터만 추출), `map()` (각 요소를 새로운 형태로 변환), `sorted()` 등.
    *   이 연산들은 파이프라인을 구성하는 과정일 뿐, 실제 처리는 최종 연산이 호출될 때만 이루어집니다.
*   **최종 연산 (Terminal Operations):**
    *   스트림이 아닌 최종 결과를 반환합니다.
    *   예: `forEach()` (요소별로 작업을 수행), `collect()` (결과를 새로운 컬렉션으로 수집), `count()`, `min()`, `max()` 등.
*   **대표적인 메서드 동작:**
    *   **`forEach`:** 각 요소를 순회하며 작업을 수행합니다.
    *   **`allMatch()` / `anyMatch()` / `noneMatch()`:** 컬렉션의 모든 요소 또는 하나라도 특정 조건을 만족하는지 여부를 불리언(boolean) 값으로 판단합니다.
    *   **`map`:** 요소를 특정 연산을 거쳐 새 값으로 변경하고 그 값을 담아주는 역할을 합니다. (예: 문자열을 정수형으로 변환)

## 6. 고급 컬렉션 활용: Collector와 Reducer
*   **리듀싱 (Reducing):**
    *   여러 요소를 단일 값으로 누적하여 연산하는 방식입니다. (예: 모든 숫자의 합계를 구하기)
    *   `reduce()` 함수를 사용하며, 시작 값(identity)을 부여하여 누적 합계 등의 계산을 할 수 있습니다.
*   **컬렉터 (Collectors):**
    *   `java.util.stream.Collectors`는 스트림의 최종 결과를 특정 컬렉션 형태(List, Set, Map 등)로 수집할 때 사용되는 유틸리티 클래스입니다.
    *   **`toList()` / `toSet()`:** 스트림의 요소를 각각 리스트나 셋으로 모아줍니다.
    *   **`toMap()`:** 스트림의 요소를 Key와 Value 쌍을 갖는 `Map` 형태로 만듭니다. (Key와 Value를 정의해야 함)
    *   **`joining()`:** 여러 요소를 하나로 합쳐서(문자열 등으로) 보여주는 역할을 합니다.

***

*   **핵심 학습 포인트:** 스트림은 코드를 간결하게 만들고, 메모리 효율성을 높이며, 지연 연산을 통해 성능을 관리하는 함수형 프로그래밍의 필수 도구입니다.
*   **필수 개념:** 람다식, 함수형 인터페이스, Intermediate/Terminal 연산 구분, Optional 처리, `collect()`를 활용한 데이터 변환 방법 이해.
*   **학습 조언:** 새로운 개념이 많으므로, 처음에는 부담을 느끼더라도 자주 사용해 볼수록 익숙해지며 활용 능력이 향상됩니다.

## 1. 스트림의 고급 활용: 그룹화와 분할 (Grouping & Partitioning)

### 1-1. 그룹화 (GroupingBy)
스트림 컬렉터의 `groupingBy()` 메서드를 활용하여 특정 조건이나 기준에 따라 데이터를 그룹화합니다.
- **기능:** 주어진 `Function` (값을 받아 값을 리턴하는 함수)을 기준으로 요소를 그룹화합니다.
- **결과:** 각 그룹의 기준이 **키(Key)**가 되고, 해당 그룹에 속한 요소들(Employee 객체 등)이 **값(Value)**으로 들어가는 `Map`을 반환합니다.
- **활용 예시:** 사원 객체를 부서별(Department Name)로 묶어(Grouping) 해당 부서별 목록을 얻어낼 때 사용합니다.
- **추가 활용:** 그룹화된 맵을 다시 스트림으로 변환하여 부서별 평균 급여와 같은 복합적인 연산을 수행할 수 있습니다.

### 1-2. 분할 (PartitioningBy)
스트림 컬렉터의 `partitioningBy()` 메서드를 활용하여 데이터를 두 가지 범주(이분법적 나눔)로 나눕니다.
- **기능:** 주어진 `Predicate` (조건식을 검사하여 `true` 또는 `false`를 리턴하는 식)를 사용하여 요소를 분리합니다.
- **결과:** `Map`을 반환하며, 키(Key)는 반드시 **Boolean (`true` 또는 `false`)**입니다.
- **활용 예시:** 사원의 급여가 특정 기준($X$ 이상)인지 아닌지(합격/불합격)를 기준으로 데이터를 두 리스트로 나누거나, 짝수/홀수 여부를 판별할 때 사용합니다.
- **핵심 차이:**
    - **`groupingBy`:** Function을 사용하며, 리턴 값이 여러 개일 수 있습니다 (예: 부서 이름).
    - **`partitioningBy`:** Predicate를 사용하며, 리턴 값이 반드시 `true` 또는 `false` 두 가지로 제한됩니다.

---

## 2. 병렬 스트림의 이해와 주의사항 (Parallel Stream)

### 2-1. 병렬 스트림의 개념
- **정의:** `parallelStream()`을 사용하면 데이터를 한 줄에서 순차적으로 처리하는 것이 아니라, 여러 스레드(Thread)에서 작업을 나누어 동시에 처리(병렬 처리)하게 합니다.
- **용도:** 계산량이 많거나 데이터셋의 크기가 **매우 클 때** 성능을 높이는 데 효과적입니다.
- **사용 방법:** 스트림을 얻어낼 때 `stream()` 대신 `parallelStream()`을 호출합니다.

### 2-2. 병렬 스트림 사용 시 주의사항 (★★ 중요)
병렬 스트림은 만능이 아니며, 사용할 때 여러 잠재적 문제에 유의해야 합니다.
- **성능 저하 (Overhead):** 데이터셋의 크기가 작을 경우, 병렬 처리를 위한 초기 설정 및 조정 비용(Overhead)이 오히려 순차 처리보다 성능을 저하시킬 수 있습니다.
- **순서 보장 불가:** 여러 스레드가 동시에 작업을 수행하므로, 결과나 처리 과정의 순서가 보장되지 않을 수 있습니다.
- **스레드 안전성 (Thread Safety):** 병렬 환경에서 공유 자원을 수정할 경우, 데이터 경쟁(Race Condition) 등 스레드 안전성 문제가 발생할 수 있으므로, 반드시 동기화 처리에 신경 써야 합니다.

---

## 3. 응용 사례 및 결론

- **다단계 처리:** 그룹화와 분할을 연달아 사용하거나(Nested Stream), 여러 필드를 조합하여 복합적인 조건을 적용하는 등(예: 부서별로 그룹화한 후, 그 안에서 급여 기준으로 다시 분할) 복잡한 비즈니스 로직을 스트림으로 구현할 수 있습니다.
- **학습 핵심:** 스트림의 강력함은 메소드 자체의 복잡함이 아니라, **'어떤 상황에서 어떤 메서드(groupingBy, partitioningBy, parallelStream)를 사용할지'**에 대한 적절한 판단에 달려 있습니다.
- **정리:** 스트림은 객체 지향 언어(Java)에서 함수형 프로그래밍 개념을 도입한 것으로, 메모리 구조나 복잡한 내부 원리를 알기보다는, **어떤 목적을 가지고 어떤 메서드를 호출해야 하는지** 기능적으로 이해하는 것이 가장 중요합니다.

## 1. 백엔드와 프론트엔드의 이해 및 웹 애플리케이션 개요
- **웹 애플리케이션 목적:** 웹 애플리케이션을 개발하는 것이 목표이며, 개발자는 백엔드 및 프론트엔드 기술을 이해해야 합니다.
- **백엔드의 역할:** 데이터를 저장하고 정제하며, 필요할 때 적절하게 데이터(데이터 조회, 처리 등)를 제공하는 역할을 담당합니다.
- **프론트엔드의 역할:** 백엔드에서 제공받은 데이터를 사용자 화면에 시각적으로 보여주고 상호작용하게 하는 역할(View)을 담당합니다.
- **HTML의 사용처:** 프론트엔드에서 화면을 보여주는 구조 자체는 주로 HTML 언어를 이용합니다.

## 2. HTML 기본 구조와 동작 원리
- **HTML의 정의:** HTML은 HyperText Markup Language의 약자로, 하이퍼링크(Hyperlink)를 클릭하여 다른 문서로 이동하는 등의 웹페이지 구조를 만드는 마크업 언어입니다.
- **기본 구조 (Element):** 모든 HTML 문서는 `시작 태그`와 `끝 태그`로 이루어진 '요소(Element)'의 구조를 갖습니다.
    - 예: `<html>` (시작 태그) ... `</html>` (끝 태그)
- **주요 엘리먼트:** 모든 HTML 엘리먼트는 크게 `<!DOCTYPE html>` (문서 유형 선언), `<head>` (메타데이터), `<body>` (실제 콘텐츠 내용)의 구조를 가집니다.
- **데이터의 추상화 (DOM Tree):**
    - 자바스크립트와 같은 언어는 HTML의 각 태그를 마치 **객체**처럼 추상화하여 다루게 되는데, 이를 **DOM (Document Object Model)** 트리를 통해 구현합니다.
    - 웹 문서는 루트(Root) 엘리먼트부터 시작하여 계층적인 트리 형태로 구성되어 있습니다.
- **핵심 디자인 개념 (박스 모델):**
    - HTML에서 가장 중요하게 이해해야 할 개념은 모든 HTML 요소는 **'박스(Box)'** 형태로 만들어진다는 것입니다.
    - 모든 웹 페이지는 거대한 박스들(요소들)이 중첩되고 배치된 구조로 이해하는 것이 웹 개발의 기본 원리입니다.

## 3. 웹 표준 및 개발 환경의 이해 (History & Modernization)
- **웹의 진화 (하이퍼링크의 도입):**
    - 초기 컴퓨터 환경은 명령 프롬프트(Command Line)처럼 어려운 환경이었습니다.
    - HTML과 하이퍼링크 개념이 도입되면서 누구나 쉽게 정보를 주고받을 수 있는 방식으로 웹이 발전했습니다.
- **브라우저의 역할:**
    - 우리가 작성한 HTML 언어로 만든 문서를 실제로 해석하고 시각적으로 보여주는 주체는 **웹 브라우저(Browser)**입니다.
    - 웹 브라우저는 HTML의 규칙(규약)을 해석하는 엔진 역할을 합니다.
- **웹 표준의 중요성 (Browser War):**
    - 초기에는 각 브라우저 제조사(Internet Explorer 등)가 독자적인 해석 규칙을 고수하며 '브라우저 전쟁'을 치렀습니다.
    - 이로 인해 페이지 제작자가 특정 브라우저 버전별로 코드를 다르게 만들어야 하는 등 심각한 문제가 발생했습니다.
    - **결론:** 모든 개발은 브라우저가 가진 **웹 표준(Web Standard)**을 준수하여, 어떤 브라우저에서도 동일하게 작동하도록 하는 것이 핵심입니다.
- **HTML5의 등장:**
    - 다양한 문서 유형 선언(DOCTYPE)이 존재했으나, 현재는 **HTML5**가 표준으로 사용됩니다.
    - HTML5는 단순히 마크업 언어의 기능을 넘어, 새로운 API 등을 포함하여 웹 애플리케이션 개발이 가능하도록 확장되었습니다.

## 4. 프론트엔드의 핵심 기술: HTML, CSS, JavaScript
- **프론트엔드 구성의 3요소 (The Trifecta):**
    * **HTML (구조/뼈대):** 페이지의 내용을 구성하고 어떤 정보가 존재하는지 정의합니다. (What)
    * **CSS (디자인/스타일):** 콘텐츠를 어떻게 보여줄지(색상, 레이아웃, 크기 등) 시각적으로 꾸며줍니다. (How it looks)
    * **JavaScript (동작/로직):** 버튼 클릭, 데이터의 표시/숨김 등 사용자 인터랙션이나 동적인 동작(Action)을 처리합니다. (How it works)

## 5. 심화 개념 정리 및 학습 방향 제시
- **자바스크립트의 확장 (Node.js):**
    - 원래 자바스크립트는 브라우저 내부에서만 작동하는 언어였습니다.
    - **Node.js**는 이 자바스크립트 언어를 브라우저 밖(서버 환경)으로 꺼내어 동작하게 만든 환경입니다.
    - 이를 통해 자바스크립트만으로 백엔드와 프론트엔드 모두를 처리하는 것이 가능해졌습니다.
- **개발 환경 관리 (NVM):**
    - Node.js는 버전별로 다양한 종류가 존재하기 때문에, **NVM (Node Version Manager)**과 같은 도구를 이용하면 여러 버전의 Node.js를 쉽게 설치하고 관리할 수 있습니다.
- **학습 방향 및 조언:**
    - 웹 개발을 배우는 과정에서 핵심은 단순히 코드를 암기하는 것이 아니라, 웹 페이지의 모든 요소가 어떻게 **'박스'** 단위로 구성되고 배치되는지 *관심을 갖고* 바라보는 시야를 기르는 것입니다.
    - 이후에는 Java (스트림, 컬렉션 등)와 자바스크립트 등 이전에 배운 개념들을 실제 예제에 적용하며 심화 학습을 진행하는 것이 중요합니다.
    - **최우선 과제:** 배운 내용을 막히지 않게, 꾸준히 실습하며 진행하는 것이 가장 중요합니다.
