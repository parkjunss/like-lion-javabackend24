## 1. 오리엔테이션 및 학습 현황 점검
- **주간 학습 분위기 공유:** 주말 이후 월요일 오전의 '월요병'을 언급하며 학생들의 공감대 형성 및 격려.
- **학습 진도 점검 및 독려:**
    - 현재까지 3주간 진행한 과정을 되돌아보며 학생들이 어려움을 느끼는 부분을 스스로 도움을 요청하는 것이 중요함을 강조.
    - 멘토, 강사, 학급 내 회고 등 도움을 요청할 수 있는 다양한 경로를 안내.
    - 지연되거나 어려운 부분을 느낀 학생들은 망설이지 말고 적극적으로 도움을 요청하여 학습 결손을 최소화해야 함.

## 2. 데이터베이스(DBMS)의 이해
- **데이터의 필요성:** 애플리케이션을 동작시키기 위해서는 수많은 데이터가 발생하며, 이 데이터들을 효과적으로 관리하고 원하는 정보를 추출하는 것이 필수적이다.
- **DBMS와 SQL의 역할:**
    - **DBMS (Database Management System):** 데이터를 저장하고 관리하는 시스템(예: MySQL).
    - **SQL (Structured Query Language):** DBMS에 저장된 데이터를 원하는 형태로 조회하고 조작하는 데 사용되는 표준 언어.
- **DBMS의 특징 및 관리:**
    - DBMS는 데이터 관리를 위해 복잡한 메커니즘을 갖추고 있으며, 일반적인 파일 저장 방식과는 다르게 비용과 전문적인 관리가 필요하다.
    - **도커(Docker) 활용:** 대규모 프로그램을 개발할 때 환경 충돌을 방지하고 깨끗한 개발 환경을 유지하기 위해 도커와 같은 컨테이너 기술을 활용하는 것이 권장됨.

## 3. 관계형 데이터 모델과 테이블 설계
- **관계형 데이터베이스(RDBMS):** 데이터를 **표(Table)** 형태로 관리하는 모델. 엑셀 시트처럼 직관적으로 데이터를 정리하는 방식이다.
- **정규화(Normalization)의 필요성:**
    - 모든 정보를 하나의 테이블에 모아 저장할 경우, 특정 정보가 중복되어 저장되고, 이는 저장 공간 낭비(비용)와 데이터 수정/관리의 어려움(불일치성)을 초래한다.
    - 따라서, 데이터의 성격에 따라 관련 있는 정보들로 테이블을 분리하고, 이 테이블 간의 **관계**를 설정하여 데이터를 관리하는 것이 효율적이다.
- **테이블 요소 및 관계:**
    - **레코드(Record):** 테이블에서 한 사람(또는 개체)에 대한 모든 정보를 의미한다.
    - **기본 키 (Primary Key):** 테이블의 각 레코드를 유일하게 식별할 수 있는 값. (예: 사원 ID).
    - **외래 키 (Foreign Key):** 한 테이블의 기본 키 값이 다른 테이블을 참조하는 역할. 테이블 간의 관계를 맺는 핵심 요소이다.

## 4. SQL의 핵심 개념: 테이블 결합 (JOIN)
- **JOIN의 목적:** 테이블 하나에 모든 정보가 들어있지 않기 때문에, 여러 테이블에 흩어져 있는 데이터를 논리적인 관계를 통해 통합하여 원하는 정보를 조회(추출)할 때 사용한다.
- **결합 조건의 중요성:**
    - JOIN을 수행할 때는 반드시 두 테이블 간의 연결 조건(Join Condition)을 명확하게 부여해야 한다. 조건이 없을 경우, 모든 데이터가 서로 조합된 과도한 결과(카테시안 조인, Cross Join)가 발생할 수 있다.
- **주요 JOIN 유형 및 사용법:**
    1. **INNER JOIN (내부 조인):**
        - 조인 조건이 *정확히 일치하는* 데이터만 결과에 포함된다. (가장 기본적인 조인)
    2. **OUTER JOIN (외부 조인):**
        - INNER JOIN의 한계점(조건이 불일치하여 데이터가 누락되는 문제)을 보완하기 위해 사용한다.
        - **LEFT OUTER JOIN:** 왼쪽(FROM 절에 먼저 명시된) 테이블의 모든 데이터를 유지하며, 오른쪽 테이블에서 일치하는 데이터가 없어도 해당 레코드를 포함시킨다.
        - **RIGHT OUTER JOIN:** 오른쪽 테이블의 모든 데이터를 유지하며, 왼쪽 테이블에서 일치하는 데이터가 없어도 해당 레코드를 포함시킨다.
        - **FULL OUTER JOIN:** 양쪽 테이블의 모든 데이터를 포함한다. (MySQL에서는 LEFT JOIN + RIGHT JOIN 결과를 UNION 연산자로 결합하여 사용)
    3. **SELF JOIN (자기 조인):**
        - 같은 테이블의 데이터를 여러 개의 역할(예: 사원 테이블 = 사원 + 매니저)로 나누어 생각하고 조인하는 방식.
    4. **NATURAL JOIN (자연 조인):**
        - 명시적인 조건 없이, 두 테이블 간에 **동일한 컬럼명**을 가진 컬럼이 있을 경우, 이들을 기준으로 자동으로 조인한다. (컬럼명 일치에 주의해야 함)
    5. **JOIN USING:**
        - 조인 조건이 되는 컬럼명을 사용자가 명확하게 지정하여 조인하는 방식. 컬럼명이 다르더라도 지정하여 조인이 가능하므로 안전성이 높다.
    6. **JOIN ON:**
        - `FROM TableA JOIN TableB ON A.col = B.col` 형태로 조인 조건을 가장 명확하고 표준적으로 지정하는 방식.
    7. **Theta Join (세타 조인):**
        - JOIN 조건이 반드시 '같음(Equality, =)'일 필요는 없으며, 범위 비교(>, <, BETWEEN 등)와 같은 일반적인 비교 연산자를 사용하여 데이터를 결합하는 조인 방식.

***

**요약 핵심 정리:**
*   **관계형 데이터 관리의 핵심:** 정보를 하나의 테이블에 모두 저장하지 않고, 필요에 따라 여러 테이블로 분리(정규화)하고, 이 테이블들을 **관계**로 엮어 관리한다.
*   **데이터 추출의 핵심:** **JOIN**을 사용하여 여러 테이블에 흩어진 정보를 원하는 조건에 맞게 조합하여 가져온다.
*   **가장 중요하게 이해할 개념:** JOIN의 종류(Inner, Outer, Self, Natural 등)와 각각의 작동 방식, 그리고 언제 어떤 유형을 사용해야 하는지에 대한 원칙을 이해하는 것이 가장 중요하다.

## 1. 서브쿼리 (Subquery)의 동작 원리 및 고급 활용
*   **서브쿼리 이해의 중요성:** 서브쿼리는 처음 접할 때 낯설고 어렵게 느껴지지만, 전체적인 실행 흐름(루(Row) 단위 처리)를 이해하면 크게 어렵지 않다.
*   **실행 원리 (기억할 핵심):** 서브쿼리는 전체 데이터셋을 한 번에 실행하는 것이 아니라, 바깥쪽 쿼리(Outer Query)가 **레코드 단위로 순차적으로** 데이터를 가져올 때마다, 해당 레코드의 값(예: 부서 번호)을 가지고 내부 서브쿼리가 실행되는 방식이다.
*   **Correlated Subquery (상관 서브쿼리):**
    *   서브쿼리가 외부 쿼리의 특정 컬럼 값을 참조하여 실행된다.
    *   예시: "자신이 속한 부서의 평균 급여보다 자신의 급여가 큰 사원"을 찾을 때, 외부 쿼리(사원)의 `DEPT_NUMBER`가 내부 서브쿼리(부서 평균 급여)의 실행 조건으로 사용된다.
    *   이 방식은 일반적인 `JOIN`만으로는 구현하기 복잡한 조건들을 처리하는 데 유용하다.

## 2. 존재 여부 확인 키워드: EXISTS
*   **용도:** 특정 조건이나 부하 직원이 존재하는지(Yes) 여부만을 확인하고 싶을 때 사용한다.
*   **동작 원리:** `EXISTS`는 서브쿼리의 결과 값 자체가 중요한 것이 아니라, **단 한 건이라도 레코드가 존재하는지(True/False)** 만 확인하여 조건을 만족하는지 검사하는 역할을 한다.
*   **활용:** 부하 직원이 없는 부서만 조회하고 싶을 때, `EXISTS`를 사용하면 부하 직원이 연결되어 있는 직원은 제외할 수 있다.

## 3. 데이터 제어 언어 (DML/DDL) 복습: 테이블 조작
*   **테이블 생성 (CREATE TABLE):** 테이블 이름, 컬럼명, 데이터 타입 등을 지정하여 테이블을 만듭니다.
*   **데이터 삽입 (INSERT INTO):** `INSERT INTO 테이블명 (컬럼명) VALUES (값)` 형태로 데이터를 추가합니다. 여러 개의 값을 한 번에 입력할 때 쉼표(,)를 사용하여 여러 쌍의 값을 나열할 수 있습니다.
*   **데이터 삭제 (DELETE FROM):**
    *   `DELETE FROM 테이블명 WHERE 조건` 구문을 반드시 사용해야 원하는 레코드만 삭제할 수 있다. `WHERE` 절이 없으면 테이블의 모든 데이터가 삭제되므로 주의해야 한다.
*   **테이블 삭제 (DROP TABLE):** 테이블 자체를 데이터와 구조와 함께 완전히 삭제하는 명령어입니다. (예: `DROP TABLE 테이블명`)
*   **워크벤치 안전 모드:** 데이터 삭제나 수정 시 실수를 방지하기 위해 워크벤치(Workbench)가 기본적으로 '안전 모드'를 활성화할 수 있으며, 실제 도스 콘솔(DOS console)에서는 이 모드가 해제되어 정상적으로 동작할 수 있다.

## 4. 집합 연산자 (Set Operators): UNION, INTERSECT, MINUS
*   **개념:** 두 개 이상의 `SELECT` 문 결과를 하나의 결과 집합으로 합칠 때 사용한다.
*   **UNION (합집합):** 두 쿼리 결과를 합치되, **중복되는 값은 제거**하고 고유한 값만 남긴다.
*   **UNION ALL:** 두 쿼리 결과를 합치며, **중복되는 값도 모두 포함**하여 출력한다.
*   **INTERSECT (교집합):** 두 쿼리 결과에서 **공통으로 존재하는 값**만 출력한다.
*   **MINUS / EXCEPT (차집합):** 첫 번째 쿼리 결과에 포함되지만, 두 번째 쿼리 결과에는 포함되지 않는 값들만 출력한다.
*   **주의사항:** 집합 연산자를 사용하려면, 모든 `SELECT` 문의 컬럼 개수와 데이터 타입이 일치해야 한다.

## 5. 윈도우 함수 (Window Function): 순위 및 계산
*   **목적:** 전체 행(Row)을 기준으로 계산하는 것이 아니라, **특정 그룹(Partition) 내의 행들을 기준**으로 연산(집계, 순위 등)을 수행하되, 그 결과를 그룹 전체에 표시할 때 사용한다.
*   **핵심 구문:** `FUNCTION() OVER (PARTITION BY 컬럼명 ORDER BY 컬럼명)`
*   **PARTITION BY:** 그룹화의 기준을 지정한다. 이 기준별로 독립적인 계산을 수행한다. (예: 부서별로 그룹을 나눈 후, 그 부서 내에서만 순위를 매긴다.)
*   **주요 함수:**
    *   **RANK():** 순위를 매기는 함수. 동점자가 있을 경우 같은 순위를 부여하고, 다음 순위는 건너뛴다.
    *   **ROW_NUMBER():** 순위를 매기는 함수. 동점자가 있어도 번호를 건너뛰지 않고 1부터 순차적으로 부여한다.
*   **로우 넘버링 (ROW_NUMBER):**
    *   MySQL은 `LIMIT` 절을 사용하여 행 번호를 가져오지만, Oracle 등 다른 DBMS에서는 별도의 `ROWNUM` 컬럼을 사용한다.

## 6. 마무리 및 다음 학습 예고
*   **요약 정리:** 강의에서 다룬 내용들(서브쿼리, EXISTS, 집합 연산, 윈도우 함수 등)은 웬만한 SQL 쿼리를 작성하는 데 필요한 핵심 지식들이다.

## 1. 데이터베이스 설계 개요 및 인덱스 이해
- **데이터베이스 설계 과정:** 테이블 설계 $\rightarrow$ 제약 조건 설정 $\rightarrow$ 인덱스 구축 $\rightarrow$ 실제로 시스템 구현의 과정을 순차적으로 진행한다.
- **인덱스(Index)의 역할:** 대규모 데이터베이스에서 데이터를 검색할 때 성능 저하를 막고 속도를 높이는 역할을 한다.
    - 데이터가 많아질수록 (예: 만 건의 데이터) 모든 데이터를 하나하나 검색하는 것보다 인덱스를 사용하는 것이 효율적이다.
    - 인덱스는 데이터의 값을 기반으로 **트리(Tree) 구조**를 형성하여, 전체 데이터를 비교할 필요 없이 원하는 값을 빠르게 찾아낼 수 있도록 돕는다. (검색 속도 향상)
- **지정 원칙:** 기본키(Primary Key) 같은 핵심 컬럼에는 기본적으로 인덱스가 설정되며, 사용자가 필요에 따라 특정 컬럼에 인덱스를 추가하여 검색 속도를 최적화할 수 있다.

## 2. SQL 명령어의 분류 및 기능 (DDL, DML, DCL, TCL)
SQL 언어는 기능에 따라 크게 네 가지로 분류된다.

- **DDL (Data Definition Language, 데이터 정의어):**
    - 테이블이나 사용자 같은 데이터베이스의 **구조**를 정의하고 제어하는 언어.
    - **주요 명령어:**
        - `CREATE`: 테이블, 사용자, 인덱스, 뷰 등을 생성한다.
        - `ALTER`: 기존 테이블이나 객체의 구조를 수정한다.
        - `DROP`: 객체(테이블, 사용자 등) 자체를 완전히 삭제한다.
        - `RENAME`: 객체의 이름(Name)을 변경한다.
- **DML (Data Manipulation Language, 데이터 조작어):**
    - 테이블 내에 있는 **실제 데이터**를 조작하고 검색하는 명령어.
    - **주요 명령어:**
        - `SELECT`: 데이터 조회 및 검색 (지금까지 사용한 주요 명령어).
        - `INSERT`: 데이터 삽입.
        - `UPDATE`: 데이터 수정.
        - `DELETE`: 데이터 삭제.
- **DCL (Data Control Language, 데이터 제어어):**
    - 사용자 권한을 관리하는 명령어.
    - **기능:** 특정 사용자에게 데이터베이스 접근 및 조작 권한을 부여하거나 박탈하는 작업(`GRANT`, `REVOKE`)을 수행한다.
- **TCL (Transaction Control Language, 트랜잭션 제어어):**
    - 데이터베이스에 여러 단계의 변경사항이 발생했을 때, 그 과정을 관리하는 언어.
    - **핵심 개념 (트랜잭션):** 데이터의 일관성을 유지하기 위해, 변경사항을 원본 데이터에 바로 반영하지 않고, **작업이 완전히 끝난 시점**에만 원본에 반영한다.
    - **주요 기능:** 트랜잭션의 시작(COMMIT)과 실행 취소(ROLLBACK)를 관리한다.
    - **격리 레벨 (Isolation Level):** 여러 세션이 동시에 작업할 때, 각 세션이 서로의 작업에 간섭받지 않도록 데이터 접근 수준을 설정하는 개념이다.

## 3. 핵심 데이터베이스 작업 비교 (삭제와 트랜잭션)
- **DELETE vs. TRUNCATE:**
    - **DELETE (DML):** 데이터만 삭제한다. 트랜잭션 제어(TCL)의 영향을 받으므로, **ROLLBACK(실행 취소)이 가능하다.** (데이터 삭제 시 트랜잭션 관리가 이루어짐)
    - **TRUNCATE (DDL):** 테이블 자체의 구조를 건드리지 않으면서 모든 데이터를 일괄 삭제한다. **트랜잭션 제어의 영향을 받지 않으므로, ROLLBACK이 불가능**하며 일반적으로 DELETE보다 속도가 빠르다.

## 4. 데이터 타입 및 제약 조건 설정
- **데이터 타입:** 컬럼에 어떤 유형의 데이터를 담을 것인지 정의하는 것(숫자, 문자, 날짜 등).
    - **숫자형:** `INT` (정수), `BIGINT` (더 큰 정수), `DECIMAL` (소수점을 포함한 숫자).
    - **문자형:**
        - **VARCHAR:** 가변 길이(Variable Length). 실제로 들어가는 길이만큼의 공간만 차지하여 효율적이다.
        - **CHAR:** 고정 길이(Fixed Length). 지정된 자릿수만큼의 공간을 항상 확보한다. (예: 주민등록번호처럼 길이나 형태가 정해진 경우 사용)
    - **기타:** `DATE`, `TIME`, `DATETIME` (날짜/시간 결합), `TIMESTAMP` (자동 시간 기록), `JSON`, `BLOB` (바이너리 파일 등).
- **제약 조건 (Constraints):** 테이블의 데이터 무결성을 보장하는 규칙.
    - **NOT NULL:** 해당 필드는 반드시 값이 입력되어야 하며, 공백(NULL)은 허용하지 않는다.
    - **UNIQUE:** 컬럼에 중복된 값(Duplicate Value)이 들어올 수 없다.
    - **PRIMARY KEY (기본키):** 테이블의 각 행(Row)을 유일하게 식별하는 핵심 컬럼. 자동으로 **NOT NULL**과 **UNIQUE** 제약 조건을 갖는다.
    - **FOREIGN KEY (외래키):** 테이블 간의 관계를 설정하는 제약 조건. 다른 테이블의 기본키(PK)를 참조하여 데이터 간의 일관성을 유지한다.
    - **DEFAULT:** 사용자가 값을 입력하지 않았을 때, 미리 설정해 둔 기본값을 자동으로 넣어주는 기능. (예: 생성 시간)
    - **CHECK:** 컬럼에 들어오는 값이 특정 조건을 충족하도록 제한하는 제약조건. (예: 스코어가 0~100 사이여야 한다.)
- **자동 증가 (Auto-Increment):** Primary Key로 사용되는 컬럼이 데이터 삽입 시 사용자가 값을 입력하지 않아도, 시스템이 자동으로 순차적인 번호를 부여하여 채워주는 기능. (Oracle의 `SEQUENCE` 객체와 원리가 유사함)

## 5. 실습 예제를 통한 개념 종합 적용
- 실제 테이블(Book, Employee 등)을 설계할 때는 위에 언급된 모든 개념(데이터 타입, 제약조건, 기본키, 외래키, 오토인크리먼트 등)을 종합적으로 적용하여 설계한다.
- **설계 과정의 중요성:** 이처럼 다양한 제약 조건을 사전에 명확하게 설정함으로써, 데이터베이스에 잘못되거나 불완전한 데이터가 들어오는 것을 원천적으로 차단할 수 있다.

## 1. 관계형 데이터베이스 모델링 및 기본 개념 복습
*   **CRUD 작업의 이해:** 데이터베이스의 기본적인 기능인 생성(Create), 읽기(Read), 수정(Update), 삭제(Delete) 과정을 숙지하고 활용할 수 있어야 합니다.
*   **테이블 설계 및 제약 조건:**
    *   데이터를 저장할 테이블을 설계할 때, 각 데이터의 무결성을 보장하기 위한 다양한 제약 조건(Constraint)을 적절하게 부여해야 합니다.
    *   테이블을 구성할 때는 식별자(ID)와 같은 기본 키(Primary Key)를 활용하며, 컬럼 이름이나 구조를 정할 때 논리적 흐름을 고려하는 것이 좋습니다.
*   **테이블 관계 설정 (멤버 - 주문 예시):**
    *   '멤버' 테이블과 '주문(Orders)' 테이블 같은 두 개의 테이블이 존재할 때, 두 테이블은 관계를 맺고 있습니다.
    *   주문 테이블에는 **누가 주문했는지**를 식별할 수 있는 **멤버 ID** 컬럼이 반드시 필요합니다.

## 2. 외래 키(Foreign Key) 제약 조건 심화 학습
*   **외래 키의 역할:** 외래 키는 한 테이블의 컬럼 값이 다른 테이블의 기본 키(Primary Key)와 반드시 연결되어 있음을 명시하는 제약 조건입니다. 이를 통해 데이터의 참조 무결성을 유지합니다.
*   **구현 방법:** 외래 키 제약 조건을 부여할 때는 `REFERENCES` 키워드를 사용하여, 어느 테이블의 어떤 컬럼과 연결되는지 명확히 지정해야 합니다.
*   **트랜잭션과 관계의 중요성:**
    *   외래 키가 설정된 경우, 부모(참조되는) 데이터가 삭제되면 자식(참조하는) 데이터의 무결성이 깨지게 됩니다. (예: 멤버 ID가 삭제되면, 그 멤버 ID를 참조하는 주문 기록도 문제가 발생).
    *   **[주요 옵션]** 외래 키 제약 조건을 설정할 때 삭제 시 어떻게 처리할지 옵션을 지정할 수 있습니다.
        *   **`ON DELETE CASCADE`:** 부모(상위) 데이터가 삭제될 경우, 연결된 자식(하위) 데이터까지 **자동으로 함께 삭제**됩니다. (예: 부서가 사라지면, 해당 부서에 속한 모든 사원도 함께 삭제)
        *   **`ON DELETE SET NULL`:** 부모 데이터가 삭제될 경우, 자식 테이블의 외래 키 컬럼 값을 **`NULL` (널)로 변경**합니다. (단, 해당 컬럼이 Null을 허용해야만 가능)
    *   *참고:* 외래 키가 설정된 컬럼은 보통 `NOT NULL` 제약 조건을 함께 적용하여 값이 항상 존재하도록 관리합니다.

## 3. 인덱스(Index)의 개념과 활용 전략
*   **인덱스의 개념 및 원리:**
    *   인덱스는 책의 목차와 유사하며, 데이터베이스가 특정 데이터를 빠르게 찾을 수 있도록 보조하는 구조(트리 형태)입니다.
    *   데이터 조회(SELECT) 시, 인덱스가 걸린 컬럼을 `WHERE` 절에 사용하여 검색하면 전체 테이블을 탐색하는 것보다 훨씬 빠르게 데이터를 가져올 수 있습니다.
    *   인덱스가 걸리면 데이터의 **입력, 수정, 삭제 시** 해당 인덱스 구조까지 모두 일관성 있게 업데이트(재조정)하는 작업이 필요합니다.
*   **인덱스 생성 방법:** `CREATE INDEX` 구문을 사용하여 특정 테이블의 특정 컬럼에 인덱스를 생성할 수 있습니다.
*   **효율적인 인덱스 사용 전략:**
    *   **가장 중요한 결정:** 어떤 컬럼을 인덱스로 걸 것인지를 결정하는 것이 가장 어렵고 중요합니다.
    *   **좋은 조건:**
        *   검색 조건(`WHERE` 절), 조인 조건(`JOIN`), 정렬 조건(`ORDER BY`)에 **자주 사용되는 컬럼**을 선택해야 합니다.
        *   **값의 분포가 다양하고 넓게 퍼져있는 (Cardinality가 높은) 컬럼**이 좋습니다. (예: 이름이나 사번 등). 같은 값이 너무 많으면 (Cardinality가 낮은 값) 인덱스 효과가 적거나 오히려 성능 저하를 일으킬 수 있습니다.
    *   **주의점:** 모든 컬럼에 인덱스를 걸면 수정/삭제 시 부하가 과도하게 발생하고 저장 공간만 많이 차지하므로, 신중하게 적용해야 합니다.

## 4. 마무리 및 프로젝트 계획 제언
*   **복습 및 학습 권장:** 데이터베이스 설계는 직접 이론만으로 숙지하기 어렵습니다. 강의에서 배운 기본 키, 외래 키, 인덱스 등의 개념을 실제로 적용해 보는 것이 중요합니다.
*   **실습과 프로젝트 제안:**
    *   가장 좋은 학습 방법은 실제 프로젝트를 진행하는 것입니다.
    *   수업에서 다루는 내용(테이블 설계, CRUD, 관계 설정 등)을 응용하여, 도서관 요소 관리 등 스스로 관심 있는 주제를 선정하여 **작은 토이 프로젝트**를 설계하고 구현해 볼 것을 권장합니다.
*   **향후 학습 방향:**
    *   다음 시간에는 트랜잭션(Transaction) 개념 등을 심화 학습할 예정입니다.
    *   개별적으로도 직접 데이터베이스 테이블을 설계하고, 이를 프로그래밍 언어(예: JDBC)와 연동하여 데이터를 저장하고 조작하는 과정을 경험해 보면 지식 습득에 큰 도움이 될 것입니다.
