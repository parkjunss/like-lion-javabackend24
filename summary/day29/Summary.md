## 목차
1. 버전 관리 시스템 (Git) 복습 및 협업의 중요성
2. 소프트웨어 설계 원칙 (SOLID) 이해
3. 개발 학습 자세 및 최종 목표 설정

---

## 1. 버전 관리 시스템 (Git) 복습 및 협업의 중요성
*   **Git의 목적 및 필요성:**
    *   개발 과정에서 발생할 수 있는 파일 이름 변경, 버전 관리의 어려움 등 비효율적인 과정을 간편하게 관리하기 위해 사용합니다.
    *   파일을 단순히 주고받거나 관리할 때 발생하는 시간 소모와 충돌(Conflict) 위험을 방지합니다.
*   **Git의 작동 원리 (내부 구조):**
    *   로컬 컴퓨터에서 동작하며, 저장소 내부의 `.git` 디렉터리가 모든 작업을 담당합니다.
    *   파일의 내용 전체를 저장하는 것이 아니라, 변경된 파일의 해시(Hash) 값 등을 기록하여 저장합니다. 따라서 같은 내용은 재저장하지 않고 기존 해시를 참고하여 매우 효율적으로 메모리를 관리합니다.
*   **협업 환경에서의 중요성:**
    *   팀 프로젝트에서는 팀원 간의 의견 공유와 동시 개발이 필수적이며, Git은 코드 통합 과정에서 발생하는 문제들을 효율적으로 해결해 줍니다.
    *   **CI/CD (Continuous Integration/Continuous Delivery):** 통합과 배포 과정을 지속적으로 반복하는 파이프라인을 구축하는 것이 중요하며, Git은 이러한 과정의 핵심 도구입니다.
*   **핵심 명령어 사용법 및 주의사항:**
    *   **`git reset`:** 현재 버전이 아닌 과거 버전으로 돌아가고 싶을 때 사용합니다. (옵션: `--hard`, `--soft`, `--mixed`를 통해 파일 상태를 제어할 수 있습니다.)
    *   **브랜치 전략:** 여러 브랜치를 나누어 개발할 때의 작업 흐름 전략을 세우는 것이 중요합니다.
    *   **충돌(Conflict) 해결:** 여러 사람이 같은 파일을 수정했을 때 발생하는 충돌은 시스템이 어느 부분이 수정되었는지 알려주므로, 개발자가 이를 보고 수동으로 해결한 후 커밋하면 됩니다.
    *   **학습 조언:** Git은 기본적으로 알아야 하는 필수 도구입니다. 단지 명령어를 암기하는 것보다는, **어떤 상황에서 이 도구가 왜 필요한지 흐름을 이해**하고 반복적으로 실습하는 것이 가장 중요합니다.

## 2. 소프트웨어 설계 원칙 (SOLID) 이해
*   **스프링 프레임워크의 이해 (최종 목표):**
    *   최종 목표는 스프링 프레임워크를 능숙하게 사용하는 것입니다.
    *   **프레임워크(Framework)의 개념:** 애플리케이션을 처음부터 모두 만드는 것이 아니라, 이미 튼튼하게 만들어져 있는 **뼈대(반제품)**를 가져와서 우리 프로그램에 필요한 기능(살)만 붙여나가는 방식입니다.
    *   따라서 단순히 동작하는 것(기능 구현)을 넘어, 이 뼈대(프레임워크)가 어떻게 만들어졌는지 그 **원리**를 이해하는 것이 중요합니다.
*   **SOLID 원칙:**
    *   좋은 객체와 시스템을 만들기 위해 지켜야 할 5가지 객체 지향 설계 원칙입니다.
    *   **S - 단일 책임 원칙 (Single Responsibility Principle, SRP):**
        *   하나의 객체나 메서드가 너무 많은 책임(기능)을 가지고 과부하가 걸리지 않도록 해야 합니다.
        *   책임이 분리되어야 특정 기능에 문제가 생겼을 때 해당 부분만 수정할 수 있습니다.
    *   **O - 개방-폐쇄 원칙 (Open-Closed Principle, OCP):**
        *   객체는 **확장(Open)**은 용이해야 하지만, **수정(Close)**은 닫혀 있어야 합니다.
        *   예: 새로운 결제 수단이 추가되어도 기존의 코드를 수정하지 않고 기능을 추가할 수 있도록 설계해야 합니다.
    *   **L - 리스코프 치환 원칙 (Liskov Substitution Principle, LSP):**
        *   자식 클래스를 부모 클래스를 대체하여 사용해도 예상했던 대로 완벽하게 동작해야 합니다.
        *   부모와 자식 간의 동작 일관성을 유지하는 것이 중요합니다.
    *   **I - 인터페이스 분리 원칙 (Interface Segregation Principle, ISP):**
        *   하나의 인터페이스가 너무 많은 기능을 포함해서는 안 됩니다.
        *   클라이언트가 필요로 하는 기능만 담은 여러 개의 작은 인터페이스로 분리하는 것이 좋습니다 (관심사 분리).
    *   **D - 의존성 역전 원칙 (Dependency Inversion Principle, DIP):**
        *   상위 모듈은 하위 모듈의 구체적인 구현체(Details)에 의존하는 것이 아니라, **추상화(Abstraction)**에 의존해야 합니다.
        *   이 원칙을 통해 모듈 간의 결합도를 낮추고 유연하게 시스템을 설계할 수 있습니다.

## 3. 개발 학습 자세 및 최종 목표 설정
*   **가장 중요한 원칙: 단순화 및 동작 구현:**
    *   초보 단계에서는 너무 많은 지식(예: 모든 디자인 패턴, 복잡한 설계)에 매몰되지 말고, **가장 단순하게 동작하는 것(Working Code)**을 만드는 것이 최우선입니다.
    *   너무 많은 이론을 주워듣기보다는, 현실의 문제를 단순하게 쪼개서 해결하는 과정(문제 단순화)에 집중하는 것이 핵심 역량입니다.
*   **배움의 자세:**
    *   지식은 휘발성이 강하므로, 한 번에 이해하고 끝내려고 하기보다는 **반복적으로 반복(Repetition)**하고 직접 손으로 써보는 경험을 쌓는 것이 중요합니다.
    *   AI 도구는 생산성을 높여주지만, 처음 학습 단계에서는 AI에 지나치게 의존하지 않고, 스스로 흐름을 파악하며 깊이 고민하는 과정이 필수적입니다.
    *   막연하게 질문하기보다는 **'스스로 생각한 가설'**을 세우고, 그 안에서 의문점을 찾아 파고들어 해결해 나가는 방식으로 AI를 활용하는 것이 효과적입니다.

5. 결론: 구조적 설계의 중요성과 스프링의 역할

## 1. 고결합도(High Coupling)의 문제점 이해 (TV 예시)
*   **상황 제시:** `TVUser` 클래스가 구체적인 `KaramaiTV` 객체에 직접 의존하며 사용하고 있다.
*   **문제점:** 만약 `TV` 객체가 고장 나거나(예: `KaramaiTV`에서 `STV`로 교체될 때), 구현체가 변경되면 `TVUser`의 사용 코드 전체를 수정해야 하는 심각한 문제가 발생한다.
*   **핵심 개념:** 한 객체가 다른 객체의 **구체적인 구현체**에 직접 의존하는 상태를 **높은 결합도(High Coupling)**라고 하며, 이는 시스템의 유연성을 떨어뜨리는 주요 원인이다.

## 2. 인터페이스를 이용한 결합도 낮추기 (추상화 활용)
*   **해결책 1: 인터페이스 정의:** 모든 TV 구현체(`KaramaiTV`, `STV` 등)가 공통의 `TV` 인터페이스를 구현하도록 구조를 변경한다.
*   **작동 원리:** `TVUser`는 특정 구현체(`KaramaiTV`)가 아닌, 오직 `TV`라는 **인터페이스(추상화)**에만 의존하게 된다.
*   **장점:** 구현체가 `KaramaiTV`에서 `STV`로 바뀌더라도, `TVUser`는 인터페이스를 기준으로 동작하기 때문에 코드를 거의 수정할 필요가 없어져 시스템의 유연성과 확장성이 높아진다.

## 3. 객체 생성 과정 분리: 팩토리 패턴 도입
*   **문제점 지속:** 인터페이스를 사용해도, 여전히 `TVUser`가 직접 객체(`new KaramaiTV()`)를 생성하는 코드를 가지고 있다면(즉, 생성 과정에 의존한다면), 구현체 교체 시 이 생성 코드 역시 수정해야 하는 문제가 남는다.
*   **해결책 2: 팩토리(Factory) 패턴 도입:** `TV` 객체를 **직접 생성하는 코드**를 `TVUser`가 가지지 않고, 별도의 **팩토리 객체**에게 위임한다.
*   **작동 원리:** 팩토리 객체는 외부의 요청(예: "K 타입 TV를 원함")을 받아 내부 로직에 따라 적절한 구현체(`KaramaiTV` 또는 `STV`)를 생성하여 반환한다.
*   **결과:** `TVUser`는 누가, 언제, 어떤 방식으로 TV 객체를 생성하는지에 대한 지식으로부터 완전히 분리된다.

## 4. 의존성 역전 원칙 (DIP, Dependency Inversion Principle)
*   **핵심 원칙:** 시스템의 상위 모듈(사용하는 쪽)이 하위 모듈(구현하는 쪽)의 구체적인 구현체에 의존해서는 안 되며, 대신 **추상화(인터페이스)**에 의존해야 한다. (이존 방향을 역전시킨다.)
*   **실제 예시 (메시징 서비스):**
    *   **비(非)DIP 구조:** `MessageSender`가 `EmailService`나 `SMSService`와 같은 구체적인 서비스 클래스에 직접 의존한다. (결합도 높음)
    *   **DIP 적용 구조:** `MessageSender`가 `MessageService`라는 추상 인터페이스에만 의존한다.
    *   **장점:** 만약 나중에 '카카오톡 메시징 서비스'가 추가되어도, `MessageSender`는 인터페이스만 보고 있으므로 코드를 수정할 필요가 전혀 없다.
*   **패키지 의존성 관점:** 모듈 간의 의존성 역시, A 패키지가 B 패키지의 구체 클래스에 의존하는 것이 아니라, **공통 인터페이스**만 의존하도록 구조를 설계해야 결합도를 낮출 수 있다.

## 5. 결론: 구조적 설계의 중요성과 스프링의 역할
*   **최종 결론:** 소프트웨어 설계를 할 때 높은 결합도를 피하고, 인터페이스를 통해 추상화하여 의존성 방향을 역전시키는 것이 시스템을 유연하고 확장 가능한 구조로 만드는 핵심이다.
*   **스프링과 DI:** 실제 개발 환경에서는 객체를 생성하고 연결하는 과정(팩토리의 역할)을 개발자가 직접 할 필요가 없다. **스프링(Spring) 프레임워크**는 이러한 객체 생성 및 의존 관계 주입(Dependency Injection, DI)을 프레임워크 차원에서 **자동으로** 처리해주어 개발자가 코드를 더욱 간결하게 만들 수 있게 한다.
*   **요약:** 개발자는 **"무엇을(What)"** 할 것인지(인터페이스)에 집중하고, **"어떻게(How)"** 할 것인지(구현체)는 프레임워크나 외부에서 주입받는 방식으로 설계하는 것이 가장 이상적이다.

2. POJO(Plain Old Java Object)의 등장 배경과 중요성
3. 단위 테스트(Unit Test)의 필요성과 구현

---

## 1. 객체 설계의 핵심 원칙 (SOLID)
- **단일 책임 원칙 (Single Responsibility Principle, SRP) 강조:**
    - 객체는 오직 하나의 책임만 가져야 합니다.
    - 예를 들어, 데이터를 저장하는 기능, 보고서를 생성하는 기능 등 여러 기능을 한 객체에 몰아넣는 것은 SRP에 위배됩니다.
- **설계 원칙의 적용:**
    - 객체를 설계할 때는 단일 책임 원칙 등 여러 디자인 원칙을 염두에 두고 적용해야 합니다.
    - 기존에 작성했던 코드도 이 원칙들을 생각하며 재점검하는 습관이 중요합니다.

## 2. POJO(Plain Old Java Object)의 등장 배경과 중요성
- **EJB와 엔터프라이즈 기술의 한계 (배경 지식):**
    - 스프링(Spring)과 같은 현대적인 프레임워크가 나오기 전에는 EJB(Enterprise Java Beans)와 같은 기술들이 엔터프라이즈급 대규모 프로젝트에서 많이 사용되었습니다.
    - EJB는 기능을 구현하는 것 외에 보안 처리, 트랜잭션 처리, 로깅 등 복잡한 기능을 담당하는 '세션빈(Session Bean)' 형태를 주로 사용했습니다.
- **EJB가 야기한 문제점 (강한 결합도):**
    - EJB를 사용하려면 인터페이스 구현, 상속, 그리고 복잡한 설정 파일(XML 등)이 필요하여 구조가 매우 복잡했습니다.
    - 이러한 기술들은 사용되는 과정에서 코드 전체가 특정 프레임워크에 강하게 종속되는(Tight Coupling) 문제를 초래했습니다.
    - 이로 인해 코드를 수정하거나 다른 기술로 교체하기가 매우 어려워졌습니다.
- **POJO의 정의 및 역할:**
    - **POJO**는 **Plain Old Java Object**의 약자입니다.
    - 이는 특정 프레임워크나 라이브러리에 의존하지 않는 **순수한 일반 자바 객체**를 의미합니다.
    - 즉, 상속 관계나 인터페이스 구현 같은 복잡한 제약 없이, 일반적인 자바 객체만으로도 설계할 수 있는 객체입니다.
- **POJO의 중요성:**
    - POJO는 객체 지향 원칙에 충실하며, **프레임워크에 구속되지 않는 순수한 로직**을 구현할 수 있게 합니다.
    - **장점:**
        * **유지보수 용이성:** 특정 프레임워크의 기술이 바뀌어도 코드를 변경할 필요성이 줄어듭니다.
        * **높은 이식성:** 다른 환경이나 프레임워크에 종속되지 않아 자유롭게 활용할 수 있습니다.
        * **테스트 용이성:** 독립적인 객체이므로 단위 테스트를 수행하기 매우 편리합니다.

## 3. 단위 테스트(Unit Test)의 필요성과 구현
- **단위 테스트의 필요성:**
    - 소프트웨어를 개발한 후, 각 기능을 가장 작은 단위(메서드나 클래스)로 분리하여 완벽하게 동작하는지 테스트해야 합니다. (테스트는 필수 과정입니다.)
    - 만약 테스트를 거치지 않으면, 코드에 숨겨진 버그나 잠재적인 결함을 놓칠 수 있습니다.
- **테스트 과정의 어려움 (JUnit 도입 배경):**
    - 초기에 개발자들은 수동으로 테스트 코드를 작성했기 때문에, 테스트를 실행할 때마다 모든 테스트 코드가 본래 서비스 코드에 함께 포함되어 배포하거나 관리하는 것이 매우 번거롭고 불편했습니다.
- **JUnit의 역할:**
    - **JUnit**은 이러한 불편함을 해소하기 위해 등장한 단위 테스트 프레임워크입니다.
    - **핵심 기능:**
        * **자동화된 실행:** 테스트할 기능을 `@Test` 어노테이션을 사용하여 표시하면, 테스트 프레임워크가 자동으로 이 코드를 찾아 실행합니다.
        * **결과 검증:** 단순하게 결과를 눈으로 보는 것을 넘어, 시스템이 자동으로 '예측했던 값'과 '실제 결과 값'을 비교하여 성공 또는 실패 여부를 알려줍니다. (예: `Assertions` 사용)
    - **구조적 이점:**
        * 빌드 도구(Maven, Gradle)를 사용하면 테스트 코드가 들어가는 `src/test` 디렉토리를 물리적으로 분리하여, 실제 운영 환경에 배포될 코드와 테스트 코드를 명확하게 분리할 수 있습니다.

## 1. 객체 설계 원칙 및 POJO 개념 이해
*   **SOLID 원칙의 중요성:**
    *   객체를 설계할 때 단일 책임 원칙(Single Responsibility Principle) 등 객체 지향 설계 원칙(SOLID)을 하나씩 정리하며 적용하는 것이 중요하다.
    *   이러한 원칙들은 단지 이론으로 끝나는 것이 아니라, 실제로 코드를 설계하고 바라볼 때 적용되어야 한다.
*   **POJO (Plain Old Java Object)의 정의 및 배경:**
    *   POJO는 외부 프레임워크나 라이브러리에 의존하지 않고, 순수한 자바 객체(독자적으로 동작 가능한 객체)로 만들어진 개체를 의미한다.
    *   과거의 웹 기술(서블릿, EJB 등)은 특정 프레임워크에 강하게 종속되어(High Coupling) 개발 및 유지보수에 어려움이 많았다.
    *   POJO 개념은 이러한 강한 종속성에서 벗어나 순수한 자바 객체 구조를 유지하려는 시도에서 출발했다.

## 2. 테스트의 중요성 및 Unit Test 학습 (JUnit, TDD)
*   **테스트의 필수성 강조:**
    *   소프트웨어 개발 과정에서 테스트는 가장 중요하게 강조되는 부분이며, 시스템이 예상치 못한 문제에서 문제가 생겼을 때 이를 파악하는 핵심 수단이다.
    *   테스트 코드를 꼼꼼하게 작성하는 것은 프로젝트 전체 아키텍처를 설계하는 기반이 되며, 프로젝트의 안정성을 보장한다.
*   **TDD (Test Driven Development)와 Unit Test:**
    *   **TDD:** '테스트 기반 개발'을 의미하며, 필요한 기능을 개발하기 전에 테스트 코드를 먼저 작성하고, 그 테스트 코드를 통과할 수 있는 최소한의 코드를 작성하는 방식이다.
    *   **Unit Test:** 코드의 가장 작은 단위(함수, 메서드 등)가 독립적으로 제대로 작동하는지 검증하는 테스트이다.
    *   JUnit은 Java 환경에서 Unit Test를 수행할 때 사용되는 대표적인 라이브러리이며, 스프링 프로젝트에서는 표준적으로 포함되어 있다.
*   **테스트 코드의 역할:**
    *   테스트 코드는 실제 배포 시에는 필요가 없는 코드(별도의 디렉토리에서 관리됨)이지만, 개발 단계에서는 시스템이 개발자가 만든 로직을 객관적으로 검증하는 기준점 역할을 한다.
    *   JUnit의 `Assertions` 같은 기능은 단순히 눈으로 결과 값을 확인하는 것 이상으로, 시스템이 자동으로 '같다(True)' 또는 '다르다(False)' 등을 판단하게 돕는다.

## 3. 스프링의 철학: POJO 기반의 개발 지원
*   **스프링 등장 배경:**
    *   초기 웹 기술들은 특정 프레임워크에 너무 많은 종속성을 가지고 있어 개발의 효율성이 떨어졌다.
    *   스프링 프레임워크를 설계한 이 로드 존슨 등은 이러한 문제점을 인지하고, **POJO를 기반으로 개발을 지원**하는 것을 주요 목적으로 삼았다.
*   **스프링의 핵심 철학:**
    *   스프링은 프레임워크의 강력한 기능을 활용하면서도, 개발자가 작성한 핵심 로직(비즈니스 로직)은 외부 프레임워크에 강하게 종속되지 않도록(POJO 기반으로) 설계하도록 돕는다.
    *   이러한 원칙 덕분에, 코드는 프레임워크에 묶이지 않는 본질을 유지하게 되어 재사용성과 테스트 용이성(Testability)이 매우 높아진다.

## 4. 최신 개발 환경 이해: 스프링 부트(Spring Boot)
*   **스프링 프레임워크 vs. 스프링 부트:**
    *   **스프링 프레임워크:** 강력하지만, 프로젝트를 구성할 때 설정해야 하는 설정(Configuration)들이 매우 복잡하고 많았다.
    *   **스프링 부트:** 스프링 프레임워크를 사용하기 편리하게 만든 도구이다.
    *   **역할:** 스프링 부트는 복잡했던 초기 설정 과정들을 개발자가 신경 쓰지 않아도 자동으로 처리해주어(Auto-Configuration) 개발 과정을 획기적으로 간소화했다.
*   **개발 현황과 전망:**
    *   과거에는 스프링 프레임워크를 직접 다루는 개발자가 많았으나, 현재 시점의 대부분의 프로젝트와 애플리케이션들은 스프링 부트를 이용하여 구축되고 있다.
    *   스프링 부트가 제공하는 편리함 덕분에, 복잡한 설정 없이도 기능을 구현하고 빠르게 결과물을 만들 수 있다.
    *   **결론:** 개발자는 스프링의 깊이 있는 원리(POJO, 테스트, 설정 방식)를 이해하는 것이 중요하며, 이 지식을 바탕으로 스프링 부트를 이용해 개발을 진행하는 것이 가장 효율적이다.

## 1. 스프링 프레임워크 소개 및 프로젝트 생성 환경
- **스프링 프레임워크의 역할:** 스프링은 애플리케이션 개발을 위한 '반제품'처럼 사용될 수 있도록 미리 구축된 거대한 프레임워크이다.
- **공식 문서 활용:** 최신 정보나 상세한 내용은 스프링 공식 페이지를 참고하는 것이 가장 좋다.
- **IDE를 통한 생성:** IntelliJ IDEA와 같은 개발 환경(IDE)은 스프링 프레임워크를 사용하여 애플리케이션이 생성되도록 도움을 준다.
    - IntelliJ Ultimate 사용자는 IDE 내에서 직접 스프링 프로젝트 생성이 가능하다는 장점이 있다.
    - 커뮤니티 버전 사용자의 경우, 프로젝트를 외부에서 생성하여 오픈하거나 수동으로 임포트하는 방식으로도 충분히 개발을 진행할 수 있다. (결국 모든 프로젝트는 외부에서 생성되어 가져올 수 있다.)

## 2. Spring Boot 프로젝트 생성 과정 및 설정 (Initializr 활용)
- **Spring Boot의 간편함:** 전통적으로 웹 애플리케이션 서버(WAS)를 사용하려면 설정 과정이 복잡했으나, Spring Boot는 이러한 복잡한 설정(WAS 포함, 위치 설정 등)을 알아서 처리해준다.
- **프로젝트 생성 단계:** 프로젝트를 생성할 때 다음과 같은 주요 설정을 진행해야 한다.
    - **위치 지정:** 프로젝트가 생성될 로컬 경로를 지정한다.
    - **언어 및 빌드 도구:** 사용할 언어(Java)와 빌드 도구(Gradle/Maven)를 선택한다.
    - **패키지 설정:** 그룹(패키지명)과 아티팩트(프로젝트 이름)를 지정하여 기본 패키지 구조를 확정한다.
    - **JDK 버전:** 사용할 자바 개발 키트(JDK) 버전을 선택한다.
    - **패키징 형식:** 웹 애플리케이션 서버에 배포하는 방식(WAR)과 독립 실행 가능한 방식(JAR) 중 선택한다. 웹 애플리케이션의 경우, Spring Boot가 WAS를 포함하여 필요한 모든 설정을 간편하게 처리해준다.
- **Spring Initializr 사용:** 공식 Initializr 서비스를 이용하면 필수적인 의존성(Dependency)을 쉽게 추가하고, 프로젝트의 기본 골격을 자동으로 생성할 수 있다.

## 3. 설정 파일 포맷의 이해: Properties vs. YAML
- **기존 방식 (Properties):** 자바의 `Properties` 객체를 활용하며, 값 사이에 `=` 기호를 사용하여 `Key=Value` 형태로 저장된다.
    - 이 파일은 스프링 프레임워크가 값을 읽어 메모리 상의 Key-Value 구조로 변환하는 역할을 한다.
    - 단점: 오타 체크나 가독성 측면에서 다소 불편할 수 있다.
- **권장 방식 (YAML):** YAML(Yet Another Markup Language) 포맷을 사용하는 것이 현재 스프링 부트 개발에서 훨씬 간편하고 선호된다.
    - YAML은 들여쓰기(Indentation)와 콜론(`:`) 형태로 값을 구조화하여 가독성을 높인다.
    - 스프링 부트는 기본적으로 Properties 형식을 YAML 형식으로 바꿔서 처리하는 경우가 많다.

## 4. 핵심 개념: 의존성 관리와 프레임워크의 역할
- **빌드 도구의 목적:** Maven이나 Gradle 같은 빌드 도구는 외부 라이브러리(Dependency)를 프로젝트에 편리하게 추가할 수 있도록 도와주는 도구이다.
- **의존성 관리의 필요성:** 스프링 프레임워크를 사용하려면 수많은 라이브러리(의존성)가 필요하지만, 개발자가 이 모든 라이브러리를 수동으로 다운로드하여 관리하는 것은 매우 복잡하다.
- **Spring Boot의 역할:** Spring Boot는 개발자가 단순한 의존성 추가(예: `spring-boot-starter-web`)만 해주면, 내부적으로 해당 기능에 필요한 모든 보조 라이브러리들(로그 라이브러리, JSON 변환 라이브러리 등)을 자동으로 찾아 포함시켜준다.

## 5. 웹 애플리케이션 구조와 RESTful 원칙
- **애플리케이션 동작 원리:** Spring Boot가 애플리케이션을 실행하면 내부에 웹 서버(일반적으로 Tomcat)가 구동되며, 특정 포트(기본 8080)에서 대기하게 된다.
- **@RestController와 @GetMapping:** REST API를 만들기 위해서는 `@RestController`와 같은 애노테이션(Annotation)을 사용하여 해당 클래스나 메서드가 "컨트롤러 역할을 한다"는 것을 프레임워크에 알려줘야 한다.
    - `@GetMapping`은 특정 URL 경로(예: `/`)로 HTTP GET 요청이 들어왔을 때 이 메서드가 실행되도록 연결하는 역할을 한다.
- **Single Responsibility Principle (SRP) 준수:** 코드를 작성할 때는 한 클래스가 하나의 역할만을 수행하도록 분리하는 것이 좋으며, 역할별로 패키지 및 클래스를 분리하는 것이 바람직하다.
- **HTTP 메서드와 REST:** 클라이언트와 서버 간의 통신은 HTTP 프로토콜을 기반으로 하며, 다음과 같은 표준 메서드를 사용한다.
    - **GET:** 데이터를 조회할 때 (읽기)
    - **POST:** 새로운 데이터를 생성할 때 (추가)
    - **PUT:** 데이터를 전체 수정할 때 (전체 수정)
    - **PATCH:** 데이터를 일부 수정할 때 (부분 수정)
    - **DELETE:** 데이터를 삭제할 때 (삭제)
    - *핵심:* URL을 통해 값을 가져오는 것은 GET 방식이 기본이다.

## 6. Spring Boot 동작 원리 및 개발 모범 사례
- **프레임워크의 약속:** 스프링 프레임워크는 개발자가 임의로 동작하는 것이 아니라, `@RequestMapping`과 같은 애노테이션을 통해 미리 정의된 '약속(Contract)'을 지켜가며 사용해야 한다.
- **자동 변환 (Auto-Conversion):**
    - Spring Boot는 클라이언트가 보내는 데이터(JSON 형태)를 자바 객체(POJO)로 자동 변환해주거나, 자바 객체를 JSON 형태로 자동 변환해주는 기능을 제공한다.
    - 이 과정에서 Jackson과 같은 라이브러리들이 내부적으로 핵심적인 역할을 수행하며, 개발자가 별도로 설정할 필요가 없다.
- **요약:** Spring Boot는 개발자가 설정해야 할 복잡한 인프라와 설정을 최대한 추상화하고, 개발자가 오직 비즈니스 로직에만 집중할 수 있도록 돕는 것이 핵심적인 장점이다.

## 1. 프레임워크의 개념 및 필요성 (Framework Concept)
- **프레임워크란?**
    - 소프트웨어 개발 시 처음부터 모든 것을 직접 만들지 않고, 기본적인 뼈대(골격)나 반제품 형태로 제공된 구조를 의미합니다. (예: 웹 애플리케이션 개발 시 사용하는 Spring, Spring Boot 등)
- **프레임워크의 필요성:**
    - **효율성:** 모든 것을 처음부터 개발하는 것보다 이미 정해진 뼈대를 이용하는 것이 훨씬 빠르고 효율적입니다.
    - **목적:** 개발 과정의 복잡성을 줄이고, 개발자가 핵심 비즈니스 로직에만 집중할 수 있도록 돕습니다.

## 2. 프레임워크 사용의 장점과 단점
- **장점 (Advantages):**
    - **빠른 개발 속도 (Speed):** 이미 구축된 구조를 사용하므로 개발 시간을 획기적으로 단축할 수 있습니다.
    - **안정성 및 신뢰성 (Reliability):** 전문적인 개발자들이 이미 구현해 놓은 어렵고 복잡한 부분들(예: 트랜잭션 관리, 보안)은 안정적이고 검증된 형태로 제공됩니다. (실력 없는 개발자가 처음부터 만들 경우 무너질 위험이 있음)
    - **유지보수 용이성 (Maintainability):** 표준화된 구조와 규칙(Standard)을 따르기 때문에, 나중에 다른 개발자나 회사에서 유지보수하더라도 일관된 방식으로 접근할 수 있습니다.
    - **표준화:** 공공기관 프로젝트 등에서는 전사적인 표준 프레임워크(예: 전자정보 프레임워크)를 사용하여, 누가 개발하든 일관된 기준을 갖추게 합니다.
- **단점 (Disadvantages):**
    - **자유도 제한:** 정해진 뼈대나 규칙을 따라야 하므로, 특정 비즈니스 요구사항에 맞춰 자유롭게 구조를 설계하거나 커스터마이징하는 데 제약이 따를 수 있습니다. (예: 원하는 창문 위치나 방 개수 조절이 어려움)

## 3. 스프링 부트의 특징과 역할
- **Spring Boot의 목표:** 복잡한 Spring Framework 사용 과정을 최대한 단순화하고 간편하게 만듭니다.
- **핵심 특징:**
    - **Convention over Configuration (관례 우선 설정):** 개발자가 별도로 설정하지 않아도, "이런 부분이 있을 것이라고 가정하고" 기본 설정(Default Setting)을 제공하여 프로젝트가 동작하게 합니다.
    - **Auto-Configuration (자동 구성):** 사용된 라이브러리나 의존성(Dependency)을 파악하여 필요한 설정 파일들을 자동으로 구성해줍니다.
    - **독립 실행 가능성 (Embedded WAS):** 별도로 웹 애플리케이션 서버(WAS)를 설치할 필요 없이, WAS 기능이 내장되어 바로 실행이 가능합니다. (이전 Spring Framework는 WAS를 별도 설치해야 했음)
- **스타터(Starter) 의존성 관리:**
    - Spring Boot는 **스타터(Starter)** 라는 형태의 라이브러리를 통해 필요한 기능(예: 웹, JPA, 보안 등)에 관련된 **필요한 모든 라이브러리와 기본 설정값**을 한 번에 묶어서 가져다줍니다.
    - 이를 통해 개발자가 직접 수많은 의존성을 찾고 버전 호환성까지 맞춰가야 하는 번거로움을 해소합니다.

## 4. 웹 애플리케이션 아키텍처와 프로젝트 구조
- **웹 서버 vs. WAS (웹 애플리케이션 서버):**
    - **웹 서버 (Web Server, 예: Apache):** 주로 정적인 자원(Static Content, 이미지, HTML 파일 등)을 처리하고 요청을 받는 역할을 합니다. 경량화되어 빠르고 가볍습니다.
    - **웹 애플리케이션 서버 (WAS, 예: Tomcat):** 비즈니스 로직과 같은 **동적인 자원**을 처리합니다. 웹 서버보다 기능이 많아 무거운 편입니다.
    - **실제 구조:** 일반적으로 웹 서버가 요청을 먼저 받아 정적인 자원 여부를 확인하고, 처리할 수 없는 동적인 요청만 WAS로 전달하는 형태로 시스템이 구축됩니다.
- **프로젝트 구조 (폴더/파일):**
    - `src/main/java`: 실제 소스 코드를 작성하는 곳입니다.
    - `src/main/resources`: 설정 파일 (Properties 파일 등)이나 리소스 파일을 넣는 곳입니다.
    - `src/main/static`: 정적인 자원(Static Resource, 이미지, HTML, JavaScript 등)을 넣는 곳입니다.
- **빌드 도구 및 의존성 관리:**
    - **Build Tool (Gradle/Maven):** 프로젝트의 의존성(어떤 라이브러리를 사용할지), 빌드 과정, 실행 명령 등을 정의하는 도구입니다.
    - **Gradle/Maven:** 프로젝트의 모든 종속성(Dependency) 정보와 버전 관리를 담당하며, 실제 애플리케이션을 실행 가능한 파일(JAR, WAR)로 만들어주는 역할을 합니다.

## 5. 핵심 개념 정리 및 다음 학습 방향
- **주요 에노테이션 (Annotations):**
    - `@SpringBootApplication`: 애플리케이션의 시작점(진입점)을 정의하며, 스프링 부트의 핵심 기능들을 포함하는 메타 태그 역할을 합니다.
    - `@ComponentScan`: 스프링이 관리해야 할 클래스(컴포넌트)를 어디서 찾을지 지정하는 역할을 합니다.
    - `@EnableAutoConfiguration`: 라이브러리 사용에 따른 설정을 자동 활성화해주는 역할을 합니다.
- **학습의 심화:**
    - 현재는 구조와 작동 원리를 **'맛보기'** 수준으로 파악하는 것이 중요하며, 다음 단계에서는 스프링의 가장 핵심적인 원리인 **IOC(Inversion of Control)와 DI(Dependency Injection)**를 집중적으로 학습해야 합니다.
    - 코드를 따라 치는 것보다, "이 코드가 어떻게 작동하는가?", "왜 이런 설정이 필요한가?"라는 관점에서 원리와 동작 원리를 이해하는 데 초점을 맞추어야 합니다.
- **학습 목표 제언:**
    - 배움을 위한 목적을 스스로 설정하는 것이 중요합니다. 배운 지식을 바탕으로 작은 **웹 애플리케이션(토이 프로젝트)**을 직접 만들어 동작시켜 보는 것을 목표로 삼는 것이 학습에 가장 효과적입니다.


1. 객체지향 설계의 핵심 : 결합도를 낮추는 것
구체 클래스에 직접 의존하면 변경에 약하고 인터페이스에 의존하면 유연해진다.

2. SOLID는 좋은 객체 설계의 기준
SRP: 단일 책임 원칙 - 하나의 클래스나 메서드는 하나의 책임만 가져야 한다.
DIP: 의존성 역전 원칙 - 상위 모듈이 하위 구현체에 직접 의존하면 안 된다. 대신 인터페이스 같은 추상화에 의존해야 한다.

3. 인터페이스만으로는 부족하고 객체 생성도 분리
객체 생성 책임을 따로 빼야 한다. 사용자는 객체를 직접 만들지 않고, 외부에서 만들어서 넣어준다.

4. 테스트 코드는 실무에서 필수
Unit Test: 메서드나 클래스 단위로 기능 검증
@Test: 테스트 대상 메서드 표시
src/test: 테스트 코드를 운영 코드와 분리하는 위치
TDD: 테스트를 먼저 작성하고, 그 테스트를 통과하는 코드를 작성하는 방식

5. Spring Boot : Spring 설정을 자동화한 도구
Auto Configuration: 자동 설정
Convention over Configuration: 관례 기반 기본 설정
Embedded WAS: Tomcat 같은 웹 서버 내장
Starter Dependency: 필요한 라이브러리를 묶어서 제공
JAR 실행 가능: 서버 설치 없이 실행 가능
