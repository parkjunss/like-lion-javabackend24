## 1. 형상 관리(Version Control)의 이해와 Git 소개
- **형상 관리의 목적:** 프로젝트 진행 시 파일의 변경 이력을 체계적이고 효과적으로 관리하여, 파일 이름 변경이나 중복 저장을 할 필요 없이 버전별로 쉽게 되돌리거나 추적할 수 있게 돕습니다.
- **Git과 GitHub의 구분:**
    - **Git:** 버전 관리를 수행하는 **도구(Tool)** 자체입니다. (로컬 컴퓨터의 형상 관리).
    - **GitHub:** Git으로 관리된 프로젝트의 저장소를 인터넷(클라우드 환경)에 올려 공유하고 관리하는 **서비스(Repository)**입니다.
- **필요성:** 개발자는 어떤 기술을 만날 때 그것이 "무엇을 위한 도구인지", 그리고 "왜 써야 하는지"를 이해하는 것이 중요합니다. Git은 프로그래밍 작업 외에도 문서 작성 등 모든 파일의 이력을 관리할 수 있습니다.

## 2. Git 내부 작동 구조 및 핵심 개념
- **로컬 저장소 설정 (`git init`):**
    - 원하는 디렉토리에서 `git init` 명령어를 실행하면 해당 디렉토리가 Git의 버전 관리 저장소 역할을 하게 됩니다.
    - 로컬 컴퓨터에는 원하는 만큼 여러 개의 Git 저장소를 만들 수 있습니다.
- **`.git` 디렉토리의 역할:**
    - Git 저장소의 모든 비밀과 데이터가 저장되는 핵심 내부 디렉토리입니다.
    - 이 안에 있는 파일들(예: `HEAD`, `refs`, `objects`)이 Git의 동작 방식을 결정합니다.
- **주요 내부 구조:**
    - **`HEAD`:** 현재 사용자가 어느 브랜치를 보고 있는지 그 위치 정보를 가리키고 있습니다. (브랜치 이동 시 이 정보가 바뀜)
    - **`objects`:** 실제 변경된 데이터(커밋 내용)가 저장되는 Git의 데이터베이스 역할을 하는 곳입니다.
    - **커밋(Commit) 과정:**
        - 커밋은 일정 단위로 변화된 파일들의 스냅샷을 기록하는 행위입니다.
        - 커밋이 발생할 때마다 고유한 **커밋 해시값(Commit Hash)**이 생성됩니다. 이 해시값만 알면 커밋 당시의 파일 상태(어떤 파일이 있었는지, 내용은 무엇이었는지)를 모두 찾아낼 수 있습니다.
        - 커밋은 이전 커밋의 정보를 가지고 있으므로, 어떤 커밋이 가장 최근 버전의 '부모'인지도 기록됩니다.
- **작업 흐름 (핵심 명령어):**
    - **Add (스테이징):** 변경된 파일을 커밋할 준비 영역(Staging Area)에 올리는 과정입니다. (✅ `git add`)
    - **Commit:** 스테이징된 파일들을 묶어 하나의 버전(스냅샷)으로 확정하고, 고유 해시값을 부여하여 로컬 저장소에 기록합니다. (✅ `git commit`)
    - **추적 흐름:** 파일을 수정 → `add`로 스테이징 → `commit`으로 버전 기록.

## 3. 원격 저장소와 협업 워크플로우
- **원격 저장소의 필요성:** Git으로 관리하는 내용이 내 로컬 컴퓨터에만 있는 것을 넘어, 여러 컴퓨터나 팀원들이 공동으로 사용해야 할 때 필요합니다. 이를 **클라우드 환경**에 올려두는 것이 원격 저장소입니다.
- **필수 명령어:**
    - **Clone:** 원격 저장소의 전체 내용과 기록을 나의 로컬 컴퓨터로 처음 복제하여 가져올 때 사용합니다.
    - **Push:** 로컬에서 작업하고 커밋한 내용을 원격 저장소(GitHub 등)에 **업로드**하여 반영할 때 사용합니다.
    - **Pull:** 원격 저장소에 누군가 올린(Push한) 최신 내용을 나의 로컬 컴퓨터로 **다운로드**하여 동기화할 때 사용합니다.
- **충돌(Conflict) 해결:**
    - 여러 사람이 같은 파일의 같은 부분을 수정하여 `Push`나 `Pull` 시 충돌이 발생할 수 있습니다.
    - Git은 충돌이 발생한 부분을 주석 형태로 파일에 남겨주므로, 개발자가 직접 그 내용을 비교하고 정리하여 해결(수동 병합)한 후 다시 커밋하고 Push해야 합니다.
- **Gitignore:**
    - 버전 관리의 대상이 되어서는 안 되는 파일들(예: 컴파일된 클래스 파일, 라이브러리 파일 등)을 미리 지정하여 Git의 추적 대상에서 제외할 때 사용합니다. (✅ `.gitignore` 파일에 작성)

## 4. 고급 Git 활용 전략 및 주요 명령어 정리
- **브랜치(Branch)의 개념:**
    - 메인 작업 흐름에 영향을 주지 않도록 **작업 공간을 분리**할 수 있게 해줍니다.
    - 새로운 기능을 개발하거나 테스트할 때 메인 브랜치에서 가지(Branch)를 뻗어 나간 후 작업하는 것이 안전합니다.
- **브랜치 전략 (Branching Strategies):**
    - 팀 프로젝트 시에는 정해진 규칙(전략)에 따라 브랜치를 관리하는 것이 좋습니다. (예: `develop` 브랜치, `master`/`main` 브랜치 등)
    - 대표적인 전략으로는 Git Flow, GitHub Flow 등이 있으며, 프로젝트 성격에 맞게 전략을 세우는 것이 중요합니다.
- **협업의 원칙 (Fork & Pull Request):**
    - **Fork:** 원격 저장소의 내용을 완전히 복사하여 **나만의 개인 저장소**로 가져오는 것입니다. (내 작업 공간 확보)
    - **Pull Request (PR):** 포크한 저장소에서 작업한 내용을 원본 저장소 개발자에게 "이 부분을 추가했는데 어떻게 생각하시나요?"라고 검토를 요청하는 행위입니다.
    - **Merge (병합):** PR을 통해 검토가 완료된 코드를 원본 프로젝트의 메인 브랜치에 실제로 적용하는 최종 단계입니다.
- **핵심 명령 정리:**
    - **`clone`:** 원격 저장소를 복제하여 로컬로 가져오기.
    - **`pull`:** 원격의 최신 변경 사항을 로컬로 받아 동기화하기.
    - **`push`:** 로컬에서 작업한 커밋을 원격 저장소에 반영하기.
    - **`add`**: 파일을 스테이징 영역에 올리기.
    - **`commit`**: 스테이징된 내용을 로컬에 스냅샷으로 기록하기.
    - **`branch`**: 브랜치를 만들거나 이동시키기.
- **개선된 관습:**
    - Git의 기본 브랜치 명칭이 과거 `master`였으나, 정치적 의미 때문에 현재는 **`main`**을 기본 브랜치로 사용하는 것이 권장되고 있습니다.
    - **가장 중요한 습관:** 협업 시에는 항상 `Pull`을 먼저 수행하여 최신 상태로 동기화한 후 작업하는 것이 충돌을 줄이는 핵심 습관입니다.

# 강의 요약: Git 버전 관리의 핵심 명령 (Commit, Reset, Restore, Diff)

## 목차
1. Git 커밋 및 히스토리 확인
2. Git Reset 개념 및 기본 동작
3. `git reset` 옵션의 이해 (Soft, Mixed, Hard)
4. Git 히스토리 복구 및 비교 (`reflog`, `diff`)
5. 변경 사항 되돌리기 및 복원 (`restore`)

---

## 1. Git 커밋 및 히스토리 확인
- **히스토리 확인:** `git status`나 `git log` 등의 도구를 사용하여 작업한 커밋들을 추적할 수 있습니다.
- **시각적 확인 도구:** Git GUI 툴이나 `git log`의 비주얼 기능을 활용하면 커밋 단위로 파일의 변화를 직관적으로 확인할 수 있습니다.
- **Changes 탭 활용:** 파일의 변화된 내용(변경된 부분)만 따로 모아 보여주는 `changes` 탭을 통해 어떤 부분이 수정되었는지 쉽게 파악할 수 있습니다.

## 2. Git Reset 개념 및 기본 동작
- **Reset의 역할:** 특정 시점의 커밋 상태로 되돌릴 때 사용하는 명령어입니다.
- **커밋의 안전성:** `git reset`을 사용하더라도 이전에 커밋했던 내용(데이터)이 영원히 삭제되는 것은 아닙니다. Git 데이터베이스는 삭제나 업데이트 기능이 아닌 입력과 조회만 하기 때문에 데이터는 보존됩니다.
- **되돌아간 커밋 추적:** 되돌린 커밋 이후에 수행한 작업 기록은 일반적인 `git log`에서는 사라지지만, `git reflog` 명령어를 통해 해당 히스토리까지 추적하고 확인할 수 있습니다.

## 3. `git reset` 옵션의 이해 (Soft, Mixed, Hard)
`git reset` 명령은 커밋을 되돌릴 때 어떤 상태로 작업 영역을 남길지 세 가지 옵션을 통해 제어할 수 있습니다.

- **`--soft`:**
    - 커밋 기록만 취소하고, 작업 디렉터리 및 스테이징 영역의 파일 내용은 그대로 유지합니다.
    - **용도:** 커밋 메시지나 커밋 단위만 수정하고 싶을 때 유용합니다.
- **`--mixed` (기본값):**
    - 커밋 기록을 취소하고, 파일 내용은 작업 디렉터리(Working Directory)와 스테이징 영역(Staging Area)에 변경 사항을 유지합니다.
    - **용도:** 변경된 파일을 다시 검토하거나 다른 커밋과 합치기(Squashing) 전에 유용합니다.
- **`--hard`:**
    - 커밋 기록과 함께 작업 디렉터리, 스테이징 영역의 모든 변경 사항을 완전히 삭제하고, 지정한 커밋 시점으로 돌아갑니다.
    - **용도:** 현재 작업 내용 전체를 버리고 특정 과거 상태로 완벽하게 되돌리고 싶을 때 사용합니다. (가장 강력한 옵션)

## 4. Git 히스토리 복구 및 비교 (`reflog`, `diff`)
- **`git reflog`:**
    - 가장 중요한 명령어 중 하나로, 과거의 모든 HEAD(작업의 흐름) 이동 기록을 보여줍니다.
    - 단순 `git log`가 현재 브랜치의 커밋 히스토리만 보여준다면, `reflog`는 사용자가 어떤 명령으로 어디로 이동했는지 그 **궤적 전체**를 보여줍니다.
    - **용도:** 실수로 커밋을 되돌리거나 특정 커밋 해시를 잊었을 때, 과거의 기록을 찾아 복구하는 데 사용합니다.
- **`git diff`:**
    - 두 시점(커밋 A와 커밋 B) 또는 특정 영역(작업 디렉터리 vs 스테이징 영역) 간의 **차이점**을 비교하는 명령어입니다.
    - **활용:** 두 개의 커밋 사이의 변경된 파일 내용, 새로 추가되거나 삭제된 파일을 구체적으로 확인할 수 있습니다.
- **`git log` 활용:**
    - 특정 커밋 간의 차이점(`git log <commit1>..<commit2>`)을 분석하여 어떤 파일이 어떻게 변경되었는지 확인할 수 있습니다.

## 5. 변경 사항 되돌리기 및 복원 (`restore`)
- **파일 복원 (Discarding Changes):**
    - **`git restore` (또는 `git checkout -- <파일명>`)**: 작업 디렉터리나 스테이징 영역에 있는 특정 파일을 이전 커밋 상태로 되돌릴 때 사용합니다.
    - **활용:** 작업한 파일의 내용이 마음에 들지 않거나, 의도치 않은 변경 사항이 생겼을 때 해당 파일을 안전하게 과거 상태로 복원할 수 있습니다.
- **스테이징 취소:**
    - 스테이징 영역에 실수로 추가한 파일이나 변경사항을 언스테이징(Un-staging)하여, 임시 저장소에서 제거하고 다시 작업 영역으로 되돌릴 수 있습니다. (예: `git restore --staged <파일명>`)

## 1. Git의 히스토리 관리 및 복원 명령어 학습
*   **버전 관리의 기능:** `reset`, `restore`, `log`, `show` 등 다양한 명령어를 사용하여 Git이 관리하는 파일의 히스토리 상태를 확인하고 원하는 시점으로 되돌리거나 복구할 수 있다.
*   **히스토리 확인:** `git log`를 통해 커밋 목록과 상세 정보를 확인하고, 특정 커밋의 해시 값을 이용하여 해당 시점의 파일 변경 내역(어떤 파일이 있었고 어떻게 변했는지)을 상세히 조회할 수 있다.
*   **옵션 활용:** 명령어에 다양한 옵션을 추가하여 결과물을 단순하게 보이게 하거나, 반대로 아주 상세하게 보이도록 조정할 수 있어 원하는 정보만 추출하는 것이 가능하다.

## 2. 브랜치(Branch) 개념과 협업 환경에서의 중요성
*   **브랜치 정의:** 특정 커밋 지점부터 독자적인 개발 흐름을 만들고 싶을 때(예: 새로운 기능 개발, 버그 수정) 분기하여 작업하는 것이 브랜치이다.
*   **협업의 필수 요소:**
    *   **충돌 방지:** 여러 사람이 같은 메인 브랜치에 직접 작업할 경우, 같은 파일이나 이름의 파일이 겹치면서 충돌이 발생할 위험이 크다. 브랜치를 사용하면 개인의 작업 공간이 분리되어 이런 문제를 방지할 수 있다.
    *   **안정적인 개발:** 개발 중인 기능이나 개선 사항을 메인 코드와 분리하여 독립적으로 개발하고 테스트할 수 있다.
*   **브랜치 구조:** `develop` (개발 브랜치), `main` (메인 브랜치), `release` (릴리스 브랜치) 등 목적에 따라 여러 브랜치를 나누어 관리한다.
*   **장점:** 브랜치를 사용하면 개발 과정에서 발생하는 문제를 메인 코드에 영향을 주지 않고 안전하게 테스트할 수 있다.

## 3. 소프트웨어 개발 방법론의 변화와 실무적 접근 전략
*   **개발 방법론의 변화:** 과거에는 폭포수형 개발 방법론처럼 기획 → 분석 → 설계 → 개발 → 배포 순서로 오랜 시간 동안 체계적이고 상세하게 진행되었다.
*   **현대의 트렌드 (애자일/민첩성):** 현대 사회는 변화 속도가 매우 빠르기 때문에, 과거처럼 긴 시간을 두고 완벽하게 설계한 프로젝트가 출시될 시점에 이미 시대가 변해 있을 수 있다.
    *   **핵심 전략:** 이제는 **빠르게 동작할 수 있는 최소 기능(MVP)을 빠르게 출시**하고, 사용자 피드백을 바탕으로 기능을 지속적으로 추가하거나 애플리케이션을 단단하게 만드는 '반복적 개선' 방식이 중요하다.
*   **실력 향상의 핵심:**
    *   **기초 공사:** AI 같은 최신 기술을 잘 활용하는 것도 중요하지만, 그보다 **전반적인 기초 개념과 기본적인 원리를 탄탄하게 다지는 것이 가장 중요하다.**
    *   **프로젝트 중심 학습:** 이론을 주구장창 공부하는 것보다, **'무언가를 만들고 싶다'라는 동기에서 주제를 정하고 실제로 만들어보는 경험**이 지식 습득과 실력 향상에 훨씬 효율적이다.
*   **취업과 커리어 관리:**
    *   **경험의 중요성:** 실제로 직접 완성하여 출시해 본 경험(프로젝트 경험)은 면접 등에서 외운 지식보다 훨씬 강력하고 신뢰성 있는 답변이 된다.
    *   **소프트 스킬 및 네트워킹:** 기술적 실력만큼이나, 적극적인 태도, 성실한 참여, 팀원들과의 원활한 소통(인맥) 같은 소프트 스킬이 직장 생활에서 매우 중요한 자산이 된다.

## 4. 브랜치 활용 심화 및 실전 Git 명령어 연습
*   **브랜치의 심화 활용 (협업):**
    *   **독립 작업 환경:** 개발자는 자신의 작업 브랜치에서 독립적으로 작업하므로, 다른 개발자의 작업에 전혀 영향을 주지 않는다.
    *   **코드 리뷰 (Code Review):** 기능을 완성한 후 바로 메인에 합치기보다는, 팀원들과 함께 브랜치 단위로 코드를 리뷰하는 과정을 거쳐야 한다.
        *   **주의사항:** 코드 리뷰 시 지적을 받았을 때 감정적으로 반응하기 쉽지만, 이러한 '지적'은 나를 지적하는 것이 아니라 **내가 미처 알지 못했던 더 좋은 방법을 알려주는 도움**이라고 긍정적으로 받아들이는 연습이 필요하다.
    *   **브랜치 관리의 이점:** 브랜치를 활용하면 버전 관리, 기능별 분리, 테스트 환경 분리 등을 쉽게 처리할 수 있다.
*   **핵심 Git 명령어 연습:**
    *   **`git branch <이름>`:** 새로운 브랜치를 생성한다.
    *   **`git switch -c <이름>`:** 브랜치를 생성함과 동시에 해당 브랜치로 이동한다. (가장 효율적인 방법)
    *   **`git checkout <이름>`:** 특정 브랜치로 이동한다.
    *   **`git branch -d <이름>`:** 브랜치를 삭제한다.
    *   **`git branch`:** 현재 존재하는 모든 브랜치 리스트를 확인한다.

## 1. Git의 활용 범위 및 기본 워크플로우
- Git은 단순히 개발자가 작성한 소스 코드뿐만 아니라, 개인적으로 일상에서 사용하는 파일이나 데이터를 관리하는 데도 활용할 수 있습니다.
- 실습의 목표는 로컬 리파지토리(repository)를 만들고, 이를 원격 서버(GitHub 등)에 연결한 후 데이터 추가(커밋) 및 동기화(클론) 과정을 반복하며 익숙해지는 것입니다.
- 기본적인 워크플로우는 **브랜치 생성 → 작업 및 커밋 → 메인 브랜치로 병합(Merge)**의 순서로 진행됩니다.

## 2. 브랜치(Branch) 생성 및 관리
- **브랜치 개념**: 브랜치는 독립적인 작업 공간을 의미하며, 메인 개발 라인(main/master)에 영향을 주지 않고 새로운 기능을 개발하거나 버그를 수정할 수 있게 해줍니다.
- **브랜치 생성 및 이동**:
    - `git switch -c [브랜치명]` 명령어를 사용하여 브랜치를 새로 생성함과 동시에 해당 브랜치로 이동할 수 있습니다.
    - 브랜치를 전환할 때마다 Git은 현재 작업 디렉터리 내의 `.git/HEAD` 파일의 내용이 바뀌면서, 어떤 브랜치 상태를 바라보고 있는지 관리합니다.
- **브랜치 관리 명령어**:
    - **전체 브랜치 확인**: `git branch -a` 명령어를 사용하여 로컬 및 원격 브랜치 목록을 확인할 수 있습니다.
    - **브랜치 삭제**:
        - `-d` (소문자): 병합되어 다른 브랜치에서 이미 사용되고 있는 브랜치만 삭제 가능합니다.
        - `-D` (대문자): 병합 여부와 관계없이 강제로 브랜치를 삭제할 때 사용합니다.
    - **브랜치 이름 변경**: `git branch -m` 옵션을 사용하여 브랜치 이름을 변경할 수 있습니다.
- **특정 커밋으로 이동**: `git checkout` 또는 `git switch` 명령어를 사용하여 브랜치 이름이 아닌, 특정 커밋 해시값으로 HEAD 포인터를 이동시킬 수 있습니다.

## 3. 메인 브랜치로 기능 병합하기 (Merging)
- **병합의 목적**: 여러 브랜치(각 팀원의 독립적인 작업 영역)에서 완성된 기능들을 공통적으로 사용하는 메인 브랜치에 통합하는 과정입니다.
- **병합의 원칙 (주의사항)**: 병합을 수행할 때는 **반드시 대상이 될 브랜치(예: 메인 브랜치)로 먼저 이동**해야 합니다.
- **작업 흐름**:
    1. 메인 브랜치로 이동(`git switch main`).
    2. 현재 메인 브랜치에 병합을 원하는 브랜치(`feature-branch`)의 내용을 병합(`git merge feature-branch`)합니다.
    3. 이 과정을 통해 메인 브랜치는 해당 브랜치가 가진 새로운 기능(파일, 코드 등)을 포함하게 됩니다.

## 4. 브랜치 병합 전략 및 충돌 해결 (Conflict Resolution)
- **Fast-Forward 병합 (간단한 병합)**:
    - 메인 브랜치와 기능 브랜치 사이에 분기된 커밋 이력이 없을 때 발생합니다.
    - 메인 브랜치 포인터(HEAD)가 단순히 기능 브랜치의 최신 커밋까지 '따라가기(Move)'만 하면서 병합이 완료됩니다. (새로운 커밋 기록이 남지 않음)
- **Recursive (Three-Way) 병합 (일반적인 병합)**:
    - 메인 브랜치와 기능 브랜치가 서로 다른 커밋을 가진 채 진행된 경우 발생합니다.
    - Git은 이 두 갈래의 이력을 모두 포함하는 **새로운 커밋(Merge Commit)**을 생성하여 병합을 완료합니다.
- **충돌 발생 (Conflict)**:
    - 같은 파일의 같은 라인을 여러 브랜치에서 다르게 수정했을 때, Git이 자동으로 해결하지 못하고 충돌이 발생합니다.
    - **해결 방법**: 병합 과정 중 충돌이 발생하면, 개발자가 직접 파일을 열어 어느 버전의 코드를 남길지 수동으로 결정하고, 이를 수정한 후 `git add`와 `git commit`을 통해 병합을 완료해야 합니다.

## 1. Git Branching 및 Merge (브랜치 작업 흐름)
- **브랜치 생성 및 작업:** 메인(main) 브랜치에서 `hello.txt` 파일을 생성하고 커밋한 후, 새로운 기능 개발을 위해 `feature-1`, `feature-2` 등의 브랜치를 생성하며 각 브랜치에서 작업을 수행합니다.
- **Merge 과정:** 각 기능 단위로 개발된 변경사항을 메인 브랜치로 합칠(Merge) 때, 충돌이 발생할 수 있음을 인지해야 합니다.
- **브랜치 이동:** Merge를 수행할 때는 항상 병합 대상 브랜치(여기서는 메인 브랜치)로 이동하는 것이 기본 원칙입니다.
- **Fast-forward:** 충돌이 발생하지 않은 경우(예: Feature 1을 Main으로 병합할 때), 메인 브랜치가 단순히 앞당겨지는 Fast-forward 방식이 발생합니다.

## 2. 충돌 해결(Conflict Resolution) 심화 과정
- **충돌 발생 원인:** 두 브랜치(예: main과 feature-2)에서 같은 파일의 같은 부분을 다르게 수정했을 때 충돌(Conflict)이 발생합니다.
- **해결 방법:** 충돌이 발생한 파일에 접근하여, 기존 버전(Original)과 새 버전(Incoming)의 내용을 모두 확인합니다.
- **충돌 해결 전략:**
    - 둘 중 하나만 남기고 선택 (한쪽 버전 승인).
    - 둘 다 필요한 경우, 두 내용을 모두 포함하도록 수동으로 코드를 수정하여 병합합니다.
    - 둘 다 필요하지 않은 경우, 삭제합니다.
- **주의사항 (필수 검토 사항):**
    - 충돌이 발생한 모든 부분을 주의 깊게 검토해야 합니다.
    - 코드를 병합하기 전에 의도한 대로 변경되었는지 서로 검토하는 과정이 필수입니다.
    - 복잡한 충돌 해결이 필요할 경우, 개발자와 충분히 협의하는 것이 가장 좋습니다.

## 3. 웹 애플리케이션 구조 및 설계 원칙 개요
- **웹 동작 방식 이해:** 웹 클라이언트가 서버에 요청을 보내고, 서버가 이를 받아 응답하는(Response) 기본적인 통신 흐름을 이해하는 것이 중요합니다.
- **스프링 프레임워크 활용:** 자바를 이용해 웹 애플리케이션을 개발할 때는 일반적으로 **스프링(Spring)**과 같은 프레임워크를 사용합니다.
- **SOLID 원칙 (SOLID Principles):**
    - SOLID는 객체 지향 설계의 5가지 핵심 원칙으로, 시스템의 **유지보수성과 확장성**을 높이기 위해 사용됩니다.
    - 개발 시 이 원칙들을 고려하여 객체들을 설계하는 것이 중요합니다.

## 4. 객체 지향 프로그래밍(OOP) 핵심 개념 복습
강의 초반부에서 다룬 네 가지 핵심 개념을 복습하며, 이 원칙들을 잘 지키는 것이 좋은 설계를 만드는 핵심입니다.

- **추상화(Abstraction):**
    - 복잡한 시스템이나 객체를 단순화하여 **핵심적인 속성이나 기능만을 추출**하는 과정입니다.
    - 예: '고객'이라는 객체를 추상화할 때, 도서대여점 고객, 신발가게 고객 등 사용처에 따라 필요한 필수 속성만 남기고 불필요한 세부사항을 제거합니다.
    - **핵심:** 객체의 필요한 인터페이스나 기능만을 노출하고, 구체적인 내부 구현은 숨기는 것입니다.
- **캡슐화(Encapsulation):**
    - 객체의 속성(데이터)과 행위(메서드)를 하나의 논리적인 단위(캡슐)로 묶는 것입니다.
    - 외부에서는 오직 객체가 제공하는 메서드를 통해서만 접근하도록 제한(접근 제한자: `private`, `public` 등)하여 내부 구현 세부사항을 보호합니다.
- **상속(Inheritance):**
    - 상위 클래스(부모)가 가진 속성과 메서드를 하위 클래스(자식)가 물려받아 사용하는 개념입니다.
- **다형성(Polymorphism):**
    - **하나의 인터페이스(껍데기)를 통해 여러 구현체가 다르게 동작하도록** 하는 것입니다.
    - 예: '동물'이라는 상위 타입으로 '소리 내기'라는 같은 명령을 내리더라도, 실제로 객체가 '비둘기'라면 구구거리고, '개'라면 멍멍거리는 식으로 각 인스턴스에 따라 다르게 동작하는 것이 다형성의 대표적인 예시입니다.

## 5. 설계 원칙과 유지보수성 확보 전략
- **핵심 목표:** 좋은 설계는 유지보수성을 높이고, 확장성(새 기능 추가 용이성) 및 유연성을 확보하는 것을 목표로 합니다.
- **낮은 결합도와 높은 응집도:**
    - **응집도(Cohesion):** 하나의 객체가 관련된 기능들로 잘 묶여 있는 정도가 높아야 좋습니다.
    - **결합도(Coupling):** 객체 간의 의존도가 낮아야 합니다. 두 객체가 서로에게 과도하게 의존하면, 한 쪽이 수정될 때 다른 쪽도 무조건 변경해야 하는 문제가 발생합니다.
- **효과적인 설계:** 설계 단계부터 미래에 발생할 변화(새로운 종류의 고객, 새로운 데이터 구조 등)를 예측하고, 중간에 인터페이스와 같은 추상화된 경계를 삽입하여 모듈 간의 의존도를 낮추고 재사용성을 높여야 합니다.
- **학습의 자세:** 모든 기술은 이론적 지식 습득을 넘어, **반복적인 실습**을 통해 익숙해지는 것이 가장 중요합니다.

## 1. 단일 책임 원칙 (Single Responsibility Principle, SRP)
- **정의:** 클래스는 단 하나의 책임만 가져야 합니다. (하나의 클래스는 단 하나의 목적을 가져야 함)
- **핵심:** 클래스나 모듈을 변경하는 이유는 오직 하나의 이유여야 합니다.
- **위반 사례:** 하나의 객체가 데이터 관리(저장), 비즈니스 로직(급여 계산), 그리고 도메인 정보(직원 정보) 등 여러 가지 역할을 동시에 수행할 때 발생합니다. (예: `Employee` 클래스가 급여 계산, DB 저장 기능까지 가지는 경우)
- **개선 방향:** 객체들이 너무 많은 책임을 지지 않도록, 각 책임별로 독립적인 클래스나 모듈을 분리하여 설계하는 것이 바람직합니다. (예: `SalaryCalculator` 클래스를 따로 분리)
- **실제 적용 예시:** 회원가입 시 비밀번호의 복잡성 검증 로직은 사용자 데이터(`User` 클래스)가 아닌, 비밀번호 검증 전용 클래스(Password Validator)를 만들어 분리해야 합니다.

## 2. 개방-폐쇄 원칙 (Open-Closed Principle, OCP)
- **정의:** 소프트웨어는 **확장에는 열려 있고(Open)**, **수정에는 닫혀 있어야(Closed)** 합니다.
- **핵심:** 새로운 기능을 추가할 때 기존에 잘 동작하던 코드를 수정할 필요가 없어야 합니다.
- **위반의 문제점:** 결제 시스템에 '애플페이'와 같은 새로운 결제 방식이 추가되었을 때, 기존의 결제 코드를 직접 수정해야 한다면 시스템의 안정성이 떨어집니다.
- **적용 방안:** 인터페이스(Interface)를 활용하여 설계합니다.
    - 결제 프로세서(Payment Processor)는 특정 인터페이스를 구현한 모든 결제 방식을 받아들이도록 설계하고, 새로운 결제 방식이 필요할 때 해당 인터페이스만 구현하여 추가합니다.
- **이점:** 기존 코드를 안정적으로 유지하면서도 새로운 요구사항을 쉽게 추가할 수 있어 시스템의 유지보수성이 높아집니다.

## 3. 리스코프 치환 원칙 (Liskov Substitution Principle, LSP)
- **정의:** 자식 클래스는 부모 클래스를 대체할 수 있어야 합니다. (상속된 자식 객체가 부모 타입으로 사용될 때 문제없이 동작해야 함)
- **위반의 문제점:** 부모 클래스(Rectangle)의 기능을 자식 클래스(Square)가 상속받았지만, 자식 클래스가 부모가 기대하는 동작 방식(예: 넓이와 높이를 독립적으로 변경)을 따르지 않을 때 발생합니다.
- **예시:** 정사각형(Square)은 넓이와 높이가 항상 같으므로, 단순히 너비만 변경한다고 해서 높이까지 자동으로 바뀌는 방식으로 동작할 수 있습니다. 이 경우, 부모가 기대하는 '넓이와 높이 분리 변경 가능'이라는 계약을 위반하게 됩니다.
- **핵심:** 상속 관계를 사용할 때, 자식 클래스가 부모 클래스의 계약을 깨지 않고 대체 가능한 형태로 기능해야 합니다.

## 4. 인터페이스 분리 원칙 (Interface Segregation Principle, ISP)
- **정의:** 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 합니다. (인터페이스를 작게 분리하여 불필요한 의존도를 줄이는 것이 목표)
- **문제점:** 하나의 인터페이스가 너무 많은 기능을 한데 묶어 놓으면 (God Interface), 그 인터페이스를 구현하는 클래스는 자신이 실제로 사용하지 않을 기능까지 모두 구현해야 하는 부담을 갖게 됩니다.
- **개선 방향:** 인터페이스를 관련 기능별로 작은 단위로 분리해야 합니다.
- **예시:** '주문 서비스'가 주문 기능과 취소 기능 등 여러 기능을 한 인터페이스로 묶고 있을 때, 단순히 주문만 하는 클라이언트가도 취소 기능을 구현해야 하는 비효율이 발생합니다. 따라서 `OrderService`와 `CancelService`와 같이 개별 인터페이스로 분리하는 것이 좋습니다.

## 5. SOLID 원칙 요약 및 설계 고려 사항
- **SOLID 원칙:** 객체 지향 설계의 품질을 높이기 위해 반드시 지켜야 하는 5가지 핵심 원칙들의 집합입니다.
    1. **S** - 단일 책임 원칙 (SRP): 클래스는 하나의 책임만 갖도록 분리
    2. **O** - 개방-폐쇄 원칙 (OCP): 확장에는 열려 있고, 수정에는 닫혀 있도록 설계
    3. **L** - 리스코프 치환 원칙 (LSP): 자식 클래스가 부모 클래스를 대체할 수 있도록 보장
    4. **I** - 인터페이스 분리 원칙 (ISP): 인터페이스를 작게 나누어 의존도를 최소화
    5. **D** - 의존성 역전 원칙 (DIP): (개념적으로) 상위 모듈이 하위 모듈에 직접 의존하는 대신, 추상화(인터페이스)에 의존하도록 설계하여 유연성을 높입니다.
- **최종 목표:**
    - **결합도(Coupling)를 낮추고, 응집도(Cohesion)를 높이는 것**이 소프트웨어 설계의 가장 핵심입니다.
    - 의존도가 낮은 객체들로 구성되어야, 새로운 기능이 추가되거나 변경될 때 시스템 전체에 미치는 영향(ripple effect)을 최소화할 수 있습니다.
- **결론:** 이 원칙들은 코드를 작성할 때 지키는 '가이드라인'으로 이해하는 것이 중요하며, 원칙에 맞추기 위해 불필요하게 코드를 복잡하게 만들 필요는 없습니다.
