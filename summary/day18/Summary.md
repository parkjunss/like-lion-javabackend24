## 1. 미니 프로젝트 진행 및 학습 태도 점검
*   **핵심 목표**: 프로젝트의 완성도보다도 **'실행'하는 과정** 자체가 중요하다.
*   **최소 기능 구현의 중요성**: 완벽하고 복잡한 구조를 만들기보다, 일단 **CRUD(생성-조회-수정-삭제)** 기능 등 최소한의 기능이라도 동작하는 것을 만드는 것이 가장 훌륭하다.
*   **학습 접근법**:
    *   **실행 기반 학습**: 이론만 공부하기보다, 직접 코드를 만드면서 경험하고 부딪히는 과정을 통해 지식이 습득되어야 한다.
    *   **반복과 확장**: 일단 가장 단순하게 만들어 본 후, 이 경험을 바탕으로 코드를 반복적으로 수정(Refactoring)하고 기능을 점진적으로 추가하며 발전시켜 나가는 것이 중요하다.
    *   **시간 관리**: 학습에 필요한 시간 할애를 우선순위에 두는 것이 가장 중요하며, 학습은 과정과 분리되어 지속적으로 진행되어야 한다.

## 2. 코드 구조 설계 원칙 (DAO와 서비스 계층)
*   **레이어드 아키텍처(Layered Architecture)**: 소프트웨어를 역할을 분리하여 설계하는 방식이다.
*   **객체의 책임 분리 (Separation of Concerns)**:
    *   하나의 객체에 너무 많은 기능을 몰아주는 것은 바람직하지 않다. 각 객체는 자신이 가장 잘하고, 맡은 업무만 수행하도록 역할을 분리해야 한다.
    *   **서비스 계층 (Service)**: 핵심 비즈니스 로직(Business Logic)을 담당하는 곳. 실제 '어떤 조건'으로 기능을 수행할지, '어떻게' 처리할지에 대한 규칙과 로직을 정의한다.
    *   **DAO (Data Access Object)**: 데이터베이스와의 통신만을 담당하는 계층. 단순하게 데이터를 저장하거나(Save), 읽거나(Retrieve), 수정하거나(Update), 삭제하는(Delete) 기능 등 **순수 데이터 조작** 역할에 국한되어야 한다.
*   **코드 설계 시 유의점**:
    *   **일단은 코딩에 집중**: 처음에 복잡한 구조나 완벽한 메서드 분리를 고민하기보다, 일단 동작하게 코드를 쭉 작성하는 데 집중한다.
    *   **개선은 나중에**: 코드가 길어지거나, 반복되는 로직이 눈에 띌 때, 그때 고민하여 기능을 별도의 메서드로 분리하거나 구조를 개선하는 것이 효율적이다.

## 3. Java 입출력 (I/O) 이해와 스트림 개념
*   **I/O의 정의**: 프로그램의 관점에서 데이터를 처리하는 과정을 말한다.
    *   **Input (입력)**: 외부에서 프로그램 안으로 흘러 들어오는 데이터 (예: 사용자 키보드 입력).
    *   **Output (출력)**: 프로그램 안에서 외부로 흘러 나가는 데이터 (예: 콘솔 출력).
*   **스트림(Stream) 개념**:
    *   스트림은 데이터의 형태(텍스트, 이미지, 동영상, 소리 등)와 무관하게, 데이터를 일정한 **흐름(Flow)**으로 취급하는 추상화된 개념이다.
    *   데이터는 한 방향으로 흘러가기 때문에, 스트림은 일반적으로 한 방향성(One-way)을 가진다.
    *   이러한 흐름을 다루기 위해 자바는 모든 종류의 데이터를 스트림 형태로 통일하여 처리할 수 있게 한다.
*   **I/O의 특징**:
    *   데이터가 입력되면(Input Stream) 처리되고, 반드시 출력을 위한 통로(Output Stream)가 필요하다.
    *   I/O API를 살펴볼 때, 클래스 이름의 접미사(`*In`, `*Out`)와 같은 규칙을 통해 해당 클래스가 어떤 기능을 담당하는지 직관적으로 파악하는 것이 좋다.
    *   `*InputStream` 계열은 입력 기능을, `*OutputStream` 계열은 출력 기능을 담당한다.

## 1. 디자인 패턴 및 자바 I/O의 배경
- **디자인 패턴의 목적:** 프로그램 개발 과정에서 발생하는 공통적인 문제점을 해결하는 구조적 '해결책(Solution)'을 제공합니다.
- **데코레이터 패턴 도입 이유:** 클래스 간의 결합도를 낮추고, 기존 클래스를 수정하지 않고 새로운 기능을 추가하거나 변형할 필요가 있을 때 사용합니다.
- **핵심 개념:** 설계된 객체를 "주인공 클래스"와 "장식 클래스"로 구분하여 생각할 수 있습니다.
    - **주인공 클래스 (Concrete):** 목적지나 근원지(Source/Destination)가 명확하게 정해져 있는 핵심 기능을 수행하는 클래스입니다. (예: 파일 입출력)
    - **장식 클래스 (Decorator):** 기능(Functionality)을 추가하거나 확장하는 역할만 합니다. 이들은 독립적으로 생성될 수 없으며, 반드시 다른 클래스(주인공 클래스)에 "끼워져서" 사용되어야 합니다.

## 2. 바이트 스트림과 문자 스트림의 개념
- **I/O의 기본 흐름:** 모든 프로그램의 데이터 입출력(I/O)은 프로그램을 기준으로 외부와의 상호작용을 합니다.
- **기본 추상화 (바이트 단위):**
    - **`InputStream`:** 외부로부터 데이터를 읽어오는 입력(Input)의 기본 메커니즘입니다.
    - **`OutputStream`:** 외부로 데이터를 쓰는 출력(Output)의 기본 메커니즘입니다.
    - **기본 단위:** 이들은 모든 데이터를 **바이트(Byte) 단위**로 처리하는 가장 기본적인 스트림입니다.
- **문자 단위 처리의 필요성 (Reader/Writer):**
    - 초기 자바는 영어권 환경을 기준으로 개발되어 바이트 단위 처리에 최적화되어 있었습니다.
    - 한국어와 같이 문자 하나를 여러 바이트(예: 2~3바이트)로 표현해야 하는 언어권에서는 바이트 단위로 데이터를 처리할 경우 데이터가 깨지거나 잘못 묶이는 문제가 발생합니다.
    - **해결책:** 이 문제를 해결하기 위해 **문자 단위**의 입출력을 전담하는 `Reader`와 `Writer`가 추가되었습니다.

## 3. I/O 아키텍처의 설계 철학: 스트림 및 디코레이터 패턴
- **스트림(Stream)의 정의:** 데이터가 한 방향으로 흐르는 '통로'를 추상화한 개념입니다. 자바 I/O는 이 스트림 형태로 모든 데이터(텍스트, 오디오, 비디오 등)를 다루는 통일된 방식을 채택했습니다.
- **I/O 시스템의 3가지 설계 원칙:** 자바 I/O가 구조적으로 견고하고 유연한 이유는 다음 세 가지 철학 덕분입니다.
    1. **유연성 (Flexibility):** 데이터 소스/목적지가 파일, 네트워크, 메모리 등 어디든 상관없이 동일한 인터페이스로 데이터를 처리할 수 있게 합니다.
    2. **일관성 (Consistency):** 데이터의 출처가 달라져도, 사용자가 데이터를 읽고 쓰는 방식(메서드 호출 방법)은 동일합니다.
    3. **확장성 (Extensibility):** 새로운 입출력 타입이나 기능을 손쉽게 추가할 수 있습니다.
- **디코레이터 패턴의 역할 (기능 확장):**
    - 아키텍처의 유연성을 극대화하기 위해, **기능(Function)**을 담당하는 클래스들을 중간에 끼워 넣는(Wrapping) 방식으로 설계되었습니다.
    - 예: 파일에서 읽는 기본 기능에 '버퍼링'이라는 추가 기능을 붙이고 싶을 때, `BufferedReader`와 같은 장식 클래스를 사용합니다.
    - **핵심:** 자바 I/O의 모든 입출력 기능은 **'핵심 로직(주인공)'**과 **'추가 기능(장식)'**을 조립하여 사용하는 구조로 이루어져 있습니다.

## 4. 핵심 I/O 클래스 구조 이해 (InputStream, Reader 등)
- **추상 클래스의 활용:** `InputStream`, `OutputStream`, `Reader`, `Writer`는 모두 추상 클래스(Abstract Class)로 정의되어 있습니다.
    - **이유:** 이들 클래스 자체가 특정 데이터 소스를 지정할 수 없기 때문입니다. (예: `InputStream`이 어디서 데이터를 읽을지는 정의할 수 없음)
    - **상속 구조:** 이 추상 클래스들을 상속받는 구체적인 클래스들이 실제 입출력을 담당하게 됩니다.
- **분류 및 역할 정리:**
    - **바이트 스트림 계열:** `InputStream`, `OutputStream` (바이트 단위 입출력).
    - **문자 스트림 계열:** `Reader`, `Writer` (문자 단위 입출력).
    - **특정 소스 지정:** `FileInputStream` (파일에서 바이트 단위 입력), `FileReader` (파일에서 문자 단위 입력) 등은 생성자에서 목적지나 근원지(Source/Destination)를 명시하는 '주인공'의 역할을 합니다.
    - **기능 추가:** `BufferedReader`는 버퍼링이라는 기능을 추가해주는 '장식'의 역할을 수행하며, 기반이 되는 `Reader`를 생성자에서 받아와 사용합니다.
- **결론:** 자바 I/O 시스템을 이해하는 핵심은, 단순히 클래스 이름을 외우는 것이 아니라, **이 클래스가 '어떤 데이터'를 '어디서', '어떻게(기능)'** 처리하는지에 초점을 맞추어 이해하는 것입니다.

## 1. Java I/O의 기본 이해 및 난이도 분석
- **전체적인 어려움:** Java I/O 코드가 처음 접할 때 복잡하게 느껴지는 주된 이유는 생성자(Constructor)와 예외 처리(Exception Handling) 등 복잡한 요소들이 한 번에 등장하기 때문입니다.
- **필수 전제:** I/O 코드를 작성하려면 ‘무엇을 하고 싶은지’(목표)가 먼저 명확해야 합니다. 예를 들어, 키보드로부터 한 줄씩 데이터를 입력받고 싶다고 정의할 수 있습니다.

## 2. Decorator 패턴을 활용한 데이터 원천(Source) 분리
- **핵심 원리:** I/O 작업의 핵심은 데이터를 *어디서 가져오는지(Source)*입니다. Java I/O의 구조는 이 데이터를 가져오는 원천(Source)과 데이터를 처리하는 방식(Reader/Writer)을 분리하여 설계되었습니다.
- **패턴의 역할 (Decorated Pattern):** `BufferedReader`, `FileReader` 등 상위 레벨 클래스는 그 아래에 있는 하위 스트림(예: `Reader`)을 감싸서(Wrapping) 기능을 추가합니다. 이를 **데코레이터 패턴**이라고 합니다.
- **연결의 중요성 (Interoperability):** 상위 스트림을 생성하려면 하위 스트림이 특정 인터페이스를 만족해야 합니다.
    *   예시: `BufferedReader`는 반드시 `Reader` 타입만 받을 수 있습니다. 따라서 키보드(`System.in`)와 `BufferedReader`를 바로 연결할 수 없으므로, **`InputStreamReader`**와 같은 "변환기(Connector)"가 필요합니다.
    *   이러한 변환기는 한 스트림 타입과 다른 스트림 타입을 연결해 주는 **브릿지(Bridge)** 역할을 수행합니다.

## 3. 표준 입력 (Keyboard) 처리 및 리소스 관리
- **키보드 입력 처리:**
    - 키보드로부터 한 줄씩 입력받을 때는 `BufferedReader` 클래스를 사용합니다.
    - `BufferedReader`를 사용하기 위해서는 `System.in` (키보드 입력)을 `InputStreamReader`로 변환하는 과정이 필수적입니다.
- **데이터 반복 처리:**
    - **단발성 입력:** `readLine()` 메서드를 한 번만 호출하여 값을 받고 프로그램을 종료할 수 있습니다.
    - **반복 입력:** 여러 번의 입력을 받고 싶을 경우, `while` 루프를 사용하여 특정 조건(예: 입력 값이 `null`이 아닐 때)이 유지되는 동안 반복적으로 데이터를 읽어와야 합니다.
- **리소스 관리 (Resource Management):**
    - I/O 작업에서 사용한 모든 스트림(Stream)이나 연결 통로는 반드시 사용이 끝난 후 **닫아주어야** 합니다.
    - `try-with-resources` 구문을 사용하면 `AutoCloseable` 인터페이스를 구현한 리소스들을 선언 블록에서 자동으로 처리하고 닫아주기 때문에(자동 클로저), 자원 관리가 간편해집니다.

## 4. 파일 및 URL 등 다양한 데이터 원천 처리
- **데이터 원천의 교체:** 가장 큰 장점은 데이터의 원천지(Source)만 바꾸면 코어 로직을 수정할 필요가 거의 없다는 점입니다.
    1. **키보드 $\rightarrow$ 파일:** 데이터 원천을 `System.in` 대신 `FileInputStream`과 `FileReader`를 통해 파일 경로를 지정해 주면, 코드 전체가 파일 내용을 읽어오는 방식으로 동작합니다.
    2. **파일 $\rightarrow$ URL:** 데이터 원천을 `System.in`이나 `FileInputStream` 대신 `java.net.URL` 객체를 사용하면, 네트워크를 통해 웹페이지의 내용을 가져와 처리할 수 있습니다.
- **핵심 구조:** 항상 "데이터를 어디서 가져올지 (Source)"만 변경해주고, "어떻게 읽고 쓸지 (Read/Write logic)"는 그대로 유지하는 것이 이 패턴의 핵심입니다.
- **파일 경로 주의사항:** 파일 경로를 지정할 때, 운영체제와 상관없이 슬래시(`/`)를 사용하는 것이 일반적이며, 역슬래시(`\`)가 Escape 문자로 인식되는 문제 때문에 경로 지정 시 주의가 필요합니다.

## 5. I/O 스트림의 작동 원리 및 결론
- **I/O의 근본:** 프로그램이 처리하는 모든 데이터는 본질적으로 "스트림(Stream)"의 형태로 흐릅니다.
- **바이트 단위 처리:** 낮은 레벨에서 데이터를 다룰 때는 `read()` 메서드를 통해 한 바이트(바이트 데이터)씩 데이터를 읽어오고, 파일 끝에 도달하면 `-1`을 반환한다는 개념을 이해해야 합니다.
- **한 줄 단위 처리:** 데이터를 읽을 때 문자로 간주하고 처리하려면, 스트림을 감싸서 `BufferedReader`를 이용해 한 줄(String) 단위로 데이터를 읽어오는 것이 일반적입니다.
- **요약:** Java I/O는 데이터를 가져오는 원천지(Source)와 데이터를 처리하는 방식(Logic)이 분리되어 있으며, 이 덕분에 어떤 종류의 데이터(키보드, 파일, 네트워크)라도 일관된 방식으로 처리할 수 있는 유연한 시스템입니다. 처음에는 어렵지만, 이 기본 개념(Source $\rightarrow$ Connector $\rightarrow$ Logic)만 파악하면 쉽게 접근할 수 있습니다.

## 1. I/O(입출력)의 기본 개념 및 스트림 추상화
- **핵심 개념 정의:**
    - I/O는 프로그램 실행 중 외부에서 데이터를 받아들이는 것(입력, Input)과 데이터를 외부로 내보내는 것(출력, Output)의 모든 과정을 의미합니다.
    - **스트림(Stream)의 역할:** 프로그램의 동작 과정에서 들어오고 나가는 모든 데이터(이미지, 텍스트, 영상 등 데이터 형식 무관)를 통일된 형태로 추상화하여 처리합니다.
- **주요 스트림 클래스:**
    - `InputStream` (입력 스트림): 데이터를 받는 통로를 담당합니다.
    - `OutputStream` (출력 스트림): 데이터를 내보내는 통로를 담당합니다.
    - `Reader` / `Writer`: 바이트(Byte) 단위의 입출력을 넘어, 사람이 인식하는 문자 단위(Character) 입출력을 위해 별도로 제공됩니다. (특히 한글과 같이 바이트가 여러 개 필요한 문자열 처리 시 필수적)
- **독립적인 통로 사용:** 입력 통로와 출력 통로는 서로 다르게 분리되어 사용해야 합니다.

## 2. I/O 클래스 구조와 데코레이터 패턴 이해
- **클래스 구조:** I/O 관련 클래스들은 크게 두 종류로 나뉩니다.
    - **주인공 클래스 (Concrete/Protagonist):** 목적지나 근원지에 직접적으로 도달할 수 있는 기본 기능을 수행하는 클래스 (예: 파일 자체, 키보드 입력).
    - **장식 클래스 (Decorator):** 주인공 클래스 위에 기능을 추가하거나 확장하는 역할을 하는 클래스.
- **데코레이터 패턴의 장점 (일관성):**
    - I/O 시스템은 데코레이터 패턴을 사용하여 객체가 어떤 근원지(키보드, 파일, URL, 네트워크 등)에서 데이터를 받든, 내부 코드를 수정할 필요 없이 항상 통일된 방식으로 데이터 처리가 가능합니다.
    - 개발자는 데이터를 다루는 *방법*에 집중할 수 있고, 데이터의 *출처*에 상관없이 코드를 재사용할 수 있습니다.

## 3. 데이터 읽기/쓰기 방법: 바이트 스트림과 버퍼링
- **데이터 단위의 선택:**
    - **바이트 단위 처리:** 바이너리 데이터(이미지, 파일 구조 등)를 다룰 때는 바이트 단위 처리를 합니다.
    - **문자 단위 처리:** 문자열을 다룰 때는 `Reader`/`Writer`를 사용합니다.
- **효율적인 데이터 전송 (버퍼링의 중요성):**
    - **문제점:** 바이트 하나하나를 읽거나 쓸 경우, 반복문(Loop)이 불필요하게 많은 횟수(예: 1,227번)를 돌게 되어 성능이 매우 비효율적입니다.
    - **해결책:** `read(byte[] b)`와 같이 **바이트 배열을 이용해 한 번에 대량의 데이터를 읽거나 쓰는 방식**을 사용해야 합니다.
- **데이터 쓰기 주의사항:**
    - 배열을 이용해 데이터를 읽어온 경우, 파일 끝 부분까지의 초기 데이터(쓰지 않은 나머지 값)가 남아있을 수 있으므로, 데이터를 쓸 때(Write) 반드시 **실제 읽어온 데이터의 개수(limit)까지만 지정하여 출력**해야 합니다.

## 4. 표준 입력/출력 및 파일 객체 활용
- **표준 시스템 스트림:**
    - `System.in`: 표준 입력 (키보드 등)
    - `System.out`: 표준 출력 (콘솔 화면)
    - `System.err`: 표준 오류 (에러 메시지)
- **표준 입력 처리 (BufferedReader):**
    - `BufferedReader`와 `readLine()` 메서드를 사용하면 키보드나 파일에서 한 줄(String) 단위로 데이터를 쉽게 읽을 수 있습니다.
    - 입력받은 문자열(String)을 숫자로 사용하려면 명시적으로 형 변환이 필요합니다.
- **Scanner 사용 시 주의사항 (Input Pitfall):**
    - `Scanner`의 `nextInt()`, `nextDouble()` 등 특정 타입 메서드는 숫자만 읽고 커서가 남기는 **줄 바꿈 문자(\n)를 비워두게 됩니다.**
    - 이후 `nextLine()`을 사용하려고 할 때, 이 남아있는 줄 바꿈 문자를 **즉시 한 번 소비해줘야(Consume) 오류 없이 정상적인 문자 입력을 받을 수 있습니다.**
- **`File` 객체의 추상화:**
    - `java.io.File` 클래스는 실제 파일이나 디렉토리 같은 운영체제의 개념을 자바 프로그램 내에서 다룰 수 있도록 추상화한 객체입니다.
    - 이 객체를 통해 파일의 존재 여부, 크기, 절대 경로, 부모 디렉토리, 읽기/쓰기 전용 여부 등 다양한 메타데이터를 조회하고, 삭제 등의 파일 시스템 작업을 수행할 수 있습니다.

## 5. 필수 주의사항: 자원 관리 및 예외 처리
- **try-with-resources 구문:**
    - I/O 자원(파일, 네트워크 연결, DB 연결 등)은 사용 후 반드시 닫아줘야 합니다.
    - 이 패턴(`try (ResourceDeclaration)`)을 사용하면 자원 선언 및 사용이 끝난 후 자동으로 자원(Resource)을 닫아주므로, 리소스 관리가 매우 용이하고 안전합니다.
- **예외 처리:**
    - I/O 관련 클래스의 생성자나 메서드는 대부분 예외(Exception)를 발생시킬 수 있습니다.
    - 특히 I/O 관련 예외는 **체크된 예외(Checked Exception)**에 속하므로, 코드를 컴파일할 때 반드시 예외 처리 블록(`try-catch`)을 사용해야 합니다.

## 1. 파일(File) 객체에 대한 이해 및 개념
- **추상화된 객체:** 자바에서 `File` 객체는 실제 물리적인 폴더를 생성하는 것이 아니라, 파일 시스템의 경로와 정보를 **추상화하여 사용하기 위한 객체**입니다.
- **파일 시스템 상호작용:** `File` 객체를 사용하면 파일의 존재 여부 확인, 디렉터리 조회(`list files`), 임시 파일 생성 및 삭제 등 파일 시스템의 다양한 기능을 코드로 구현하고 추상화하여 활용할 수 있습니다.
- **핵심 원리:** 파일 입출력의 모든 과정은 이 추상화된 `File` 객체를 중심으로 이루어집니다.

## 2. 파일 입출력(File I/O)의 핵심 스트림 개념 (Stream)
- **스트림의 역할:** 스트림(Stream)은 데이터를 읽거나 쓰는 통로 또는 흐름 그 자체를 의미합니다. 데이터를 처리하는 방식에 따라 여러 종류로 분류됩니다.
- **바이트 단위 스트림 (Binary Streams):**
    - **`BufferedInputStream`/`BufferedOutputStream`:** 가장 기본적인 단위인 바이트(byte) 단위로 데이터를 처리할 때 사용합니다. 데이터를 한 번에 묶어서(버퍼링) 처리하여 효율성을 높입니다.
- **문자 기반 스트림 (Character Streams):**
    - **`Reader`와 `Writer` 계열:** 문자 단위(Character)로 데이터를 처리할 때 사용합니다. 바이트 스트림에서 문자로 변환하는 과정의 복잡성을 추상화하여 개발자가 쉽게 사용할 수 있도록 합니다.
    - **`BufferedReader`:** 파일이나 스트림으로부터 **한 줄씩(Line by Line)** 문자열(String)로 데이터를 읽어올 때 가장 편리합니다.
    - **`PrintWriter`:** 문자를 파일이나 출력 스트림으로 쓸 때(출력할 때) 가장 편리한 클래스입니다.
- **데이터 타입 스트림 (Data Streams):**
    - **`DataOutputStream`/`DataInputStream`:** 문자열이나 일반적인 텍스트가 아니라, **원시 데이터 타입(Primitive Data Type)**(예: `double`, `int`, `boolean` 등)을 파일에 저장하거나 읽어올 때 사용합니다.
    - **특징:** 데이터를 쓸 때부터 시스템이 인식하는 데이터 타입 형태로 저장하므로, 읽어올 때 별도의 형변환(Casting) 과정 없이 해당 타입으로 바로 읽어낼 수 있습니다.
    - **주의사항:** 데이터를 기록할 때의 순서와 데이터를 읽어올 때의 순서가 반드시 일치해야 합니다.

## 3. 텍스트 기반 파일 처리 및 자원 관리
- **작동 원리:** 파일을 읽고 쓰는 과정은 데이터의 출처(InputStream)와 목적지(OutputStream)를 연결하는 과정입니다.
- **최적의 처리 흐름:** 파일에서 한 줄씩 읽어와서(예: `BufferedReader` 사용) 내용을 가공한 후, 원하는 포맷으로 파일에 쓰는 과정(예: `PrintWriter` 사용)이 일반적입니다.
- **자원 해제 (Resource Closing):**
    - **필수 절차:** 파일을 열어 사용하는 모든 스트림(Stream)은 사용을 마친 후 반드시 닫아주어야 합니다.
    - **이유:** 운영체제는 데이터를 즉시 쓰지 않고 내부 버퍼에 모아두었다가 한 번에 쓰려는 경향이 있습니다. 파일을 제대로 닫지 않으면(Close) 버퍼링된 내용이 외부로 쓰여지지 않아 데이터가 손실될 수 있습니다.
    - **권장 방법:** `try-with-resources` 구문(자원 블록)을 사용하여 스트림을 선언하면, 코드가 정상적으로 종료되든 예외가 발생하든 자동으로 스트림을 닫아줍니다.

## 4. 객체 직렬화(Object Serialization)
- **목적:** 파일 I/O의 일반적인 스트림들은 텍스트 형태(문자) 또는 원시 데이터 타입(Primitive)만 다룰 수 있습니다. 만약 **객체(Object) 자체**를 파일에 저장하고 싶을 때가 있습니다.
- **개념:** 객체 직렬화는 메모리상의 복잡한 객체 구조 자체를 파일에 저장할 수 있는 바이트 흐름(Stream)으로 변환하는 과정입니다.
- **활용 시점:** 데이터베이스(DB)를 사용할 수 없는 환경에서, 프로그램 내부의 복잡한 객체 상태(예: 친구 정보 목록)를 파일에 임시 저장해야 할 때 사용합니다.
- **흐름:**
    1. 객체를 저장할 스트림을 생성합니다.
    2. 해당 스트림을 통해 객체 자체를 씁니다 (`writeObject`).
    3. 나중에 읽을 때도 동일한 방식으로 객체 스트림을 사용해 객체 구조를 복원합니다.
- **결론:** 객체 직렬화는 시스템 메모리에 있는 복잡한 데이터를 그대로 보존하여 저장 및 읽어오는 고급 기술입니다.

## 1. 파일 입출력의 기본 이해 및 데이터 구조화
- **기본 파일 읽기:** `BufferedReader`를 사용하여 파일에서 한 줄씩 데이터를 읽어오는 방법을 학습합니다.
- **데이터 파싱 및 전처리:**
    - 파일에서 읽어온 한 줄의 문자열(String) 데이터를 필요한 정보(이름, 전화번호, 이메일) 단위로 분리해야 합니다.
    - 이때 `String` 클래스의 `split()` 메서드를 사용하여 공백(Space)을 기준으로 문자열을 배열(String Array)로 쪼개는 원리를 이해합니다.
- **데이터 구조화:** 파일에서 읽어온 분리된 값들을 단순 문자열이 아닌, `Friend`와 같은 객체(Object)의 속성으로 결합하여 관리하는 것이 효율적입니다.
- **데이터 저장:** 객체화된 친구들(Friend Object)을 `List`나 `Map`과 같은 자바의 자료 구조(Collection)에 담아 관리합니다.

## 2. 객체 직렬화(Serialization)의 필요성 및 원리
- **문제 제기 (상태 저장의 어려움):**
    - 일반적인 텍스트 파일에 데이터를 저장할 경우, 프로그램이 다시 시작될 때 저장된 텍스트를 읽어와서(예: `readLine()` -> `split()`) 모든 데이터를 다시 객체로 재구성하는 과정(파싱)을 거쳐야 합니다.
    - 이는 복잡하고 번거로운 과정입니다.
- **해결책 (객체 직렬화):**
    - 애플리케이션이 종료되었다가 다시 시작할 때, 메모리상의 객체(예: 친구 목록 `List<Friend>`) 자체를 파일에 그대로 저장하고, 재시작 시 객체 자체로 불러와 사용하는 방법이 훨씬 간단하고 효율적입니다.
- **핵심 메커니즘:**
    - **직렬화(Serialization):** 자바 객체의 상태(데이터)를 파일에 저장하기 위해 일련의 바이트(byte) 흐름(Stream) 형태로 변환하는 과정입니다.
    - **역직렬화(Deserialization):** 파일에 저장된 바이트 스트림을 읽어와 원래의 객체 타입으로 복원하는 과정입니다.
    - **사용 클래스:** 객체 스트림(`ObjectOutputStream`, `ObjectInputStream`)을 사용하며, 저장하려는 객체는 반드시 `java.io.Serializable` 인터페이스를 구현해야 합니다.
    - **결과:** 이를 통해 객체 자체가 파일에 저장되고 읽어와지므로, 객체 고유의 메서드나 속성을 그대로 사용할 수 있습니다.

## 3. 자바 I/O의 심화 개념 및 디자인 패턴 (Decorator)
- **I/O (Input/Output)의 정의:** I/O는 자바 프로그램 기준으로 외부에서 데이터가 들어오거나(Input) 프로그램상에서 나가는 것(Output)을 통칭합니다.
- **스트림(Stream) 개념:**
    - 데이터가 마치 강물처럼 연속적으로 흘러들어오거나 흘러나가는 형태로 추상화되어 표현됩니다.
    - 스트림의 특성상, 한 번 흘러간 데이터는 되돌아올 수 없습니다.
- **데코레이터 패턴(Decorator Pattern) 적용:**
    - 자바의 I/O 클래스들은 이 데코레이터 패턴을 활용합니다.
    - 이는 기본적인 I/O 기능을 제공하는 클래스(주인공 클래스)에 추가적인 기능(예: 데이터 형식 변환, 자원 관리, 특정 포맷팅)을 쉽게 '덧붙여(장식하여)' 사용할 수 있게 해줍니다.
    - **예시:** `PrintWriter`와 같이, 기본적인 파일 쓰기 기능을 수행하면서도 사람이 읽기 쉬운 형태로 출력할 수 있는 기능을 추가해준 클래스가 있습니다.
- **주요 I/O 클래스 특징:**
    - I/O 클래스들은 목적지(Destination)나 근원지(Source)를 유연하게 선택할 수 있도록 설계되어 있어, 파일뿐만 아니라 네트워크 등 다양한 환경에 적용 가능합니다.
    - `Scanner`, `PrintWriter`와 같은 유틸리티 클래스들은 개발자가 어려운 부분을 쉽게 사용할 수 있도록 기능이 확장되어 있습니다.

## 4. I/O 처리의 핵심 원칙 및 정리
- **자원 관리 원칙:**
    - I/O 작업을 수행할 때 사용한 자원(Resource)은 반드시 닫아주어야 합니다(Close).
    - 최신 자바에서는 `try-with-resources` 구문을 사용하여 자원 관리가 자동으로 이루어지게 하므로 이 방식을 습관화하는 것이 중요합니다.
- **학습 정리의 중요성:**
    - 자바의 I/O와 예외처리 코드는 여러 요소가 결합되어 있어 처음 접하면 복잡하게 느껴지는 것이 일반적입니다.
    - 중요한 것은 각 개념을 하나씩 이해하고, 전체적인 흐름(데이터가 들어오고 나가는 과정)을 파악하는 것입니다.
    - I/O에 대한 이해도가 높아지면 네트워크 프로그래밍과 같은 더 복잡한 분야 학습도 수월하게 진행할 수 있습니다.
