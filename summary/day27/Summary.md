## 1. 개발 환경 설정 및 터미널 명령어 숙달
*   **환경 설정:**
    *   개발 시에는 Git Bash와 같은 터미널 환경을 사용하는 것이 CMD(Command Prompt)보다 명령어나 운영체제 명령어(리눅스 명령어)를 익히는 데 훨씬 도움이 된다.
    *   터미널 환경을 통해 리눅스 기반의 다양한 명령어들을 사용하게 된다.
*   **기본 명령어 숙지:**
    *   **`pwd`**: 현재 사용자의 경로(위치)를 알려준다.
    *   **`cd`**: 다른 디렉토리로 이동할 때 사용한다.
    *   **`ls`**: 디렉토리 내부의 파일 목록을 보여준다.
        *   `ls -a`: 숨겨진 파일(`.`)까지 포함하여 목록을 보여준다.
        *   `ls -la`: 상세 정보와 숨겨진 파일을 모두 포함하여 보여준다.
    *   **`mkdir`**: 디렉토리(폴더)를 생성한다.
    *   **`cat`**: 파일의 내용을 화면으로 읽어와 보여준다.
    *   **`touch`**: 빈 파일을 생성한다.
    *   **디렉토리 삭제 시 주의:** 디렉토리를 지울 때는 `-r` 옵션(재귀적으로 지움)을 붙여야 내부까지 완전히 지울 수 있다.

## 2. 웹 애플리케이션과 서버 환경 이해
*   **웹 애플리케이션의 동작 원리:**
    *   웹 애플리케이션은 라이트 서버(웹 서버)가 존재하고, 클라이언트가 요청하면 서버가 파일(HTML 등)을 응답하는 방식으로 동작한다.
    *   개발 초기에는 로컬 컴퓨터(Local)에서 테스트하지만, 실제 서비스에서는 네트워크에 연결된 외부 서버에서 동작한다.
*   **네트워크 주소의 이해:**
    *   로컬 주소 대신 IP 주소를 사용하면 전 세계 누구에게나 페이지에 접근이 가능해진다.
    *   IP 주소를 외우기 힘들기 때문에, 도메인(예: fullstack.com)이라는 개념을 사용한다.
    *   도메인은 도메인 센터에서 관리하며, 사용자가 도메인을 요청하면 도메인 센터가 해당 서비스의 IP 주소를 알려주는 방식으로 동작한다.
*   **서버 운영체제(OS) 선택의 중요성:**
    *   **Windows 사용 지양:** Windows는 운영체제 자체가 복잡하여 서버 운영 중 예기치 않은 문제(멈춤 현상 등)를 일으킬 수 있다.
    *   **Linux 권장:** 서버 구동에는 리눅스(Linux)와 같은 운영체제가 가장 안정적이다.
    *   **비용 효율성:** 서비스 규모가 커지면 서버가 여러 대 필요할 수 있는데, Windows는 대당 사용료가 발생하지만, 리눅스는 비용 부담이 없다.
    *   **개발 환경 측면:** 맥(Mac)은 기본적으로 리눅스와 기반이 같기 때문에 프로그래머들이 사용하기에 유리한 측면이 있다.

## 3. 버전 관리 시스템(VCS)의 필요성과 Git의 역사
*   **버전 관리의 근본적인 문제:**
    *   문서를 작성하거나 코드를 개발하다 보면, 수정을 거듭하면서 '이전 버전'으로 되돌아가거나, '추가된 기능이 문제의 원인'인지 확인해야 하는 경우가 빈번하게 발생한다.
    *   이러한 상황을 관리하기 위해 파일을 단순 덮어쓰지 않고 여러 버전을 보존하는 것이 필수적이다.
*   **VCS의 역할:**
    *   버전 관리 시스템(VCS)은 코드나 데이터의 변경 이력을 체계적으로 관리하는 도구이다.
*   **Git의 배경:**
    *   Git은 리눅스를 개발한 리누스 토르발즈(Linus Torvalds)에 의해 개발되었다.
    *   과거에는 CVS나 Subversion 같은 형상 관리 시스템이 주로 사용되었으나, Git은 이러한 문제점을 해결하기 위해 개발된 혁신적인 도구이다.
    *   Git은 오픈 소스 철학에 따라 개발되었으며, 현존하는 형상 관리 시스템의 표준처럼 자리 잡았다.

## 4. Git의 로컬 사용 환경 구축 및 설정
*   **Git vs. GitHub (개념 구분):**
    *   **Git:** 파일을 추적하고 버전을 관리하는 '도구' 자체를 의미한다. (사용자 로컬 컴퓨터에서 동작)
    *   **GitHub:** Git으로 관리되는 프로젝트 파일들을 네트워크에 연결된 '저장소(Repository)'에 올려두는 웹 서비스 플랫폼을 의미한다.
    *   GitHub를 사용하면 네트워크를 통해 전 세계 어디서든 동일한 프로젝트를 공유하고 사용할 수 있다.
*   **Git 초기 설정 (`git config`):**
    *   Git을 사용하기 전에 전역(Global) 및 로컬(Local) 환경 설정을 반드시 해줘야 한다.
    *   **전역 설정 (`--global`):** 모든 프로젝트에서 사용할 사용자 이름(user.name)과 이메일(user.email) 등의 정보를 설정한다.
    *   **로컬 설정:** 특정 폴더 내에서만 적용할 설정을 할 수 있다.
*   **Git 저장소 초기화 (`git init`):**
    *   `git init` 명령을 실행하면, 현재 디렉토리가 버전 관리를 할 수 있는 Git 저장소로 초기화된다.
    *   이 과정에서 **`.git`**이라는 숨겨진 디렉터리가 생성되며, 이 안에 모든 버전 관리 메타데이터가 담긴 파일들(Objects, Hooks 등)이 존재한다.

## 5. 핵심 Git 워크플로우 및 관리 전략 (Add, Commit, Ignore)
*   **Git의 핵심 3단계 워크플로우:**
    1.  **파일 생성 및 수정:** 프로젝트 파일을 작성한다.
    2.  **스테이징 (Staging Area - `git add`):** 변경된 파일 중 '이번 커밋에 포함할 파일'을 선택하여 Git에게 알려주는 단계이다.
    3.  **커밋 (Commit - `git commit`):** 스테이징된 파일들을 특정 시점의 '버전'으로 영구 기록하는 행위이다. 커밋할 때는 반드시 **커밋 메시지(Commit Message)**를 작성해야 기록을 추적하기 용이하다.
*   **GitIgnore 파일 사용:**
    *   모든 파일을 Git이 관리할 필요는 없다. 프로젝트를 진행하면서 생성되는 로그 파일, IDE의 빌드 폴더(`.idea`), 환경 설정 파일 등은 버전 관리에서 제외해야 한다.
    *   이때 제외할 파일 목록을 담는 파일이 **`.gitignore`**이며, 이 파일을 Git에 등록하여 관리 대상에서 제외하도록 지시한다.

---
***[요약의 핵심]***
Git은 단지 파일 백업 도구가 아니라, "언제, 누가, 무엇을, 왜 변경했는지"에 대한 이력을 관리하는 강력한 버전 관리 시스템이다. 이 과정은 `Init` (저장소 생성) $\rightarrow$ `Add` (추적 파일 선택) $\rightarrow$ `Commit` (버전 기록)의 순서로 이루어진다.

## 1. 원격 저장소(Remote Repository)의 이해
- **개념:** 로컬 컴퓨터에만 저장된 소스 코드를 인터넷상의 서버(원격지)에 업로드하여 다른 사람과 공유하거나 백업할 수 있게 해주는 공간입니다.
- **주요 도구:** 가장 많이 사용되는 도구는 GitHub입니다.
- **사용 목적:** 공유, 협업, 백업 등 여러 목적으로 사용되며, 기업이나 프로젝트 성격에 따라 공개(Public) 또는 비공개(Private) 저장소를 선택하여 사용할 수 있습니다.

## 2. GitHub 저장소 생성 및 설정
- **접속 및 생성:** GitHub 계정으로 로그인한 후, 원하는 저장소(Repository) 이름과 설명(Description)을 입력하여 새로 만듭니다.
- **비용 및 접근성:** 과거에는 비용 지불이 필요했던 프라이빗(Private) 저장소도 현재는 무료로 제공됩니다.
- **가시성 선택:**
    *   **Public (공개):** 다른 모든 사람이 접근하고 사용할 수 있습니다.
    *   **Private (비공개):** 초대한 사람만 접근할 수 있도록 제한할 수 있습니다.

## 3. 로컬 저장소와 원격 저장소 연결 (Push 워크플로우)
- **초기화 (새 저장소의 경우):** 처음 시작할 때는 저장소를 초기화하고, 필요하다면 README 파일 등을 만들고 커밋을 진행합니다.
- **연결 (기존 저장소의 경우):** 이미 존재하는 로컬 저장소를 원격 저장소에 연결할 때는 `git remote` 명령어를 사용합니다.
    - **`git remote add origin [URL]`:** 원격 저장소의 주소(URL)를 `origin`이라는 별칭(Nickname)으로 설정합니다. (별칭은 주로 `origin`을 사용하며, 이 명령을 통해 로컬이 원격지를 인지하게 합니다.)
- **푸시 (Push):** 로컬에 커밋된 내용을 원격 저장소로 올려주는(업로드) 작업입니다.
    - **기본 명령어:** `git push`를 사용합니다.
    - **첫 푸시 시:** 원격 브랜치와 로컬 브랜치를 연결해 줄 때 `git push -u origin main`과 같이 옵션을 사용하여 **"앞으로는 항상 `origin`의 `main` 브랜치로 푸시해 줘"**라는 약속을 설정해 주는 것이 효율적입니다.

## 4. Git 핵심 명령어와 작업 흐름 (Add, Commit, Push)
*Git의 작업 흐름은 '로컬 저장소에 기록(Commit) -> 원격 저장소에 배포(Push)' 순서로 진행됩니다.*

- **`git add` (Stage):** 작업 디렉토리에서 변경되거나 추가된 파일을 **스테이징(Staging Area)** 영역에 추가합니다.
    - `git add .` : 현재 디렉토리의 모든 변경 사항을 스테이징합니다.
- **`git commit` (Local Save):** 스테이징 영역에 있는 파일들을 묶어 로컬 저장소에 기록합니다. (버전 기록 생성)
- **Commit과 Push의 차이:**
    - **Commit:** 내 로컬 컴퓨터에만 저장된 버전 기록입니다.
    - **Push:** 내 로컬의 커밋 기록을 원격 서버로 실제로 전송(업로드)하여 다른 팀원과 공유합니다.
- **작업 순서:** 로컬에 파일을 추가/수정 $\rightarrow$ `git add` $\rightarrow$ `git commit` $\rightarrow$ `git push` (원격 업데이트)

## 5. 협업 고급 기능: Clone, Fork, Conflict Resolution
- **`git clone` (복제):** 원격지에 존재하는 전체 저장소의 사본을 내 컴퓨터의 새로운 위치(디렉토리)에 가져오는 명령어입니다. (가장 일반적인 환경 구축 방법)
- **`git pull` (가져오기):** 원격 저장소에 올라온 최신 변경 내용을 내 로컬 저장소로 내려받아 통합(Merge)하는 명령어입니다. (협업 시 가장 중요하게 사용)
- **`git fork` (분기 및 사본 만들기):**
    - **`clone`과의 차이:** `clone`은 *저장소 자체*를 복사하는 것이고, `fork`는 *원격지 서버 상*에서 원본 프로젝트를 내 계정의 저장소로 복사하는 것입니다.
    - **용도:** 다른 사람의 프로젝트를 기반으로 삼아 자신만의 독립적인 사본을 만들 때 사용합니다.
- **Conflict Resolution (충돌 해결):**
    - **발생 원인:** 같은 파일의 같은 부분을 여러 사람이 동시에 수정하여 병합(Merge)할 때, 어떤 내용을 최종적으로 남길지 Git이 판단할 수 없을 때 발생합니다.
    - **해결 방법:** 사용자가 직접 충돌이 발생한 파일을 열어, 어느 쪽의 수정 내용이 맞는지 판단하고, 필요한 내용만 남기고 나머지 내용을 수동으로 정리(삭제/통합)해야 합니다.

## 6. 원본 저장소에 기여하기: Pull Request
- **Pull Request (PR):** 내가 (Fork 받은 사본 저장소에서) 수정한 내용이 원본 프로젝트에 반영되기를 '요청'하는 행위입니다.
- **흐름:** 다른 사람의 저장소를 Fork $\rightarrow$ 사본 저장소에서 수정 및 커밋 $\rightarrow$ Pull Request 생성 $\rightarrow$ 원본 저장소의 관리자가 코드 검토 후 병합(Merge)을 진행합니다.

## 1. 버전 관리 시스템 (Version Control)의 이해
- **필요성:** 프로젝트 진행 중 파일의 변경 이력을 체계적으로 관리하고, 이전 버전의 내용을 필요할 때 불러와 사용할 수 있게 합니다.
- **장점:** 단순 복사 및 덮어쓰기 방식으로 관리할 때 발생하는 저장 공간의 낭비를 막고, 실제 변경된 부분(차이점)만을 효율적으로 기록하고 관리합니다.
- **활용 범위:** 전문 개발 영역뿐만 아니라 보고서(레포트) 작성 등 문서 작업에서도 버전 관리 개념을 적용할 수 있습니다.

## 2. Git과 GitHub의 역할 구분
- **Git (도구):** 실제 형상 관리(버전 관리)를 수행하는 로컬 도구 자체입니다. 사용자의 컴퓨터 내에서 버전 기록을 관리합니다.
- **GitHub (저장소):** Git으로 관리된 코드를 네트워크상(클라우드 환경)에 저장하고 공유할 수 있는 원격 저장소(Repository)를 제공하는 서비스입니다.
- **기본 관계:** Git은 로컬에서 동작하는 기능이며, GitHub는 이 로컬 저장소를 외부에 보관하고 여러 사용자와 공유하는 역할을 합니다.

## 3. 로컬 Git 저장소 관리 방법
- **저장소 초기화:**
    - 특정 디렉토리를 버전 관리 대상(저장소)으로 설정해야 합니다.
    - 명령어: `git init` (사용 시 해당 폴더 내에 `.git`이라는 숨겨진 디렉토리가 생성됨).
- **파일 추적 및 기록 과정 (순서):**
    1. **추가 (Staging):** Git이 추적할 파일들을 지정합니다. 모든 파일이 자동으로 추적되는 것이 아니므로, `git add` 명령을 이용해 특정 파일을 스테이징 영역에 올려줘야 합니다.
    2. **커밋 (Commit):** 스테이징 영역에 있는 파일들을 기반으로 변경 사항을 저장소에 반영하고 기록합니다. (버전 기록 생성).
- **설정 관리:**
    - **글로벌 설정:** 컴퓨터의 모든 저장소에 공통적으로 적용되는 설정을 할 때 사용합니다.
    - **로컬 설정:** 특정 저장소 내에서만 다른 설정을 적용할 때 사용합니다.

## 4. 원격 저장소와 협업 (Push & Clone)
- **원격 저장소의 필요성:** 로컬 컴퓨터에만 존재하는 버전 기록을 다른 컴퓨터나 팀원들과 공유하고 동기화할 때 필요합니다.
- **원격 연결:** GitHub 등의 원격 저장소에 로컬 저장소를 연결해야 합니다. (`git remote` 명령어 사용).
- **변경 사항 전송 (Push):** 로컬에서 커밋된 버전 기록을 원격 저장소로 업로드하여 공유합니다. (커밋만으로는 원격에 올라가지 않습니다.)
- **저장소 복제 (Clone):** 원격 저장소에 있는 전체 버전 기록을 내 로컬 컴퓨터로 통째로 가져와서 작업할 수 있게 합니다. (PC 위치와 관계없이 동일한 환경으로 작업할 때 사용).

## 5. 포크(Fork)와 풀 리퀘스트(Pull Request)의 이해
- **포크 (Fork):** 원본 저장소(Repository)를 기반으로 **나만의 사본**을 나의 계정 공간 안에 복제하는 행위입니다. (원본과는 독립적인 공간에서 개발을 시작할 때 유용합니다.)
- **풀 리퀘스트 (Pull Request):** 포크본이나 특정 브랜치에서 작업한 내용을 원본 저장소의 메인 브랜치에 **'이 부분을 반영해 주면 안 될까요?'** 라고 요청하는 기능입니다.
- **용도:** 협업 과정에서 다른 사람의 코드에 변화를 제안하고, 해당 변경 사항에 대해 리뷰를 거쳐 반영될 수 있도록 하는 과정입니다.

## 6. 브랜치(Branch)를 이용한 독립적 작업 흐름
- **개념:** 브랜치는 메인 코드(Main Branch)에서 분기되어 나가는 독립된 가지라고 생각할 수 있습니다.
- **목적:** 메인 브랜치가 안정적인 운영 코드인 경우, 새로운 기능을 개발하거나 오류를 수정할 때 메인 코드를 건드리지 않고 독립된 브랜치에서 작업을 진행할 수 있습니다.
- **작업 흐름:**
    1. 메인 브랜치에서 **브랜치 생성** (예: 기능 개발 브랜치 생성).
    2. 새로운 브랜치로 **이동**하여 개발을 진행합니다. (브랜치에만 반영된 파일은 다른 브랜치에서는 보이지 않음).
    3. 작업 완료 후, **병합(Merge)** 명령을 이용해 변경된 내용을 메인 브랜치에 합칩니다.
- **브랜치 관리 명령:**
    - **생성:** `git branch [브랜치 이름]`
    - **이동:** `git switch [브랜치 이름]` (최신 명령어이며, 과거에는 `git checkout`을 사용했음)
    - **삭제:** `git branch -d [브랜치 이름]` (변경 사항이 병합되지 않은 브랜치는 대문자 `-D` 옵션을 사용해 강제 삭제해야 함)
- **실제 활용:** 대형 프로젝트에서는 개발 기능별, 팀원별 등으로 브랜치를 체계적으로 관리하는 규칙을 정하여 사용합니다.

---
**✅ 핵심 요약 정리:**
*   **버전 관리의 목표:** 코드를 안전하고 체계적으로 관리하여, 오류나 변경 사항을 추적하고 이전 상태로 되돌릴 수 있게 하는 것입니다.
*   **작업 흐름 순서:** `git init` (설정) $\rightarrow$ `git add` (추적 지정) $\rightarrow$ `git commit` (로컬 저장) $\rightarrow$ `git branch` (작업 공간 분리) $\rightarrow$ `git push` (원격 공유) $\rightarrow$ `git merge` (최종 반영)

- **학습 목표:** 복잡한 기능을 한 번에 알기보다는, **저장소 생성(init) → 파일 추가(add) → 커밋 → 원격 저장소 연결 → 클론** 등 기본적인 핵심 흐름을 반복적으로 연습하는 데 집중해야 합니다.
- **학습 난이도:** 처음 Git을 접하는 것은 어려울 수 있지만, 반복적인 연습을 통해 익숙해지는 것이 중요합니다.
- **최신 환경 변화:** AI 도구(CLI)가 발전하면서 과거처럼 모든 오류를 수동으로 해결할 필요가 줄어드는 추세이나, 기본 동작 원리를 이해하는 것이 여전히 중요합니다.

## 2. Git 브랜치 이름 변경 및 기본 설정 이해
- **기본 브랜치 이름의 변화:** Git의 초기 기본 브랜치 이름은 'master'였으나, 이 용어의 연관성 문제로 인해 현재는 'main'으로 변경되었습니다.
- **설정 방법:** 로컬 Git 설정(`git config --global init.defaultBranch`)을 통해 기본 브랜치를 'main'으로 지정할 수 있습니다.
- **개념 이해:** Git의 기본 동작 원리(어떤 구조를 가지고 동작하는지)를 파악하고 넘어가는 것이 중요합니다.

## 3. Git 저장소의 핵심 구조 분석 (.git 디렉터리)
- **`.git` 디렉터리의 중요성:** 이 디렉터리가 Git 저장소의 모든 정보(히스토리, 설정, 객체 데이터 등)를 관리하는 핵심 공간입니다.
- **주요 서브 디렉터리별 역할:**
    - **`objects`:** 모든 파일의 데이터가 객체 형태로 저장되는 핵심 데이터베이스입니다.
    - **`refs`:** 브랜치나 태그 등 포인터 정보를 담고 있습니다. (예: `refs/heads/main`)
    - **`HEAD`:** 현재 작업 중인 브랜치(포인터)가 어디를 가리키고 있는지 보여주는 가장 중요한 파일입니다.
    - **`hooks`:** 특정 이벤트(커밋 전, 푸시 전 등)가 발생했을 때 자동으로 실행되는 스크립트 파일들입니다. (고급 사용 영역)
    - **`info`:** 로컬 저장소에만 특정한 설정을 저장하는 공간입니다.
- **원리적 이해:** Git이 모든 정보를 이 내부 파일 시스템을 통해 관리한다는 구조를 파악하는 것이 가장 중요합니다.

## 4. 실습: Git 기본 명령어 활용 및 상태 확인
- **워크플로우 실습:** 새로운 디렉터리를 만들고, `git init`을 통해 Git 저장소로 설정한 후, 파일 생성 및 수정 과정을 거칩니다.
- **상태 확인 (`git status`):** 현재 작업 디렉토리의 파일들이 어떤 상태인지(추적 대기 중, 수정됨, 추적되지 않음 등) 시각적으로 보여줍니다.
- **핵심 명령어 활용:**
    - **`git add`**: 변경된 파일을 스테이징 영역(Staging Area)에 올리는 과정입니다. (`git add .`는 현재 디렉토리 전체를 추가)
    - **`git rm`**: 스테이징 영역에서 파일을 제거하거나, 작업 공간에서 파일을 삭제하는 등의 작업을 할 수 있습니다.
- **시각적 도구의 역할:** VS Code나 SourceTree 같은 GUI 도구는 내부적으로 Git의 이 복잡한 과정을 눈으로 보기 쉽게 보여주는 보조 수단입니다.

## 5. `.gitignore`를 활용한 관리할 파일 제외 설정
- **목적:** Git이 형상 관리해서는 안 되는 파일이나 폴더(예: 임시 로그 파일, IDE 설정 파일, 빌드 결과물 등)를 명시적으로 제외하기 위해 사용합니다.
- **필수성:** 이러한 파일들을 무시하지 않으면, 불필요한 정보가 커밋 이력에 쌓여 저장소의 용량을 늘리고 관리 효율을 떨어뜨립니다.
- **사용 방법:**
    - **규칙 정의:** `.gitignore` 파일에 제외할 파일명이나 경로를 작성합니다.
    - **와일드카드 사용:** `*.log` 와 같이 패턴 매칭을 사용하여 특정 확장자를 가진 모든 파일을 무시할 수 있습니다.
    - **예외 처리:** 만약 전체 로그 파일을 무시하되, 특정 로그 파일은 추적하고 싶다면 느낌표(`!`)를 사용하여 예외 규칙을 지정할 수 있습니다.

## 1. Git 작업 흐름 이해: 스테이징 영역과 커밋
- **스테이징 영역(Staging Area)의 역할:** 커밋 직전에 파일의 변경 사항을 임시로 모아놓는 공간입니다.
- **선택적 변경 관리:** 작업 디렉토리에서 발생한 모든 변경 사항을 한 번에 커밋할 필요 없이, 필요한 변경 사항만 세밀하게 선택하여 커밋할 수 있도록 돕습니다.
- **작업 흐름 순서:** 변경된 파일은 스테이징 영역 $\rightarrow$ 커밋(Commit) $\rightarrow$ 레포지토리(Repository) 순서로 관리됩니다.
- **효율적인 커밋 단위 설정:**
    - 커밋은 너무 작거나(한 줄 수정 시마다) 너무 큰 단위로 묶기보다는, **논리적으로 의미 있는 기능 단위**로 묶는 것이 중요합니다.
    - 커밋 단위가 명확할수록 히스토리를 추적하거나 특정 시점으로 되돌릴 때 관리하기 용이합니다.

## 2. Git 내부 동작 원리: 객체 저장소(Objects) 이해
- **내용 기반 주소 지정(Content-Addressable):** Git은 파일의 이름이나 위치가 아닌, **파일 내용 자체**를 기반으로 고유한 해시 코드(Hash Code)를 생성하여 저장합니다.
- **Objects 디렉터리:** Git 저장소 내의 `objects` 디렉터리는 모든 커밋의 데이터베이스 역할을 합니다. 이 안에는 모든 파일 내용이 해시 값으로 저장됩니다.
- **핵심 요소 및 데이터 구조:**
    - **Blobs (Binary Large Objects):** 실제 파일의 내용물 자체를 저장하는 객체입니다.
    - **Trees:** 디렉터리 구조를 나타냅니다. 특정 디렉터리 안에 어떤 파일(블롭)들이 어떤 이름으로 존재하는지 구조를 지정합니다.
    - **Commits:** 커밋 단위의 스냅샷을 저장하며, **해당 시점의 전체 트리(Tree)의 해시값**과 **부모 커밋(Parent Commit)의 해시값**을 포함하여 연결 고리를 만듭니다.
- **불변성 및 효율성:** 파일이 변경되지 않았다면, Git은 새로운 파일을 만들지 않고 기존의 해시 값을 재사용하여 저장 공간을 효율적으로 관리합니다.

## 3. 커밋 히스토리 추적 및 버전 관리
- **히드(HEAD) 포인터:** 현재 작업 중인 가장 최신 커밋을 가리키는 포인터입니다.
- **버전 관리의 핵심:** Git의 가장 큰 장점은 '시간' 단위로 작업 내용을 관리할 수 있다는 것입니다.
    - 커밋을 진행할 때마다 새로운 커밋 해시가 생성되고, 이전 커밋의 해시를 부모로 연결하여 **선형적인 이력(Linear History)**을 만듭니다.
- **과거 시점 복구:**
    - **`git log`:** 프로젝트의 커밋 기록(해시 값, 커밋 메시지, 시간 등)을 보여줍니다.
    - **`git show` / `git cat-file`:** 특정 커밋 해시를 이용하면, 그 커밋이 생성되었을 때의 **파일 상태 전체**를 찾아내어 볼 수 있습니다.
    - **HEAD 이동:** 원한다면 HEAD 포인터만 이전 버전의 커밋 해시로 되돌릴 수 있어, 물리적인 파일 시스템을 건드리지 않고도 과거 상태를 복원할 수 있습니다.
- **전체 구조 파악:** Git은 모든 커밋, 트리를 연결하여 하나의 거대한 네트워크 구조(DAG, Directed Acyclic Graph)를 형성하며, 이 해시 값들을 추적함으로써 프로젝트의 전 과정이 기록됩니다.

## 4. 효율적인 Git 사용을 위한 권장 사항
- **커밋 메시지 작성의 중요성:**
    - 커밋 메시지는 프로젝트의 기록을 남기고, 팀원 간의 효율적인 커뮤니케이션을 가능하게 하는 핵심 요소입니다.
    - 단순히 "수정함"이 아니라, **"무엇을(What)" 수정했고, "왜(Why)" 수정했는지**를 명확하게 기술해야 합니다.
- **커밋 메시지 규칙 통일:**
    - 팀 전체가 사용하는 명확한 커밋 메시지 규칙(예: 스코프, 타입 등)을 정하고 이를 지키는 것이 중요합니다.
- **명령어 수준 이해:**
    - `git status`와 같은 **고수준 명령(High-level)**을 숙지하는 것이 일상적인 작업에 필수적입니다.
    - 반면, `git cat-file`, `git hash`와 같은 **저수준 명령(Low-level)**들은 Git이 내부적으로 데이터를 어떻게 관리하고 있는지 원리를 깊이 이해하고자 할 때 활용할 수 있습니다. (실제 사용 빈도는 낮으나 원리 이해에 도움).


1. Git 데이터 생성 시점

git add - 스테이징:
파일 내용을 읽어 고유한 해시값을 만들고, Blob 객체로 Git 저장소에 저장하는 단계이다.

git commit - 기록:
스테이징된 Blob들을 바탕으로 파일/폴더 구조를 나타내는 Tree 객체를 만들고,
작성자, 시간, 메시지, 부모 커밋 정보 등을 담은 Commit 객체를 만드는 단계이다.


2. 파일 내용이 같으면 해시값 동일, 저장은 하나, 보이는 건 여러 개

Git은 파일 이름이 아니라 파일 내용으로 Blob을 구분한다.
따라서 파일 내용이 같으면 같은 해시값을 가진다.

같은 내용의 파일이 여러 위치에 있어도 실제 Blob 데이터는 하나만 저장될 수 있고,
각 파일의 이름과 경로는 Tree 객체에서 따로 관리된다.

Git은 변경된 코드 한 줄만 저장하는 방식이 아니라,
각 커밋마다 프로젝트 전체 구조를 스냅샷처럼 기록한다.
다만 같은 객체는 재사용하므로 저장 공간을 효율적으로 사용한다.


3. Branch

새 브랜치 생성 후 이동:

git switch -c newBranch

원격 저장소에 새 브랜치 올리기:

git push -u origin newBranch

현재 브랜치 이름과 같은 원격 브랜치로 push하도록 기본값 설정:

git config --global push.default current

다른 브랜치 가져오기:

git fetch
git branch -r
git switch otherBranch

or

git switch -c otherBranch origin/otherBranch


4. Pull Request

Pull Request는 내가 작업한 브랜치의 변경 내용을 main, develop 같은 대상 브랜치에 병합해 달라고 요청하는 것이다.
코드 리뷰, 테스트 확인, 변경 내용 토론 후 병합할 수 있다.
