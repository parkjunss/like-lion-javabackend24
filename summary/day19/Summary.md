## 목차
1. I/O의 기본 개념 및 데이터 흐름
2. 디자인 패턴으로서의 I/O: 데코레이터 패턴
3. 객체 직렬화 (Serialization)와 입출력 처리
4. I/O 스트림과 자바 디자인 원칙 복습

---

## 1. I/O의 기본 개념 및 데이터 흐름
- **I/O의 정의:** 프로그램(프로그래밍)을 기준으로 외부에서 데이터가 들어오는 것(Input)과 프로그램에서 외부로 데이터가 나가는 것(Output)을 통칭합니다.
- **데이터의 추상화 (Stream):** I/O 과정에서 모든 데이터(문자, 이미지, 소리, 영상 등)는 본질적으로 '스트림(Stream)'이라는 형태로 추상화됩니다.
    - 스트림은 물이 흘러가는 통로처럼 데이터를 순차적으로 전달하며, 한번 흘러간 데이터는 되돌릴 수 없습니다.
- **스트림의 두 가지 특성:**
    1. **단방향성 및 순차성:** 데이터는 한 방향으로만 흘러가기 때문에, 데이터를 읽거나 쓸 때는 반드시 스트림을 닫고 다시 열어야 합니다.
    2. **분리성:** 입력을 위한 스트림(Input Stream)과 출력을 위한 스트림(Output Stream)은 반드시 별개로 존재해야 하며, 하나의 통로로 입력과 출력이 동시에 일어날 수는 없습니다.
- **개념적 접근:** 어려운 I/O 객체들은 일상생활의 복잡한 기계나 카페의 전문 장비에 비유할 수 있습니다. 처음에는 어렵지만 원리와 구조를 이해하고 반복적으로 사용하다 보면 익숙해질 수 있습니다.

## 2. 디자인 패턴으로서의 I/O: 데코레이터 패턴
- **패턴의 목적:** 자바 I/O 시스템은 데코레이터(Decorator) 패턴을 활용하여 설계되어 있어, 사용자가 어떤 종류의 데이터 원천(키보드, 파일, 네트워크 등)에서 데이터를 가져오는지에 관계없이 일관된 방식으로 입출력을 처리할 수 있게 합니다.
- **데코레이터 패턴의 역할:**
    - **주인공 클래스 (Component):** 실제 데이터의 근원지나 목적지 역할을 수행합니다. (예: 실제 파일, 실제 콘솔 출력)
    - **장식 클래스 (Decorator):** 기능만을 추가하는 층(Layer) 역할을 하며, 주인공 클래스를 감싸서(Wrap) 사용 방법의 통일성을 제공합니다.
- **구현 효과:** 개발자가 데이터를 읽거나 쓸 때, 근원지/목적지(예: 파일에서 읽기 → 네트워크에서 읽기)만 교체해 주면 코드의 사용 방식(API 호출)은 완전히 동일하게 유지될 수 있습니다.

## 3. 객체 직렬화 (Serialization)와 입출력 처리
- **직렬화의 필요성:** 객체(Object)는 메모리에 존재하는 복잡한 데이터 구조(필드, 메소드 등)입니다. 이를 파일이나 네트워크 등 프로그램 외부에 저장하거나 다른 시스템으로 전달하려면, 객체 자체로 전달할 수 없고 단순한 **바이트(Byte)의 흐름**으로 변환되어야 합니다.
- **개념 설명:**
    - **직렬화 (Serialization):** 메모리상의 객체를 바이트 단위의 스트림 형태로 변환하는 과정입니다. (알약 속 알갱이들을 꺼내 한 줄로 세우는 과정과 유사)
    - **역직렬화 (Deserialization):** 전달받은 바이트 스트림 데이터를 다시 원래의 객체 형태로 복원하는 과정입니다.
- **구현 요구사항:**
    - 직렬화가 가능한 객체는 반드시 `java.io.Serializable` 인터페이스를 구현(Implement)해야 하며, 이는 자바 가상 머신(JVM)에게 "나를 바이트 스트림으로 변환해도 안전하다"고 승인하는 표식(Marker Interface)의 역할을 합니다.

## 4. I/O 스트림과 자바 디자인 원칙 복습
- **추상 클래스와 인터페이스의 이해:**
    - `InputStream`과 `OutputStream` 같은 I/O 기본 클래스들은 실제 기능이 정해져 있지 않고 공통적인 추상적 개념을 정의합니다. 이는 다른 곳에서 구현하는 구체적인 개체들이 따라 할 수 있도록 일반화한 것입니다.
    - **추상 클래스:** 공통적으로 필요한 기능(추상 메서드)은 정의하지만, 세부 구현은 하위 클래스에 맡기는 형태입니다.
    - **Decorator의 활용:** I/O 과정에서 문자 단위의 처리가 필요할 때 (바이트 단위의 한계 극복), `Reader`나 `Writer` 같은 장식 클래스들이 `InputStream`을 받아들여 사용성을 높이는 '커넥터(Connector)' 역할을 수행합니다.
- **주요 클래스 사용법:**
    - **문자 처리 시:** `BufferedReader` (읽기)와 `PrintWriter` (쓰기)를 주로 사용합니다. (문자 단위로 한 줄씩 읽고 쓰는 데 가장 효과적입니다.)
    - **바이트 단위 전송 시:** `FileInputStream`, `FileOutputStream` 등 기본적인 스트림 클래스를 바로 연결하여 사용합니다.

## 1. 멀티스레드가 필요한 이유 (단일 스레드 환경의 한계)
- **기본 동작 방식 (단일 스레드):** 프로그램은 `main` 함수에서 시작하여 하나의 흐름(Single Thread)을 가지고 순차적으로 진행됩니다.
- **I/O 블로킹 문제:** 파일 입출력(I/O)이나 사용자 입력 (`BufferedReader.readLine()`)과 같은 작업을 할 때, 해당 작업이 완료될 때까지 프로그램의 **전체 수행 흐름이 멈추게(Blocking)** 됩니다.
- **예시 (채팅 프로그램):** 채팅 프로그램의 경우, 사용자가 메시지를 입력하는 동안 프로그램의 모든 기능(다른 사람의 메시지 출력, 알림 등)이 멈춘다면 사용성이 극도로 떨어집니다.
- **필요성:** 복잡하거나 사용성이 높은 응용 프로그램(예: 채팅, UI 애플리케이션)을 구현하기 위해서는 하나의 흐름이 아닌, 여러 흐름이 동시에 동작해야 할 필요가 생깁니다.

## 2. 병렬 처리의 이해 (프로세스 vs. 스레드)
- **멀티태스킹의 원리:**
    - 실제로 우리가 동시에 여러 일을 하는 것처럼 느껴지는 것은, 운영체제가 **시간을 매우 짧은 단위로 쪼개서(Time-slicing)** 여러 작업을 번갈아 가며 실행하기 때문입니다. (Context Switching)
    - 이처럼 여러 일을 동시에 처리하는 개념이 바로 '멀티태스킹'이며, 이는 멀티스레딩의 기본 원리가 됩니다.
- **프로세스 (Process):**
    - 실행 중인 프로그램의 인스턴스입니다. 운영체제는 여러 개의 독립적인 프로세스들을 관리합니다.
    - **특징:** 프로세스들은 메모리 공간이 완전히 격리(독립적)되어 있어 안정성이 높고, 코드나 데이터를 공유하기 어렵습니다.
    - **비용:** 새로운 프로세스를 생성하고 관리하는 비용(Overhead)이 높습니다.
- **스레드 (Thread):**
    - **정의:** 프로세스 내에서 실제 작업을 수행하는 가장 최소 단위의 흐름입니다.
    - **특징:** 하나의 프로세스 내에서 여러 개의 스레드가 존재하며, **힙(Heap)과 같은 메모리 영역을 공유**할 수 있습니다.
    - **장점:** 프로세스에 비해 리소스 사용량이 적고, 통신(데이터 공유)이 비교적 간단합니다.
    - **단점:** 여러 스레드가 공유된 데이터에 동시에 접근하여 문제를 일으킬 수 있습니다 (동시성 문제, Race Condition).

## 3. 자바에서 스레드 구현 방법 및 생명 주기
- **스레드 생성의 두 가지 방법:**
    1. **상속(Inheritance):** `Thread` 클래스를 상속받아 사용합니다.
    2. **구현(Implementation):** `Runnable` 인터페이스를 구현하여 사용합니다.
    - *(참고: 자바는 다중 상속을 지원하지 않기 때문에, `Runnable`을 이용하는 방법이 더 권장되기도 합니다.)*
- **핵심 메서드:**
    - **`run()`:** 스레드가 실제로 수행하고자 하는 작업(코드)을 구현하는 메서드입니다.
    - **`start()`:** 스레드를 시작하기 위해 필요한 준비 작업(자원을 할당하고 흐름을 준비하는 과정)을 수행하는 메서드입니다. **(주의: `start()`를 호출해야 스레드가 생성되고 `run()`이 호출됩니다. 단순히 `run()`을 직접 호출해서는 병렬성이 확보되지 않습니다.)**
- **스레드의 생명 주기 (Lifecycle):** 스레드는 상태 변화를 거치며 동작합니다.
    1. **생성 (New):** `new` 키워드를 사용해 객체를 생성합니다.
    2. **준비 (Runnable):** `start()` 메서드를 호출하면 준비 상태가 되며, 실행 기회를 노립니다.
    3. **실행 (Running):** 운영체제에 의해 실행 권한이 주어지면 코드가 실행됩니다.
    4. **블록 (Blocked):** I/O 작업이나 자원 할당 대기 등으로 인해 일시적으로 멈추는 상태입니다.
    5. **종료 (Terminated):** 모든 작업을 완료하고 흐름을 마칩니다.
- **결론:** 스레드는 메인 스레드와 별개로 존재하며, 각 스레드는 독립적으로 작업을 수행함에도 불구하고, 운영체제에 의해 매우 짧은 시간 단위로 작업을 나누어 번갈아 실행되어 마치 동시에 일하는 것처럼 동작합니다.

2. 쓰레드 구현 방식: 상속 vs. Runnable 인터페이스
3. 간편한 쓰레드 작성: 람다 표기법 (Lambda Expression)
4. 쓰레드 자원 관리: 동기화(Synchronization)와 문제 해결

---

## 1. 멀티 스레드와 동시 동작의 이해
*   **쓰레드(Thread)의 역할:** 쓰레드는 여러 작업을 동시에(Concurrency) 수행할 수 있는 실행 단위입니다.
*   **동시성 구현:** 여러 개의 독립적인 쓰레드를 생성하고 실행할 수 있으며, 각 쓰레드는 서로 영향을 받지 않고 독자적으로 작업을 수행합니다.
*   **운영 원칙:** 복잡한 쓰레드 구현 과정에서도 개발자는 두 가지 약속만 지키면 됩니다.
    *   수행하고자 하는 핵심 로직(Task)을 쓰레드 내부에 구현한다.
    *   쓰레드를 실행할 때는 반드시 `start()` 메서드를 사용한다.

## 2. 쓰레드 구현 방식: 상속 vs. Runnable 인터페이스
쓰레드를 만드는 방법은 크게 두 가지가 있으며, 각각의 개념과 사용법을 이해하는 것이 중요합니다.

### 🟢 방법 1: `Thread` 클래스 상속
*   **원리:** 직접 `Thread` 클래스를 상속받아 커스텀 쓰레드를 만듭니다.
*   **장점:** 자신이 쓰레드 자체가 되므로 별다른 설정 없이 `start()` 메서드를 실행하여 즉시 동작시킬 수 있습니다.
*   **단점:** 자바는 **다중 상속**이 불가능하기 때문에, 만약 쓰레드 외에 다른 기능(예: 특정 인터페이스 구현)을 추가해야 한다면 이 방식은 사용이 제한적입니다.

### 🟢 방법 2: `Runnable` 인터페이스 구현 (권장 방식)
*   **원리:** 쓰레드 자체가 아닌, **수행할 로직(작업)**만을 담는 `Runnable` 인터페이스를 구현합니다.
*   **실행 방식:**
    1.  `Runnable` 구현체를 생성합니다. (실제 동작 로직)
    2.  이 `Runnable` 객체를 `Thread` 클래스의 생성자에 전달합니다. (실행 주체)
    3.  생성된 `Thread` 객체를 `start()` 메서드로 실행시킵니다.
*   **장점:** 쓰레드 자체를 상속받을 필요가 없기 때문에 다중 상속의 제약에서 자유로우며, 객체 지향 설계상 재사용성이 높아 가장 권장되는 방식입니다.
*   **주의사항:**
    *   `Runnable` 구현체만 만든다고 해서 쓰레드가 되는 것이 아닙니다. (쓰레드가 아님)
    *   반드시 `Thread` 객체를 생성하고 `start()` 메서드를 호출해야 비로소 동시 수행 흐름이 시작됩니다.
    *   **절대** `run()` 메서드를 직접 호출해서는 안 됩니다. `run()`을 직접 호출하면 쓰레드가 생성되지 않고 일반 메서드 호출로 종료됩니다.

## 3. 간편한 쓰레드 작성: 람다 표기법 (Lambda Expression)
*   **배경:** `Runnable`과 같은 인터페이스는 본질적으로 기능을 하나만 가진 **펑셔널 인터페이스(Functional Interface)**입니다.
*   **문제점:** 펑셔널 인터페이스를 구현할 때마다 클래스를 정의하거나 익명 내부 클래스를 작성하는 과정이 번거롭고 코드가 길어집니다.
*   **해결책:** **람다 표기법**을 사용하면 이러한 간접적인 클래스 정의 과정을 생략하고 코드를 훨씬 간결하게 만들 수 있습니다.
*   **원리:** 람다 표기법은 특정 기능을 수행하는 코드 블록(람다식)을 마치 *'런어블(Runnable)' 구현체*인 것처럼 자동으로 인식하여 `Thread`의 생성자에 넣어주는 역할을 합니다.
*   **결론:** 람다 표기법은 사용자의 편의성을 극대화하기 위해 자바에서 추가된 기능입니다.

## 4. 쓰레드 자원 관리: 동기화(Synchronization)와 문제 해결
*   **동기화가 필요한 이유:** 여러 쓰레드가 동시에 하나의 공유 자원(Shared Object, 예: 카운터)에 접근하여 값을 수정할 때(예: `count++`), 처리 순서가 꼬이면서 예상치 못한 오류가 발생할 수 있습니다. (이를 **데이터 경쟁 문제**라 합니다.)
*   **동기화 개념:** 쓰레드 여러 개가 동시에 공유 자원에 접근할 때, 특정 기능(메서드)이 완전히 종료될 때까지 다른 모든 쓰레드는 대기하게 만드는 메커니즘입니다.
*   **구현 방법:** 해당 공유 자원 객체의 메서드 앞에 `synchronized` 키워드를 붙여줍니다.
*   **작동 방식:**
    1.  `synchronized`가 붙은 메서드가 호출되면, 해당 객체에 대한 접근 권한을 최초로 얻은 쓰레드가 **독점적으로** 자원을 사용합니다.
    2.  다른 쓰레드가 같은 자원에 접근하려고 시도하면, 먼저 사용 중인 쓰레드가 작업을 끝낼 때까지 기다리게 됩니다.
    3.  이를 통해 공유 자원(예: 카운트 값)이 예상대로 정확하게 증가(1, 2, 3...)하도록 보장할 수 있습니다.
*   **실습 예시 (카운터):** `synchronized` 키워드를 사용하여 카운터 객체의 `increment()` 메서드를 보호함으로써, 여러 스레드가 동시에 접근하더라도 정확하게 10,000이라는 최종 카운트 값을 얻을 수 있게 됩니다.

## 1. 멀티스레딩의 개념 및 필요성
- **멀티스레딩의 필요성:** 프로그램이 단일한 수행 흐름(메인 스레드)만으로 동작할 경우, 특정 작업(예: I/O 입력)에서 멈추는(Blocking) 문제가 발생합니다.
- **문제점:** 채팅 프로그램처럼 상대방의 입력과 사용자의 입력이 동시에 이루어져야 하는 서비스에서는 단일 흐름으로는 처리할 수 없습니다.
- **해결책 (멀티스레딩):** 하나의 수행 흐름이 아니라 여러 개의 수행 흐름을 동시에 처리해야 합니다.
- **기술 원리:** Java는 `Thread`라는 객체를 제공하며, 실제 동작 시에는 **시간 분할(Time Sharing)** 기법을 통해 짧은 시간 단위로 자원을 나누어 공유함으로써 여러 흐름을 처리할 수 있게 합니다.

## 2. 스레드 구현 방법 및 문법 간소화
- **구현 방법 (전통적):**
    1.  **`Thread` 상속:** 클래스가 `Thread`를 상속받고, 실제로 수행할 로직을 `run()` 메서드에 오버라이드(재정의)합니다.
    2.  **`Runnable` 인터페이스 구현:** `Runnable`을 구현한 객체를 생성하여 `Thread` 생성자에 전달하고, 이 `Thread` 객체를 통해 시작합니다. (객체의 중요한 기능을 상속받는 경우 유리합니다.)
- **실행 원칙:** 새로운 수행 흐름을 만들려면 `Thread` 객체를 생성하고 `start()` 메서드를 호출해야 합니다. (직접 `run()`을 호출해서는 안 됩니다.)
- **람다식(Lambda Expression) 활용:**
    *   람다식은 코드를 간편하게 작성할 수 있도록 자바에 추가된 문법입니다.
    *   기존에 클래스를 정의하여 구현하던 과정을 최소화하여, 특히 단일 추상 메서드를 가진 **펑셔널 인터페이스(Functional Interface)**를 구현할 때 강력하게 사용됩니다.
    *   람다식을 사용하면 코드가 매우 간결해지지만, 내부적으로는 기존의 객체 정의 방식과 동일하게 동작합니다.

## 3. 동기화 문제와 해결책 (Synchronization & Join)
- **공유 자원 문제:** 여러 스레드가 동시에 공유되는 자원(변수, 객체 등)에 접근하여 값을 수정할 경우, 데이터 불일치(Race Condition) 문제가 발생하여 값이 예상과 다르게 저장될 수 있습니다.
- **`synchronized` 키워드 (상호 배제):**
    *   공유 자원에 접근하는 메서드나 블록에 `synchronized` 키워드를 사용하면, 해당 구역에 **오직 하나의 스레드만 접근**할 수 있도록 강제합니다. (락킹 메커니즘)
    *   **메서드 레벨 동기화:** 메서드 전체를 동기화할 때 사용합니다.
    *   **블록 레벨 동기화:** 메서드 내 특정 코드 블록만 동기화하여 성능 최적화를 할 때 유용합니다.
- **`join()` 메서드 (대기):**
    *   스레드는 기본적으로 독립적으로 동작하므로, 메인 스레드가 다른 스레드가 끝날 때까지 기다려야 할 때 문제가 발생합니다.
    *   `thread.join()` 메서드를 호출하면, 현재 스레드(메인 스레드)가 해당 스레드가 작업을 완료할 때까지 **강제로 대기**하게 됩니다. 이는 스레드 간의 실행 순서를 제어하는 중요한 방법입니다.

## 4. 고급 동기화 및 락(Lock) 메커니즘
- **동기화의 목적:** 여러 스레드가 공동으로 사용하는 데이터의 무결성(Integrity)을 보장하는 것입니다.
- **`synchronized` 블록을 사용할 때 락 객체:**
    *   `synchronized` 블록을 사용할 때 락(Lock)을 거는 객체를 지정해야 합니다. 이 객체를 **락 객체**라고 합니다.
    *   이 락 객체는 일종의 '비밀 열쇠' 역할을 하며, 이 열쇠를 가진 스레드만이 동기화된 영역에 접근할 수 있습니다.
    *   **권장 사항:** 가급적 `this`를 사용하기보다는, 동기화에 사용되는 별도의 `private` **락 객체**를 생성하여 사용하는 것이 더 안전하고 분리된 제어를 할 수 있어 바람직합니다.

## 1. 쓰레드(Thread)의 비순차적 동작 이해
- **쓰레드의 실행 순서 비보장:** 여러 쓰레드를 동시에 사용할 경우, 개발자가 원하는 특정 순서(예: 스레드 1 → 스레드 2 → 스레드 3)대로 동작한다는 보장이 없습니다.
- **비결정적 순서 (Non-deterministic):** 쓰레드는 시스템의 상황에 따라 알아서(임의로) 동작하며, 어떤 쓰레드가 먼저 실행될지 예측하거나 보장할 수 없습니다.
- **Join의 역할:** `join()` 메서드는 특정 쓰레드가 완료될 때까지 **메인 쓰레드가 기다리게** 하는 기능만 수행할 뿐, 실행 순서를 제어하는 기능은 아닙니다.

## 2. 생산자-소비자 패턴과 동기화 (Wait / Notify)
- **목적:** 생산자와 소비자와 같이, 자원(데이터)을 주고받는 작업이 특정 순서와 조건에 따라 동작해야 할 때, 쓰레드 간의 동기화(Synchronization)가 필수적입니다.
- **문제점:** 일반적인 쓰레드 환경에서는 순서 제어가 어렵고, 소비자가 데이터가 없을 때 작업을 멈추거나, 생산자가 자원이 가득 찼을 때 추가 생산을 할 수 없습니다.
- **해결책: Wait와 Notify 사용:**
    *   **`wait()`:** 자원이 부족하거나 작업 수행을 할 조건이 안 될 때, 현재 쓰레드를 강제로 **대기 상태(잠자기)**에 빠지게 합니다. (다른 쓰레드가 깨워주기 전까지 아무것도 못 함)
    *   **`notify()`:** 대기 상태에 있는 특정 쓰레드를 깨워 작업을 재개하게 합니다.
    *   **`notifyAll()`:** 대기 상태에 있는 **모든 쓰레드**를 깨워 재시도하도록 합니다.
- **핵심 로직:**
    *   **생산자 (Producer):**
        *   버퍼가 가득 찼는지 확인합니다. 만약 가득 찼다면 `wait()`를 호출하여 대기합니다.
        *   데이터를 추가하고, 소비자가 대기 중일 수 있으므로 `notify()` 또는 `notifyAll()`을 호출하여 소비자를 깨웁니다.
    *   **소비자 (Consumer):**
        *   버퍼에 데이터가 남아있는지 확인합니다. 만약 비어 있다면 `wait()`를 호출하여 대기합니다.
        *   데이터를 소비하고, 데이터가 남아있는지 지속적으로 확인합니다.

## 3. 쓰레드 생명주기 관리 및 제어 메서드
- **쓰레드 상태:** 쓰레드는 생성(New) → 준비(Runnable) → 실행(Running) → 대기/블로킹(Waiting/Blocked) → 종료(Terminated)의 생명주기를 가집니다.
- **주요 제어 메서드:**
    *   **`sleep(시간)`:** 지정된 시간 동안 쓰레드 실행을 멈춥니다. (자원 할당을 포기하고 일정 시간 쉬는 것)
    *   **`wait()`:** 쓰레드를 특정 조건(조건 충족)이 될 때까지 대기 상태로 만듭니다. (Wake-up은 반드시 `notify()`를 통해서만 가능)
    *   **`yield()`:** 쓰레드가 자발적으로 CPU 제어권을 다른 준비 상태의 쓰레드에게 양보합니다.
- **쓰레드 종료 처리:**
    *   **`join()`:** 특정 쓰레드가 작업을 완료할 때까지 호출한 쓰레드가 기다리게 합니다.
    *   **`interrupt()`:** 강제로 쓰레드를 중단시키는 방법입니다. (사용 권장)
    *   **(주의)** 과거의 `stop()` 메서드는 안전 문제로 사용이 권장되지 않습니다.

## 4. 쓰레드 우선순위 설정 및 주의사항
- **우선순위 (Priority):** 쓰레드에게 최소(MIN)부터 최대(MAX)까지의 우선순위를 부여할 수 있습니다.
- **우선순위의 효과:** 우선순위가 높을수록 CPU 자원을 얻을 확률이 높아지고, 같은 조건에서 더 먼저 접근할 수 있습니다.
- **최대 주의사항:** 우선순위는 **'권장 사항'**일 뿐, 어떤 쓰레드가 무조건 먼저 실행될지 **절대적으로 보장하지 않습니다.**
- **순서 보장을 위한 방법:** 만약 반드시 특정 순서로 작업을 진행해야 한다면, `wait()`와 `notify()` 같은 동기화 메커니즘을 사용하여 제어해야 합니다.

## 1. 쓰레드(Thread)의 기본 개념과 역할
*   **쓰레드의 정의와 중요성:** 쓰레드는 프로그램이 동시에 여러 작업을 처리하게 만드는 핵심 요소입니다. 처음에는 복잡하게 느껴질 수 있지만, 웹 애플리케이션의 동작 원리를 이해하는 데 가장 중요합니다.
*   **웹 서버의 동작 원리:** 사용자가 웹 페이지를 요청할 때, 웹 애플리케이션 서버는 요청이 들어올 때마다 새로운 객체를 생성하는 것이 아니라, 핵심 객체는 하나만 유지합니다.
*   **쓰레드의 역할:** 대신, 요청이 들어올 때마다 **쓰레드를 생성하여 각각의 요청 처리를 담당**하게 합니다. 이를 통해 서버가 효율적으로 대규모 트래픽을 처리하고 메모리 과부하를 방지할 수 있습니다.
*   **실제 사용 시점:** 스프링 프레임워크나 웹 애플리케이션을 개발할 때, 개발자가 쓰레드를 직접 생성하거나 관리할 일은 많지 않습니다. 이미 프레임워크들이 내부적으로 쓰레드 개념을 이용해 동작하고 있으므로, 개념을 이해하는 것이 중요합니다.

## 2. 동시성 제어와 자원 관리
*   **동기화 필요성:** 웹 애플리케이션이 여러 쓰레드에 의해 동시에 접근될 때, 데이터의 일관성을 유지하고 상태를 관리하는 것이 중요합니다.
*   **쓰레드 로컬(ThreadLocal)의 개념:** 각 쓰레드가 사용할 독립적인 공간(변수)을 제공하는 기능입니다. 여러 쓰레드가 공유하는 일반 메모리 공간을 사용하는 대신, 자신만의 격리된 저장 공간을 가질 수 있게 하여 데이터 충돌을 막습니다.
    *   *활용 예:* 웹 환경에서 사용자의 인증 정보(로그인 정보 등)를 현재 쓰레드에만 안전하게 저장하고 관리할 때 유용하게 사용됩니다.

## 3. 교착 상태(Deadlock) 이해 및 해결 방법
*   **교착 상태란?** 두 개 이상의 쓰레드가 각자 필요한 자원(리소스)을 획득한 후, 서로가 가진 자원을 기다리며 **무한정 멈추는 상태**를 의미합니다. (자원 A를 쥔 쓰레드 1이 B를 기다리고, 자원 B를 쥔 쓰레드 2가 A를 기다리는 경우)
*   **발생 원리:** 여러 쓰레드가 여러 자원을 순차적으로 사용하며 락(Lock)을 걸 때, 서로의 락을 기다리다가 아무도 진전하지 못하는 순환 구조가 형성될 때 발생합니다.
*   **해결 방법 1: 락 획득 순서 통일 (가장 중요)**
    *   모든 쓰레드가 자원을 획득할 때 항상 **동일하고 일정한 순서**로 락을 걸도록 코드를 설계해야 합니다. (예: 항상 A 자원 락 → B 자원 락 순으로 접근)
*   **해결 방법 2: 타임아웃(Timeout) 사용**
    *   락을 획득할 때 무한정 기다리지 않고, 일정 시간(타임아웃)이 지나면 실패하고 다른 처리를 하도록 설정할 수 있습니다.

## 4. 쓰레드 풀(Thread Pool) 및 쓰레드 로컬(ThreadLocal)
*   **쓰레드 풀의 필요성:** 쓰레드를 새로 생성하고 종료하는 과정은 시간적, 자원적 비용(오버헤드)이 매우 큽니다. 따라서 웹 서버 등에서 요청이 들어올 때마다 쓰레드를 처음부터 만드는 대신, 미리 일정 수량의 쓰레드를 만들어 저장해 두고 재사용하는 풀(Pool) 시스템을 사용합니다.
*   **작동 원리:** 커넥션 풀(Connection Pool)이 데이터베이스 연결을 미리 만들어 재사용하듯이, 쓰레드 풀은 미리 생성된 쓰레드들을 요청에 따라 가져와 사용합니다.
*   **장점:** 쓰레드 생성에 드는 시간을 절약하고 시스템의 전반적인 성능을 크게 향상시킵니다.
*   **활용:** 실제 웹 애플리케이션 서버는 내부적으로 이 쓰레드 풀 메커니즘을 이용합니다.
*   **주의할 점:** 공유되는 자원(Resource)을 사용할 때는 각 쓰레드가 독립적으로 사용하며 데이터가 유실되거나 예측치 못한 상태가 되지 않도록 주의해야 합니다.

## 5. 고급 쓰레드 활용 및 요약 정리
*   **비동기 처리 및 결과 반환:** 쓰레드는 단순히 작업을 수행하는 것을 넘어, 작업 결과를 나중에(비동기적으로) 꺼내어 사용할 수 있도록 관리하는 객체들(Future, Coroutine 등)을 제공합니다.
*   **주요 패턴:**
    *   **생산자-소비자 패턴:** 쓰레드가 자원을 생성(생산)하고, 다른 쓰레드가 해당 자원을 소비하는 방식입니다. (시행어, 웨이트, 노티파이 등의 개념 사용)
    *   **ExecutorService:** 쓰레드 풀을 사용하여 쓰레드를 효율적으로 관리하고 작업을 수행할 수 있게 해주는 객체입니다.
*   **성능 최적화 고려 사항:**
    *   쓰레드 풀의 적정 크기(Size)는 서버의 CPU 성능과 예상되는 트래픽 부하 등을 종합적으로 고려하여 설정해야 합니다.
    *   하나의 락이 여러 작업을 독점하는 것은 성능에 매우 좋지 않으므로, 작업을 세분화하여 관리하는 것이 권장됩니다.
