## 1. 강의 시작 및 생활 습관 점검 (Intro & Wellness Check)
*   **학습 동기 부여 및 공감대 형성:** 학생들이 매일의 일상 속 루틴(목요일의 장점 등)과 컨디션 관리에 대해 이야기하며 심리적 연결고리를 형성함.
*   **생활 패턴의 중요성:** 습관은 21일 이상 꾸준히 유지해야 고정되며, 특히 수면 패턴은 매우 중요합니다.
    *   *Tip:* 수면 시간은 개인에게 최적화된 시간이 존재하므로, 본인에게 가장 기분 좋고 최적인 수면 패턴을 찾아보는 것이 중요합니다.
*   **능동적 참여의 중요성:** 과정에 대한 만족도와 학습 효과를 높이기 위해서는 학생들이 적극적으로 움직이고 주도적으로 참여하려는 노력이 필요함을 강조.

## 2. 예외 처리의 이해와 구현 (Exception Handling)
### 📌 예외(Exception)란 무엇인가?
*   **정의:** 프로그램 실행 도중 개발자가 예상치 못한 문제(예: 객체가 NULL인 상태에서 메서드 호출, 존재하지 않는 인덱스로 배열 접근 등)가 발생했을 때 이를 **예외**라고 정의합니다.
*   **에러(Error) vs. 예외(Exception):**
    *   **에러(Error):** 시스템적으로 처리할 수 없는 심각한 오류입니다. (예: 메모리 부족 등)
    *   **예외(Exception):** 프로그래머가 사전에 예측하고 적절하게 처리할 수 있는 상황입니다.
*   **예외 발생 시 문제:** 예외가 발생하면 기본적으로 프로그램이 강제로 종료(Crash)됩니다. 따라서 안정적인 프로그램을 만들기 위해 예외 처리 기법이 필수적입니다.
### 📌 예외 처리의 방법 및 종류
*   **예외 처리 방법 (2가지):**
    1.  **직접 처리:** `try-catch` 구문을 사용하여 예외를 잡아 처리합니다.
    2.  **예외 전가 (Throwing):** 예외를 발생시키는 쪽(호출자)에게 책임을 떠넘기거나(Throws), 특정 로직 내에서 강제로 발생시킵니다(Throw).
*   **예외의 두 가지 유형:**
    1.  **체크드 익셉션 (Checked Exception):** 컴파일 단계에서부터 반드시 개발자가 예외 처리를 강제해야 하는 예외입니다. (예: 파일 입출력, 네트워크 연결 등 외부 자원과 관련된 예외).
    2.  **언체크드 익셉션 (Unchecked Exception):** 컴파일 시 오류는 발생하지 않지만, 런타임 시점에 발생합니다. 개발자의 실수나 논리 오류(예: NullPointerException)로 인해 발생하는 경우가 많습니다.
*   **사용자 정의 예외 (Custom Exception):**
    *   프로젝트의 특성에 맞게 특정 오류 상황을 정의할 수 있습니다.
    *   **구현 방법:** 반드시 `Exception`을 상속받기만 하면 사용자 정의 예외 객체가 됩니다.
    *   **권장사항:** 예외 객체 설계 시 적절한 이름 부여와 함께, 부모 클래스(Parent Class)의 생성자에 오류 메시지(`String message`)를 전달하여 사용하는 것이 좋습니다.

## 3. 자원 관리의 필수 요소: `finally`와 `try-with-resources`
*   **`finally` 블록의 역할:**
    *   예외 발생 여부나 처리 여부와 관계없이 **무조건적으로 반드시 실행되어야 하는** 코드를 작성하는 곳입니다.
    *   **사용 예:** 데이터베이스 연결 해제(Close), 파일 닫기 등 자원 해제가 필수적일 때 사용됩니다.
*   **`finally` 사용 시 문제점:**
    *   여러 개의 자원(예: DB 연결 3개)을 관리할 때, `if` 문과 `try-catch` 구문을 반복적으로 사용해야 하므로 코드가 매우 길어지고 복잡해집니다.
*   **`try-with-resources`의 등장 (해결책):**
    *   자바가 제공하는 고급 문법으로, **리소스(Resource)**를 선언할 때 자동으로 자원 해제(Close)를 수행해줍니다.
    *   **원리:** `try (...)` 괄호 안에 자원을 정의하면, 블록이 종료될 때 자원 해제(`close()`)를 자동으로 처리해줍니다.
    *   **필수 조건:** 자동 클로징이 가능하려면, 해당 객체는 `AutoCloseable` 인터페이스를 구현하고 있어야 합니다.

## 4. 객체 지향 설계의 핵심: 인터페이스와 추상 클래스
*   **인터페이스 (Interface)의 중요성:**
    *   인터페이스는 특정 기능(메서드)의 **규약(Contract)**을 정의하는 역할을 합니다.
    *   서로 다른 개체들이 이 인터페이스를 구현하면, 비록 내부 구현이나 이름은 달라도 공통적으로 약속된 기능을 사용할 수 있게 되어 코드의 **통일성과 확장성**이 높아집니다 (다형성 구현).
*   **설계적 사고의 중요성:**
    *   단순히 문법만 익히기보다는, **"왜 이 코드가 이렇게 설계되었을까?"**라는 근본적인 질문을 던지며 코드를 바라보는 시야를 확장해야 합니다.
    *   객체를 볼 때 상속(Inheritance) 구조와 인터페이스(Interface)가 어떻게 사용되고 있는지 끊임없이 생각하는 것이 깊은 이해로 이어집니다.
*   **학습 태도:**
    *   현재 배우는 개념에 국한하지 않고, 모든 객체와 코드를 볼 때마다 **'이 부분에 상속이 쓰였나?', '이 부분에 인터페이스가 쓰였나?'**를 질문하며 개념들을 확장적으로 적용하는 노력이 가장 중요합니다.

## 1. 데이터 관리의 필요성 및 변화 과정
- **데이터 발생의 문제:** 프로그램이 수행되는 과정에서 대량의 데이터가 지속적으로 발생하므로, 이 데이터들을 효율적으로 관리하는 것이 매우 중요합니다.
- **데이터 저장 방식의 발전:**
    - 초기에는 종이, 펀치 카드 등 아날로그적인 방식으로 데이터를 기록했습니다.
    - 이후 파일 시스템(File System)을 이용해 데이터(예: 엑셀 파일)를 저장했지만, **동시성 문제**가 발생했습니다.
    - 파일 시스템은 여러 사용자가 동시에 접근하거나 데이터를 수정할 때, 최신 정보가 실시간으로 반영되지 않는 **데이터의 일관성 및 무결성** 문제가 발생했습니다.
- **데이터베이스 시스템(DB)의 등장:** 데이터베이스는 이러한 문제를 해결하고, 데이터를 효율적으로 저장, 검색, 갱신하며 여러 데이터 집합 간의 연관성을 조직화하는 시스템이 필요해졌습니다.

## 2. 데이터베이스(DB)와 DBMS의 정의 및 핵심 개념
- **데이터베이스 (DB):**
    - 말 그대로 '데이터의 집합'을 의미합니다.
    - 특정 응용 시스템에 통합되어 저장하고 운영할 수 있는 거대한 정보의 집합입니다. (예: 전화번호부, 개인 다이어리 기록)
- **DBMS (Database Management System):**
    - 데이터베이스를 관리하는 **시스템** 자체를 의미합니다.
    - 사용자가 데이터를 정의(DDL), 조작(DML: 검색, 삽입, 삭제, 갱신), 제어(데이터의 정확성 및 일관성 유지)하는 기능을 제공합니다.
- **DB와 DBMS의 구분:**
    - **데이터베이스 (DB):** 관리 대상이 되는 '데이터의 내용' 그 자체입니다.
    - **DBMS:** 데이터베이스를 체계적으로 관리해 주는 '도구 또는 시스템'입니다.
- **관계형 모델 (Relational Model):**
    - 데이터베이스가 가장 일반적이고 표준적으로 데이터를 관리하는 형태입니다.
    - 데이터를 **테이블(Table)**이라는 '표(Grid)' 형태로 저장하며, 테이블 간의 관계를 통해 데이터를 연결합니다.
    - 관계형 DB는 표준화된 질의어인 **SQL(Structured Query Language)**을 통해 데이터에 접근하고 조작합니다.

## 3. 관계형 데이터베이스의 구조와 원리
- **핵심 구성 요소:**
    - **테이블 (Table):** 데이터를 저장하는 표 형태의 전체 구조입니다.
    - **컬럼 (Column):** 테이블의 열(세로축)에 해당하며, 데이터가 갖는 속성(Attribute)을 정의합니다. (예: 학생 ID, 이름, 등급)
    - **레코드 (Record) / 행 (Row):** 하나의 개체에 대한 모든 정보를 담고 있는 한 줄 전체를 의미합니다. (예: 한 학생의 ID, 이름, 등급 정보 전체)
    - **필드 (Field):** 레코드 내의 개별적인 값 하나를 말합니다.
- **데이터 무결성(Integrity) 확보:**
    - DB는 각 행(레코드)을 고유하게 식별할 수 있는 **키(Key)**를 필수적으로 갖습니다. (주로 기본키 Primary Key가 사용됨)
    - DBMS의 제어 기능을 통해 데이터 입력 시 중복 값이 저장되지 않도록 막거나, 특정 범위의 값만 입력되도록 **제약 조건**을 설정하여 데이터의 일관성을 보장합니다.
- **주요 DB 시스템 선택:**
    - **Oracle:** 오랜 기간 사용되어 온 상용 데이터베이스입니다.
    - **MySQL:** 오픈소스 기반으로, 안정성과 접근성이 높아 많은 웹 애플리케이션에서 사용되며, 교육용으로 많이 채택됩니다.
    - *💡 참고: 어떤 DBMS를 사용하든 표준 SQL을 사용하기 때문에 큰 어려움 없이 개념을 적용할 수 있습니다.*

## 4. 환경 문제 해결을 위한 컨테이너(Docker) 활용
- **DBMS 설치의 어려움 (전통적 방식):**
    - DBMS와 같은 복잡한 소프트웨어는 운영체제(OS)의 깊숙한 영역(레지스트리 등)을 건드리며 설치됩니다.
    - 이 과정에서 발생하는 '찌꺼기(Residual data)'나 충돌 등으로 인해 재설치나 운영체제 환경 구성이 매우 어렵고 복잡합니다.
- **컨테이너(Container)와 도커(Docker)의 등장:**
    - **도커의 역할:** 로컬 PC에 직접 설치하는 대신, 시스템의 영향을 받지 않는 **‘가상의 독립된 환경(컨테이너)’**을 생성하여 그 안에 소프트웨어를 설치하고 사용하는 방식입니다.
    - **장점:**
        1. **격리성:** 호스트 OS(사용자의 본체 OS)에 아무런 변경 없이 깨끗한 환경에서 프로그램을 실행할 수 있습니다.
        2. **재현성:** 어떤 환경에서 설치하든 동일한 환경을 보장하여 개발 및 테스트가 용이합니다.
        3. **관리 용이성:** 사용이 끝난 컨테이너만 삭제하면 되므로, 시스템이 깨끗하게 유지됩니다.
- **이미지(Image)와 컨테이너(Container)의 관계:**
    - **이미지 (Image):** 컨테이너를 만들기 위한 **'설계도' 또는 '설정 파일'**입니다. (예: MySQL 8 버전의 필수 구성을 담은 청사진)
    - **컨테이너 (Container):** 이미지를 기반으로 실제로 실행되는 **'인스턴스'**입니다.
    - 한 개의 이미지를 여러 개의 컨테이너로 실행할 수 있습니다.
- **결론:** 개발 환경의 복잡성과 버전 충돌 문제를 해결하기 위해, 현재 데이터베이스를 포함한 모든 소프트웨어 개발은 도커 기반의 컨테이너 환경을 활용하는 것이 표준이 되었습니다.
- **학습 방향:** 따라서 우리는 복잡한 설치 과정을 직접 다루기보다는, 도커 컨테이너 안에 MySQL 데이터베이스를 설치하여 실습하는 방식으로 학습을 진행합니다.

## 1. 도커를 활용한 MySQL 데이터베이스 구축 환경 설정
*   **목표:** 도커(Docker)를 이용해 MySQL 데이터베이스를 컨테이너 환경에 설치하고 관리하는 방법을 습득합니다.
*   **작업 환경 준비:**
    *   도커 프로젝트를 위한 전용 디렉터리를 생성합니다. (예: `C:\docker\mysql`)
    *   이 디렉터리 내에서 `docker-compose.yml` 파일을 작성하여 서비스를 정의할 준비를 합니다.
*   **핵심 도구:**
    *   **Docker:** 애플리케이션을 격리된 환경(컨테이너)에서 실행할 수 있게 해주는 플랫폼입니다.
    *   **Docker Compose:** 여러 개의 컨테이너와 서비스들을 하나의 YAML 파일로 정의하고 한 번에 관리할 수 있게 돕는 도구입니다.

## 2. Docker Compose 파일 상세 설정 및 이해
`docker-compose.yml` 파일은 컨테이너가 어떻게 동작해야 하는지 상세하게 정의하는 설계도 역할을 합니다.

*   **서비스 정의 (`service`):** 생성할 서비스의 이름을 정의합니다. (예: `mysql-db`)
*   **이미지 지정 (`image`):** 사용할 도커 이미지를 명시합니다. (예: `mysql:8.0` 등, 도커 허브에서 다운로드)
*   **컨테이너 이름 (`container_name`):** 컨테이너가 실행될 때의 이름을 지정합니다.
*   **자동 재시작 설정 (`restart`):** 컨테이너가 비정상 종료되거나 서버가 재부팅될 경우 자동으로 다시 시작하도록 설정합니다. (MySQL과 같은 DB는 지속적 동작이 필수적이므로 이 설정을 주로 사용합니다.)
*   **환경 변수 설정 (`environment`):** 컨테이너 초기 실행 시 필요한 설정 값들을 지정합니다.
    *   **루트 비밀번호 (Root Password):** 시스템 관리자 계정(ROOT)의 비밀번호를 설정합니다. (안전을 위해 기본값을 사용하는 것을 권장합니다.)
    *   **데이터베이스/유저 생성:** 초기 접속에 사용할 데이터베이스 이름(예: `liondb`)과 전용 사용자(예: `lion`)의 이름 및 비밀번호를 정의할 수 있습니다.
*   **데이터 영속성 설정 (`volumes`):**
    *   **개념:** 도커 컨테이너는 휘발성(Ephemeral) 환경이어서, 컨테이너를 중지하거나 삭제하면 내부 데이터가 사라집니다.
    *   **필요성:** 컨테이너가 동작하는 동안 생성된 중요한 데이터를 호스트(실제 PC) 환경에 저장하여 데이터가 유지되도록 연결하는 것이 필수적입니다.
    *   **설정:** 호스트 PC의 특정 디렉터리(`./data/db`)와 컨테이너 내부의 데이터 경로를 연결합니다.
*   **포트 포워딩 설정 (`ports`):**
    *   **원리:** 네트워크의 포트는 집 주소(IP)에 비유되며, 포트 번호는 방 번호와 같습니다. 하나의 IP에 여러 서비스가 구동되기 위해서는 포트를 분리해야 합니다.
    *   **설정:** 외부(호스트 PC)의 포트를 컨테이너 내부의 포트와 연결합니다. (예: `[호스트 포트]:[컨테이너 포트]`)
    *   **주의사항:** 포트 충돌을 피하기 위해 외부 포트와 내부 포트를 명확히 인지하고 사용해야 합니다.

## 3. MySQL 접속 및 관리 도구 사용법 (MySQL Workbench)
*   **접속 도구 사용:** CLI(명령줄) 접속도 가능하지만, GUI 환경인 **MySQL Workbench**를 사용하는 것이 훨씬 편리하고 직관적입니다.
    *   **권장 사항:** 데이터베이스 관리를 위해 워크벤치만 설치하는 것을 권장하며, MySQL 전체 서버를 로컬에 설치할 필요는 없습니다.
*   **연결(Connection) 설정:**
    *   **필수 정보:** 연결 이름(Display Name), 사용자명(User Name), 비밀번호(Password), 그리고 **포트 번호(Port)**를 정확하게 입력해야 합니다.
    *   **테스트:** 'Test Connection' 기능을 통해 설정된 정보가 올바른지 확인합니다.

## 4. 데이터베이스 기본 개념 및 SQL 쿼리 실습
*   **사용자(User)와 데이터베이스(Database)의 개념 분리:**
    *   MySQL은 **User**라는 개념과 **Database**라는 개념을 분리하여 관리합니다.
    *   **권한(Privilege):** 사용자가 어떤 데이터베이스에 접근하고 어떤 작업을 수행할 수 있는지를 제어합니다.
        *   `ROOT` 계정: 모든 권한을 갖는 최상위 관리자 계정입니다.
        *   전용 사용자(`lion`): 특정 데이터베이스(`liondb`)에 대해서만 접근 권한을 가진 계정입니다.
    *   **원리:** 사용자 계정을 생성할 때, 특정 데이터베이스에 대한 접근 권한을 부여하는 것이 보안적으로 바람직합니다.
*   **SQL 기본 쿼리:**
    *   **데이터 조회:** `SELECT` 문법을 통해 데이터를 조회합니다.
    *   **데이터베이스 확인:** `SHOW DATABASES;`
    *   **DB 선택/사용:** `USE [데이터베이스명];` (선택한 DB 내부에서 작업할 수 있게 합니다.)
    *   **테이블 목록 확인:** `SHOW TABLES;` (현재 선택된 데이터베이스 내부의 테이블 목록을 보여줍니다.)
*   **고급 쿼리 고려 사항:**
    *   **접속 대상 지정:** `CURRENT_USER`와 같은 함수를 통해 현재 접속 사용자와 IP 주소 정보를 확인할 수 있습니다.
    *   **보안 강화:** 사용자 생성 시, 특정 IP 주소만 접근할 수 있도록 제약 조건을 걸어 보안성을 높일 수 있습니다.
    *   **접속 최적화:** 워크벤치에서 각 환경(Root, User)별로 별도의 커넥션을 설정하고 사용하는 것이 효율적입니다.

1.  **도커의 핵심 기능:** 컨테이너를 이용해 애플리케이션을 격리시키고, **`volumes`**를 통해 데이터를 영속화한다.

## 1. 프로젝트 기반 학습을 통한 실력 향상 전략
- **실력 향상의 핵심은 프로젝트 경험:** 자바 학습 후 이론만으로는 실력 향상에 한계가 있습니다. 실제 프로젝트(예: 학생 관리 프로그램, 은행 계좌 시스템)를 진행하면서 기능을 추가하고 확장하는 과정에서 지식이 체화됩니다.
- **효율적인 학습 방식:** 개별 챕터 지식을 순서대로 복습하는 것보다, 직접 무언가를 만들어보고(Build), 이를 확장하고 개선해나가는(Expand) 과정이 가장 효율적입니다.
- **협업의 중요성:** 가능하다면 짝 프로그래밍이나 팀 프로젝트를 통해 지식을 공유하고 토론하는 과정이 학습에 큰 도움이 됩니다.
- **목표 설정:** 주제가 없다면, 투두리스트(To-do list)나 메모장 앱과 같은 간단한 주제부터 시작하여 필요성을 느끼는 기능을 스스로 추가하며 학습 범위를 넓히는 것이 좋습니다.

## 2. 데이터베이스(DB)의 이해와 중요성
- **DB의 필요성:** 애플리케이션이 종료된 후에도 데이터가 영구적으로 남아있어야 할 필요성(예: 은행 고객 정보) 때문에 데이터베이스가 필요하게 되었습니다.
- **DB의 역할:** DB는 데이터를 안정적이고 일관되게 관리할 수 있도록 돕는 도구입니다. 과거 파일 저장 방식은 데이터 불일치, 관리 오류 등의 문제점이 있었습니다.
- **데이터 구조의 발전:** 데이터를 '표(Table)' 형태로 관리하는 방식이 간편하고 편리했기 때문에, 이를 체계화하여 **관계형 데이터베이스(Relational Database)**의 개념이 발전하게 되었습니다.

## 3. 관계형 데이터베이스(RDB)의 구조와 특징
- **기본 구성 요소:**
    - **테이블 (Table):** 데이터를 저장하는 기본 구조물입니다.
    - **컬럼 (Column):** 테이블의 속성(Attribute)을 정의합니다.
    - **행 (Row):** 특정 개체의 하나의 데이터 묶음입니다.
- **핵심 개념:**
    - **무결성(Integrity) 및 일관성(Consistency):** DB는 기본키(Primary Key)와 같은 제약 조건을 통해 데이터의 중복 저장이나 잘못된 정보를 아예 입력할 수 없도록 막아주는 강력한 특징을 가집니다.
- **시간적 배경:** 관계형 데이터베이스의 개념은 이미 1970년대부터 존재했으며, 오늘날 사용하는 MySQL은 1996년에 공식화된 역사 깊은 기술입니다.

## 4. 데이터베이스 관리 시스템(DBMS) 및 사용자 권한 개념
- **DBMS의 정의:** 데이터베이스(DB)라는 데이터를 관리하고 조작하는 시스템 자체를 의미합니다. (예: MySQL, Oracle)
- **MySQL 구조:**
    - **Server:** 하나의 MySQL 서버 안에는 여러 개의 데이터베이스(Database)를 가질 수 있습니다.
    - **User:** 각 사용자는 데이터베이스를 사용하기 위해 접속 권한과 역할을 부여받는 '유저' 객체를 가집니다.
    - **권한 관리:** 유저가 DB에 접근하려면 단순히 계정이 있는 것 외에도 '접근 권한', '읽기(SELECT) 권한', '쓰기(INSERT/UPDATE) 권한' 등 적절한 권한이 부여되어야 합니다.
- **접속의 중요성:** 프로그램(자바)이나 GUI 도구(Workbench)로 DB 작업을 하든, 반드시 **'DB에 접속하는 과정'**이 선행되어야만 작업을 수행할 수 있습니다.
- **보안 측면 (Root vs. 일반 유저):**
    - **Root 계정:** 모든 권한을 가지므로 매우 편리하지만, 만일의 사태에 대비하여 보안상 관리자(DBA)는 Root 계정 자체를 배포하거나 직접 사용하는 것을 지양하고, 최소한의 권한을 가진 사용자 계정을 만들어 쓰는 것이 원칙입니다.

## 5. 개발 환경 구축 도구: Docker와 Docker Compose
- **Docker의 등장 배경:** 컴퓨터에 프로그램을 설치할 때 찌꺼기가 남아 문제가 발생하거나, 특정 환경(버전) 간에 간섭이 생기는 것을 방지하기 위함입니다.
- **컨테이너 (Container) 개념:** 운영체제 위에 가상의 공간(컨테이너)을 만들어 필요한 프로그램을 독립적으로 설치하고 사용할 수 있게 합니다. 이는 가상 머신(Virtual Machine, VM)보다 훨씬 가볍고 효율적입니다.
- **Docker의 장점:** 어떤 시스템 환경(OS)과도 간섭을 덜 받고 필요한 환경만 정확하게 구축할 수 있습니다. (예: Python의 가상 환경, Virtual Environment)
- **Docker Compose의 역할:**
    - 복잡한 개발 환경(여러 서비스와 DB)을 설정할 때 필요한 여러 도커 명령을 하나의 YAML 파일로 통합하여 한 번에(One-shot) 환경을 구축할 수 있게 도와줍니다.
    - **Volumes:** 컨테이너가 종료되어도 데이터가 삭제되지 않도록 지속적으로 데이터를 보존할 수 있게 해줍니다.

## 6. 학습 방법론 및 결론
- **지속적인 학습 태도:** IT 기술 분야는 빠르게 변화하기 때문에, 특정 기술을 배운 후에 멈추지 않고 **스스로 공부할 수 있는 능력**을 기르는 것이 가장 중요합니다.
- **'왜?'라는 질문 습관:** 새로운 기술이나 개념을 접할 때, 단순히 작동 방식만 외우지 말고 "이것이 왜 만들어졌는가?"라는 질문을 던지며 근본적인 배경(맥락)을 이해하는 데 집중해야 합니다.
- **간격 벌어짐 방지:** 학습 과정에서 이해가 안 되는 부분이나 모르는 부분이 생겨도 주저하지 말고 적극적으로 질문하거나 (AI, 친구, 멘토 등) 찾아보려는 노력이 중요합니다. 지식의 간극(Gap)이 커지기 전에 즉각적으로 채워야 합니다.

## 1. SQL 기본 구조 및 데이터 모델링 개념

*   **SQL의 표준화:**
    *   SQL(Structured Query Language)은 표준화된 언어이며, 여러 데이터베이스(MySQL 등)가 유사한 문법을 공유하지만, 각 DB마다 특정 부분에서 차이가 있을 수 있습니다.
    *   사용자 정의 이름(테이블명, 컬럼명 등)은 대소문자를 구분하지 않으나, 데이터베이스의 핵심 명령어와 예약어는 문법을 따라야 합니다.
*   **테이블 생성 (`CREATE TABLE`):**
    *   데이터를 저장하기 위해 테이블을 생성합니다.
    *   **문법 구성 요소:** `CREATE TABLE [테이블명] (컬럼1 [데이터 타입] [제약조건], 컬럼2 [데이터 타입] ...)` 형태를 가집니다.
    *   **주요 설정:**
        *   **데이터 타입:** 저장할 데이터의 형태(예: `INT` - 숫자, `VARCHAR` - 문자열)를 지정합니다.
        *   **AUTO_INCREMENT:** MySQL이 제공하는 기능으로, ID와 같은 값이 사용자가 입력하지 않아도 자동으로 증가하게 설정합니다.
        *   **PRIMARY KEY:** 테이블의 기본 키(Primary Key)를 설정합니다.
        *   **Null 허용/필수 여부:** 컬럼의 값이 NULL을 허용할지 여부를 지정할 수 있습니다.
*   **기본 SQL 명령어:**
    *   **데이터 삽입:** `INSERT INTO [테이블명] (컬럼명...) VALUES (값...)`
    *   **데이터 조회:** `SELECT * FROM [테이블명] WHERE 조건;`
    *   **데이터 변경/삭제/제거:** (강의에서 언급된 `UPDATE`, `DELETE`, `DROP` 등 기본적인 CRUD 개념을 포함합니다.)
*   **추가 기능:**
    *   **뷰(View):** 테이블 데이터를 기반으로 가상의 테이블을 생성하는 기능입니다. (나중에 상세 설명 예정)
    *   **실습 환경:** MySQL Workbench나 IntelliJ 같은 개발 도구를 사용하면 ERD 생성, 데이터 인포트/익스포트, 성능 분석 등 다양한 기능을 사용할 수 있습니다.

## 2. 데이터베이스 사용자(User) 및 권한(Permission) 관리

*   **데이터베이스 접속의 원리 (핵심 개념):**
    *   데이터베이스는 사용자가 '누구'인지, '어디에서' 접속하는지에 따라 접근 권한이 달라집니다.
    *   실제 데이터베이스 서버는 네트워크에 연결된 원격 서버에 존재하는 경우가 많습니다.
    *   **IP와 호스트:** 사용자가 어느 위치(IP 주소)에서 접속하는지(호스트)를 지정하여 접근 범위를 제한할 수 있습니다.
*   **사용자 생성 (`CREATE USER`):**
    *   사용자(User)를 생성하는 것은 필수적인 첫 단계입니다.
    *   **사용자 생성 문법:** `CREATE USER '유저명'@'호스트' IDENTIFIED BY '비밀번호';`
        *   호스트를 `%` (와일드카드)로 설정하면, 어느 위치(어디서든)에서 접속하는 것이 가능해집니다. (테스트 환경에서 주로 사용)
        *   특정 IP 주소로 제한할 경우, 해당 IP에서만 접속 가능합니다.
*   **권한 부여 (`GRANT`) 및 권한 취소 (`REVOKE`):**
    *   사용자가 생성되었다고 해서 모든 행동을 할 수 있는 것은 아니며, 반드시 적절한 권한을 부여받아야 합니다.
    *   **권한 부여:** `GRANT [권한 목록] ON [데이터베이스] TO '사용자명'@'호스트';`
        *   **`ALL PRIVILEGES`:** 해당 DB의 모든 권한을 부여합니다.
        *   **필수 권한:** 연결(CONNECT), 조회(SELECT), 생성(CREATE), 수정(INSERT/UPDATE), 삭제(DELETE) 등의 개별 권한을 지정할 수 있습니다.
    *   **권한 취소:** `REVOKE [권한 목록] ON [데이터베이스] FROM '사용자명'@'호스트';`
    *   **적용:** 권한을 부여하거나 취소한 후에는 `FLUSH PRIVILEGES` 명령을 실행하여 변경 사항을 즉시 적용해줘야 합니다.
*   **사용자 권한 순서:**
    1.  유저 생성 (`CREATE USER`)
    2.  적절한 권한 부여 (`GRANT`)
    3.  필요에 따라 권한 취소 (`REVOKE`) 또는 사용자 삭제 (`DROP USER`)

## 3. 실습을 통한 SQL 명령어 실습 및 데이터 준비

*   **실습 환경 구축:**
    *   강의에서는 `hrdb`라는 연습용 데이터베이스를 생성하고, 실습용 테이블들을 생성했습니다.
    *   사용자는 이 과정을 통해 **테이블 생성부터 데이터 삽입 및 조회**까지의 전 과정을 경험합니다.
*   **핵심 명령어 실습 과정:**
    *   **테이블 생성:** `CREATE TABLE` 구문을 사용하여 여러 컬럼과 데이터 타입, 기본 키 등을 지정하여 테이블을 만듭니다.
    *   **데이터 삽입:** `INSERT INTO` 구문을 사용하여 샘플 데이터를 삽입합니다.
    *   **데이터 조회:** `SELECT * FROM [테이블명]`을 사용하여 데이터를 조회합니다.
*   **실습 결과물:**
    *   `hrdb`라는 데이터베이스 내에 연습용 테이블들(예: `emp`, `dept` 등)이 생성되었으며, 이 테이블들은 실제 SQL 학습에 필요한 샘플 데이터를 포함하고 있습니다.
    *   이후 사용자는 이 실습 DB를 바탕으로 SQL 문법을 연습하게 됩니다.

## 1. 데이터베이스 접근 및 권한 관리 (DBMS Setup & Security)
- **DB 구조의 이해:** MySQL과 같은 DBMS는 여러 개의 데이터베이스(Database)를 포함할 수 있으며, 각 데이터베이스는 데이터를 저장하는 독립적인 공간 역할을 합니다.
- **권한 관리의 중요성 (Principle of Least Privilege):**
    - 루트(Root) 사용자는 관리자 권한이 매우 높아 모든 데이터베이스에 접근하고 모든 작업을 수행할 수 있습니다.
    - 하지만 실제 운영 환경에서는 루트 계정을 직접 사용하는 것은 바람직하지 않습니다.
    - 반드시 목적에 맞는 **적절한 사용자(User)를 생성**하고, 해당 유저에게 필요한 최소한의 데이터베이스 접근 및 작업(CRUD: 생성, 조회, 수정, 삭제 등) 권한만을 부여하여 사용하는 것이 보안상 안전합니다.
- **권한 부여 예시:** 특정 유저(`HR` 유저)가 특정 DB(`HRDB`)에 접속하여 테이블을 만들고(CREATE), 모든 종류의 작업을 수행할 수 있도록 권한을 명시적으로 부여하는 과정이 필요합니다.

## 2. SQL의 기본 구조와 기본적인 데이터 조회 (Basic SELECT Statements)
- **SQL의 정의:** SQL(Structured Query Language)은 관계형 데이터베이스에 접근하고 데이터를 조작하기 위한 표준 언어입니다. (질문하는 언어의 역할을 합니다.)
- **기본 SELECT 구문 구조:**
    - **`SELECT [컬럼 이름]`:** 조회하고자 하는 컬럼을 지정합니다.
    - **`FROM [테이블 이름]`:** 데이터를 조회할 원본 테이블을 지정합니다.
    - **예시:** `SELECT first_name, last_name FROM employee;`
- **전체 데이터 조회:** 모든 컬럼을 조회할 때는 `*` (별표)를 사용합니다. (`SELECT * FROM 테이블명;`)
- **컬럼 정보 확인:** 테이블 구조를 확인하고 싶을 때는 `DESCRIBE` (DESC) 명령어를 사용합니다. (예: `DESC employee;`)
- **필수 문법 요소:**
    - **세미콜론(;)**: 문장의 끝을 알리는 필수 문법 요소입니다.
    - **예약어 (Keywords):** `SELECT`, `FROM` 등 데이터베이스가 특별히 정의한 명령어는 대소문자를 구분하지 않으나, 규칙을 따라야 합니다.

## 3. 데이터 조회 심화 기능: 가공 및 집계 함수 활용
- **DISTINCT (중복 제거):** 특정 컬럼의 값이 중복될 때, 중복된 값은 제거하고 고유한 값들만 조회할 때 사용합니다. (예: 특정 직군(Job)의 종류만 알고 싶을 때)
- **별칭 부여 (Alias):** 조회 결과로 나오는 컬럼의 이름이나, 복잡한 계산 결과를 사용자 친화적인 이름으로 변경하여 보여줄 때 사용합니다. (예: `SELECT salary * 1.8 AS annual_salary FROM employee;`)
    - *팁:* 별칭을 포함한 복잡한 구문이나 공백이 포함된 별칭을 사용하려면 반드시 **큰따옴표(`"`)**를 사용해야 합니다.
- **데이터 가공 및 함수 활용:**
    - **연산자:** 단순 계산 (`*`, `+` 등)을 통해 새로운 값(예: 연봉)을 계산하여 조회할 수 있습니다.
    - **문자 연결 (CONCAT):** 여러 컬럼의 값이나 여러 문자열을 하나의 문자열로 합칠 때 사용합니다. (예: 성과 이름 합치기)
- **집계 함수와 그룹핑:**
    - **`GROUP BY`:** 특정 컬럼을 기준으로 데이터를 그룹화하여 집계함이 필요한 경우 사용합니다. (예: 부서별 평균 급여를 계산하고 싶을 때)
    - **집계 함수:** `SUM` (합계), `AVG` (평균), `MAX` (최댓값), `MIN` (최솟값) 등이 있습니다.
    - **`HAVING`:** `WHERE` 절은 개별 행 단위의 조건을 걸지만, `HAVING`은 **`GROUP BY`로 그룹화된 그룹 자체**에 조건을 걸 때 사용합니다. (예: 평균 급여가 특정 값 이상인 그룹만 조회)

## 4. WHERE 절을 사용한 조건 검색 및 필터링
- **WHERE 절의 역할:** 데이터가 방대할 때, 원하는 조건을 충족하는 데이터만 골라내어 조회할 때 사용합니다.
- **비교 연산자:** 값의 비교(크기, 범위 등)를 할 때 사용합니다. (`>`, `<`, `>=`, `<=`, `!=` 또는 `<>`)
- **논리 연산자 (Logical Operators):** 여러 개의 조건을 조합하여 복잡한 필터링을 할 때 사용합니다.
    - **AND:** 모든 조건이 **모두** 참일 경우에만 결과를 반환합니다. (가장 까다로운 조건)
    - **OR:** 조건 중 **하나라도** 참이면 결과를 반환합니다.
    - **NOT:** 특정 조건이 **아닌** 경우를 필터링합니다.
- **쿼리 구조 유의 사항:** 복합적인 조건을 처리할 때는 괄호를 사용하여 **조건의 우선순위**를 명확히 지정해 주는 것이 좋습니다.

## 5. SQL 구문의 표준 이해 및 실습 팁
- **SQL의 표준성 vs. 구현의 차이:**
    - SQL은 본래 표준 언어이지만, MySQL, Oracle 등 각 DBMS마다 내부적인 구현 방식(사투리, Dialect)이나 기본 문법에서 차이점이 있습니다.
    - 따라서 특정 DBMS에서 배운 기능이 다른 DBMS에서는 다르게 동작할 수 있음을 유의해야 합니다.
- **데이터 타입 및 대소문자:**
    - **SQL 명령어 (Keyword):** 대소문자를 구분하지 않는 경우가 많습니다.
    - **데이터 값 (Value):** 기본적으로 대소문자를 구분합니다.
    - **MySQL의 특징:** MySQL은 특이하게도 값에 대해서도 대소문자를 구분하지 않고 조회하는 경우가 많습니다.
- **DDL (데이터 정의어) 복습:**
    - **`CREATE TABLE`:** 테이블을 생성할 때, 컬럼명, 데이터 타입, 제약조건 등을 명시합니다.
    - **`SHOW TABLES` / `DESCRIBE`:** 데이터베이스 내의 테이블 목록이나 구조를 확인하는 명령어는 자주 사용되므로 숙지해야 합니다.
- **학습 태도:** SQL은 반복적인 실습을 통해 가장 빠르게 익숙해지며, 처음에는 낯설고 복잡하게 느껴질 수 있지만, 꾸준히 연습하다 보면 자연스럽게 감각이 생기게 됩니다.
