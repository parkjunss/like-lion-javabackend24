## 1. 지난 시간 복습 및 학습 목표 설정
*   **핵심 요약:** 이전 시간에는 Java 프로그램과 데이터베이스(DBMS)를 연결하여 데이터를 입력하는 과정을 배웠습니다.
*   **학습 목표 설정의 중요성:** 배우는 개념(JDBC, DBMS 등)들이 단절된 지식이 아니라, 궁극적으로 **웹 애플리케이션**을 구축하는 데 필수적으로 연결되어 있음을 이해하는 것이 중요합니다.
*   **DBMS의 역할:** 웹 환경에서 발생하는 수많은 데이터를 체계적으로 저장하고 관리할 수 있게 해주어, 애플리케이션의 동작을 가능하게 하는 핵심 기반 기술입니다.

## 2. 트랜잭션(Transaction) 이해 및 개념 심화
*   **트랜잭션 정의:** 트랜잭션은 여러 개의 작업을 묶어 처리하는 **논리적인 작업 단위**입니다.
    *   **핵심 원칙:** 이 작업 단위 내의 모든 작업이 성공하면 모두 반영되고(Commit), 단 하나라도 실패하면 처음 상태로 모두 되돌아가야(Rollback) 합니다.
    *   **예시:** 계좌 이체처럼 'A 계좌 출금'과 'B 계좌 입금'이라는 두 가지 작업이 하나의 논리적 단위로 묶여야 합니다.
*   **데이터 무결성 유지:** 트랜잭션 개념은 데이터의 일관성(Consistency)을 유지하는 데 필수적이며, 이를 통해 데이터 오류를 방지합니다.
*   **DBMS의 처리 메커니즘:**
    *   **자동 커밋(Auto-Commit):** 기본적으로 데이터 변경(INSERT, UPDATE, DELETE)이 발생하면 DBMS는 기본적으로 트랜잭션이 시작된 것으로 간주합니다.
    *   **직접 제어:** 개발자는 `commit`이나 `rollback` 같은 메서드를 사용하여 트랜잭션의 최종 반영 시점을 명시적으로 제어할 수 있습니다.
*   **동시성 제어 (Concurrency Control):**
    *   **세션(Session):** 여러 클라이언트가 동시에 접속하여 데이터를 사용할 때 발생하는 상황을 가정합니다.
    *   **격리 레벨(Isolation Level):** DBMS는 동시에 발생하는 여러 작업이 데이터를 오염시키지 않도록 일종의 격리 레벨을 갖고 처리합니다.
    *   **데드락(Deadlock):** 여러 세션이 서로가 작업한 리소스를 기다리느라 무한 대기 상태에 빠지는 현상입니다. DBMS는 이러한 상황을 감지하고 작업을 취소(Rollback)하여 해결합니다.

## 3. JDBC 및 SQL 문장 처리 기법
*   **프로그래밍 관점의 JDBC:** JDBC는 Java 프로그램에서 데이터베이스에 접속하여 데이터를 조작(CRUD)할 수 있게 해주는 기술입니다.
*   **연결 과정:** DBMS에 접근하려면 **커넥션(Connection)** 객체를 얻어와야 하며, 이를 이용해 **스테이트먼트(Statement)** 객체를 생성합니다.
*   **Prepared Statement의 활용 (필수):**
    *   일반 Statement는 SQL 문장이 완성된 형태를 다루므로, 값이 변경될 때마다 DBMS가 매번 구문 분석(번역)을 해야 합니다.
    *   **Prepared Statement**는 SQL 템플릿(물음표로 값 대치) 형태로 먼저 전처리(Pre-compile)해 둡니다.
    *   **장점:** 재사용성으로 성능이 향상되며, 외부 값을 직접 문자열로 결합하는 방식의 보안 취약점(SQL Injection)을 방지할 수 있습니다.
*   **구현의 분리 원칙:** JDBC를 사용할 때 데이터베이스 접속, SQL 생성, 실행 등의 과정은 라이브러리(Driver)를 통해 구현되어야 하며, 이 과정이 라이브러리에 의존성을 갖는 것이 일반적입니다.

## 4. 객체지향 설계와 아키텍처 이해 (추상화 원리)
*   **추상화의 중요성:** 코딩할 때 모든 것을 직접 구현하지 않고, 공통적으로 필요한 기능(예: 데이터베이스 접속, 통장 관리)은 공통의 형태로 정의하는 것이 효율적입니다.
*   **인터페이스(Interface)의 역할:**
    *   인터페이스는 '규약' 혹은 '약속'을 정의하는 도구입니다. (예: `java.sql.Connection` 인터페이스)
    *   실제 DBMS(MySQL, Oracle 등)마다 내부 구현 방식이 다르더라도, 이 인터페이스라는 **규칙**만 지켜서 구현체를 제공하면, 개발자는 구현체가 어떤 종류인지 알 필요 없이 일관되게 코드를 작성할 수 있습니다.
    *   이를 통해 시스템의 유연성과 확장성이 크게 높아집니다.
*   **ER 다이어그램 및 UML:**
    *   **ER 다이어그램:** 요구 사항을 정의하고 테이블 구조를 설계할 때 사용하며, 시스템의 관계를 그림으로 표현합니다.
    *   **UML (Unified Modeling Language):** 클래스 다이어그램, 시퀀스 다이어그램 등 다양한 형태로 시스템의 구조와 동작을 그림으로 표현하여 개발자 간의 의사소통을 돕는 도구입니다.

## 5. 개발 환경 관리 도구 및 데이터 교환 방식의 진화
*   **라이브러리 관리의 필요성:** 대규모 프로젝트에서는 다양한 외부 라이브러리가 사용되는데, 이 라이브러리들의 버전을 직접 다운로드하여 프로젝트에 추가하고 경로를 설정하는 과정이 매우 복잡합니다.
*   **빌드 도구의 등장 (Maven/Gradle):**
    *   Maven이나 Gradle 같은 빌드 도구는 이러한 라이브러리 관리의 복잡성을 해결해 줍니다. 필요한 라이브러리를 명령어 하나로 다운로드하고 프로젝트에 자동으로 포함시킵니다.
*   **데이터 교환 형식의 발전:**
    *   **XML:** 과거에는 데이터를 구조화하여 주고받을 때 XML 형태를 사용했습니다. 이 방식은 태그 구조가 명확하지만, 데이터 전송량이 많고 파싱(Parsing) 과정이 복잡할 수 있습니다.
    *   **JSON/객체:** 시간이 지나면서 데이터 필드와 값에 대한 약속을 더욱 간편하게 전달할 수 있는 JSON 형태를 거쳐, 최종적으로 프로그래밍 언어의 **객체(Object)** 형태로 변환되어 데이터를 주고받는 방식이 일반화되었습니다.

## 1. JDBC 기본 CRUD 복습 및 목표 설정 (SQL 실행 및 연결)
*   **복습 내용:** 지난 시간에는 데이터베이스 접속 및 `INSERT`, `UPDATE`, `DELETE`와 같은 기본적인 CRUD(Create, Read, Update, Delete) 작업을 진행함.
*   **학습 목표:** 이번 시간에는 SELECT 쿼리 실행 과정에 초점을 맞추고, 특히 **예외 처리(Exception Handling)**를 염두에 두고 코드를 작성하는 방법을 학습함.

## 2. JDBC 예외 처리 기법: Try-Catch 구문 활용
*   **JDBC 연결 과정의 위험성:** 데이터베이스 접속(`Connection`)과 같은 외부 자원 작업은 예외가 발생할 가능성이 높음.
*   **기본 예외 처리:**
    *   `try` 블록을 사용하여 예외가 발생할 수 있는 코드를 감싸고, `catch` 블록에서 특정 예외(`SQLException` 등)를 잡아서 처리하는 방식을 사용함.
    *   **리소스 해제(Close):** `Connection` 객체를 사용한 후에는 반드시 `close()` 메서드를 호출하여 자원을 해제해야 함.
    *   **주의 사항:** `close()` 호출 또한 예외가 발생할 수 있으므로, `try-catch`를 통해 안전하게 처리해야 함.
    *   **안전한 코드 작성 팁:** 리소스를 닫는(`close()`) 과정은 해당 객체가 `null`이 아닐 때만 실행하도록 조건을 추가하는 것이 가장 바람직함.

## 3. JDBC SELECT 구문 동작 순서 및 객체 관리
*   **SELECT 쿼리 실행의 순서:**
    1.  **접속 (Connection):** `DriverManager`를 이용해 데이터베이스에 접속 객체(`Connection`)를 획득함.
    2.  **쿼리문 작성 (Query):** `SELECT` 쿼리문을 작성함.
    3.  **Prepared Statement 생성:** `Connection` 객체로부터 `prepareStatement()` 메서드를 사용하여 준비된 구문 객체(`PreparedStatement`)를 얻어옴.
    4.  **실행 (Execution):** `PreparedStatement`의 `executeQuery()` 메서드를 호출하여 쿼리를 실행함.
    5.  **결과 수신 (Result Set):** 쿼리 실행 결과는 `ResultSet` 객체로 반환됨.
*   **객체 선언 및 해제 순서 (중요):**
    *   **프로그래밍 순서:** 필요한 객체(`Connection`, `PreparedStatement`, `ResultSet`)를 먼저 선언하는 것이 구조적으로 좋음.
    *   **리소스 해제 순서:** 리소스를 닫을 때는 **획득한 역순**으로 닫아줘야 함.
        (예: `ResultSet` $\to$ `Statement` $\to$ `Connection`)

## 4. ResultSet 처리 및 결과 값 추출 방법
*   **ResultSet의 개념:** `ResultSet`은 DBMS가 쿼리 결과를 프로그램에 직접 주는 것이 아니라, **결과 전체를 추상화하여 담아두는 객체**임.
*   **커서(Cursor)의 역할:** 데이터를 한 건씩 읽어오기 위해서는 결과 셋 전체를 가리키는 `ResultSet`에서, 읽을 **첫 번째 레코드(Row)**를 가리키는 커서(Cursor)를 이동시켜야 함.
*   **데이터 이동 (next()):**
    *   `rs.next()` 메서드를 호출하면, 커서가 다음 레코드로 이동함.
    *   **반환 값:** 다음에 읽어올 레코드가 있으면 `true`를 반환하고, 더 이상 읽을 레코드가 없으면 `false`를 반환함.
*   **반복문 활용:** `next()` 메서드는 데이터가 남아있는 동안 반복적으로 실행되어야 하므로, `while (rs.next())` 구문을 사용하여 반복문을 구성함.
*   **데이터 값 추출 (get*()):**
    *   반복문 내부에서 데이터를 추출할 때는 `rs.get*()` 메서드를 사용함.
    *   **필수 요소:** 가져올 데이터의 타입(예: `getInt()`, `getString()`)을 지정해야 함.
    *   **인수 지정 방법:**
        1.  **컬럼 인덱스:** `rs.getString(1)`처럼 숫자로 순서를 지정할 수 있음.
        2.  **컬럼 라벨 (이름):** `rs.getString("column_name")`처럼 컬럼명을 지정할 수 있음.
        *   (*권장 사항:* 코드의 명확성을 위해 컬럼 이름(라벨)을 사용하는 것이 좋음.)

## 5. 안전한 리소스 관리를 위한 Try-With-Resources 구문
*   **문제점:** 예외 처리(`try-catch-finally`)를 사용하면 코드가 길어지고 복잡해지는 단점이 있음.
*   **해결책 (Java 7+):** **Try-With-Resources** 구문을 사용하면 리소스 관리가 훨씬 간편해짐.
*   **작동 원리:**
    *   리소스 객체를 `try(...)` 구문 내에서 선언하는 것만으로도, 이 객체가 `try` 블록을 벗어날 때 **자동으로(`AutoCloseable` 인터페이스를 구현했기 때문에)** `close()`가 호출됨.
    *   따라서 명시적으로 `finally` 블록에서 `close()`를 처리할 필요가 없음.
*   **조건:** 이 구문을 사용하려면 선언되는 모든 리소스 객체(`Connection`, `Statement`, `ResultSet` 등)가 반드시 `AutoCloseable` 인터페이스를 구현하고 있어야 함.
*   **장점:** 코드의 가독성이 높아지고, 자원 누수(Resource Leak)를 방지하여 훨씬 안전하고 간결하게 JDBC 코드를 작성할 수 있음.

## 1. JDBC 프로그래밍의 기본 원리와 구조 이해
JDBC는 복잡한 알고리즘을 요구하는 것이 아니며, 그 동작 순서와 과정이 정해져 있는 비교적 쉬운 분야입니다.
*   **일반적인 과정:** 데이터베이스 접속(Connection) → 쿼리문 생성(Statement) → 실행(Execute) → 결과값(ResultSet) 처리.
*   **핵심 원리:** 데이터베이스와의 상호작용 순서(순차적 절차)가 정해져 있으며, 개발자는 이 정해진 틀에 맞춰 기능을 구현하는 데 집중합니다.
*   **CRUD의 중요성:** 실무에서 개발하는 대부분의 애플리케이션 기능은 **CRUD** (Create, Read, Update, Delete: 생성, 읽기, 수정, 삭제)의 기본 기능을 기반으로 합니다.

## 2. 자원 관리 및 SQL 실행 심화 (try-with-resources)
JDBC 코드를 효율적이고 안정적으로 작성하는 방법을 다룹니다.
*   **자원 자동 관리:** `try-with-resources` 구문을 사용하면 `Connection`, `Statement`, `ResultSet`과 같은 자원 객체들이 명시적으로 `close()`를 호출하지 않아도 자동으로 자원을 해제하여 누수(Resource Leak)를 방지할 수 있습니다.
*   **파라미터 처리:** SQL 쿼리문 내의 값(변수)은 반드시 `?` 와 같은 플레이스홀더(Placeholder)를 사용해야 합니다. 실제 값을 바인딩(Binding)할 때는 `PreparedStatement`를 이용하여 안전하게 값을 넣어줘야 합니다.
*   **예외 처리의 분리:** 데이터베이스 작업은 실패할 가능성이 높으므로, 예외 처리 전략을 고민해야 합니다.
    *   **개별 처리 vs. 전파:** 기능 구현 객체 내부에서 예외를 처리할지, 아니면 호출하는 상위 레벨(메인 로직)에서 예외가 발생했을 때 처리할지 결정해야 합니다. (일반적으로 상위 레벨에서 처리하는 것이 바람직함)
    *   **비즈니스 로직 예외:** 데이터 무결성 위반(예: 중복된 이메일 주소) 같은 경우, 예외(Exception)를 던져서 상위 레이어에서 사용자에게 알림을 주는 방식으로 처리하는 것이 좋습니다.

## 3. 애플리케이션 아키텍처 설계 패턴: DAO (Data Access Object)
단순히 `main` 메소드 안에 모든 기능을 구현하는 방식은 유지보수성이 떨어지므로, 역할을 분리하여 체계적으로 설계해야 합니다.
*   **메인 메소드의 한계:** `main` 메소드에 로직을 구현하면, 해당 기능이 한 번만 사용되는 것처럼 보이게 되어 재사용성이 떨어지고, 코드가 복잡해지며 비효율적입니다.
*   **클래스 단위로 기능 분리:** 반복적이고 재사용 가능한 기능(예: 회원 추가, 회원 조회)은 각각 독립적인 클래스로 분리하여 구현해야 합니다.
*   **DAO 패턴 적용:**
    *   **정의:** **DAO (Data Access Object)** 패턴은 특정 테이블(엔티티)에 대한 모든 데이터 접근 로직(CRUD)을 전담하는 객체를 만드는 디자인 패턴입니다.
    *   **장점:** 특정 테이블에 대한 모든 기능을 하나의 객체(예: `MemberDAO`)에 묶어 관리함으로써, 애플리케이션 코드를 간결하고 체계적으로 유지할 수 있습니다.
    *   **구현 방향:** 각 기능을 독립적인 메서드(예: `insertMember()`, `selectMemberByEmail()`, `updateMember()`)로 구현하고, 이 메서드들을 해당 DAO 객체를 통해 호출합니다.

## 4. 데이터 전달 객체 설계: DTO/VO (Value Object)
DAO 클래스의 메서드를 호출할 때 필요한 데이터를 효과적으로 전달하는 방법을 다룹니다.
*   **문제 인식 (매개변수의 과부하):** 여러 개의 필드(이름, 이메일, 비밀번호, 전화번호 등)를 메서드의 매개변수로 나열하면, 코드가 지저분해지고 어떤 값이 어떤 필드에 매칭되는지 파악하기 어렵습니다.
*   **해결책 (데이터 묶음):** **VO (Value Object) 또는 DTO (Data Transfer Object)** 패턴을 사용합니다.
    *   **개념:** DTO/VO는 단순히 여러 데이터를 **담는 가방(Container)**과 같은 역할을 합니다. 데이터 자체의 값(Value)을 담는 데 초점을 맞추며, 로직을 담기보다는 데이터를 전송하는 역할만 수행합니다.
    *   **설계:** 필요한 모든 필드(예: `name`, `email`, `password`, `hireDate`)를 필드로 선언하고, 이 필드들을 담기 위해 생성자(Constructor)와 접근자(Getter/Setter)를 제공하는 클래스를 만듭니다.
    *   **효율성:** DAO 메서드 호출 시 개별 변수를 넘기는 대신, 이 DTO/VO 객체 하나만 넘겨주면 코드가 매우 깔끔하고 가독성이 높아집니다.

## 1. 데이터베이스 접근의 구조 설계 (DAO 패턴)
- **객체지향 설계의 원칙 적용:** 프로그램 내의 모든 로직을 메인 클래스에서 처리하기보다, 각 기능을 전담하는 독립적인 객체(DAO: Data Access Object)를 만들어야 한다.
- **관심사 분리 (Separation of Concern):** DAO는 특정 테이블(예: `Member`)에 데이터에 접근(입력, 수정, 조회, 삭제)하는 역할만을 전담하도록 설계하여, 코드를 구조적으로 분리하고 유지보수성을 높인다.
- **구조적 이점:** `MyGame`과 같은 메인 로직 객체는 실제 데이터베이스 접근 방법을 알 필요 없이, 오직 `MemberDAO` 같은 DAO 객체를 통해 필요한 기능만을 호출한다. (클라이언트와 데이터 접근 계층의 분리)

## 2. 데이터 전달을 위한 DTO의 활용
- **DTO (Data Transfer Object)의 역할:** DTO는 데이터 자체를 담는 '가방'의 역할을 한다. 함수가 여러 개의 인자(매개 변수)를 사용하거나 반환할 때, 개별 변수를 전달하는 것이 복잡해지는 것을 방지하고 값을 묶어서 전달하는 효과적인 매개체이다.
- **사용의 용이성:** 함수를 호출할 때도 값을 전달할 때도, 이 DTO 객체 하나만 전달하면 되므로 코드의 가독성과 관리 용이성이 크게 향상된다.
- **구현 요소:** DTO 내부의 필드에 안전하게 값을 담고 꺼낼 수 있도록 **Getter와 Setter** 메서드를 반드시 구현해야 한다.

## 3. JDBC 구현 및 리소스 관리
- **JDBC 핵심 원리:** 데이터베이스와 연결(Connection)하고, SQL 문(Statement)을 준비(PreparedStatement)하여 실행하는 과정을 거친다.
- **리소스 관리의 중요성 (Try-with-Resources):**
    - 데이터베이스 접속(Connection, Statement 등)은 반드시 사용 후 닫아주어야 한다. 이를 누락할 경우 **리소스 누수**가 발생하여 시스템 전체가 멈추는 심각한 문제(DB 자원 고갈)를 초래할 수 있다.
    - Java 7+에서 도입된 `try-with-resources` 구문을 사용하면, 해당 블록을 벗어날 때 자원(Resource)이 자동으로 닫히므로 안전성이 보장된다.
- **CRUD별 처리 방식:**
    - **INSERT (입력):** `Connection` $\rightarrow$ `PreparedStatement`를 얻고, DTO의 Getter 메서드를 사용하여 SQL 문장의 `?` 위치에 순차적으로 값을 바인딩한 후, `executeUpdate()`를 호출한다.
    - **UPDATE (수정):** 입력과 마찬가지로 `PreparedStatement`를 사용하며, 수정할 값들을 DTO를 통해 바인딩하여 `executeUpdate()`를 호출한다. (단, 모든 필드를 대상으로 업데이트문을 구성하는 것이 일반적이다.)
    - **DELETE (삭제):** 삭제의 경우 ID 값만 있으면 충분하므로, 복잡한 DTO 없이 필요한 최소한의 매개 변수(ID)만으로 처리할 수 있다.
    - **SELECT (조회):** 데이터를 한 번에 조회하는 경우, 메서드의 반환 타입은 단일 타입이 아닌 **DTO 객체(또는 DTO의 리스트)**가 되어야, 조회된 여러 필드(ID, 이름, 이메일 등)를 하나의 '가방'에 담아 반환할 수 있다.

## 4. 체계적인 애플리케이션 설계 원칙 및 개선점
- **모듈화 원칙:** 사용자 입력 처리나 특정 기능을 수행하는 과정이 메인 클래스(Main)에 길게 늘어지면 가독성이 떨어진다. 따라서 **사용자 입력 처리(예: 스캐너 이용)**와 같은 기능은 별도의 전용 메서드로 분리하여 모듈화하는 것이 바람직하다.
- **접속 코드 중복 처리 고민:** DAO를 구현하며 Connection을 얻는 과정이 각 메서드마다 반복되어 코드가 중복되는 문제가 발생한다. (→ **개선점 고민:** 이 부분이 가장 먼저 고민되어야 할 부분이다.)
- **Connection 공유 금지:** 데이터베이스 작업 시 **Connection을 공유하여 사용하는 것은 매우 위험**하다. 모든 트랜잭션(입력, 수정, 삭제 등)은 반드시 독립적인 Connection을 열고 닫아 사용해야 한다.
- **메서드 역할 명확화:**
    - **읽기 전용 작업:** 단순히 데이터를 조회만 하고 아무것도 변경하지 않는 경우 (예: `findByID`), 반환 값만 가지는 것이 좋다.
    - **수정/삭제 작업:** 데이터를 변경하는 작업 (예: `insert`, `update`, `delete`)은 성공/실패 여부를 나타내는 `boolean`이나 영향을 받은 행의 수를 나타내는 `int`를 리턴하는 것이 일반적이다.

## 1. JDBC 기본 구조 및 단일 데이터 조회 (Single Record)
- **JDBC 핵심 과정:** 애플리케이션에서 ID와 같은 파라미터 값을 받아와 커넥션(Connection)을 얻고, Prepared Statement를 통해 SQL을 실행한다.
- **쿼리 실행 및 파라미터 처리:** `PreparedStatement`를 사용하여 SQL에 값을 안전하게 바인딩한다. (예: `ps.setInt(1, id)`)
- **데이터 조회:** 쿼리 실행 후 `ResultSet`을 얻게 되며, 이를 통해 결과 데이터를 추출한다.
- **단일 데이터 처리 (예: getMember):**
    - `ResultSet`에서 데이터를 가져올 때는 `rs.getColumnName()` 또는 `rs.getInt()` 등의 Getter 메소드를 사용한다.
    - 조회한 값을 미리 선언한 **DTO(Data Transfer Object)** 객체에 Setter 메소드를 통해 담아 반환한다.

## 2. 데이터 타입 매핑 및 주의사항 (Date/Time 처리)
- **컬럼명 사용 권장:** 결과를 가져올 때 컬럼의 순서(1번, 2번)보다는 **컬럼명(Column Name)**을 명시적으로 사용하는 것이 가독성이 높고 안정적이다.
- **날짜/시간 타입 매핑 (Date/Time):**
    - Java의 `LocalDateTime`이나 `LocalDate`와 같이 최신 객체 타입은 JDBC 드라이버가 자동으로 맵핑해주지 못할 수 있다.
    - 이 경우, `rs.getObject()`를 사용하여 데이터를 일단 객체(Object)로 꺼낸 다음, 필요한 Java 타입으로 명시적으로 변환(Casting 또는 변환 로직)하여 DTO에 담아줘야 한다.

## 3. 결과 집합(Result Set) 반복 처리 원칙
- **`next()` 메소드의 중요성:** `ResultSet` 객체 자체는 전체 결과 집합을 가리킨다. 실제 레코드 데이터에 접근하고 순회하기 위해서는 반드시 `resultSet.next()`를 호출하여 커서(Cursor)를 다음 레코드로 이동시켜야 한다.
- **JDBC 반복 구조:** `while (resultSet.next())` 구문을 사용하여 결과 레코드가 존재하는 동안 반복문을 돌며 데이터를 추출하는 것이 기본적인 패턴이다.
- **반복문 내부 로직 구성 (핵심):**
    - 여러 개의 데이터를 가져와야 할 경우, **반복문 외부**에서 **빈 리스트(List)**를 선언한다.
    - **반복문 내부**에서 데이터를 조회할 때마다 새로운 DTO 객체를 생성한다.
    - 생성된 DTO 객체에 데이터를 채운 후, 이 DTO 객체를 선언된 리스트에 반드시 **`add()`** 메소드를 이용해 추가해야 한다. (반복문이 끝나면 로컬 변수 범위가 해제되므로, 리스트에 추가하지 않으면 데이터가 소실된다.)

## 4. 다중 데이터 조회 및 리스트 활용 (Multiple Records)
- **반환 타입 변경:** 단일 데이터 대신 여러 데이터를 반환할 경우, 반환 타입을 `DTO`가 아닌 **`List<DTO>`**와 같은 컬렉션 형태로 변경해야 한다.
- **예시 메서드:** `getMembers()`와 같이 여러 데이터를 담을 수 있는 컬렉션을 반환하도록 설계한다.
- **데이터 접근:** 서비스 레이어(DAO)에서 여러 결과를 조회한 후, 이를 상위 계층(Controller 등)으로 전달할 때 리스트 구조를 유지하며 사용한다.

## 5. 코드 최적화 및 아키텍처 고민 (자원 관리)
- **중복 코드 문제 인식 (Anti-Pattern):** 예제 코드를 따라가다 보면, 매번 데이터 접근 메서드마다 `Connection`을 얻고 `PreparedStatement`를 설정하는 로직이 반복되어 중복성이 발생한다.
- **개선 방향 (Resource Pooling / DAO 패턴):**
    - 이러한 반복되는 리소스 획득 및 해제 로직은 상위 개념으로 묶어 관리하는 방법을 고민해야 한다. (예: 트랜잭션 관리, 커넥션 풀(Connection Pool) 사용, 또는 DAO/Repository 패턴을 적용하여 반복적인 코드를 공통으로 처리하는 방안)

## 1. 공통 리소스 관리 및 클래스 설계 (Connection Manager Pattern)
*   **문제 제기:** 여러 DAO(Data Access Object)에서 공통적으로 필요한 '커넥션(Connection)' 획득 로직이 반복되고 중복되는 문제를 해결해야 함.
*   **해결 방안:**
    *   커넥션 획득 로직을 별도의 독립적인 클래스(혹은 유틸리티 클래스)로 분리하여 관리하는 것이 효율적이다.
    *   모든 DAO가 이 독립된 클래스를 통해 커넥션을 요청하도록 설계해야 한다.
*   **고도화된 설계:**
    *   커넥션 획득 메서드를 오버로딩하여, 단순히 기본 자격 증명뿐만 아니라 URL, 사용자명, 패스워드 등 다양한 조건에 맞춰 커넥션을 생성할 수 있도록 설계할 수 있다.

## 2. JDBC 아키텍처 이해 및 자바의 역할 (Interface 기반의 추상화)
*   **JDBC의 정의:** 자바 애플리케이션과 데이터베이스를 연결하는 프로그래밍 기술.
*   **인터페이스 기반의 장점:**
    *   데이터베이스 벤더(DB Vendor)들은 자바가 정의한 표준 JDBC 인터페이스를 구현한다.
    *   자바 개발자는 실제 사용하는 데이터베이스 종류(Oracle, MySQL 등)에 관계없이 **표준 인터페이스만 알고 사용**하면 되기 때문에 높은 이식성(Portability)을 확보할 수 있다.
    *   *주요 객체들:* `Connection` (연결), `Statement` (쿼리 실행), `ResultSet` (결과값), `DriverManager` (연결 관리). 이들 대부분은 인터페이스를 기반으로 한다.
*   **프로젝트 빌드 도구:**
    *   외부 라이브러리 종속성 관리의 어려움을 해결하기 위해 Maven이나 Gradle 같은 빌드 도구를 사용한다.
    *   이 도구들은 필요한 라이브러리(JAR 파일)를 자동으로 다운로드하고 프로젝트 구조에 포함시켜 개발 편의성을 극대화한다.

## 3. JDBC 기본 동작 구조 및 자원 처리 (ResultSet, Statement, Transaction)
*   **데이터 처리 플로우 (순서):** JDBC를 사용할 때는 일정한 순서를 따른다.
    1.  `DriverManager`를 이용해 `Connection` 객체 획득.
    2.  `Connection` 객체를 이용해 `Statement` (또는 `PreparedStatement`) 객체 획득.
    3.  `Statement` 객체를 이용해 SQL 쿼리 실행.
    4.  결과값(`ResultSet`)을 받아와서 처리.
*   **실행 메서드별 특징:**
    *   **`executeQuery()`:** `SELECT`문 실행에 사용하며, 결과를 담는 `ResultSet`을 반환한다.
    *   **`executeUpdate()`:** `INSERT`, `UPDATE`, `DELETE` 등 수정 작업에 사용하며, 영향을 받은 행의 수를 정수(int)로 반환한다.
*   **자원 관리 및 예외 처리:**
    *   JDBC 사용 과정에서 발생하는 모든 객체(Connection, Statement, ResultSet)는 반드시 리소스를 해제(`close()`)해야 한다.
    *   **권장 방식:** `try-with-resources` 구문을 사용하면 자동으로 자원 해제를 처리해주어 코드를 간결하고 안전하게 만들 수 있다.
*   **트랜잭션(Transaction) 처리:**
    *   여러 데이터베이스 작업(A, B, C)을 하나의 단위로 묶어 처리할 때 사용한다.
    *   `conn.setAutoCommit(false)`로 자동 커밋을 비활성화한 후, 모든 작업이 성공하면 `conn.commit()`을 호출하여 최종 반영하고, 중간에 예외(Error)가 발생하면 `conn.rollback()`을 호출하여 모든 변경사항을 취소한다.

## 4. 개발 효율성 및 안정성 향상 기법 (Prepared Statement, DAO 패턴)
*   **PreparedStatement 사용의 중요성 (권장):**
    *   **`Statement`:** SQL 쿼리문이 완성된 상태로 사용되어 사용이 번거롭고 성능이 떨어질 수 있다.
    *   **`PreparedStatement`:** 쿼리문 내에 `?` 형태의 플레이스홀더(Placeholder)를 사용하여 SQL 구문을 미리 준비한다.
        *   **장점 1 (보안):** SQL 인젝션(Injection) 공격을 방지할 수 있다.
        *   **장점 2 (성능):** 재사용성이 높아 실행 속도가 향상된다.
        *   **장점 3:** 파라미터 값 바인딩을 통해 안전하게 값을 전달한다.
*   **자동 생성 키 처리 (Generated Keys):**
    *   데이터를 삽입(INSERT)할 때, DB가 자동으로 생성하는 기본 키(Primary Key) 값을 즉시 얻어와야 하는 경우에 사용한다.
    *   `PreparedStatement`를 사용하여 쿼리 실행 시 `Statement.RETURN_GENERATED_KEYS` 상수를 지정하고, 이를 통해 `ResultSet`으로 자동 생성된 ID 값을 가져올 수 있다.
*   **DAO 패턴 적용:**
    *   데이터베이스와의 모든 상호작용(데이터 액세스)을 전담하는 계층(DAO: Data Access Object)을 분리한다.
    *   이는 비즈니스 로직과 데이터 액세스 로직을 분리하여 시스템의 유지보수성과 응집도를 높이는 좋은 설계 패턴이다.

## 5. 데이터베이스 성능 최적화 기법 (Connection Pool, DataSource)
*   **Connection Pool의 필요성:**
    *   애플리케이션이 실행되는 동안 요청이 발생할 때마다 데이터베이스에 접속(Connect)하고 끊는(Close) 과정은 비용(시간 지연, 시스템 자원 소모)이 높다.
    *   Connection Pool은 DB에 미리 여러 개의 연결 객체를 확보(Pool)해 두고, 필요한 시점에 이 객체들을 빌려(Acquire) 쓰고, 사용 후에는 끊지 않고 반납(Release)하여 재사용하는 방식이다.
*   **표준 인터페이스:**
    *   Connection Pool을 관리하는 공통의 자바 인터페이스는 `DataSource`이다.
    *   애플리케이션은 이 `DataSource` 인터페이스를 통해 연결을 요청하면, 내부적으로는 HikariCP, DBCP 같은 다양한 구현체가 관리하고 있게 된다.
*   **주요 설정:**
    *   풀 사이즈(최소/최대 연결 개수) 설정.
    *   커넥션 타임아웃 등 성능에 영향을 주는 다양한 설정을 통해 데이터베이스 연결 풀을 최적화할 수 있다.

## 6. 실습 및 다음 과정 안내
*   **실습 권장:** 배운 개념들을 바탕으로 실제로 테스트할 수 있는 미니 웹 애플리케이션(예: 게시판 CRUD)을 구축하며 전체 과정을 복습하는 것이 가장 효과적이다.
*   **결론:** 이번에 학습한 모든 개념(JDBC 연결 -> 자원 관리 -> 트랜잭션 -> Prepared Statement -> Connection Pool)을 통합하여 하나의 작동하는 애플리케이션을 완성하는 것을 목표로 한다.
