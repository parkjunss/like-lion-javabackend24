## 1. 학습 목표 및 개념 정리 (JDBC와 DBMS)
*   **DBMS의 필요성:** 애플리케이션(자바 프로그램)을 개발할 때, 사용자 정보, 게시글, 도서 정보 등 발생하는 데이터를 체계적으로 저장하고 관리할 필요가 있습니다.
*   **DBMS 역할 (CRUD):** 데이터베이스 관리 시스템(DBMS)은 데이터를 **생성(Create)**, **조회(Read)**, **수정(Update)**, **삭제(Delete)**하는 기능을 제공하여 데이터 관리를 용이하게 합니다.
*   **JDBC의 정의:** Java 프로그래밍 언어에서 DBMS와 같은 외부 데이터베이스 시스템과 통신할 수 있도록 해주는 기술 표준입니다. (JDBC Driver)
*   **JDBC의 역할:** 매번 DBMS에 접속하고 SQL 쿼리를 실행하는 과정을 프로그래밍 언어 차원에서 수행할 수 있게 도와주는 다리(Bridge) 역할을 합니다.

## 2. 데이터 영속성 확보와 JDBC의 역할
*   **프로그래밍의 한계 (휘발성):** 일반적인 자료구조(예: Java List)에 데이터를 저장할 경우, 프로그램이 종료되면 모든 데이터가 사라집니다.
*   **영속성(Persistence)의 필요성:** 애플리케이션이 종료되었다가 다시 시작해도 이전에 저장했던 데이터가 계속 남아있도록 보장하는 것이 중요합니다.
*   **DBMS의 해결책:** DBMS는 데이터가 디스크에 저장되므로, 시스템이 꺼졌다 켜져도 데이터가 영구적으로 유지되게 해줍니다.
*   **JDBC를 통한 연결:** JDBC는 자바 프로그램이 이러한 영속적인 저장 공간인 DBMS에 데이터를 저장, 조회, 수정, 삭제할 수 있도록 구체적인 구현 방법을 제공합니다.

## 3. JDBC 사용 시 핵심 기술 및 설계 원칙
*   **데이터베이스의 추상화 및 표준화:** 각 DBMS 벤더(Oracle, MySQL 등)마다 작동 방식이 다름에도 불구하고, JDBC는 표준화된 **인터페이스(Interface)**를 제공합니다.
    *   **효과:** 개발자는 특정 DBMS에 종속되지 않고, 인터페이스를 통해 코드를 작성할 수 있어 결합도를 낮추고(Low Coupling) 데이터베이스 종류가 바뀌어도 코드 수정 범위를 최소화할 수 있습니다.
*   **JDBC의 기본 동작 원리:**
    1.  데이터베이스에 **연결(Connection)**합니다.
    2.  DBMS가 이해할 수 있는 **쿼리문(Query)**을 작성합니다.
    3.  쿼리를 **실행(Execute)**하고, 필요한 결과를 얻어옵니다.
*   **결과 처리 (ResultSet):** SELECT 쿼리 등의 결과는 `ResultSet`이라는 형태로 반환되며, 결과를 가져올 때는 반드시 `next()` 메소드를 사용하여 다음 레코드로 커서를 이동시키며 처리해야 합니다.
*   **필수 예외 처리 (Exception Handling):**
    *   JDBC 코드는 연결 및 쿼리 실행 등 여러 단계에서 예외가 발생할 수 있습니다.
    *   `try-catch-finally` 구조를 사용하여 예외 발생에 대비하는 것이 필수적입니다.
    *   **리소스 관리:** 데이터베이스 자원(Connection, Statement, ResultSet)은 사용 후 반드시 닫아주는(Close) 과정이 필요합니다.
*   **자원 자동 관리 (Try-with-resources):** `try (자원선언)` 구문을 사용하면, 수동으로 `finally` 블록에서 자원을 닫아주지 않아도 자원이 자동으로 닫히게 되어 코드가 매우 간결해지고 안전합니다.

## 4. 소프트웨어 설계 패턴 및 최적화 기법 (DAO, DTO, Connection Pool)
*   **공통 코드 재사용 (패턴화):** 프로그램 개발 과정에서 반복적으로 사용되는 로직(예: 회원 정보 등록, 비밀번호 변경)이나 중복되는 코드가 생기면 구조적인 재사용이 필요합니다.
*   **DAO 패턴 (Data Access Object):**
    *   **목적:** 데이터베이스에 접근하는 모든 기능(CRUD)을 하나의 독립적인 클래스(객체)로 모아서 관리하는 패턴입니다.
    *   **이점:** 데이터베이스 접근 로직이 애플리케이션의 나머지 부분과 분리되어, 유지보수성과 응집도가 높아집니다. (예: `MemberDAO` 클래스)
*   **DTO 패턴 (Data Transfer Object):**
    *   **목적:** 데이터를 한 레이어(계층)에서 다른 레이어로 전달할 때, 데이터를 담는 용도로만 사용되는 객체입니다.
    *   **특징:** 비즈니스 로직을 포함하지 않고, 단순하게 필드(Field)와 Getter/Setter만을 가집니다. (단순히 '가방'의 역할만 수행)
*   **고급 JDBC 기능:**
    *   **자동 생성 키 활용:** 데이터베이스에 데이터를 입력(Insert)할 때, 자동으로 생성된 기본 키(Primary Key) 값을 얻어와서 다른 테이블의 외래 키(Foreign Key)로 사용하는 기능이 필요할 수 있습니다. (Prepared Statement 사용)
    *   **Connection Pool (연결 풀):** DB에 접속(Connection)하고 해제하는 과정(Open/Close)은 시간이 소요되고 자원 소모가 큰 작업입니다. Connection Pool은 미리 접속된 객체들을 풀(Pool)에 만들어두었다가, 필요할 때마다 가져와 사용하고 반납하는 방식으로 성능을 최적화합니다.
    *   **응용:** 실제 대규모 시스템에서는 Spring 같은 프레임워크가 이 Connection Pool(예: HikariCP) 기능을 기본적으로 제공하여 사용합니다.

## 1. 프로젝트 기획 및 기능 설계의 중요성
*   **간단한 시작점:** 코드를 작성하기에 앞서, 만들고자 하는 애플리케이션(예: 친구 관리 프로그램, 은행 시스템)의 핵심 기능을 먼저 정의하는 것이 중요합니다.
*   **핵심 기능 목록화:** 구현할 기능을 CRUD(Create, Read, Update, Delete) 관점에서 목록화합니다. (예: 친구 추가/조회/수정/삭제, 계좌 생성/조회/입금/출금 등)
*   **메뉴 설계:** 사용자 입장에서 자연스러운 메뉴 흐름을 설계합니다. (예: 1번 계좌 생성, 2번 전체 조회 등)
*   **구체적 기획의 단계:** 실제 코딩에 바로 돌입하기보다, 필요한 기능, 메뉴 흐름, 데이터베이스의 테이블 구조(스키마)를 먼저 종이에 작성하거나 기획하는 시간을 거쳐야 합니다.

## 2. 계층형 아키텍처 (Layered Architecture)의 이해
*   **레이어 분리 개념:** 애플리케이션을 목적에 따라 여러 개의 독립적인 레이어(층)로 나누어 구조화합니다. 하나의 클래스에서 모든 일을 처리하기보다, 각 레이어에 맡겨야 할 역할(책임)을 명확히 구분하는 것이 중요합니다.
*   **주요 레이어 구분:**
    *   **프레젠테이션 레이어 (Presentation Layer):** 사용자에게 보여지는 화면이나 메뉴 구조를 담당합니다. (예: 콘솔 메뉴, 웹 화면)
    *   **서비스 레이어 (Service Layer) / 비즈니스 레이어 (Business Layer):** 애플리케이션의 핵심 로직(비즈니스 규칙)이 존재하는 곳입니다. '무엇을 할 것인가'에 대한 판단과 처리를 담당합니다.
    *   **데이터 액세스 레이어 (DAO, Data Access Object):** 데이터베이스와의 연결 및 데이터 조작(CRUD)만 전담합니다. 데이터 자체를 어떻게 저장할지(SQL 실행 등)를 처리합니다.

## 3. 비즈니스 로직과 데이터 접근의 분리 (Service와 DAO)
*   **책임 분리 원칙:** DAO는 '데이터를 저장하는 일'만 합니다. 서비스 레이어는 '비즈니스를 수행하는 일'을 합니다. 이 둘의 경계가 명확해야 합니다.
*   **서비스 레이어의 역할 (비즈니스 로직):**
    *   친구 등록 과정처럼 단순한 데이터 저장 외에, 특정 조건(예: "친구가 되기 위해서는 재밌어야 한다", "욕설 여부를 검사해야 한다")을 검증하거나, 여러 단계의 복합적인 절차를 거쳐야 하는 **'판단'**과 **'흐름 제어'** 로직이 담당됩니다.
    *   또한, 여러 개의 데이터베이스 작업(INSERT, UPDATE, DELETE)이 하나의 업무 단위로 묶일 때, **트랜잭션(Transaction)** 처리(실패 시 롤백 등)는 서비스 레이어에서 담당해야 합니다.
*   **DAO의 역할:** 서비스 레이어로부터 필요한 데이터 객체(DTO)를 받고, 이를 데이터베이스에 저장(혹은 조회/수정/삭제)하는 *단순한 실행* 역할만 수행합니다.

## 4. 객체 간 결합도를 낮추는 방법 (Interface 활용)
*   **인터페이스 사용의 목적:** 시스템 구성 요소 간의 **결합도(Coupling)**를 낮추기 위함입니다.
*   **문제점 (높은 결합도):** A 모듈이 B 모듈의 구체적인 구현체에 의존할 경우, B 모듈이 바뀌면 A 모듈 전체를 수정해야 하는 단점이 생깁니다.
*   **해결책 (인터페이스 사용):** 서비스 레이어와 DAO 레이어 사이의 상호작용 시, 구체적인 구현체 대신 **인터페이스**를 사용합니다.
*   **장점:** 앞쪽 레이어(서비스)는 인터페이스만 바라보게 되므로, 뒤쪽 레이어(DAO 구현체)가 JDBC에서 JPA로 바뀌거나, 다른 기술로 대체되더라도 서비스 레이어의 코드를 수정할 필요가 없어 유연성이 극대화됩니다.

## 5. 개발 프로세스 및 구조화 전략
*   **개발 전 가장 중요한 단계: 기획 및 논의 (Group Discussion):**
    *   코드를 작성하기 전에, 팀원들과 충분히 아이디어를 공유하고, 어떤 기능을 구현할지, 어떤 데이터가 필요할지, 어떤 메뉴 순서로 진행할지 논의하는 과정(회고)을 충분히 거쳐야 합니다.
    *   **주의사항:** 개발 초기 단계부터 코딩에 손을 대기보다, 먼저 **설계도(Plan)**를 완성하는 데 집중해야 합니다.
*   **코드 구조화 (패키징):**
    *   프로젝트가 복잡해지면 관련된 클래스들을 패키지별로 분리하여 관리하는 것이 효율적입니다.
    *   **일반적인 권장 구조:** `common` (공통 모듈), `dao` (DAO 구현체), `dto` (데이터 전달 객체), `service` (서비스 구현체), `app` (메인 실행 클래스) 등으로 구분합니다.
    *   **유연성 확보:** 처음에는 간단한 구조로 시작해도 좋으나, 프로젝트 규모가 커질 가능성을 염두에 두고 처음부터 설계하는 것이 좋습니다.

*   **기획 > 코딩:** 개발을 시작하기 전, 기획과 설계가 가장 중요합니다.
*   **관심사 분리:** 기능별로 책임(Role)을 나누어 클래스를 설계합니다. (서비스 = 로직 판단, DAO = 데이터 저장)

## 1. 팀 프로젝트 진행 방법 및 그룹핑 조언
*   **프로젝트 아이디어 구상:** 당장 어떤 것을 만들고 싶은지 자유롭게 생각하며 주제를 정하는 것이 중요합니다.
*   **협업 방법 조언:**
    *   **팀원 구성:** 혼자 진행하기 부담스러울 경우, 짝을 이루거나 3인 이하의 소규모 팀으로 진행하는 것을 추천합니다.
    *   **협업 방식:** 비디오 콜을 켜놓고 화면 공유 기능을 활용하여 함께 코드를 작성하고 생각하는 방식으로 진행하면 효율적입니다.
    *   **협업의 이점:** 짝과 함께 작업하면 생각의 폭을 넓히고 아이디어를 더 많이 확장할 수 있습니다.

## 2. 코딩 및 AI 활용 원칙 (핵심 가이드)
*   **주도적 코딩 연습:** 코딩 과정에서는 가급적 본인이 직접 코드를 작성하려고 노력하는 것이 좋습니다.
*   **AI 활용의 범위:**
    *   **학습 목표:** AI의 도움을 받아 코드를 작성하더라도, **그 코드가 무엇을 하는지 설명할 수 있도록** 이해하는 것이 중요합니다.
    *   **지양해야 할 습관:** "이거 짜줘"처럼 기능을 통째로 AI에게 요구하고 의존하는 방식은 지양해야 합니다.
    *   **효율적인 사용법:** 본인이 특정 클래스나 기능을 구현해야 할 때(예: "여기서 기능을 만들려면 어떻게 해야 할까요?") 구체적인 도움을 요청하는 방식으로 사용하는 것이 바람직합니다.
*   **코드 재활용 및 참고:** 학습 과정에서 기존에 배웠던 코드나 다른 팀의 코드를 참고하여 활용하되, 단순히 복사-붙여넣기만 하는 것은 지양합니다.

## 3. 개인별 프로젝트 구상 및 발표
*   **구상 단계:** 단순히 주제만 정하는 것이 아니라, 첫 화면에 들어갈 메뉴 구성, 어떤 클래스를 만들지 등 **디테일한 구상**을 팀원들과 함께 이야기하며 진행해야 합니다.
*   **개별 프로젝트 사례 공유:**
    *   **민태님:** 운동 기록 조회 기능을 구현할 예정입니다.
    *   **준성님:** 상품 등록 기능 등 쇼핑몰 관련 기능을 구현할 예정입니다.
    *   **본인 (혹은 다른 팀원):** 간단한 투두리스트(To-do list)를 작성하고 관리하는 기능을 구현할 예정입니다.
*   **재확인:** 개발을 진행하면서 막히는 부분이나 자신이 어려움을 겪는 부분이 있다면, 혼자 고민하기보다 짝과 함께 화면을 열고 이야기하며 진행하는 것이 도움이 됩니다.

## 4. 마무리 및 다음 일정 안내

## 1. 과제 구상 및 목표 설정의 중요성
*   **만들기의 목적 명확화:** 프로젝트를 다른 사람에게 보여주기 위한 것이 아니라, 스스로의 실력 향상과 학습을 목적으로 해야 합니다.
*   **난이도 조절:** 처음에는 복잡하게 만들 필요가 없으며, 정말 간단한 아이디어나 기능을 가지고 시작해도 무방합니다.
*   **접근 방식:** 당장 뭘 해야 할지 감이 오지 않는 경우, 강사(혹은 진행자)와 함께 고민하며 작은 것부터 구상을 시작하는 것이 좋습니다.

## 2. 학습 원칙: 코딩 과정과 협업 방법
*   **AI 활용의 주의점 (핵심 강조):**
    *   요구하는 기능을 AI에게 전체적으로 "만들어줘"라고 요청하는 것은 학습에 도움이 되지 않습니다.
    *   문법이나 특정 로직을 질문할 때도, 단순히 코드를 얻는 것에만 의존하기보다는 스스로 구상을 하고 원리를 파악하려는 노력이 중요합니다.
*   **효율적인 학습 방식 (관찰 지양):**
    *   팀원 중 가장 잘하는 사람이 혼자 코딩하는 것을 옆에서 지켜보는 것만으로는 실력이 늘지 않습니다.
    *   **최적의 방법은 '페어 프로그래밍(Pair Programming)'**과 같이, 2~3명이 함께 스크린을 공유하며 논의하고 역할을 분담하며 코드를 작성하는 방식입니다.
    *   서로의 코드와 논리를 공유하며 생각하는 과정 자체가 가장 큰 도움이 됩니다.

## 3. 실습 진행 및 팀 구성 가이드
*   **팀워크 방향:**
    *   혼자 프로젝트를 진행하는 것도 좋고, 2인 또는 3인 팀을 구성하여 진행하는 것도 모두 좋은 방법입니다.
    *   가장 중요한 것은 '함께 논의하며 배우는 과정' 자체에 집중하는 것입니다.
*   **실습 진행 (현장 지침):**
    *   (선정된 그룹) 참여 인원별로 별도의 비디오 콜(Video Call)을 개설하여 진행합니다.
    *   페어 프로그래밍을 위해 화면 공유를 주기적으로 번갈아가며 진행하며, 진행 과정 중에는 자유롭게 의견을 주고받도록 합니다.
    *   **핵심 요약:** 과정에 대한 이해가 가장 중요하므로, 부담 없이 쉬운 기능을 정하여 실습에 임할 것을 재강조했습니다.
