### 🔑 **핵심 개념 5가지**  
1. **`main` 메서드 구조**  
   ```java
   public static void main(String[] args) {
       // args[0] = 첫 번째 인수, args[1] = 두 번째 인수...
   }
   ```
   - `args`는 **문자열 배열** (`String[]`) → 실행 시 전달된 값들이 순서대로 저장됨.

2. **인수는 **항상 문자열**로 들어온다!**  
   - `80` → `"80"`, `A` → `"A"`로 인식됨 → **숫자로 변환 필수**  
   - **변환 방법**: `Integer.parseInt(args[0])` (숫자 문자열 → `int`)

3. **실행 시 인수 전달 방법 (IntelliJ 예시)**  
   - `Run` → `Edit Configurations` → `Application` → `Program arguments`에 값 입력  
     (예: `80 100 A B` → `args[0]="80"`, `args[1]="100"`, `args[2]="A"`, `args[3]="B"`)

4. **변경해도 코드는 안 바뀌고, 인수만 바꾸면 동작 달라짐**  
   ```java
   // args[0] = "80" → "pass" (통과)
   // args[0] = "90" → "fail" (불통과)
   ```
   → 프로그램을 수정하지 않고도 **값만 바꾸면 다른 결과**를 얻을 수 있음.

5. **주의사항**  
   - `args[0]`을 가져오면 **인덱스 오류** 발생 가능 (인수가 없을 때)  
     → `if (args.length > 0)`로 체크 필수  
   - 숫자 변환 시 **예외 처리** 필요 (`NumberFormatException` 발생 가능)  
     ```java
     try {
         int num = Integer.parseInt(args[0]);
     } catch (NumberFormatException e) {
         System.out.println("숫자만 입력하세요!");
     }
     ```

## 1️⃣ 오늘 강의에서 핵심적으로 언급된 **조건문** 정리  

| 구분 | 언제 쓰면 좋은가? | 대표적인 문법 (Java 기준) | 주의해야 할 점 |
|------|-------------------|---------------------------|----------------|
| **if‑else** | *조건이 복합적이고 순서가 중요한 경우* (예: “점수가 90점 이상이면 A, 80점 이상이면 B …”) | ```java<br>if (cond1) { … } else if (cond2) { … } else { … }``` | `else if`를 과도하게 쌓으면 가독성이 떨어짐 → **switch** 로 옮기기 |
| **switch** | *같은 연산자를 여러 **값**에 적용하고 싶을 때* (예: 메뉴 번호에 따라 메뉴 이름 출력) | ```java<br>switch (value) {<br>    case 1: … break;<br>    case 2: … break;<br>    default: …<br>}``` | *break* 를 빠뜨리면 **fall‑through** 가 발생 → 실수로 여러 케이스를 실행할 위험 |
| **ternary (조건 연산자)** | *간단한 “값 선택”을 한 줄에 쓰고 싶을 때* (예: `max = a > b ? a : b;`) | `result = (cond) ? exprIfTrue : exprIfFalse;` | 복잡한 로직을 한 줄에 넣으면 가독성이 급락 → **가능하면 if‑else** 로 풀어라 |
| **switch‑expressions (Java 14+)** | *switch 문을 값으로 바로 반환하고 싶을 때* (값 반환을 바로 지원) | ```java<br>int result = switch (day) {<br>    case MON, TUE, WED, THU, FRI -> 5;<br>    case SAT, SUN -> 6;<br>};``` | 새 문법은 아직 **IDE**와 **빌드 도구**가 최신 버전이어야 사용 가능 |

> **핵심 포인트**  
> - **조건이 *많고* 복합적이면 `if‑else`** → 순서를 자유롭게 바꿀 수 있다.  
> - **값이 “같은 연산자”에 대해 여러 경우에 걸쳐 달라질 때** → `switch` (특히 `switch‑expression` 사용하면 더 깔끔).  
> - **한 줄에 “값을 고르기”만 필요하면 `?:`** – 너무 복잡해지면 즉시 `if` 로 전환.

## 2️⃣ 실습 예시 – “두 정수와 연산자를 입력받아 결과를 출력하는 작은 계산기”

아래 코드는 **조건문을 여러 가지**로 활용해 보는 연습용 프로그램입니다.

```java
import java.util.Scanner;

public class SimpleCalculator {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        System.out.print("첫 번째 정수: ");
        int a = sc.nextInt();
        System.out.print("두 번째 정수: ");
        int b = sc.nextInt();
        System.out.print("연산자 (+, -, *, /, %): ");
        String op = sc.next();

        // 1️⃣ switch‑expression 사용 (Java 14+)
        int result = switch (op) {
            case "+" -> a + b;
            case "-" -> a - b;
            case "*" -> a * b;
            case "/" -> {
                if (b == 0) {               // Divisão por zero 방어
                    yield "Error: divide by zero";
                } else {
                    yield a / b;
                }
            }
            case "%" -> {
                if (b == 0) {
                    yield "Error: modulo by zero";
                } else {
                    yield a % b;
                }
            }
            default -> "Error: unknown operator";
        };

        System.out.println("결과: " + result);
        sc.close();
    }
}
```

### 코드 포인트

| 라인 | 왜 이런 구문을 썼을까? |
|------|------------------------|
| `switch (op) { … }` (값 기반) | 연산자를 **값**으로 직접 스위치하고, 연산별 결과를 한 줄에 `->` 로 반환 → 가독성 ★★★★★ |
| `if (b == 0)` 내부에 `yield` | **나눗셈**은 **예외** 처리가 필요함. `switch` 안에서 직접 `if` 로 방어코드를 넣어야 함. |
| `default` → 오류 메시지 | 사용자가 예상치 못한 문자(예: `x`)를 줬을 때 **안전한** fallback 제공 |

> **연습 포인트**  
> 1. `switch` 를 **값 반환형**(`->`) 형태로 쓰면 `break` 를 잊어도 오류가 나지 않는다 (fall‑through 방지).  
> 2. **복잡한 연산**은 `case` 안에 일반 `if` 를 넣어 **조건 검증**을 별도 처리한다.  
> 3. `String` 변수를 직접 `switch` 에 넘겨도 OK (Java 12+ 부터 `switch` 에 문자열 허용).

## 3️⃣ 언제 **`switch`** 를 쓰면 좋은가? – 실전 체크리스트

| 상황 | 추천 `switch` (or switch‑expression) |
|------|----------------------------------------|
| **값이 몇 개 정해진 경우** (예: `enum` 상태, `int` 1~12) | `switch` – 가독성 ↑, `enum` + `switch` 은 컴파일러가 `case` 누락 시 **경고**를 내준다. |
| **동일한 연산을 여러 입력에 적용** (예: 메뉴 코드 → 가격) | `switch` – `case`마다 **반환값**을 다르게 지정하면 한 줄로 구현 가능. |
| **다양한 객체 유형을 구분** (예: `Object` instanceof 체크) | `instanceof` + `switch` (Java 16+) – `switch` 에서 **패턴 매칭**을 지원 → `case String s -> …` 등. |
| **조건이 3~5개 정도** (조건 순서가 중요하고, **동일한 연산**이 아니라면) | `if‑else` → 조건이 서로 겹치지 않게 관리하기 쉬움. |
| **조건이 10개 이상** 그리고 **값 비교만** 하면 | `switch` → **가독성**이 급격히 떨어지는 `else if` 체인을 피함. |

> **Tip**: `switch` 를 사용할 때는 **항상 `break`** 를 넣을지, 혹은 **`switch`‑expression** (값 반환) 형태를 취할지를 미리 결정한다. `break` 를 깜빡이면 **알 수 없는 실행 흐름**이 생겨 디버깅이 힘들다.

## 4️⃣ 프로그래밍 연습 팁 (조건문 마스터하기)

1. **“조건문 1개, 2개, 3개 …” 로 점진적으로 늘리기**  
   - 단계 1: `if‑else` 로 간단히 `if (a > b) … else …` 구현.  
   - 단계 2: 같은 로직을 **값**을 바꾸어 `switch` 로 변환.  
   - 단계 3: 복합 조건은 **논리 연산자**(`&&`, `||`) 로 추가하고, `switch` 로 재구성.

2. **“조건문을 읽어보라”**  
   - 실제 코드(예: Spring MVC 컨트롤러 메서드)에서 `if (status == 200) … else if (status == 404) …` 와 같은 구문을 찾아서 **왜 `switch` 로 바꾸지 않았는가**를 고민한다.  
   - `switch` 로 바꿀 수 있는지를 직접 체크해 본다.

3. **“패턴 매칭”** (Java 17+)  
   - `switch` 에서 **`case List<Integer> l -> …`** 와 같이 **구조적 패턴**을 활용하면, 단순 값이 아니라 **데이터 구조**까지 한 번에 검사할 수 있다.  
   - 처음엔 `instanceof` 와 `instanceof` 조합이 복잡해 보이지만, **실제 프로젝트**에서 오브젝트 타입을 나눌 때 유용함.

4. **오류 로그를 남겨라**  
   - 조건 검증이 실패했을 때 **왜** 실패했는지 로그를 남겨두면 디버깅이 훨씬 쉬워진다.  
   - 예: `if (b == 0) logger.warn("Division by zero: a={}, b={}", a, b);`

5. **리팩터링 연습**  
   - `if (cond1) { … } else if (cond2) { … }` 를 **함수** 로 분리하고, **조건값**을 변수에 담아 `switch` 로 바꾸어 본다.  
   - 예: `int state = (cond1) ? 1 : (cond2) ? 2 : 0; switch(state) { … }`

## 5️⃣ 다음에 해볼 **미니 프로젝트** (조건문 활용)

| 프로젝트 아이디어 | 구현에 필요한 주요 조건문 | 기대 효과 |
|-------------------|---------------------------|------------|
| **날씨 기반의상 추천기** <br>입력: 온도, 강수 여부, 바람세기 → 옷 (코트, 가디건, 양복, 후드) 추천 | - 온도 구간(`if (temp < 0)` … `else if (temp < 15)` …) <br>- `switch (rain)` 로 강수 효과 적용 | **다중 조건**을 자연스럽게 조합하고, `enum` 로 날씨 상태를 정의하면 더 깔끔해짐. |
| **주식 투자 시뮬레이터** <br>입력: 투자 등급(초보, 중급, 전문가) → 매수 전략(보수적, 중립적, 공격적) | - `switch` 로 등급에 따라 **다른 전략 로직** 호출 <br>- 전략 로직마다 `if` 로 **일일 변동** 검증 | `switch` + `enum` 으로 **전략을 캡슐화**하고, 전략마다 별도 클래스를 만들면 OOP와 조건문을 연결할 수 있다. |
| **단어 검사기** <br>입력: 영어 단어 → 대소문자 구분, 특정 접두사, 길이에 따라 출력 | - `switch` 로 단어 길이(`case "short": …`) <br>- `if` 로 **대소문자 검사** (`startsWithIgnoreCase`) | 짧은 문자열이면 `switch`, **다양한 문자열 검사**는 `if` 로 분리. |
| **간단한 “명령어 인터프리터”** <br>입력: `move north`, `move south`, `pick up 1`, `drop 2` → 해당 동작 실행 | - 문자열을 `split` 후 **명령어**를 `switch` 로 분기 <br>- 좌표값 업데이트는 `if (x >= 0)` 로 경계 검사 | **명령 파싱**과 **조건 검사**를 한 번에 연습. `switch` 에서 `case "move": …` 안에 **좌표 검증**을 `if` 로 넣는다. |

> **목표**: *조건문을 바꾸는 과정에서 “왜 `if` 로 두는가?” 혹은 “왜 `switch` 로 바꾸는가?”를 스스로 답할 수 있게** 된다면, 어느 정도 “조건문 마스터” 수준에 도달한 것입니다.

## 6️⃣ 마무리 한 마디

> “조건문은 **‘무엇을 할지’** 를 정하는 **‘문’이 아니라 ‘**조건**’을 **‘전환점’** 으로 만든다”는 생각을 갖고 코드를 살펴보세요.  
> - `if‑else` → **조건이 복잡하고 흐름이 변할 때**  
> - `switch` → **동일한 연산에 여러 값**을 한 번에 처리하고 싶을 때  
> - `?:` → **값 선택이 단순할 때**  

조건을 **깨끗하게 정렬**하고, **불필요한 중복을 없애**면 코드가 길어지는 만큼 유지보수가 쉬워집니다.  

궁금한 점이 있으면 언제든 물어보세요! 🚀
