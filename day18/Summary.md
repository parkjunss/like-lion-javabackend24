## 1. 미니 프로젝트 진행 및 학습 태도 점검
*   **핵심 목표**: 프로젝트의 완성도보다도 **'실행'하는 과정** 자체가 중요하다.
*   **최소 기능 구현의 중요성**: 완벽하고 복잡한 구조를 만들기보다, 일단 **CRUD(생성-조회-수정-삭제)** 기능 등 최소한의 기능이라도 동작하는 것을 만드는 것이 가장 훌륭하다.
*   **학습 접근법**:
    *   **실행 기반 학습**: 이론만 공부하기보다, 직접 코드를 만드면서 경험하고 부딪히는 과정을 통해 지식이 습득되어야 한다.
    *   **반복과 확장**: 일단 가장 단순하게 만들어 본 후, 이 경험을 바탕으로 코드를 반복적으로 수정(Refactoring)하고 기능을 점진적으로 추가하며 발전시켜 나가는 것이 중요하다.
    *   **시간 관리**: 학습에 필요한 시간 할애를 우선순위에 두는 것이 가장 중요하며, 학습은 과정과 분리되어 지속적으로 진행되어야 한다.

## 2. 코드 구조 설계 원칙 (DAO와 서비스 계층)
*   **레이어드 아키텍처(Layered Architecture)**: 소프트웨어를 역할을 분리하여 설계하는 방식이다.
*   **객체의 책임 분리 (Separation of Concerns)**:
    *   하나의 객체에 너무 많은 기능을 몰아주는 것은 바람직하지 않다. 각 객체는 자신이 가장 잘하고, 맡은 업무만 수행하도록 역할을 분리해야 한다.
    *   **서비스 계층 (Service)**: 핵심 비즈니스 로직(Business Logic)을 담당하는 곳. 실제 '어떤 조건'으로 기능을 수행할지, '어떻게' 처리할지에 대한 규칙과 로직을 정의한다.
    *   **DAO (Data Access Object)**: 데이터베이스와의 통신만을 담당하는 계층. 단순하게 데이터를 저장하거나(Save), 읽거나(Retrieve), 수정하거나(Update), 삭제하는(Delete) 기능 등 **순수 데이터 조작** 역할에 국한되어야 한다.
*   **코드 설계 시 유의점**:
    *   **일단은 코딩에 집중**: 처음에 복잡한 구조나 완벽한 메서드 분리를 고민하기보다, 일단 동작하게 코드를 쭉 작성하는 데 집중한다.
    *   **개선은 나중에**: 코드가 길어지거나, 반복되는 로직이 눈에 띌 때, 그때 고민하여 기능을 별도의 메서드로 분리하거나 구조를 개선하는 것이 효율적이다.

## 3. Java 입출력 (I/O) 이해와 스트림 개념
*   **I/O의 정의**: 프로그램의 관점에서 데이터를 처리하는 과정을 말한다.
    *   **Input (입력)**: 외부에서 프로그램 안으로 흘러 들어오는 데이터 (예: 사용자 키보드 입력).
    *   **Output (출력)**: 프로그램 안에서 외부로 흘러 나가는 데이터 (예: 콘솔 출력).
*   **스트림(Stream) 개념**:
    *   스트림은 데이터의 형태(텍스트, 이미지, 동영상, 소리 등)와 무관하게, 데이터를 일정한 **흐름(Flow)**으로 취급하는 추상화된 개념이다.
    *   데이터는 한 방향으로 흘러가기 때문에, 스트림은 일반적으로 한 방향성(One-way)을 가진다.
    *   이러한 흐름을 다루기 위해 자바는 모든 종류의 데이터를 스트림 형태로 통일하여 처리할 수 있게 한다.
*   **I/O의 특징**:
    *   데이터가 입력되면(Input Stream) 처리되고, 반드시 출력을 위한 통로(Output Stream)가 필요하다.
    *   I/O API를 살펴볼 때, 클래스 이름의 접미사(`*In`, `*Out`)와 같은 규칙을 통해 해당 클래스가 어떤 기능을 담당하는지 직관적으로 파악하는 것이 좋다.
    *   `*InputStream` 계열은 입력 기능을, `*OutputStream` 계열은 출력 기능을 담당한다.
