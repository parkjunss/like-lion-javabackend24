## 1. 데이터베이스를 학습하는 동기 및 중요성
*   **데이터 증가의 시대:** 프로그래밍을 통해 개발되는 모든 애플리케이션(웹페이지, SNS 등)은 막대한 양의 데이터를 발생시키며, 이러한 데이터를 효율적으로 관리하는 것이 핵심 과제입니다.
*   **시스템의 중요성:** 데이터가 어떻게 관리되느냐에 따라 정보력과 영향력이 결정되며, 이는 기업의 존폐와 직결되는 중요한 자원입니다.
*   **역사적 배경:** 데이터베이스의 필요성은 어플리케이션 개발 초창기부터 꾸준히 고민되어 온 문제였으며, 단순히 파일에 저장하는 방식으로는 한계가 있어 점진적으로 발전해왔습니다.

## 2. 데이터베이스의 기본 개념 및 역사
*   **데이터베이스(DB)의 정의:** 체계적으로 모아진 데이터의 집합체입니다.
*   **관계형 DB(Relational DB)의 등장 배경:** 데이터를 표(Table) 형태로 관리하는 것이 가장 효과적이라는 인식에서 시작된 방식입니다.
*   **데이터 모델링의 핵심:** 데이터를 저장할 때 **테이블(Table)**을 사용하며, 테이블은 여러 개의 **컬럼(Column)**으로 구성됩니다.
*   **DBMS의 역할 및 중요성:**
    *   데이터의 무분별한 저장을 막기 위해 컬럼 정의 시 **데이터 타입과 제약 조건(Constraint)**을 설정할 수 있게 합니다.
    *   DBMS는 이러한 제약 조건을 활용하여 데이터가 효율적이고 일관성 있게 관리되도록 보장합니다.

## 3. 데이터베이스 접근 및 개발 환경 구축 (기술적 관점)
*   **DBMS 학습 목표:** 현재는 관계형 DB를 기준으로 학습하며, 주로 MySQL을 사용합니다. (다른 DBMS와 개념은 유사하여 학습에 도움이 됩니다.)
*   **개발 환경 구축의 어려움:** 데이터베이스는 시스템 레지스트리 등에 복잡하게 영향을 미치기 때문에, 일반 개인 컴퓨터에 설치하고 운영하기 까다롭습니다.
*   **가상 환경의 필요성:**
    *   **Virtual Machine (VM):** 기존에 복잡한 시스템 충돌 문제를 해결하는 데 사용되었으나, 무거운 경향이 있습니다.
    *   **Docker와 컨테이너:** VM보다 훨씬 가볍고 효율적으로 격리된 환경을 만들 수 있습니다. 애플리케이션을 어떤 환경에든 동일하게 배포하고 테스트하는 데 최적의 도구입니다.
*   **Docker/Docker Compose 활용:**
    *   **Docker:** 컨테이너라는 독립적인 가상 환경을 제공합니다.
    *   **Docker Compose:** 여러 개의 컨테이너(예: MySQL 데이터베이스 컨테이너와 애플리케이션 컨테이너)를 하나의 정의 파일로 관리하고, 복잡한 초기 설정을 자동화(예: 초기 비밀번호, 사용자 및 DB 생성)할 수 있습니다.
*   **데이터베이스 접근 방법:**
    *   **터미널 접근:** `isql` 명령을 통해 사용자 ID와 비밀번호를 입력하여 접근합니다.
    *   **GUI 도구 (Workbench):** 코드를 작성하고 결과를 확인하는 것이 훨씬 편리하여, MySQL Workbench와 같은 전용 편집기를 사용하는 것이 권장됩니다.
    *   **접속 정보의 이해 (IP, Port, User):**
        *   **Host Name (IP):** 로컬(`127.0.0.1`) 외에 외부 네트워크(AWS 등)의 IP 주소 사용이 가능합니다.
        *   **Port:** 데이터베이스 서버가 동작하는 포트 번호를 지정합니다. (MySQL의 기본값은 3306입니다.)
        *   **사용자/권한:** 접속 시 설정한 사용자 계정(권한)을 사용하며, 관리자 권한(Root)을 직접 사용하는 것은 보안상 지양하고, 적절히 제한된 사용자를 만들어 사용하는 것이 원칙입니다.

## 4. 데이터 보안 및 권한 관리 원칙
*   **사용자 계정 생성의 원칙:** 모든 사용자는 최소한의 권한만을 가진 전용 계정으로 생성되어야 합니다.
*   **접근 제어 (Host Restriction):** 사용자가 특정 IP 주소(Host)에서만 접속하도록 제한할 수 있습니다. 이는 외부로부터의 무단 접근을 차단하여 보안을 강화합니다.
*   **권한 부여 (GRANT):**
    *   사용자는 기본적으로 아무 권한도 갖고 있지 않으므로, 필요한 권한(SELECT, INSERT, UPDATE, DELETE 등)을 부여해야 합니다.
    *   **`GRANT`** 명령어를 사용하여 구체적인 권한(어떤 리소스에 대한 어떤 작업)만 부여하며, 모든 권한을 주는 것은 매우 위험합니다.
    *   **`FLUSH PRIVILEGES`:** 권한 설정 변경 사항을 즉시 DBMS에 반영하기 위해 사용합니다.

## 5. 핵심 SQL 조회 구문 (SELECT 문)
*   **SELECT 문의 기본 구조:** 원하는 정보를 조회하는 가장 기본적인 구문입니다.
*   **주요 구문 및 기능:**
    *   **`SELECT * FROM 테이블명`:** 모든 컬럼의 내용을 조회합니다.
    *   **`SELECT 컬럼1, 컬럼2 FROM 테이블명`:** 특정 컬럼만 선택합니다.
    *   **`DISTINCT`:** 결과 값에서 중복된 항목을 제거하고 고유한 값만 조회할 때 사용합니다.
    *   **`AS` (Alias):** 컬럼 이름의 별칭을 부여하여 결과를 보기 쉽게 가공합니다.
    *   **`WHERE`:** 데이터 조회 조건(필터링)을 설정합니다. (예: 부서 번호가 10번인 사람만 조회)
    *   **`ORDER BY`:** 조회된 데이터를 특정 컬럼을 기준으로 오름차순 또는 내림차순으로 정렬합니다.
*   **쿼리 작성 시 유의사항:**
    *   컬럼명이나 테이블명은 전적으로 **팀 차원의 규칙**을 정하여 일관성 있게 사용하는 것이 가장 중요합니다. (언어의 표준이 항상 정답일 수는 없습니다.)

## 1. SQL 학습 환경 및 기초 문법 복습 (논리 연산자)
- **SQL 학습 환경:** 프로그래머스, W3schools.com 등 온라인 사이트를 활용하여 데이터베이스 설치 없이도 직접 SQL 쿼리문을 작성하고 결과를 확인하며 실습할 수 있다.
- **기본 문법 복습:**
    - `AND` (논리곱): 두 조건이 모두 참일 때만 만족한다.
    - `OR` (논리합): 두 조건 중 하나라도 참이면 만족한다.
    - **복합 조건문:** 여러 개의 조건을 묶어 사용하며, 이전에 학습한 조건식(크다, 작다 등)과 논리 연산자를 조합하여 복잡한 조건을 만들 수 있다.

## 2. IN 및 BETWEEN 연산자 활용
- **IN 연산자:**
    - 여러 개의 값을 지정하여, 특정 컬럼의 값이 주어진 목록 중 하나에 속하는 행만 조회할 때 사용한다.
    - `dept_number IN (10, 20)`와 같이 사용하며, 이 경우 `OR` 연산자를 여러 번 사용하는 것과 동일한 결과를 가져온다.
    - `NOT IN`을 사용하면 주어진 목록에 포함되지 않는 값만 조회할 수 있다.
- **BETWEEN AND 연산자:**
    - 특정 컬럼의 값이 지정된 범위(Start 값과 End 값 사이)에 속하는 행을 조회할 때 사용한다.
    - **주의사항:** `BETWEEN` 연산자를 사용할 때는 반드시 **작은 값이 먼저, 큰 값이 나중**에 오도록 순서를 지켜야 올바른 결과를 얻을 수 있다.
    - 날짜(Date) 데이터나 숫자(Numeric) 데이터 등 다양한 데이터 타입에 적용 가능하다.

## 3. LIKE 연산자와 와일드카드 사용법
- **LIKE 연산자:**
    - 특정 문자열과 완전히 일치하는 값만 찾는 것이 아니라, 패턴에 맞는 값을 검색하고 싶을 때 사용한다. (예: 'S'로 시작하는 이름 검색)
- **와일드카드(Wildcard) 문자와의 활용:**
    - **퍼센트 (`%`):** 0개 이상의 임의의 문자를 대체할 수 있다. (가장 흔하게 사용됨)
    - **언더바 (`_`):** 정확히 한 개의 임의의 문자를 대체한다.
    - **활용 예시:**
        - `LIKE 'S%'`: 'S'로 시작하는 모든 문자열을 찾는다.
        - `LIKE 'S_%'`: 'S'로 시작하고 두 번째 글자가 반드시 존재하는 문자열을 찾는다.
        - **팁:** 패턴 검색 시, 뒤에 값이 있을지 없을지 모르는 경우 `%`를 사용하고, 글자의 개수나 구조가 중요한 경우 `_`를 사용한다.

## 4. NULL 값 처리 및 함수 활용
- **NULL 값의 이해:** `NULL`은 알 수 없는 값, 또는 값이 비어 있음을 의미한다.
- **NULL 비교의 특수성:** SQL에서 `NULL`은 일반적인 비교 연산자(`=`, `!=`)로 비교할 수 없다.
- **NULL 조회 방법:** 반드시 `IS NULL` 또는 `IS NOT NULL` 구문을 사용해야 한다.
- **NULL 연산 시 문제:** `NULL` 값과 숫자나 다른 데이터 타입을 연산할 경우, 결과 값 자체가 `NULL`이 되어 연산 결과가 무효화된다.
- **NULL 처리 함수:** 연산에 사용되는 `NULL` 값을 특정 기본값(Default Value)으로 대체하여 연산을 가능하게 하는 함수가 필요하다.
    - 예: `IFNULL` (또는 DBMS별 함수 `COALESCE`, `NVL` 등)을 사용하여, 만약 특정 컬럼이 `NULL`이면 0이나 10 같은 값을 대신 사용하도록 지정할 수 있다.

## 5. 데이터 정렬 및 결과 제한 (ORDER BY, LIMIT)
- **ORDER BY (정렬):**
    - 쿼리 결과를 특정 컬럼을 기준으로 정렬할 때 사용한다.
    - **다중 정렬:** 여러 개의 기준 컬럼을 지정할 수 있으며, 첫 번째 기준과 같은 값이 나오면 두 번째 기준으로 정렬하는 방식(순차적 정렬)으로 작동한다.
    - **정렬 방향:** 기본적으로 오름차순(`ASC`)이며, 내림차순은 `DESC` 키워드를 사용한다.
    - **Alias 사용:** 컬럼 별칭(Alias)을 이용해 정렬 기준을 지정할 수 있다.
- **LIMIT (결과 제한):**
    - 한 번에 조회할 데이터의 최대 개수를 제한할 때 사용한다.
    - **필요성:** 전체 데이터가 너무 많을 경우, 한 번에 모든 데이터를 가져오는 것은 시스템 부하를 유발하므로, 필요한 만큼만(페이지 단위로) 데이터를 가져오기 위해 사용한다.
    - **페이징 처리:** 데이터베이스는 내부적으로 순번(번호)을 매겨 순차적으로 데이터를 가져오는 원리를 이용하며, 이는 성능 최적화에 필수적이다.

## 6. SQL 함수 및 데이터 타입 개요
- **함수의 학습:** SQL은 데이터 조작을 넘어 다양한 기능을 수행할 수 있는 내장 함수들을 제공한다.
- **함수의 분류:** 함수들은 주로 **문자열 함수, 숫자 함수, 날짜/시간 함수** 등 기능별로 분류되어 있다.
- **학습 방법:** 각 DBMS(데이터베이스 관리 시스템)별 공식 문서를 참조하여 어떤 타입의 데이터가 어떤 함수와 연산(Operation)이 가능한지 확인하는 것이 중요하다.

## 1. SQL 함수 사용의 이해 및 실습 (String, Math, Date 함수)
- **SQL 함수 개요:** SQL에서는 데이터 조회를 위해 다양한 내장 함수(String, Numeric, Date 등)를 제공하며, 이 함수들은 `SELECT` 절이나 `WHERE` 절 등 원하는 곳에서 사용될 수 있습니다.
- **데이터 전처리 함수:**
    *   `TRIM()`: 문자열의 **양쪽** 공백을 제거합니다. (사용자 입력 데이터에서 의도치 않은 공백이 포함되는 경우 데이터 비교 시 오류를 방지하는 데 유용합니다.)
    *   `LOWER()`, `UPPER()`: 문자열을 소문자 또는 대문자로 통일하여 검색의 일관성을 높입니다. (데이터베이스 시스템에 따라 대소문자를 구분하는 경우가 있으므로, 검색 전에 통일하는 것이 권장됩니다.)
    *   `REPLACE()`: 특정 문자열을 다른 문자열로 대체할 때 사용합니다.
- **패딩 및 포맷팅 함수:**
    *   `LPAD()`, `RPAD()`: 특정 컬럼의 값이 정해진 길이(예: 10자)에 미달할 경우, 왼쪽 또는 오른쪽 공백을 특정 문자(예: 별표 `*`)로 채워 넣을 때 사용합니다.
- **날짜 및 숫자 함수:**
    *   **날짜 함수:** 현재 날짜 및 시간을 확인하거나(`NOW()`), 날짜를 특정 형식(Format)으로 변환하거나(`DATE_FORMAT`), 날짜 간의 기간 계산(예: 한 달 후 날짜 계산)을 할 수 있습니다.
    *   **수학 함수:** 숫자를 다루는 함수들(예: `FLOOR()`: 내림, `CEIL()`: 올림)을 사용하여 수학적인 연산 결과를 얻을 수 있습니다.
- **학습 가이드:** 함수 사용법을 익힐 때는 DBMS 공식 문서(Documentation)를 참조하는 것이 가장 중요하며, 실제 샘플 예제들을 통해 다양한 활용 방법을 숙지하는 것이 좋습니다.

## 2. 집계 함수(Aggregate Function)와 그룹핑
- **함수 유형 구분:** SQL 함수는 처리하는 레코드의 개수에 따라 크게 두 가지로 나뉩니다.
    *   **단일 행 함수 (Single-Row Function):** 레코드 **하나당** 하나의 결과값을 반환하는 함수입니다. (예: `LOWER()`, `TRIM()`)
    *   **그룹 함수 (Group Function):** 여러 레코드(행)의 데이터를 처리하여 **하나의 요약 결과값**을 도출해내는 함수입니다. (예: `COUNT()`, `SUM()`, `AVG()`, `MAX()`, `MIN()`)
- **집계 함수 사용 예시:**
    *   `COUNT(*)`: 전체 행의 개수를 세는 데 사용합니다.
    *   `COUNT(컬럼명)`: 해당 컬럼에 값이 존재하는 행의 개수를 셉니다. (NULL 값은 개수에서 제외됩니다.)
    *   `SUM()`: 특정 컬럼의 합계를 구합니다.
- **GROUP BY 절 (그룹핑):**
    *   데이터를 특정 기준(예: 부서별, 직업별)에 따라 묶어 그룹으로 나누어 집계 함수를 적용할 때 사용합니다.
    *   그룹 함수를 일반 컬럼과 함께 사용하려면, 반드시 그룹핑의 기준이 되는 컬럼들을 명시해야 합니다. (예: `SELECT dept_number, COUNT(*) FROM EMP GROUP BY dept_number`)

## 3. 데이터 필터링 조건: WHERE vs. HAVING
- **데이터 필터링의 원리 이해:** SQL이 쿼리를 수행할 때, 우리가 생각하는 것처럼 한 번에 결과를 추출하는 것이 아니라 **모든 행을 한 줄씩** 검사하며 조건을 적용한다는 원리를 이해하는 것이 매우 중요합니다.
- **WHERE 절 (Filtering Rows):**
    *   **실행 시점:** 그룹핑(`GROUP BY`)이 **진행되기 전**, 전체 데이터 레코드 중에서 특정 조건에 맞지 않는 행을 걸러낼 때 사용합니다.
    *   **사용 목적:** 데이터를 그룹핑하기 전에 기본적인 조건을 걸러내고 싶을 때 사용합니다.
- **HAVING 절 (Filtering Groups):**
    *   **실행 시점:** `GROUP BY`를 통해 그룹핑이 **완료된 후**, 그룹 전체의 결과값(집계값)을 기준으로 필터링할 때 사용합니다.
    *   **사용 목적:** 그룹 함수(`SUM`, `AVG` 등)의 결과가 특정 조건을 만족하는 그룹만 보고 싶을 때 사용합니다.
    *   **주의:** `HAVING` 절은 집계된 값을 기준으로 조건을 걸러내는 역할을 하며, `WHERE`와는 실행되는 시점과 대상이 다릅니다. (예: "평균 급여가 10,000 이상인 부서만 보고 싶을 때" `HAVING` 사용)

**[정리 요약]**
*   SQL은 함수를 사용하여 데이터를 전처리하거나 원하는 방식으로 변형할 수 있습니다.
*   데이터를 집계할 때는 `GROUP BY`를 통해 그룹을 나눈 후, `COUNT`, `SUM`, `AVG` 등의 집계 함수를 사용합니다.
*   조건 필터링은 단계별로 구분됩니다. 개별 행을 필터링할 때는 `WHERE`를, 그룹 결과를 필터링할 때는 `HAVING`을 사용해야 합니다.
*   SQL은 반복적인 실습과 공식 문서 참조를 통해 익숙해지며, 모든 문법을 이해하는 것보다 동작 원리(어떤 순서로 실행되는지)를 이해하는 것이 중요합니다.

## 1. SQL의 개념 및 기본 구문
- **SQL 정의**: SQL(Structured Query Language)은 데이터를 요청하고 조작할 때 사용하는 일종의 '약속된 언어'입니다. DBMS(Database Management System)에게 특정 요청(예: 데이터를 조회해 줘)을 할 때 사용하는 언어입니다.
- **SQL 실행 과정**: 사용자가 SQL 문장을 실행하면, DBMS는 이를 사람이 이해하는 언어로 해석하는 '파싱(Parsing)' 과정을 거칩니다. (이는 프로그래밍에서 컴파일하는 과정과 유사합니다.)
- **기본 조회 구문**:
    - 모든 조회는 `SELECT` 키워드로 시작합니다.
    - `SELECT` 뒤에 조회하고자 하는 컬럼 이름을 나열하고, `FROM` 예약어 뒤에 테이블명을 지정합니다.
    - 여러 개의 테이블에서 데이터를 조회할 경우, 여러 테이블명을 사용할 수 있습니다.

## 2. 데이터 필터링 및 비교 연산자 심화
- **WHERE 절 (Row Filtering)**:
    - 테이블의 모든 데이터에서 원하는 조건에 맞는 데이터만 골라낼 때 사용합니다.
    - **조건 비교**: `=` (같음), `!=` 또는 `<>` (다름) 등으로 비교할 수 있습니다.
    - **논리 연산자**: `AND`, `OR`, `NOT` 등을 사용하여 여러 조건을 결합할 수 있습니다.
    - **`IN` 연산자**: 특정 컬럼 값이 주어진 여러 값들 중 하나에 해당하는지 확인할 때 유용합니다. (예: 부서 번호가 10번이거나 20번인 경우)
    - **`NOT` 연산자**: 특정 값에 포함되지 않은(제외되는) 데이터를 검색할 때 사용합니다.
- **`BETWEEN AND` 연산자**: 특정 컬럼 값이 지정된 두 값 사이(작은 값부터 큰 값까지)에 포함될 때 사용합니다.
    - *주의사항*: 반드시 작은 값이 먼저, 큰 값이 뒤에 오도록 작성해야 합니다.
- **`LIKE` 연산자 (패턴 검색)**:
    - 완벽하게 일치하지 않아도 유사한 값을 검색할 때 사용합니다.
    - **와일드카드 문자**:
        - **`%` (퍼센트)**: 0개 이상의 모든 문자를 대체합니다. (가장 포괄적)
        - **`_` (언더바)**: 정확히 한 글자를 대체합니다.
- **널(NULL) 값 처리**:
    - `NULL` 값은 일반적인 비교 연산자(`=`)를 사용해도 비교가 되지 않습니다. (`NULL = NULL`은 참이 아닙니다.)
    - `NULL` 값 여부를 확인할 때는 반드시 `IS NULL` (NULL인 경우) 또는 `IS NOT NULL` (NULL이 아닌 경우) 구문을 사용해야 합니다.
- **널 값 대체 함수**:
    - **`COALESCE`**: 여러 컬럼 값 중 첫 번째로 `NULL`이 아닌 값을 반환합니다.
    - **`IFNULL`**: 특정 컬럼의 값이 `NULL`일 때, 지정된 대체 값으로 처리하여 연산에 사용합니다.

## 3. SQL 함수 및 그룹 처리
- **내장 함수 (Built-in Functions)**:
    - 주로 숫자 값, 문자 값, 날짜 값에 대해 처리하는 함수들입니다. (예: `UPPER`, `LENGTH`, 날짜 함수 등)
    - 이러한 함수들은 DBMS 별로 제공되는 문서(Documentation)를 참고하여 확인하는 것이 가장 정확합니다.
- **그룹 함수 (Group Functions)**:
    - 여러 레코드(행)를 묶어 하나의 결과값으로 집계하는 함수입니다. (예: 합계(`SUM`), 평균(`AVG`), 개수(`COUNT`) 등)
- **그룹화 및 필터링**:
    - **`GROUP BY`**: 데이터를 특정 기준(컬럼)별로 그룹 묶음을 만들 때 사용합니다.
    - **주의사항**: 그룹 함수를 사용할 때, 그룹화에 참여하지 않은 컬럼은 결과로 출력할 수 없습니다. (반드시 `GROUP BY` 대상 컬럼만 출력 가능)
    - **`WHERE` 절**: 전체 데이터 레코드(로우)를 필터링하여 그룹화가 **되기 전**에 조건부로 제거합니다.
    - **`HAVING` 절**: `GROUP BY`로 묶인 그룹 자체를 필터링합니다. (즉, 그룹화된 **결과**에 조건을 걸 때 사용)
    - *차이점*: `WHERE`는 개별 레코드를, `HAVING`은 그룹 자체를 대상으로 조건을 걸어 필터링합니다.

## 4. SQL의 내부 처리 원리 이해
- **로우 바이 로우 처리 (Row by Row Processing)**:
    - 모든 SQL 쿼리는 '한 번에 모든 결과를 계산한다'는 오해를 버려야 합니다.
    - 실제로는 데이터베이스가 모든 데이터 레코드(Row)를 **첫 번째 레코드부터 순차적으로 하나씩** 읽어와서, 각 조건(WHERE, GROUP BY 등)과 비교하며 처리합니다.
    - 이 과정이 데이터의 양만큼 반복됩니다.
- **결과 도출 과정**:
    1. `FROM` 절을 통해 테이블 데이터를 하나씩 읽어옵니다.
    2. `WHERE` 절의 조건이 만족하는지 비교합니다. (만족하면 임시 결과에 추가)
    3. 모든 로우를 다 돌고 난 후, 조건에 만족하는 로우만 임시 결과 테이블에 모입니다.
    4. 최종적으로, `SELECT` 절에서 필요한 컬럼만 선택하여 사용자에게 결과로 보여줍니다.

## 5. 데이터베이스 설계 원칙 (정규화와 조인)
- **관계형 데이터베이스 설계의 필요성**:
    - 실제 데이터베이스는 하나의 테이블에 모든 정보를 담기보다, 관련된 정보를 여러 개의 테이블로 분리하여 관리합니다.
- **정규화 (Normalization)**:
    - 데이터 중복을 최소화하고 데이터의 무결성(Integrity)을 유지하기 위해 테이블을 분할하고 재설계하는 과정입니다. (예: 부서 정보가 사원 테이블에 반복되는 것을 방지)
- **테이블 관계와 조인 (JOIN)**:
    - 여러 테이블로 정보를 분산하여 관리했기 때문에, 원하는 하나의 결과를 얻으려면 여러 테이블 간의 **관계(Relationship)**를 연결하여 데이터를 조회해야 합니다.
    - **조인(Joining)**이란, 관계를 맺고 있는 여러 개의 테이블에서 필요한 정보를 조합하여 하나의 결과셋을 얻는 작업을 의미합니다.

## 1. 데이터 모델링 및 관계형 데이터베이스 기본 개념

*   **객체와 데이터베이스의 연결:** JPA(Java Persistence API)와 같은 기술들은 자바 객체와 관계형 데이터베이스의 테이블을 매핑하여, 개발자가 직접 복잡한 쿼리문을 작성하지 않아도 데이터를 처리할 수 있게 돕습니다.
*   **테이블 설계의 중요성 (정규화):**
    *   하나의 테이블에 모든 데이터를 모아 관리할 경우, 데이터가 비대해지고 중복 데이터가 발생합니다.
    *   중복 데이터는 공간 비효율을 가져올 뿐만 아니라, 수정이나 변경이 발생했을 때 데이터의 무결성(Integrity)을 깨뜨릴 위험이 있습니다.
    *   데이터를 효과적으로 관리하기 위해 데이터를 관련성별로 쪼개는 과정(정규화)을 거쳐 테이블을 설계하게 됩니다.
*   **핵심 키 개념:**
    *   **기본 키 (Primary Key, PK):** 테이블 내에서 각 레코드를 유일하게 식별할 수 있는 식별자입니다. (예: `EMP_NUMBER`)
    *   **외래 키 (Foreign Key, FK):** 테이블 간의 관계를 설정하는 필드로, 한 테이블의 PK가 다른 테이블의 FK로 참조됩니다. (예: `EMP` 테이블의 부서 번호가 `DEPT` 테이블의 PK를 참조)

## 2. JOIN을 사용하는 이유: 데이터 중복성 해결 (정규화)

*   **필요성:** 우리가 원하는 정보는 일반적으로 하나의 테이블에 모두 들어 있지 않고, 여러 테이블에 분산되어 존재합니다.
*   **JOIN의 역할:** JOIN은 여러 개의 테이블에 흩어져 있는 관련 데이터를 하나의 쿼리문에서 연결하여(연결하여) 필요한 정보를 한 번에 조회할 수 있게 해줍니다.
*   **결론:** 복잡한 정보를 얻기 위해 여러 테이블을 연결하는 작업 자체가 'JOIN'이며, 이 과정에서 적절한 **조인 조건(JOIN Condition)**을 설정하는 것이 가장 중요합니다.

## 3. JOIN의 기본 메커니즘 및 문법

*   **JOIN의 개념:** 두 개 이상의 테이블에서 연관된 데이터를 검색해 오는 방법을 의미하며, `SELECT` 구문과 `FROM` 절에서 여러 테이블을 나열합니다.
*   **조인 조건의 필수성:**
    *   테이블을 단순히 나열(FROM A, B)만 할 경우, 각 레코드 수만큼 곱해져 과도하게 많은 결과(Cartesian Product, 교차 조인)가 발생합니다.
    *   반드시 테이블 간의 관계를 명시하는 **조인 조건(ON 절, WHERE 절)**을 주어야 의미 있는 결과가 나옵니다.
    *   (예: `FROM EMP JOIN DEPT ON EMP.DEPT_NO = DEPT.DEPT_NO`)
*   **쿼리 작성 팁:**
    *   **Alias 사용:** 여러 테이블에 같은 컬럼 이름(예: `DEPT_NO`)이 있을 경우, 쿼리가 혼동되는 것을 막기 위해 테이블명에 별칭(Alias)을 부여하여 사용합니다.
    *   **여러 테이블 조인:** 테이블이 3개 이상인 경우에도, 각 쌍을 연결할 때 필요한 조인 조건만 순차적으로(테이블 개수 - 1개만큼) 설정해주면 됩니다.

## 4. 다양한 JOIN의 종류와 사용 예시

JOIN은 조건의 만족 여부에 따라 여러 종류로 나뉩니다.

*   **INNER JOIN (내부 조인):**
    *   **원리:** 조인 조건을 **모두 만족하는** 데이터 쌍(튜플)만을 결과로 보여줍니다.
    *   **특징:** 조건에 맞지 않는 데이터는 결과에서 완전히 제외됩니다.
*   **OUTER JOIN (외부 조인):**
    *   **INNER JOIN과의 차이:** 한쪽 테이블에는 데이터가 있지만, 조인 조건이 맞지 않아 짝을 찾지 못한 데이터도 결과에 포함하여 보여주고 싶을 때 사용합니다.
    *   **종류:**
        *   **LEFT OUTER JOIN:** 좌측 테이블의 모든 데이터를 유지하고, 우측에서 매칭되는 데이터가 있으면 가져오고, 없으면 NULL로 처리합니다.
        *   **RIGHT OUTER JOIN:** 우측 테이블의 모든 데이터를 유지하고, 좌측에서 매칭되는 데이터가 있으면 가져오고, 없으면 NULL로 처리합니다.
        *   **FULL OUTER JOIN:** 양쪽 테이블의 모든 데이터를 유지하며, 매칭되지 않는 부분은 NULL로 처리합니다.
*   **SELF JOIN (자기 조인):**
    *   **원리:** 하나의 테이블을 마치 두 개의 독립된 테이블인 것처럼 취급하여 자기 자신과 조인하는 방식입니다.
    *   **사용 예:** 사원 정보 테이블(`EMP`)에 매니저(`MGR`) 번호가 기록되어 있을 때, 이 `MGR` 번호를 다시 `EMP` 테이블의 사원 번호와 조인하여 **매니저의 이름**을 알아낼 때 사용합니다.
    *   **개념화:** 복잡하게 생각하기보다 'A라는 사원'과 'B라는 사원'의 관계를 알고 싶어, 테이블이 두 개 있다고 생각하면 헷갈리지 않습니다.
*   **CROSS JOIN (카테시안 조인):**
    *   **원리:** 조인 조건을 아예 주지 않고, 모든 테이블의 레코드들을 무작위로 결합(곱)시키는 방식입니다.
    *   **사용:** 이론적으로는 존재하지만, 실제로는 데이터가 기하급수적으로 늘어나기 때문에 특정 목적(예: 모든 경우의 수 계산)이 아니라면 거의 사용하지 않습니다.
*   **Theta Join (세타 조인):**
    *   **원리:** 등호(`=`) 같은 특정 조건을 사용하지 않고, 'A 필드가 B 필드보다 크다($>$)'와 같이 **다른 조건**을 사용하여 테이블을 조인할 수 있습니다. (조인 조건의 범위를 확장)

## 5. 데이터베이스 트랜잭션 제어 및 심화 학습 내용

*   **트랜잭션(Transaction):** 데이터베이스에 발생하는 일련의 작업 단위입니다.
*   **ACID 속성:** 트랜잭션이 성공적으로 수행되어야 하는 네 가지 속성입니다.
    *   **원자성(Atomicity):** 트랜잭션 내의 작업은 전부 성공하거나, 아니면 전부 실패해야 합니다.
    *   **일관성(Consistency):** 트랜잭션 전후에 데이터가 항상 일관성을 유지해야 합니다.
    *   **격리성(Isolation):** 여러 트랜잭션이 동시에 실행되어도 서로 영향을 주지 않아야 합니다.
    *   **영속성(Durability):** 성공적으로 커밋된(Commit) 결과는 시스템 장애가 있어도 영구적으로 보존되어야 합니다.
*   **Commit과 Rollback:**
    *   **Commit:** 트랜잭션이 성공적으로 완료되었을 때, 데이터 변경 사항을 데이터베이스에 *최종 반영*하는 명령입니다.
    *   **Rollback:** 트랜잭션 과정에서 문제가 발생했거나 취소해야 할 경우, 변경 사항을 반영하지 않고 *이전 상태로 되돌리는* 명령입니다.
*   **Auto Commit:** 워크벤치 등 개발 환경에서 기본적으로 설정되어 있어, 명령을 입력하는 순간마다 자동으로 커밋(반영)되는 기능입니다. 이 개념을 인지하고 트랜잭션 제어(Commit/Rollback)의 필요성을 이해하는 것이 중요합니다.

## 1. 데이터베이스 조인(JOIN)의 기본 개념
- **조인의 필요성:** 데이터베이스는 여러 개의 테이블(테이블 분리 저장)로 정보를 보관하므로, 필요한 데이터를 추출하려면 여러 테이블 간의 연결(JOIN) 방법이 필수적입니다.
- **조인의 위험성 (필수 조건):** 테이블 간의 조인 조건(JOIN condition)을 명시하지 않으면, 의도하지 않게 방대한 양의 데이터(데카르트 곱)가 생성될 수 있습니다. 반드시 최소한의 조인 조건이 필요합니다.
- **관계 이해 (키):** 테이블 간의 연결 관계는 주로 **외래 키(Foreign Key)**와 **기본 키(Primary Key)**를 통해 성립하며, 이를 기준으로 조인을 수행합니다.
- **명시적 조인의 중요성:** 테이블을 여러 개 조인할 때, 컬럼 이름이 중복될 경우 컴퓨터가 어떤 컬럼을 기준으로 삼아야 할지 판단할 수 없어 오류가 발생하므로, 반드시 어떤 테이블의 어떤 컬럼을 사용할지 명시해 주는 것이 좋습니다.

## 2. 다양한 JOIN 구문 비교 및 활용 (JOIN ON, USING, NATURAL)
조인을 조건에 따라 명시하는 문법은 세 가지가 있으며, 각각의 용도와 특징을 이해해야 합니다.

- **JOIN ON (가장 권장되는 방식):**
    * 가장 명확하게 조인 조건을 지정하는 방식입니다.
    * `JOIN` 구문과 `ON` 구문을 사용하여 "어떤 테이블과 어떤 테이블을, 어떤 컬럼을 기준으로 연결하겠다"고 명시합니다.
    * **장점:** 조인 조건을 분리하여 명시하므로 가장 이해하기 쉽고 오류 발생 위험이 적습니다.
- **JOIN USING:**
    * 조인하려는 두 테이블에 조인 조건으로 사용되는 **컬럼 이름이 동일할 때** 사용하기 편리합니다.
    * 예: 두 테이블 모두에 `departmentId`라는 컬럼이 있다면, `JOIN USING(departmentId)`로 간결하게 사용할 수 있습니다.
    * **주의사항:** 반드시 컬럼명이 동일해야만 사용 가능하며, 이 경우 `JOIN ON`보다 제약이 많습니다.
- **NATURAL JOIN:**
    * 조인하려는 모든 테이블에서 **컬럼명이 완전히 일치하는 컬럼**이 발견되면, 그 컬럼들 모두를 자동으로 조인 조건으로 사용합니다.
    * **위험성:** 컬럼명이 같은 컬럼이 여러 개 있을 경우, 모든 컬럼을 조인 조건으로 간주하여 의도치 않은 결과를 초래할 수 있습니다.
    * **결론:** 이름이 같은 컬럼이 여러 개 존재할 위험이 크므로, 명시적으로 조건을 지정할 수 있는 **`JOIN ON` 구문 사용이 가장 안전하고 권장**됩니다.

## 3. 특수 목적의 JOIN 유형
일반적인 등가 조건(A=B) 외에 특수한 데이터 추출 목적을 가진 조인 방법들입니다.

### A. Non-Equi Join (비등가 조인)
- 두 테이블의 값이 **완벽히 일치(Equal)할 필요가 없는** 경우에 사용합니다.
- 사원의 급여가 특정 범위(예: 700에서 1200 사이)에 속할 때 해당 등급(Grade)을 찾아낼 때처럼, **범위(Range) 또는 조건을 이용**하여 조인할 때 사용합니다.
- 일반적으로 `BETWEEN`과 같은 범위 비교 연산자를 사용합니다.

### B. Outer Join (외부 조인)
- 기본 조인(INNER JOIN)은 조인 조건에 **딱 맞는 데이터만** 보여줍니다. 하지만 어느 한쪽에만 데이터가 존재하는 경우도 있습니다.
- **상황 예시:** 부서가 없는 사원(테이블 A에만 존재)이나, 사원이 없는 신규 부서(테이블 B에만 존재)의 정보를 함께 보고 싶을 때 사용합니다.
- **종류:**
    * **LEFT OUTER JOIN (LEFT JOIN):** 왼쪽 테이블(FROM 절의 기준)의 모든 행을 유지하고, 오른쪽 테이블에서 매칭되는 정보만 가져옵니다.
    * **RIGHT OUTER JOIN (RIGHT JOIN):** 오른쪽 테이블의 모든 행을 유지하고, 왼쪽 테이블에서 매칭되는 정보만 가져옵니다.
    * **FULL OUTER JOIN (FULL JOIN):** 양쪽 테이블 중 한쪽에만 존재하는 모든 행을 포함하여 결과를 출력합니다.
- **DB 별칭의 차이:**
    * **SQL 표준/Oracle:** `LEFT OUTER JOIN`과 같은 키워드를 사용합니다.
    * **MySQL:** `LEFT JOIN` 또는 `RIGHT JOIN`을 사용하고, `FULL OUTER JOIN`은 지원하지 않으므로 **`LEFT JOIN` + `UNION` + `RIGHT JOIN`** 방식으로 대체해야 합니다.

### C. Self Join (셀프 조인)
- 테이블 하나를 대상으로, 마치 두 개의 테이블처럼 간주하여 조인하는 방식입니다.
- **핵심 원리:** 물리적으로 테이블이 하나라도, 논리적으로는 **두 개의 독립적인 테이블**처럼 간주하고 조인하는 것이 핵심입니다.
- **실습 방법:** `FROM` 절에 테이블 이름을 **두 번** 사용하여 (`FROM Employee E JOIN Employee M`) 가상의 두 테이블로 만듭니다.
- **사용 구문:** 이 경우 컬럼명이 같더라도 `JOIN ON`을 사용하여 조인 조건을 명시하는 것이 가장 명확하고 안전합니다.

## 4. 결론 및 핵심 정리
- **가장 중요한 학습 포인트:** 데이터베이스 설계 구조를 이해하는 것이 가장 중요하며, 원하는 데이터가 어떤 논리적 연결고리를 가지고 있는지 파악해야 합니다.
- **실습 과정:** 복잡한 개념일지라도, 항상 **'어떤 정보를 얻고 싶은가?'**라는 질문을 던지며 시나리오 기반으로 접근하는 것이 어렵지 않게 학습하는 핵심입니다.
- **요약 정리:**
    * **기본:** `JOIN ON`을 사용하여 조인 조건을 명확하게 지정한다.
    * **범위:** 범위 조인은 `NON-EQ JOIN`을 사용한다.
    * **불완전 데이터:** 한쪽에만 존재하는 정보도 보고 싶다면 `OUTER JOIN`을 사용한다.
    * **셀프:** 테이블 하나를 두 개의 가상 테이블로 보고, `JOIN ON`으로 연결한다.
