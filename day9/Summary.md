## 1. 강좌 오리엔테이션 및 복습
- **(도입부)** 학습자 간의 안부 확인 및 학습 동기 부여를 하며 강의를 시작함.
- **핵심 목표 제시:** 자바라는 프로그래밍 언어는 컴퓨터에게 일을 시키기 위해 필요한 '컴퓨터가 알아들을 수 있는 언어'의 일종이며, 현재는 자바를 중심으로 학습을 진행함.

## 2. 자바(Java)의 탄생 배경과 설계 철학
- **자바의 역사와 특징:**
    - 자바는 1990년대에 설계되었으며, 비교적 짧은 기간 동안 급속도로 발전하여 여러 기업의 교육 언어로 사용될 만큼 성숙한 언어임.
    - 자바는 애초에 '플랫폼 독립성'을 염두에 두고 설계되었음. (어떤 운영체제에서 컴파일하더라도 어디서든 동일하게 동작하는 것을 목표)
    - 가전제품에 소프트웨어를 탑재하는 것이 주된 목적이었기에, 네트워크 기능에 강하다는 특징을 가지고 있음.
- **다른 언어와의 비교 (Python 예시):**
    - Python은 데이터 분석 및 마케팅 등 특정 분야의 API(예: 파일 처리, 전파)가 필요해지면서 급부상한 언어임.
    - 반면 자바는 처음부터 '어떤 규모의 회사에서든' 폭넓게 활용할 수 있는 체계적이고 일관성 있는 언어라는 특징이 있음.

## 3. 객체 지향 프로그래밍(OOP)의 핵심 개념 복습
- **OOP의 목적:** 우리의 물건이나 개념을 생각하는 것처럼, 자바 세상의 객체들을 이해하고 사용법을 익히면 처음 접하는 객체도 감(感)으로 사용할 수 있게 됨.
- **기본 문법:** 변수, 연산자, 반복문, 조건문 등 기본 문법은 모든 프로그래밍 언어가 공통적으로 사용하며, 핵심 원리는 유사함.
- **객체 지향의 3대 요소:**
    1. **추상화 (Abstraction):** 객체에서 가장 중요한 속성이나 기능만 남기고 나머지 세부 사항은 가리는 작업.
    2. **캡슐화 (Encapsulation):** 여러 필드(속성)나 기능들을 하나의 그릇(클래스/캡슐) 안에 묶어 관리하는 것.
    3. **상속 (Inheritance):** 부모(Parent) 클래스가 가진 속성이나 기능을 자식(Child) 클래스가 물려받아 사용하는 개념. (다만 자바는 **단일 상속**만 가능함)

## 4. 객체 지향 심화: 상속과 다형성
- **상속의 원칙 및 주의사항:**
    - **관계 성립:** 상속 관계는 명확해야 함. (예: '헬리콥터가 차'인지 아닌지 등 명확한 분류 필요).
    - **단방향성:** 부모(조상)는 자식(자손)을 가리킬 수 있지만, 자식에게서 부모를 추론하는 것은 어렵거나 불가능할 수 있음.
- **가장 중요한 원칙: 메서드 오버라이딩 (Overriding):**
    - 부모로부터 물려받은 메서드를 자식 클래스에서 재정의(덮어쓰기)하는 것이 핵심임.
    - 이는 자바가 가능한 **다형성(Polymorphism)**을 구현하는 핵심 원리이며, 실제 웹 애플리케이션 서버 등에서 사용자가 만든 페이지를 부모 타입으로 받아서 처리할 수 있게 만드는 기반이 됨.
- **메모리 관리 (가비지 컬렉션):**
    - 자바에서는 개발자가 메모리를 직접 관리할 필요가 없으며, **가비지 컬렉터(Garbage Collector)**가 사용하지 않는 메모리(객체)를 자동으로 정리해 줌.
    - *참고:* 직접 GC를 호출하는 것(GC 명시적 호출)은 시스템 성능 저하를 일으킬 수 있으므로 지양해야 함.

## 5. 추상 클래스와 인터페이스 (Abstraction & Interface)
- **추상화의 필요성:**
    - 클래스 설계 시 코드가 중복되는 부분이 발생하면, 이를 '일반화'하여 하나의 기능으로 묶어내는 고민이 필요함.
    - 이 과정에서 공통 기능이 명확하지 않을 때, 부모 클래스로 만들 수 없는 메서드는 **추상 메서드**로 선언함.
- **추상 클래스 (Abstract Class):**
    - 일부 기능만 '애매하게' 구현해 놓고(추상 메서드), 자손 클래스들이 나머지 구현이나 확장을 할 수 있도록 기반을 제공함.
    - 장점: 실제 구현체(예: Gregorian Calendar)를 숨기고 부모 추상 클래스(Calendar)만 외부에 노출할 수 있어, 내부 구현체가 변경되어도 사용자 코드는 영향을 받지 않는 **유연한 설계**가 가능함.
- **인터페이스 (Interface):**
    - 특정 기능(행위)들만을 묶어 '규격(Type)'을 정의할 때 사용됨. (예: 날개 달린 것들 $\rightarrow$ `Flyable` 기능).
    - 장점: 상속(Inheritance)과 달리, 기능의 최소한의 규격만 강제함으로써 서로 관계없는 클래스들(비행기, 새, 드론)도 특정 기능을 갖도록 **통일화**시킬 수 있음.
    - **차이점:** 추상 클래스는 클래스 그 자체의 형태를 가질 수 있지만, 인터페이스는 순수한 '계약' 또는 '기능의 목록'만을 제공함.

## 6. 인터페이스의 발전: 디폴트/정적 메서드
- **인터페이스의 다중 상속:**
    - 인터페이스는 껍데기(구현체가 없음)만 갖고 있기 때문에, 상속 제한이 없는 **여러 개의 인터페이스를 구현(implements)**하는 것이 가능함.
    - 또한, 인터페이스끼리도 서로를 상속받는 것이 가능함.
- **최신 기능 (Default/Static Method):**
    - **디폴트 메서드 (Default Method):** 인터페이스에 기본 구현체를 제공하는 기능. 기존 구현 클래스들이 수많은 인터페이스를 사용하다가 새로운 기능을 추가할 때, 기존 코드의 오류를 막기 위해 사용됨.
    - **정적 메서드 (Static Method):** 객체(인스턴스)가 생성되었는지 여부와 상관없이, 인터페이스 이름만으로 독립적으로 사용할 수 있는 기능 (예: API 라이브러리).
    - **제한:** 만약 두 개의 디폴트 메서드가 같은 이름의 기능을 구현하게 되면, 컴파일 단계에서 오류를 발생시켜 반드시 오버라이드(재정의)하도록 강제함으로써 일관성을 유지함.

## 1. 접근 제한자 및 키워드 복습 (protected, private, final)
*   **접근 제한자 복습:**
    *   **`protected`:** 해당 패키지 내는 물론, **상속 관계**에 있는 클래스에서는 접근을 허용한다.
    *   **`private`:** 해당 클래스 내부에서만 접근 가능하며, 상속 관계를 포함해 외부에서는 접근할 수 없다.
*   **`final` 키워드의 역할:**
    *   **`final` 클래스:** 상속받을 수 없다 (클래스의 구조가 변경되는 것을 방지).
    *   **`final` 메서드:** 오버라이딩(Override)할 수 없다 (메서드의 동작을 강제적으로 고정).
    *   **`final` 필드:** 상수가 되어 한 번 값이 할당되면 변경할 수 없다.
*   **활용 목적:** 시스템 설계 시 불필요한 변경이나 오용을 방지하여 객체의 안정성을 높인다.

## 2. 타입 검사 및 디자인 패턴 이해 (instanceof, Design Pattern)
*   **`instanceof` 연산자:**
    *   특정 변수가 지정된 클래스 또는 인터페이스의 인스턴스(객체)인지를 동적으로 검사할 때 사용한다.
    *   예: `object instanceof ClassName`을 통해 현재 인스턴스의 실제 타입을 확인할 수 있다.
*   **디자인 패턴 (Design Pattern):**
    *   소프트웨어 개발 과정에서 발생하는 일반적인 문제에 대해 검증된 해결 방법을 제시하는 설계 방식이다.
    *   **대표 패턴:** 싱글턴 패턴(Singleton), 팩토리 메서드 패턴(Factory Method) 등이 있다.
    *   **싱글턴 패턴:** 어플리케이션 전체에서 특정 객체가 오직 하나만 생성되도록 제어하는 설계 방식이다. (Private 생성자, 정적(static) 변수 사용 등 구현)
    *   **팩토리 메서드 패턴:** 객체를 생성하는 과정을 공장(Factory)처럼 분리하여, 사용자가 구체적인 클래스를 몰라도 객체를 생성하고 사용할 수 있도록 고도화한 패턴이다.
*   **클래스 생성 방법:**
    *   일반적인 `new` 연산자 사용 외에도, `java.lang.Class` 클래스를 활용하여 메서드를 이용해 인스턴스를 생성하는 방법이 존재한다.

## 3. 기본 객체 클래스와 메서드 오버라이딩 (Object 클래스, toString)
*   **`Object` 클래스의 중요성:**
    *   자바에서 모든 클래스는 `Object` 클래스를 상속받는 것이 기본 원칙이다. 따라서 모든 객체는 `Object`가 제공하는 기본 메서드를 자동으로 가지게 된다.
    *   `Object`가 제공하는 기본 메서드: `toString()`, `equals()`, `hashCode()`, `wait()`, `notify()` 등.
*   **`toString()` 메서드:**
    *   **정의:** 객체의 정보를 사람이 읽기 쉬운 형태의 문자열로 반환하기 위해 사용되는 메서드이다.
    *   **필요성:** 단순히 객체 이름만 출력했을 때 주소값 같은 의미 없는 값이 나오는 것을 방지하고, 해당 객체가 가진 의미 있는 정보를 보여주기 위해 필수적으로 재정의(Override)해야 한다.
    *   **작동 원리:** 객체 이름을 출력하거나, `object.toString()`을 호출하면 내부적으로 `Object`의 `toString()`을 호출하게 된다.
    *   **설계의 관점:** 설계자는 사용자가 객체 정보를 알기 쉽도록 통일된 방법(예: `toString()`)을 약속하여 사용자의 편의성을 높인다.
*   **`toString()` 오버라이딩의 효과:**
    *   개발자가 직접 해당 클래스에 맞게 `toString()` 메서드를 오버라이딩함으로써, 객체 출력 시 필요한 정보(예: 이름과 나이)만을 추출하여 가독성 높은 문자열로 반환할 수 있게 된다.

## 4. 객체 비교의 개념 이해 (equals 메서드)
*   **기본 비교 연산자 (`==`):**
    *   객체 참조 변수 간에 `==` 연산자를 사용하면, **실제 내용물(값)**을 비교하는 것이 아니라 **메모리 주소(참조지)**가 같은지 비교한다.
    *   따라서 내용물이 같더라도 메모리 주소가 다르면 `false`가 반환된다.
*   **`equals()` 메서드의 필요성:**
    *   우리가 일상생활에서 '같은 사물'을 판단할 때는 내용물(값)을 기준으로 판단한다.
    *   `equals()` 메서드는 기본적으로 주소값을 비교하도록 구현되어 있지만, 개발자가 이를 오버라이딩하여 **'논리적 동등성(Logical Equality)'**, 즉 객체가 가진 필드 값들을 비교하여 같음을 판단하도록 커스터마이징해야 한다.
*   **핵심 정리:**
    *   `toString()`: 객체의 **표현 방식**을 정의 (어떻게 보여줄 것인가?).
    *   `equals()`: 객체의 **같음의 기준**을 정의 (내용물이 같은가?).

> **💡 Tip:** 개발자가 객체를 설계할 때, 객체의 정보를 보여주는 `toString()`과 같음을 비교하는 `equals()`는 반드시 오버라이딩하여 재정의하는 것이 일반적이고 중요합니다.

## 1. 객체 비교와 식별을 위한 메서드 오버라이딩
*   **기본 동작의 한계:** 자바 객체는 기본적으로 메모리 주소(Identity)를 기준으로 비교합니다. 따라서 내용물(속성값)이 같더라도 주소가 다르면 같지 않다고 판단합니다.
*   **`equals()` 메서드의 목적:** 객체가 실제로 담고 있는 **내부 속성값(State)**을 비교하여 논리적으로 같은지 판단하기 위해 `Object` 클래스의 `equals()` 메서드를 재정의(Override)해야 합니다.
    *   **구현 시 핵심 로직:**
        *   `null` 검사 및 타입 체크 (비교할 가치가 있는지 확인).
        *   비교할 모든 필드를 순회하며, 각 필드별로 논리적 비교를 수행합니다. (예: `this.fieldName.equals(obj.fieldName)`).
        *   **유연성:** 객체가 같은지 판단하는 기준은 개발자가 설계 단계에서 임의로 정할 수 있습니다. (예: 제조사가 달라도 색깔만 같으면 같은 것으로 간주).
*   **`hashCode()`의 역할:** `equals()`를 오버라이드했다면, **반드시 `hashCode()`도 함께 오버라이드** 해야 합니다.
    *   `hashCode()`는 객체를 고유한 정수 값(해시 값)으로 변환합니다.
    *   이 해시 값은 `HashMap`, `HashSet` 등 해시 기반 자료구조에서 객체를 빠르게 검색하고 저장할 수 있게 하는 핵심적인 역할을 합니다.
    *   **⚠️ 규칙:** `equals()`가 참이면 `hashCode()`도 같아야 하지만, 그 반대는 아닙니다.
*   **`toString()` 메서드의 목적:** 객체를 문자열로 출력할 때, 메모리 주소와 같은 불필요한 값이 아닌, 객체의 **내용 정보를 사람이 읽기 쉬운 형식**으로 표현하기 위해 오버라이드합니다.

## 2. 키워드 심화 학습: `final`
`final` 키워드는 특정 요소의 변경을 막거나 제약을 가하는 데 사용됩니다.
*   **`final` 클래스:** 해당 클래스는 상속(Inheritance)이 불가능합니다.
*   **`final` 변수 (필드):** 변수가 한 번 할당된 후에는 그 값을 변경할 수 없습니다 (상수처럼 사용).
*   **`final` 메서드:** 해당 메서드는 자식 클래스에서 오버라이드(Override)할 수 없습니다.

## 3. 제한된 값 관리를 위한 열거형 데이터 타입 (`Enum`)
*   **필요성:** 특정 필드가 가질 수 있는 값이 정해진 목록(예: 요일, 배송 상태)으로 제한될 때 사용됩니다. 일반적인 `String` 타입은 잘못된 값(예: '5타')이 입력될 가능성이 있지만, `Enum`은 이를 막아 **타입 안정성**을 확보합니다.
*   **작동 방식:** `Enum`은 개발자가 직접 정의하는 유한한 값들의 집합체입니다.
*   **장점:**
    *   **컴파일 시점 안전성:** 잘못된 값이 들어오기 전에 컴파일러가 오류를 감지합니다.
    *   **정보 통합:** 각 상수(열거 항목)에 대한 고유한 값(코드 값)이나 상세 설명(`Description`)을 함께 저장하고 관리할 수 있습니다.
    *   **가독성 향상:** `switch` 문 등을 사용하여 코드를 더 직관적이고 가독성 높게 작성할 수 있습니다.

## 1. 모든 객체의 조상, `Object` 클래스 심화 이해
- **`Object` 클래스의 역할:** Java에서 사용되는 모든 객체의 근원(조상)이 되는 클래스입니다. 우리가 만드는 수많은 클래스들은 모두 `Object`를 상속받는 구조입니다.
- **`Object`가 제공하는 핵심 기능 (메서드 오버라이딩):**
    - **`toString()`:** 객체의 내부 정보를 문자열 형태로 보여주는 역할을 합니다. (객체에 대한 정보를 보기 쉽게 출력)
    - **`equals()`:** 두 객체가 단순히 메모리상에 같은 주소(참조)를 가지는지 비교하는 것 외에, **내용적인 속성(값)**까지 동일한지를 비교할 때 사용됩니다.
    - **`hashCode()`:** 객체의 값이나 속성을 정수(int) 코드값으로 변환하는 역할을 합니다. (컬렉션 구조 사용 시 중요)
- **중요성:** 이 메서드들이 정의된 이유는 객체가 구체적인 클래스가 아니기 때문에, **모든 객체가 공통적으로 사용해야 하는 일반화된 개념**을 제공하여 일관성 있는 기능을 보장하기 위함입니다.
- **활용:** 개발자는 필요에 따라 이 메서드들을 오버라이딩하여 객체에 특화된 기능을 구현할 수 있습니다.

## 2. 데이터 안정성을 위한 Enum(열거형) 사용
- **Enum의 필요성:** 요일이나 주문 상태(대기, 배송 중 등)처럼 가질 수 있는 값의 종류가 한정적일 때, 단순히 `String`이나 `int`를 사용하는 것은 오타나 잘못된 값이 들어가도 컴파일러가 잡아낼 수 없어 오류 발생 위험이 높습니다.
- **Enum의 역할:** 데이터 타입을 정의할 때, 미리 정해진 값(열거형)만을 사용하도록 **값을 강제화**하여 코드의 안전성을 높입니다.
- **장점:**
    - **컴파일 시점 강제성:** 정의된 값 외에는 사용할 수 없어 오류를 미연에 방지합니다.
    - **가독성:** 코드에 명확한 이름으로 상태나 종류를 나타낼 수 있습니다.
    - **확장성:** 값 외에 코드값, 설명(Description) 등 추가적인 메타데이터를 함께 정의하여 사용할 수 있습니다.

## 3. 클래스 안에 클래스를 정의하는 내부 클래스 (Inner Class)
- **정의:** 클래스 안에 또 다른 클래스를 정의하는 것을 의미합니다.
- **사용 목적:** 해당 내부 클래스가 오직 **특정 외부 클래스 내에서만 사용될 경우**, 이를 분리하여 외부로 노출할 필요 없이 캡슐화하여 관리하기 위함입니다.
- **내부 클래스의 종류 및 특징:**
    - **멤버 내부 클래스 (Member Inner Class):** 외부 클래스의 필드나 메서드 레벨에 정의됩니다. 외부 클래스의 인스턴스와 밀접한 관계를 가지며, 외부 클래스의 인스턴스가 있어야 사용이 가능합니다.
    - **정적 내부 클래스 (Static Nested Class):** `static` 키워드를 사용하여 정의되며, 외부 클래스의 인스턴스 생성 여부와 무관하게 독립적으로 존재할 수 있습니다.
    - **로컬 내부 클래스 (Local Inner Class):** 특정 메서드(지역 변수) 내부에서만 정의되어 사용되는 클래스입니다. 해당 메서드 내에서만 유효합니다.
    - **익명 내부 클래스 (Anonymous Inner Class):** 클래스 이름을 부여하지 않고 즉시 사용하는 클래스입니다. 구현이 필요할 때 사용됩니다.

- **고급 개념:**
    - **람다 표현식 (Lambda Expression):** 익명 클래스를 간편하게 구현하는 문법입니다. 특히 인터페이스가 단 하나의 메서드만 가진 경우 (함수형 인터페이스, Functional Interface)에 유용하며, 코드를 매우 간결하게 만들어줍니다.

## 4. 종합 복습 및 학습 방향 제시
- **핵심 복습 개념:**
    - **상속(Inheritance):** 부모가 가진 속성을 자식이 물려받아 쓰는 개념. (`extends` 키워드)
    - **다형성(Polymorphism):** 껍데기(상위 타입)는 같지만, 실제 구현체(하위 타입)에 따라 다르게 동작하는 원리입니다. (예: `Animal` 타입 변수에 `Dog` 객체를 담아도 `Dog`의 소리만 내게 할 수 있는 것)
    - **추상화(Abstraction):** 복잡한 시스템에서 핵심 기능만 드러내고 세부 구현은 숨기는 개념 (추상 클래스, 인터페이스 사용).
    - **인터페이스(Interface):** 클래스가 반드시 구현해야 할 '규칙(메서드들의 집합)'만 정의하는 청사진입니다.
- **학습 태도:**
    - 앞서 배운 개념들은 당장 100% 완벽하게 이해하기 어렵습니다.
    - 개념적 이해와 반복적인 실습(특히 코드 작성)을 통해 원리를 파악하는 것이 중요합니다.
    - **기억할 점:** 이 복잡한 개념들을 이해하려 하기보다, "이런 기능을 구현하기 위해 이런 도구가 있구나" 정도로 가볍게 받아들이면서 나중에 프로젝트를 진행하며 다시 깊이 있게 파고드는 것이 효율적입니다.

## 1. 자바 핵심 클래스 개요 및 Object 클래스 심화
*   **학습 목표 전환:** 기본적인 문법 학습을 넘어, 자바 개발에서 빈번하게 사용되는 핵심 클래스(Core Classes)들을 학습하는 것이 중요합니다.
*   **주요 학습 클래스:**
    *   `Object` 클래스: 모든 클래스의 최상위 부모 클래스입니다. (명시적 상속 없이도 자동으로 상속받음)
    *   `String` 클래스: (자세한 내용은 섹션 5에서 다룸)
    *   `StringBuilder`/`StringBuffer`: 문자열을 효율적으로 조작하기 위한 클래스입니다.
    *   `Map`, `System`, `Date`/`Time` 등: 시스템 자원 접근, 날짜/시간 처리, 데이터 매핑 등 다양한 기능 수행을 위한 핵심 클래스들입니다.

## 2. 자료구조와 객체 비교의 필수 요소: `equals()`와 `hashCode()`
*   **`Object` 클래스 주요 메서드:** 모든 자바 객체가 기본적으로 가지고 있는 메서드들이 존재합니다.
    *   `toString()`: 객체의 정보를 문자열 형태로 출력할 때 사용합니다. (강제적으로 호출되지 않아도 사용 시 알아서 호출되는 특징이 있습니다.)
    *   `equals()`: 단순히 주소값(메모리상의 위치)이 같은지 비교하는 것이 아니라, **실제 객체의 내부 필드 값**이 같은지 비교할 때 사용합니다.
    *   `hashCode()`: 객체의 값을 정수(int) 값으로 변환해 주는 기능입니다. (주로 해시 기반 자료구조에서 사용됨)
*   **오버라이딩(Overriding)의 필요성:** 기본 `Object` 클래스의 동작 방식(주소값 비교 등)이 개발자가 원하는 비즈니스 로직과 다를 때, `equals()`와 `hashCode()`를 재정의(Overriding)하여 객체 비교 기준을 커스터마이징할 수 있습니다.
    *   **응용 지식:** 객체 비교 시 필드 중 일부 값(예: 오차 범위가 있는 값)을 제외하거나, 완전히 다른 로직(예: 두 점의 넓이가 같으면 같다고 정의)을 기반으로 같음을 판단할 수 있습니다.
*   **핵심 개념 이해:** `equals`와 `hashCode`를 구현하는 방법론 자체를 이해하는 것이 중요하며, 실제 구현은 IDE의 도움을 받아 진행할 수 있습니다.

## 3. Java의 컬렉션 활용 및 인터페이스의 중요성
*   **Set 자료구조의 특징:**
    *   Set은 여러 개의 데이터를 담는 자료구조 중 **중복 저장이 허용되지 않습니다.**
    *   Set이 중복을 판별할 때 내부적으로 `equals()`와 `hashCode()`를 사용하여 객체가 동일한지 판단합니다. 따라서 Set을 사용할 때는 해당 객체에 `equals()`와 `hashCode()`를 반드시 오버라이딩해야 합니다.
*   **인터페이스 타입 사용의 장점 (다형성):**
    *   컬렉션 타입을 구체적인 구현체(예: `HashSet<Point>`)로 지정하기보다, 상위 타입의 인터페이스(예: `Set<Point>`)로 선언하여 사용하는 것이 좋습니다.
    *   **이점:** 만약 내부적으로 구현체를 `HashSet`에서 `TreeSet` 등으로 교체하더라도, 전체 코드의 사용 부분(사용부)을 수정할 필요가 없습니다. 이는 코드의 유연성과 확장성을 극대화합니다.

## 4. 객체 복제(클로닝) 메커니즘 이해
*   **`Cloneable` 인터페이스:** 객체를 복제(Clone)하기 위해서는 해당 클래스가 `java.lang.Cloneable` 인터페이스를 구현해야 합니다.
*   **`Object.clone()` 메서드:** `Object` 클래스에 내장된 메서드로, 객체의 사본(복제본)을 생성합니다.
*   **메커니즘 원리:**
    *   자바 시스템은 어떤 객체가 복제가 가능한지 판단할 때, 해당 클래스가 `Cloneable`을 구현했는지 여부를 체크합니다.
    *   만약 복제를 허용하고 싶지 않다면, `Cloneable`을 구현하지 않거나, `clone()` 메서드 호출 시 `CloneNotSupportedException`과 같은 예외를 발생시켜 복제를 원천적으로 막을 수 있습니다.
    *   **결과:** 단순히 주소값을 복사하는 것이 아니라, 메모리상에 **실제로 새로운 객체를 생성**하여 값을 복사해주는 '깊은 복사(Deep Copy)'의 개념을 가집니다.

## 5. 문자열(`String`) 클래스의 특성과 메모리 관리 원리
*   **불변 클래스 (Immutable Class):** `String`은 매우 특별한 클래스로, 한번 생성되면 **절대로 내용을 변경할 수 없습니다.** (불변성)
    *   만약 `String`이 가변적이라면, 변수를 변경할 때마다 메모리의 값이 덮어쓰여져 다른 참조 변수들이 원래의 값까지 잃어버리는 심각한 문제가 발생할 수 있습니다.
*   **`String`의 효율적인 메모리 관리 (String Pool):**
    *   문자열은 메모리 효율성을 위해 'String Pool'이라는 특정 메모리 영역에 저장됩니다.
    *   `"Hello"`라는 문자열을 여러 번 선언하더라도, JVM은 같은 문자열 값은 메모리에 중복으로 저장하지 않고, Pool 내의 같은 객체를 참조하도록 관리합니다.
*   **문자열 값 변경 시:**
    *   문자열에 값을 추가하거나 변경해야 할 경우 (예: `str.toUpperCase()`, `str + "World"`), 원본 객체는 그대로 두고, **변경된 값을 가진 완전히 새로운 `String` 객체가 메모리에 생성됩니다.**
    *   따라서 `String`은 불변성이 필수적이며, 값을 '바꾸는' 것처럼 보일지라도 실제로는 **새로운 객체를 만들고 변수를 재할당(Reassign)**하는 과정이 필요합니다.
*   **주요 메서드 정리:**
    *   `length()`: 문자열의 길이를 반환합니다.
    *   `charAt(index)`: 특정 인덱스의 문자를 가져옵니다.
    *   `substring(start, end)`: 지정된 범위의 부분 문자열을 추출합니다.
    *   `indexOf(str)`: 특정 문자열이 몇 번째 인덱스에 있는지 찾습니다.
    *   `contains(str)`: 문자열 포함 여부를 확인합니다.
    *   `trim()`: 문자열의 **양 끝**에 있는 공백만 제거합니다.
    *   `replace(old, new)`: 특정 문자열을 다른 문자열로 치환합니다.
    *   `split(delimiter)`: 지정된 구분자를 기준으로 문자열을 여러 개로 잘라 배열 형태로 반환합니다.
    *   `format(...)`: 포맷 지정자(예: `%s`, `%d`)를 사용하여 가변적으로 문자를 구성합니다.

## 1. Java String 객체의 기본 특징과 메모리 관리
- **자료 구조적 특성**: String은 '문자 열'을 다루는 자료 구조입니다.
- **메모리 관리의 효율성**: String은 객체 생성 시 메모리 효율성을 위해 특별한 방식으로 관리됩니다.
    - 같은 문자열 값(예: "Hello")이 반복적으로 생성될 때, 매번 새로운 객체를 만들지 않고 이미 생성된 공간(메모리 풀)을 공유하여 사용합니다.
    - 이로 인해 같은 문자열 값은 하나만 만들어져서 사용됩니다.
- **객체 비교**: 같은 문자열 값이라도 생성 방식에 따라 메모리 상의 주소 값(주소값)이 다르게 나타날 수 있습니다.

## 2. String의 불변성(Immutability)과 비효율성
- **불변성(Immutable Class)**: String은 불변 클래스입니다. 즉, 일단 생성된 String 객체의 내용은 변경될 수 없습니다.
- **조작 방식의 원리**: String의 메서드를 이용해 내용을 변경하는 것처럼 보이는 연산(예: `+` 연산자 사용)을 수행할 때, 실제로 기존 객체가 수정되는 것이 아닙니다.
    - 연산이 발생할 때마다 **내용이 변경된 새로운 String 객체가 메모리에 생성**됩니다.
- **비효율성**: 따라서 문자열을 빈번하게(반복적으로) 연결하거나 조작할 경우, 원본 문자열의 길이만큼 수많은 새로운 객체가 메모리에 계속 쌓이게 되어 메모리 낭비가 발생할 수 있습니다.

## 3. 효율적인 문자열 조작 클래스: StringBuilder와 StringBuffer
- **문제 해결책**: String의 비효율적인 메모리 낭비를 해결하기 위해, **가변(Mutable)**한 문자열을 다루는 클래스가 제공됩니다.
- **핵심 원리**: 이 클래스들은 내부적으로 '버퍼(Buffer)'와 같은 공간을 가지고 있어, 문자열을 연결하거나 수정할 때 매번 새 객체를 만들지 않고 같은 공간을 재사용하며 내용을 변경합니다.
- **사용 메서드**:
    - `append()`: 문자열을 버퍼에 추가할 때 사용합니다. (가장 빈번하게 사용됨)
    - `toString()`: 모든 조작이 끝난 후, 최종 결과를 다시 일반 String 타입으로 반환할 때 사용합니다.

## 4. 클래스별 특징 비교 및 사용 권장 사항
| 클래스 | 주요 특징 | 동기화(Synchronization) 지원 여부 | 사용 환경 및 성능 |
| :--- | :--- | :--- | :--- |
| **String** | 불변(Immutable) | 해당 없음 | *문자열 변경이 거의 없을 때* 가장 편하고 빠름. |
| **StringBuilder** | 가변(Mutable), **비동기화** | 지원하지 않음 | **단일 스레드 환경**에서 성능이 가장 빠름. 메모리 효율성이 높음. |
| **StringBuffer** | 가변(Mutable), **동기화** | 지원함 (안전성 확보) | **멀티스레드 환경**에서 안전하게 사용할 때 적합함. (단, 동기화 과정 때문에 오버헤드가 발생하여 일반적으로 `StringBuilder`보다 느릴 수 있음) |

- **성능 요약**:
    - **단일 스레드 환경**: `StringBuilder`가 가장 빠르고 효율적입니다.
    - **멀티스레드 환경**: `StringBuffer`가 안전합니다.
- **사용 시점**:
    - 문자열의 결합(연산)이 빈번하게 발생하는 경우 (루프 내에서 반복 결합 등)에는 String 대신 `StringBuilder`나 `StringBuffer`를 사용하여 메모리 낭비를 막고 성능을 확보해야 합니다.
    - 문자열 자체가 영구적이며 수정이 필요 없는 경우에는 String을 사용하는 것이 가장 좋습니다.
